# 玩家基础 UI 与日常闭环分阶段执行方案

## 1. 目标

建立以下可运行的玩家体验闭环：

`进入游戏 -> 新游戏或读档 -> 与 NPC 正常交流 -> 昼夜推进 -> 主动或强制睡眠 -> 午夜结算 -> 次日继续交流 -> 保存并返回菜单或退出桌面`

交付优先级：完整度 > 视觉品质 > 开发速度。

本方案只定义实施阶段和验收门槛。开始实际实现前，仍需按项目规则在 `docs/AIChanges/FrontendArchitecture/` 创建互相索引的 plan / execution 文档。

## 2. 架构硬约束

1. 时间 HUD、地点 HUD、背包、存档管理、系统消息、暂停和睡眠分别建立独立 MVC 模块。
2. View 只管理外观、控件和展示格式，不直接读取 Model。
3. Controller 只操作自己的 Model；跨模块只能通过事件或接口通信。
4. Model 可管理自身数据校验、排序和状态变更规则，不直接引用其他功能域 Model。
5. 全局暂停由独立模块持有，必须等全部暂停来源释放后才能恢复。
6. 睡眠跨日由独立睡眠流程 Controller 统筹，不得塞入暂停模块或 `GameManager`。
7. 不继续扩张已经过重的 `GameManager` 和 `GameStateStore`；实现前需取得用户明确的“减重”指令。
8. 场景、Prefab、UI 层级、床位触发区、Shader 配置和 SerializeField 连线必须使用 Unity MCP。

## 3. 阶段总览

| 阶段 | 主题 | 核心结果 |
|---|---|---|
| 0 | 基线与边界 | 固化现有回归基线，确认资产和协议入口 |
| 1 | 暂停基础设施 | 多来源暂停与 Python 即时同步 |
| 2 | Unity 权威时间 | 时间迁移、HUD、Python 镜像和昼夜 Shader |
| 3 | 系统消息 | 普通提醒、阻塞弹窗和加载遮罩 |
| 4 | 地点与背包 | 大区域 HUD 和只读分类背包 |
| 5 | 存档管理 | 自动槽、无限手动槽、截图及管理操作 |
| 6 | 暂停菜单 | 保存、加载、设置、返回和退出流程 |
| 7 | 睡眠跨日 | 主动睡眠、强制睡眠、午夜结算和自动存档 |
| 8 | 致命错误与回归 | 断线/结算失败处理及完整 Play 验收 |

## 4. 阶段 0：基线确认与实施准备

### 目的

在改变时间、存档和 UI 前建立可重复的现有功能基线。

### 工作内容

1. 创建本轮正式 plan / execution 文档。
2. 取得用户对 `GameManager`、`GameStateStore` 减重的明确授权。
3. 使用 Unity MCP 盘点开始菜单、`Town_Main`、喫茶店、Canvas、现有 Shader 和输入模块。
4. 确认玩家出生点为街道中公园通往街道的传送点附近。
5. 确认喫茶店当前场景内可放置玩家专用床和醒来 Anchor。
6. 跑通现有新游戏、NPC 对话、传送、保存、加载和重连流程。

### 通过门槛

1. 现有流程可重复运行。
2. Unity Console 无编译错误。
3. 已记录本轮开始前的已知问题，避免把旧问题误判为新回归。

## 5. 阶段 1：独立暂停基础设施

### 目的

统一背包、暂停菜单、阻塞弹窗、睡眠确认和加载遮罩的暂停行为。

### MVC 边界

1. Pause Model 保存当前暂停来源集合。
2. Pause Controller 负责申请、释放、计算最终暂停状态，并通过协议接口同步 Python。
3. 业务模块只提交自己的暂停来源，不直接修改全局时间或 `Time.timeScale`。

### 行为要求

1. 多个暂停来源可同时存在。
2. 只有全部暂停来源释放后才恢复游戏。
3. 暂停与恢复必须立即同步给 Python 镜像时钟。
4. NPC 对话期间继续沿用时间暂停语义。

### Play 验收

1. 同时打开两个暂停来源，关闭其中一个后世界仍保持暂停。
2. 最后一个暂停来源释放后 Unity 与 Python 同时恢复。
3. 不出现重复恢复、提前恢复或永久卡暂停。

## 6. 阶段 2：Unity 权威时间与昼夜表现

### 目的

将游戏时间权威从 Python 正式迁移到 Unity，同时保留 Python 镜像时钟供 NPC 日程使用。

### MVC 边界

1. Game Time Model 持有天数、小时、分钟、天气引用、流速和推进状态。
2. Game Time Controller 推进时间、响应暂停、处理关键节点同步。
3. Game Time View 格式化并显示 `[第X天]-am/pm-HH:MM`。
4. 昼夜 View 只负责将当前时间转换成 Shader 显示参数。

### 协议要求

1. Unity 为唯一权威时间源。
2. Python 保留本地镜像时间，继续驱动分钟级 NPC 日程。
3. Unity 在整点、跨日、读档、任务完成等关键节点同步时间。
4. 暂停与恢复即时同步。
5. 两端不一致时，无条件用 Unity 时间覆盖 Python。

### 时间配置

1. 新游戏从第 1 天 `08:00` 开始。
2. 测试配置保持一天约 5 分钟。
3. 正式配置为完整 24 游戏小时约 30 分钟。
4. 两种流速通过开发配置切换，不散落硬编码。

### 昼夜表现

1. 复用现有 Shader，只接入时间驱动并配置颜色范围。
2. 每个整点作为颜色节点，整点之间平滑插值。
3. 白天按时间产生冷暖变化。
4. 夜间叠加暗蓝色调，但玩家、NPC 和可交互区域必须清晰可辨。

### Play 验收

1. HUD 的 12 小时制、`am` / `pm` 和跨日显示正确。
2. 时间跨整点时色调连续变化，不发生瞬切。
3. 暂停期间两端时间停止；恢复后继续同步。
4. 人为制造两端偏差后，Python 被 Unity 时间修正。

## 7. 阶段 3：系统消息与加载遮罩

### 目的

为后续存档、读档、睡眠和致命错误提供统一反馈通道。

### 普通提醒

1. 单条显示 3 秒。
2. 多条消息按请求顺序逐条显示。
3. 显示期间不暂停游戏。
4. 阻塞弹窗存在时延后显示。

### 阻塞弹窗

1. 按请求顺序排队。
2. 显示期间统一暂停游戏时间与世界行为。
3. `Esc` 无效，只能使用弹窗提供的按钮。
4. 本阶段提供单按钮和双按钮 Prefab，结构允许未来扩展更多按钮。
5. 按钮文字由具体业务请求提供。

### 加载遮罩

1. 保存和加载期间显示同类阻塞遮罩。
2. 遮罩期间禁止玩家输入和其他 UI 操作。

### Unity MCP 工作

1. 创建普通提醒、阻塞弹窗和加载遮罩 Prefab。
2. 建立独立消息 Canvas 层级和 SerializeField 连线。
3. 中文 TMP 文本使用 `Assets/Fonts/MSYH SDF.asset`。

### Play 验收

1. 普通消息和阻塞消息各自排队正确。
2. 阻塞弹窗关闭前世界不会恢复。
3. 保存和加载期间无法移动、交互或打开其他菜单。

## 8. 阶段 4：地点 HUD 与只读背包

### 地点 HUD

1. 显示面向玩家的大区域名称，不显示内部 `location_id`。
2. 复用现有玩家位置解析和传送语义。
3. 找不到名称映射时显示“未知”。

### 背包 Model

1. 建立正式运行时背包模型并接入现有 `InventoryEntrySaveData`。
2. 新游戏初始背包为空，容量无限。
3. 每个物品必须有且只有一个用于背包展示的主分类。
4. 排序顺序为：主分类、名称、数量、稳定 ID，全部升序。
5. 本阶段只读，预留未来使用和摧毁接口，不实现对应行为。

### 背包 View 与输入

1. 按主分类分组并显示分类标题。
2. 条目只显示名称和数量。
3. 空背包显示“暂时没有物品”。
4. `B` 打开，`B` 或 `Esc` 关闭。
5. 背包打开期间申请全局暂停。

### Play 验收

1. 空背包、单分类和多分类均正确显示。
2. 排序规则在同名、同数量情况下仍稳定。
3. 背包打开和关闭不会破坏其他暂停来源。
4. 保存和加载后背包状态一致。

## 9. 阶段 5：存档管理产品层

### 存档规则

1. 全局只有一个自动存档槽。
2. 手动存档数量不限。
3. 手动存档支持新建、覆盖、重命名和删除。
4. 覆盖和删除前必须二次确认。
5. 手动存档名称不得重复。
6. 自动存档只允许读取，不允许手动覆盖、重命名或删除。

### 名称规则

1. 手动存档默认名称：`手动存档：YYYY-MM-DD-[NNN]`。
2. 日期使用现实系统日期，编号用于防重复。
3. 自动存档名称包含当前游戏日、次日醒来时间和继承的手动存档名。
4. 示例：`第四天 [上午 6:00] [无存档]`。

### 新游戏

1. 允许无存档进入，但开始前必须提醒。
2. 提供“快速创建”和“无视，继续进入”。
3. 快速创建预填默认手动存档名，并允许玩家修改。
4. 初始状态为第 1 天 `08:00`、空背包和街道出生点。

### 存档截图

1. 每个存档条目显示名称和最后存档截图。
2. 截图只包含游戏世界，不包含 HUD 或菜单。
3. 截图失败不阻止保存，列表使用默认占位图。

### 异常处理

1. 损坏或版本不兼容的存档不显示在列表中。
2. 读档会丢失未保存进度时必须二次确认。
3. 普通读档失败显示阻塞错误弹窗，并保持当前游戏状态不变。
4. 手动保存成功显示 3 秒普通提醒。
5. 读档成功不额外显示提醒。

### Play 验收

1. 验证快速创建、无存档进入、新建、覆盖、重命名和删除。
2. 验证重复名称、截图失败、损坏存档和读档失败。
3. 验证自动存档不能被编辑。
4. 验证存档截图不包含 UI。

## 10. 阶段 6：暂停菜单与退出流程

### Esc 主菜单

必须提供：

1. 直接退出。
2. 保存并退出。
3. 保存。
4. 加载。
5. 设置。
6. 回到游戏。
7. 退出到桌面。

### 行为规则

1. `Esc` 主菜单显示期间申请全局暂停。
2. 保存和保存并退出共用同一存档管理界面。
3. 保存并退出在保存成功后返回开始菜单。
4. 保存失败时提供“仍然退出”和“返回游戏”。
5. 直接退出需要二次确认，确认后返回开始菜单。
6. 游戏内退出到桌面需要二次确认。
7. 开始菜单退出到桌面不需要二次确认。
8. 设置页只显示背景和“暂无内容”，按 `Esc` 返回主菜单。

### Play 验收

1. 验证所有菜单入口、返回层级和确认弹窗。
2. 验证保存成功、保存失败和取消保存后的退出行为。
3. 验证关闭嵌套页面时不会提前恢复游戏。

## 11. 阶段 7：睡眠、午夜结算与自动存档

### 床位资产

1. 床位于喫茶店当前场景内，不新增卧室场景。
2. 床只供玩家使用，不进入 NPC affordance 或日程目标。
3. 使用 Unity MCP 创建床位触发区和次日醒来 Anchor。

### 主动睡眠

1. 玩家可在任何游戏时间主动睡觉。
2. 进入床的触发范围后自动弹出“确认睡到明天”。
3. 弹窗期间暂停游戏时间与世界行为。
4. 取消后必须离开触发范围再进入，才会再次弹出。

### 强制睡眠

1. 游戏时间到 `23:30` 时，在任意地点触发。
2. NPC 对话期间时间暂停，因此不会在对话中途触发。
3. 玩家在任何地点强制睡眠后，次日仍在喫茶店床上醒来。

### 跨日流程

1. 独立 Sleep Controller 申请暂停。
2. 中止 NPC 当前行为，并将当日剩余日程视为完成。
3. 请求 Python 执行完整午夜结算。
4. Unity 等待结算完成，最长现实 10 分钟。
5. 成功后推进至次日 `06:00`。
6. 将玩家放置到床位醒来 Anchor。
7. 覆盖全局自动存档，保存的是次日 `06:00` 状态。
8. 自动存档失败时正常醒来，并显示简单错误弹窗。

### Play 验收

1. 验证主动睡眠、取消后重新进入和白天提前睡眠。
2. 验证 `23:30` 在喫茶店外强制睡眠。
3. 验证 NPC 行为终止和次日日程重新开始。
4. 验证次日 `06:00`、床位醒来和自动存档内容。
5. 验证自动存档失败不会阻断醒来。

## 12. 阶段 8：致命错误与完整 Play 回归

### 致命错误条件

1. Python 后端断线。
2. Python 午夜结算明确失败。
3. 午夜结算等待超过现实 10 分钟。

### 处理方式

1. 不允许降级继续游戏。
2. 不允许仅保存 Unity 世界数据。
3. 显示致命错误阻塞弹窗。
4. 玩家确认后关闭到桌面。

### 最终 Play 回归路径

1. 无存档进入新游戏。
2. 快速创建手动存档进入新游戏。
3. 验证时间、地点、背包和昼夜 Shader。
4. 与 NPC 正常交流。
5. 验证所有暂停来源和系统消息。
6. 主动睡眠并生成自动存档。
7. 次日继续与 NPC 交流。
8. 手动保存、覆盖、重命名、删除和读档。
9. 保存并返回开始菜单。
10. 重新读取自动存档和手动存档。
11. 验证 `23:30` 强制睡眠。
12. 验证 Python 断线、午夜失败和午夜超时。
