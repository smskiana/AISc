# 冷启动初始知识投影与他人记忆衍生草案

## 1. 状态与范围

本文是待确认的设计草案，不是实施 plan，不代表已经授权修改代码。

本草案只解决一个方向：冷启动时，每个 NPC 的独立记忆图应该包含哪些关于世界、玩家和其他 NPC 的初始认知，以及这些认知如何从统一事实源投影为不同观察者的记忆节点。

本草案不调整玩家对话检索算法，不决定 LLM 与本地图搜索的职责分工。检索改造见 `LlmGuidedLocalDeepRetrievalDraft.md`。

## 2. 当前问题

当前冷启动会为每个 NPC 建立：

1. `self` 节点。
2. 玩家和所有 NPC 的 `person` 节点。
3. `self -> person` 的 `relationship` 边。
4. 四条玩家 / 商店街共享背景。
5. 该 NPC 自己的 `core_memories`。

但 `initial_impression_traits` 只进入 `npc_impressions`，没有形成可沿图检索的事件、观察或关系依据。其他 NPC 的设定也不会按“这个观察者实际知道什么”投影到当前 NPC 的图中。

因此人物节点往往只是名字节点。检索可以找到“千早”，却不一定能继续找到“她经营面包店”“观察者为什么这样评价她”“双方共同经历过什么”等可回答内容。

## 3. 目标与非目标

### 3.1 目标

1. 为每个 NPC 构建足够但不过量的初始世界认知。
2. 支持关于其他人物的公开事实、共同经历、传闻和主观印象形成可检索节点。
3. 同一事实按观察者知识范围生成不同表述，不把角色私密记忆复制给所有人。
4. 记录来源、可信度、涉及实体和知识范围，为回答口径与后续传播提供依据。
5. 继续保持“每个 NPC 独立维护图”的 ADR-0001 边界。

### 3.2 非目标

1. 不建立全局共享运行时记忆图。
2. 不让 NPC 自动知道其他 NPC 的全部 `core_memories`。
3. 不在冷启动时用 LLM 自由补写世界设定。
4. 不在本草案中设计对白检索深度、候选排序或 Prompt。
5. 不扩张为完整认知本体或全图 `similar_to` 网络。

## 4. 核心概念

应将“客观设定”“谁知道”“观察者如何记得”拆成三层：

```text
初始事实定义 InitialKnowledgeFact
→ 可见性判断 KnowledgeVisibility
→ 观察者记忆投影 ProjectedMemory
→ 写入该观察者的独立图与向量层
```

### 4.1 初始事实定义

初始事实是配置层的稳定输入，不直接等于任何 NPC 的第一人称记忆。建议至少表达：

| 字段 | 含义 |
|---|---|
| `fact_id` | 跨配置稳定 ID |
| `fact_type` | 人物事实、关系事实、共同事件、地点事实、传闻等 |
| `subject_ids` | 事实涉及的人物 |
| `location_ids` | 涉及地点，可为空 |
| `canonical_summary` | 事实源的中性描述 |
| `knowledge_scope` | 公开、熟人可知、指定知情者、亲历、秘密 |
| `knower_ids` | 指定知情者；不适用时为空 |
| `source_type` | 亲历、对方告知、街坊传闻、公开常识 |
| `confidence` | 初始可信度 |
| `importance` | 初始重要度 |
| `created_day` | 时间语义 |

`fact_type`、`knowledge_scope` 和 `source_type` 应使用统一枚举或规则表，禁止随着人物增加继续堆人物专属 `if / elif`。

### 4.2 可见性判断

统一的投影规则模块根据事实和观察者判断是否生成节点：

```text
can_observer_know(fact, observer_id, relationship_context) -> visibility_result
```

第一版建议支持以下范围：

1. `public`：所有在世界设定中正常生活的 NPC 可知。
2. `community`：商店街熟人默认可知，但允许排除新来者或封闭角色。
3. `participants`：只有事件参与者知道。
4. `explicit_knowers`：只有配置列出的知情者知道。
5. `private`：只进入事实主体自己的图，除非另有明确知情者。

五种范围必须走统一规则，不按 NPC 名称分支。

### 4.3 观察者记忆投影

投影结果应是观察者视角的记忆，不是原事实文本的机械复制。建议输出：

| 字段 | 含义 |
|---|---|
| `projection_id` | 可稳定重建的投影 ID |
| `observer_id` | 图的拥有者 |
| `source_fact_id` | 来源事实 ID |
| `memory_type` | event、reflection、person_fact 等现有或收敛后的节点类型 |
| `value` | 观察者能表达的记忆文本 |
| `subject_ids` | 用于与人物节点建边 |
| `location_ids` | 用于与地点节点建边 |
| `source_type` | 观察者如何知道 |
| `confidence` | 观察者对此事的可信程度 |
| `importance` | 对观察者的重要程度 |

第一版优先使用策划提供的观察者模板或确定性文本组合。只有内容规模证明人工模板不可维护后，才讨论离线 LLM 投影；即使使用 LLM，也必须由事实字段约束并经过一致性校验。

## 5. 他人记忆衍生规则

“衍生”不是读取另一个 NPC 的运行时记忆数据库，而是从同一初始事实源生成各自有权知道的视角化节点。

示例：

```text
事实源：千早小时候受欺负时，商店街的人曾保护她。

千早：
“小时候受欺负时，商店街的大家保护过我。”

樱（参与者或明确知情者）：
“小时候千早受欺负时，街上的大家一起护着她。”

九条（仅听过传闻）：
“听街坊提过，千早小时候受过欺负，因此很护着这条街。”

不知情角色：
不生成节点。
```

必须遵守：

1. 私密事实默认不传播。
2. 传闻节点必须保留“听说”等不确定口径。
3. 主观印象不能升级成客观事实。
4. 观察者不知道的因果、时间、地点不得在投影文本中补齐。
5. 同一事实可以投影到多个图，但每份投影都保留同一 `source_fact_id`，便于诊断与重建。

## 6. 初始图结构建议

每份投影至少连接：

1. 观察者的 `self` 节点。
2. 所有 `subject_ids` 对应的 `person` 节点。
3. 已存在的地点节点；若当前图没有稳定地点节点，第一版可先把地点 ID 保存在元数据中，不为本任务扩张图本体。

建议边语义继续复用现有 `involved`、`relationship` 等规则。若确需表达 `heard_from`、`observed` 等来源边，应先确认现有边类型无法承载，再评估是否需要 ADR 修订；不能在实现阶段临时扩张边本体。

`clarity`、`confidence`、`importance` 必须保持分离：

1. `clarity`：沿边是否容易联想到。
2. `confidence`：观察者认为内容是否可信。
3. `importance`：内容对观察者是否重要。

## 7. 配置与职责边界

建议把初始知识从 `state_manager.py` 内的共享常量和散落 profile 字段中抽成独立配置入口，由一个投影模块统一消费。

职责建议：

1. 配置层：定义事实、范围、知情者和稳定 ID。
2. 投影模块：判断可见性并生成观察者记忆 DTO。
3. 图初始化：只负责把 DTO 写入 SQLite 图和向量层。
4. impression 初始化：可引用投影事实形成主观基线，但不得成为唯一事实来源。

如果实施时发现 `StateManager` 继续承担配置解释、权限判断、文本投影、图构建和持久化，将触发类减重要求，必须先取得用户的“减重”指令。

## 8. 旧存档与重建

冷启动投影只自然影响新图。实施 plan 必须明确选择：

1. 仅新开局生效；或
2. 提供显式、可审计的初始图重建工具。

不得静默把新初始知识灌入旧存档，因为旧游戏中这些事实可能已被后续事件推翻，或角色已经通过运行时互动获得不同版本的信息。

## 9. 诊断与控制要求

实施后 `aisc_debug` 至少应能回答：

1. 某 NPC 冷启动生成了哪些投影节点。
2. 每个节点来自哪个 `source_fact_id`。
3. 为什么该观察者有权知道，或为什么被排除。
4. 节点的 `source_type`、`confidence`、`subject_ids`。
5. 节点实际连接到哪些人物 / 事件边。

若提供重建能力，应通过编辑器专有 `aisc_control` 暴露显式操作，并明确它会重建哪些数据；不得用普通运行时代码自动重置存档。

## 10. 验证场景

至少覆盖：

1. 公开人物事实进入所有合理观察者的图。
2. 双方共同经历进入参与者图，不进入无关角色图。
3. 私密核心记忆只进入本人和明确知情者图。
4. 传闻投影保留来源和不确定措辞。
5. impression 中关于某人的评价有对应事实或明确标为纯主观印象。
6. 问九条“千早是做什么的”时，图中存在可回答事实，而不只是千早人物节点。
7. 问无权知道的秘密时，检索结果不出现秘密节点。
8. 重复冷启动或显式重建不会生成重复投影。

## 11. 预计实现影响面

确认实施后，预计至少涉及：

1. 初始知识共享配置或 NPC profile 结构。
2. 新的初始知识投影模块。
3. `backend/src/npc/state_manager.py` 的初始化职责。
4. 图 / 向量写入和必要的数据库元数据。
5. 冷启动投影单元测试与竞争型对话检索测试。
6. `aisc_debug` DTO、provider、action 文档及测试。
7. Memory Workstream、代码目录 README 和必要的设计入口。

该方向预计必然达到复杂实现阈值。确认实施时应创建唯一 plan，并在独立执行对话中实施。

## 12. 风险与待确认项

1. 初始事实量过大可能让每个 NPC 图迅速膨胀；需要控制投影粒度，优先保留能支持对白的稳定事实和关键经历。
2. `community` 范围可能过宽；应允许事实级排除或明确知情者覆盖。
3. 视角文本若完全手写会有维护成本，若交给 LLM 又有事实漂移风险；第一版建议确定性模板优先。
4. 地点是否需要成为稳定图节点尚需结合现有图模型确认，不在草案中预设扩张。
5. `initial_impression_traits` 是继续保留为独立主观配置，还是逐步改为引用事实投影，需要在实施 plan 前确认。

## 13. 验收标准

1. 每个 NPC 对主要人物至少拥有与其知识范围相符的基础可回答信息，而不是只有名字和 bond。
2. 他人私密记忆不会因投影机制泄露给无权知道的角色。
3. 投影节点能追溯到稳定事实 ID、来源和可见性判断。
4. 同一事实能形成不同观察者口径，但不产生互相矛盾的新事实。
5. 冷启动图规模、构建时间和向量写入量有明确预算与诊断数据。
