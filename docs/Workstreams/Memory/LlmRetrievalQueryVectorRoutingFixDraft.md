# LLM 检索想法向量路由修复草案

## 1. 草案定位

本文针对 [LLM 方向合法误判与最终重排丢失实体记忆](../../DesignDocs/errors/2026-07-17_llm_direction_and_final_rerank_memory_loss.md) 提出修复方向，当前只作为设计来源，不直接修改运行代码。唯一执行案已创建：[2026-07-17_LLM检索想法向量路由修复_plan.md](../../AIChanges/Memory/2026-07-17_LLM检索想法向量路由修复_plan.md)。后续必须在独立执行对话中引用该 plan，并先取得“减重并执行”授权。

本草案拟沿既有 [三策略可配置记忆路由与本地深层检索方案](../../AIChanges/Memory/2026-07-17_LLM定向本地深层记忆检索_plan.md) 增补，不改变每个 NPC 独立记忆图、固定图起点、权限 / archived 过滤和“向量只作锚点或内容兜底”的边界。

## 2. 问题复述

问题“你知道千早在哪里吗，我在找她”中，LLM 能识别“千早”，但可能只返回结构合法的 `general_recall`。图搜索即使命中千早节点，最终固定容量重排仍可能让高先验 `event` 节点挤掉 `person` / `identity` 记忆。

本轮不要求注入 `npc_states.current_location`，也不要求回答一定正确；只要求已有的千早记忆证据不会因为方向改写或最终截断而消失。

## 3. 核心方案

### 3.1 一次 LLM 调用生成检索想法

LLM 不直接输出 embedding 数字，也不直接决定权重。它输出一段受约束的 `retrieval_query`，表达要查找的记忆方向：

```json
{
  "retrieval_query": "想知道千早最近可能在哪里，以及与她当前位置相关的记忆证据",
  "explicit_entities": ["千早"],
  "query_constraints": ["person_location", "recent"]
}
```

其中：

- `retrieval_query` 是语义检索文本；
- `explicit_entities` 是本地保护和诊断元数据，至少应由原始用户问题确定性提取；
- `query_constraints` 是受控枚举，不允许 LLM 生成任意规则或数值权重。

LLM 失败、超时或输出不可信时，使用原始问题和本地方向改写，不中断对话。

### 3.2 组成单次向量 query

不直接使用 LLM 改写句子覆盖原问题，而是按稳定顺序组成一段文本：

```text
当前问题：你知道千早在哪里吗，我在找她
检索想法：想知道千早最近可能在哪里，以及与她当前位置相关的记忆证据
相关近期对白：玩家上次提到正在寻找千早
```

对这段 `retrieval_query` 执行一次 embedding / ANN 查询。近期对白只保留与当前问题或明确实体相关的少量内容，避免玩家背景和当前场景噪声压过目标人物。

这一阶段只改变向量查询文本，不改变图搜索固定起点。向量命中仍然只能作为目标锚点或内容兜底，不能直接变成最终答案。

### 3.3 本地使用检索想法参与图路由

向量候选用于引导固定起点的图扩张：

```text
self + 对话对象
→ 图邻接扩张
→ 检索想法向量命中作为锚点相关性
→ 路径评分与权限过滤
→ 最终记忆选择
```

本地可以根据向量相似度、图路径分、importance 和明确实体命中进行综合排序，但不允许 LLM 返回任意浮点权重。

推荐将权重放入已校验的 retrieval policy；第一版优先使用固定 profile，不新增运行时设置 UI 或协议字段。

### 3.4 最终选择使用通用语义排序

最终选择不为“千早”“李二狗”或任何特定人物设置硬编码保底，也不要求每个被点名实体必然占用一个结果槽位。查询实体只作为通用语义信号参与排序：

- 原问题和 `retrieval_query` 的向量相似度是主要相关性信号；
- 图路径方向分、importance 和近期性作为辅助信号；
- 原文实体命中可以作为轻量加分或诊断字段，但不能单独越过所有其他证据；
- 节点仍必须先通过图路径、权限和 archived 过滤。

`final_memory_limit` 的计数单位是完整记忆条目，而不是字符片段。截断只能发生在条目边界：

- 不得把单条记忆文本从中间截断；
- 不得只保留证据句子的前半段；
- 同一条记忆的节点内容、必要的来源 / 时间摘要应作为一个不可拆分单元；
- 若完整条目超过 Prompt 字符预算，应淘汰整条低优先级记忆，而不是破坏高优先级记忆的语言完整性；
- 相关证据若被设计为证据束，应整体计数和整体保留。

这不是把所有 `person` 节点置顶，也不是创造缺失事实；如果图中没有与当前检索想法相关的证据，仍返回 `knowledge_absent` 或现有失败语义。

## 4. 受约束输出契约

建议保留以下最小字段：

```text
retrieval_query       必填，短文本，不得新增事实
explicit_entities     原文人物 / 地点 mention，用于校验和诊断，不作为强制保留规则
query_constraints     受控枚举，如 person_location、recent
```

`explicit_entities` 与 `query_constraints` 不一定直接拼入 embedding，它们用于：

1. 检查检索想法是否丢失用户点名实体；
2. 记录实体相关性，不强制保留实体节点；
3. 解释向量和图路由为何选择某些节点；
4. 在 LLM 改写偏离时触发本地校准。

禁止 `retrieval_query` 引入原问题没有的人物、地点、精确时间或确定事实。发现新增实体时只保留原文实体，并记录 `retrieval_query_entity_mismatch`。

## 5. 诊断要求

`aisc_debug.memory_retrieval_snapshot` 需要区分：

- 原始用户问题；
- LLM 生成的 `retrieval_query` 安全摘要；
- 实际送入 embedding 的 query 摘要 / 字符计数；
- 向量命中节点及其用途；
- 图搜索命中节点；
- 最终进入对白 Prompt 的节点；
- 实体保护、预算截断和失败原因。

不保存完整 Prompt 或未裁剪的原始 LLM 输出，不向 Unity 泄露无关长文本。

## 6. 回归测试草案

1. LLM 正常输出检索想法时，单轮只执行一次 embedding 查询。
2. “你知道千早在哪里吗，我在找她”生成的 query 同时保留原问题和千早，不被当前 `flower_shop.counter` 替换。
3. LLM 输出泛化想法、漏掉千早或加入不存在人物时，本地实体提取和校验能发现并纠正。
4. 向量命中某个查询实体但 event 节点 importance 更高时，排序由综合相关性决定，不依赖特定人物保底规则。
5. `final_memory_limit` 截断后，每条保留的记忆文本和证据束均保持完整，不出现半句、半条来源或断裂语义。
6. Prompt 字符预算不足时淘汰完整低优先级条目，而不是截短高相关记忆。
7. 没有查询相关图证据时，不因检索想法生成虚构地点。
8. 无实时位置注入时，诊断不把“记忆推断不确定”误报成检索系统异常。
9. “喫茶店最近怎么样”等无明确人物问题不被实体相关性信号改变原有语义排序。
10. LLM 不可用、非法 JSON、超长 `retrieval_query` 时，回退到原始问题 / 本地方向并记录原因。

## 7. 预算与边界

预算可以在开发阶段调整，但仍需配置化并设置单请求上限：

- LLM 调用次数；
- embedding 查询次数；
- 向量候选数；
- 图扩张深度、frontier 和展开边数；
- `retrieval_query` 与近期对白字符数；
- 最终记忆条数。

本草案默认仍采用一次 embedding 查询。若后续要分别对原问题、检索想法和近期对白执行多次 ANN 查询，应另行评估延迟、合并策略和诊断字段，并更新原检索 plan。

## 8. 已由执行案收口的事项

1. 允许加入至多一条由本地规则确定性选出的相关近期对白；它作为独立 query 段，不授权 LLM 引入事实。
2. 第一版坚持每轮最多一次 embedding / ANN，不做多 query 合并。
3. 最终评分采用执行案定义的六分量配置 profile，query semantic similarity 权重为 0.42，节点类型先验硬上限为 0.05。
4. 当前问题中的明确实体全部由本地提取并作为权威；LLM 只提供可校验的检索想法与方向补充。

具体模块 seam、三策略差异、文件范围、诊断字段和验收标准以唯一执行案为准，不得直接按本草案进入实现。
