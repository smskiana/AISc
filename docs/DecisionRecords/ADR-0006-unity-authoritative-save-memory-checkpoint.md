# ADR-0006: Unity 权威世界存档与 Python 记忆检查点分离

## 状态

Accepted

## 背景

项目当前由 Python `SaveManager` 备份完整 SQLite、LanceDB 和部分世界快照，但 SQLite 同时包含玩家位置、时间、NPC 状态、背包与 AI 记忆。随着单机玩法扩展，把全部玩法状态继续交给 Python 会扩大跨端同步成本，并使 Unity 不能直接拥有其高频使用的游戏事实。

与此同时，完全独立且无协调地保存两端数据，会产生 Unity 世界与 NPC 记忆来自不同时间点的错配。

## 决定

1. Unity 是单机游戏世界持久状态的唯一权威。
2. Python 是 AI 记忆、记忆关系和向量索引的唯一权威。
3. 两端使用独立存储格式，不建设统一数据库。
4. 同一个逻辑存档由 `slot_id` 和 `checkpoint_id` 关联两端检查点。
5. Unity 主导保存与加载事务，Python 只提供记忆检查点 prepare / commit / abort / load 能力。
6. Python 可保留世界事实缓存用于推理，但重连或读档后必须由 Unity 世界快照覆盖。
7. 运行时临时状态默认不存档；读档后取消并按持久事实重建。
8. 现有 Python 世界状态表按功能逐项迁移，在迁移完成前作为兼容数据，不立即删除。

## 放弃的方案

1. 放弃由 Python 统一保存所有游戏数据：职责过重，增加同步和恢复复杂度。
2. 放弃两端完全无协调地各自保存：无法判断两个文件组是否属于同一时刻。
3. 放弃把 Unity Transform、GameObject、UI 或进行中协程直接序列化：这些不是稳定业务语义。

## 影响

1. Unity 必须新增主存档仓储、schema migration 和保存协调器。
2. Python `SaveManager` 将收缩为记忆检查点服务。
3. 协议必须支持版本、请求关联、幂等、sequence 和统一错误。
4. 保存与加载必须验证 `checkpoint_id` 一致，任一端失败都不得提交半份存档。
5. 玩家位置、时间、背包和 NPC 世界状态将从 Python 权威逐项迁回 Unity。

## 重新讨论条件

只有出现多人服务器权威、跨设备云存档或 Python 进程升级为正式游戏服务器时，才重新评估世界状态权威边界。
