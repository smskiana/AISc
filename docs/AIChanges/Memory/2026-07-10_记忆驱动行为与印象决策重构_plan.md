# 记忆驱动行为与印象决策重构 — 方案

## 完成时间
2026-07-10

## 这次方案要解决什么

当前项目里：

1. `NPC 对话`已经能部分吃到记忆
2. `NPC 行为决策`仍然主要由 routine / personality / 当前状态驱动
3. `bond`、`印象`、`图中的 relationship` 都在不同程度上表达“关系”，语义开始重叠

这次讨论后的目标，不是继续堆新字段，而是把“关系和记忆到底由谁表达”收敛清楚。

## 本次已敲定的核心决议

### 1. 取消 `bond` 作为独立长期关系真源

原因：

- 它与图中的人物节点邻域 / relationship 边语义重叠
- 它与印象系统也容易冲突
- 后续会出现“三套关系系统同时存在”的维护问题

本次定案：

- **长期关系的唯一真源是图**
- 不再把 `bond` 视为需要长期独立维护的核心数据层
- 后续行为、对话、搭话意愿，不再以 `bond` 作为首要依据

### 2. 改用“基准印象 + 白天微调”

新的关系表达方式：

- **基准印象**：夜间批处理生成，来源是图 + 当天短期记忆
- **白天微调**：白天根据最新互动做轻量漂移，影响当天搭话和语气

这样分层后：

- 夜间负责“沉淀后的总体看法”
- 白天负责“刚发生完事情之后的即时波动”

### 3. 图仍然是唯一事实层，印象只是决策视图

职责拆分：

- **图**：长期事实、关系结构、共同经历、重要事件、熟悉度来源
- **印象**：给白天快速读取的压缩判断结果
- **状态**：当天即时情绪/需求/短时波动

换句话说：

- 图是“记忆事实库”
- 印象是“夜间生成的轻量决策缓存”
- 状态是“白天瞬时上下文”

### 4. 反思继续并入融合 / 图演化

此前已敲定：

- “反思”保留为设计术语
- 但实现层面不再要求独立系统
- 后续统一作为融合 / 图演化中的一环处理

本方案继续沿用这个口径。

## 与之前已敲定方向的合并总结

以下内容视为本轮方案的前置共识：

1. **图演化要继续做**
   - 但反思不再独立拆模块

2. **天气系统保留，但优先级不高**
   - 先不抢占原型主资源

3. **流言 / 传闻系统暂缓**
   - 等原型主链路跑通后再做

4. **多 NPC 同时对话要做**
   - 仍然保留在后续能力清单中

5. **独立 `impression.py` / `reflection.py` 不是目标**
   - 重点是能力落地，不是目录对齐

6. **玩家选项生成器要做**
   - 属于对话体验闭环的一部分

7. **API 分层先留接口**
   - 原型阶段不为了拆分而拆分

8. **测试体系、图可视化调试工具延后**
   - 等原型跑通后再补

9. **Unity 侧未明确降级的项默认仍然要做**
   - 优先补 HUD / 睡眠 / 存档 / 对话状态管理 / 配置同步

## 新架构口径

### 一、夜间重处理层

夜间处理的职责：

1. 当天短期记忆汇总
2. 短期记忆提取入图
3. 图演化 / 融合
4. 为熟人生成新的“基准印象”
5. 为次日行为生成“计划上下文摘要”

夜间产物先控制在两类，避免派生层过多：

1. `baseline_impressions[target_id]`
2. `next_day_plan_context`

说明：

- 这里不再生成独立 `bond`
- 如果后续确实需要一个数值提示，也只能作为**图派生缓存**，不能重新变成真源

### 二、白天轻决策层

白天实时阶段只读取轻量结果，不深走图：

1. 行为计划读取 `next_day_plan_context`
2. 搭话判断读取：
   - 基准印象
   - 白天微调
   - 当前情绪 / 需求 / 位置
3. 对话语气读取：
   - 基准印象
   - 白天微调
   - 最近短期记忆

白天的原则：

- 不在每个 tick 做重检索
- 不在玩家眼前做高成本图遍历
- 只维护当天的轻量波动

### 三、白天微调的定位

白天微调不承担“重写长期关系”的职责，只表达：

- 刚刚聊得很开心
- 刚被冷淡对待
- 刚收到帮助
- 刚发生尴尬或冲突

用途：

- 改变今天是否愿意主动搭话
- 改变当天说话语气
- 改变短时回避 / 亲近倾向

这些微调会在夜间被吸收进新的基准印象。

## 状态模型口径

### 一、状态需要几个参数

当前阶段采用：

- `4 个主参数 + 1 个文本`

分别是：

1. `energy`
   - 精力
   - 决定能否继续活动、是否需要休息、是否容易烦躁

2. `emotion`
   - 当前情绪
   - 建议使用有限标签表达，而不是完全自由文本

3. `sociability`
   - 当前社交开放度
   - 决定更偏主动接近、正常应答，还是回避社交

4. `current_need`
   - 当前需求
   - 例如：饿、累、想说话、无聊、想独处

5. `lingering_concern`
   - 当前心事 / 挂念
   - 自由文本，主要夜间生成，白天只在重大事件下改动

### 二、设计层与实现层的区别

设计层维持“4+1”的简洁表达。

实现层允许拆成更适合计算的字段，例如：

- `emotion_baseline`
- `emotion_delta_today`
- `sociability_baseline`
- `sociability_delta_today`
- `energy`
- `current_need`
- `lingering_concern`

也就是说：

- 对外仍然叫“情绪 / 社交开放度”
- 对内可以拆成“夜间基准 + 白天漂移”

### 三、状态不负责表达什么

为了避免和图、印象重叠，状态不承担以下职责：

1. 不表达长期关系真相
2. 不替代图中的共同经历和 relationship 结构
3. 不替代“基准印象”
4. 不再引入 `bond` 这类长期关系数值

状态只表达：

- 我今天现在怎么样
- 我此刻更想做什么
- 我脑子里现在挂着什么事

## 状态所有权与更新时间

### 一、总原则

状态更新采用：

- `3 个时机`
- `1 个最终写入者`

责任边界如下：

1. `StateManager`
   - 负责状态语义
   - 负责最终写库

2. `BehaviorEngine`
   - 负责读取状态做行为决策
   - 不再直接定义完整情绪语义

3. `DialogueService / NpcDialogueManager`
   - 负责产出“互动结果”
   - 不直接长期维护状态逻辑

4. `Runtime.on_midnight()`
   - 负责触发夜间状态结算

### 二、时机 1：夜间结算

夜间负责更新“次日基准状态”。

主要更新：

1. `emotion_baseline`
2. `sociability_baseline`
3. `lingering_concern`
4. 必要时 `fatigue carryover`（如果实现层需要）

依据：

1. 当天短期记忆
2. 图中的近期相关记忆
3. 当天最后的白天微调
4. 当天互动轨迹

谁触发：

- `Runtime.on_midnight()`

谁执行并落库：

- `StateManager`

### 三、时机 2：白天周期刷新

白天周期刷新负责规则驱动的即时状态变化。

主要更新：

1. `energy`
2. `current_need`
3. 小幅 `emotion_delta_today`
4. 小幅 `sociability_delta_today`

依据：

1. 时间推进
2. 当前行为
3. 是否休息 / 吃饭 / 工作 / 长时间移动
4. 是否处于疲劳或过社交状态

谁触发：

- `BehaviorEngine.tick()`

谁执行并落库：

- `StateManager`

建议实现形式：

- `BehaviorEngine` 负责把“当前过了一个游戏分钟，NPC 正在做什么”交给状态层
- `StateManager` 负责计算状态怎样变化

### 四、时机 3：事件结束后即时更新

事件结束后，允许对白天波动做立刻更新。

主要更新：

1. `emotion_delta_today`
2. `sociability_delta_today`
3. 必要时 `current_need`
4. 少数重大事件下可直接改 `lingering_concern`

事件包括：

1. 玩家 ↔ NPC 对话结束
2. NPC ↔ NPC 对话结束
3. 收到帮助
4. 被冷淡回应
5. 发生尴尬或冲突

谁触发：

- `DialogueService`
- `NpcDialogueManager`

谁执行并落库：

- `StateManager`

### 五、各参数由谁负责

1. `energy`
   - 白天按规则推进
   - 睡眠 / 吃饭 / 休息恢复
   - 由 `StateManager` 更新

2. `current_need`
   - 白天按阈值或规则生成 / 清除
   - 由 `StateManager` 更新

3. `emotion`
   - 夜间更新 baseline
   - 白天事件改 delta
   - 白天逐步回落到 baseline
   - 由 `StateManager` 更新

4. `sociability`
   - 夜间更新 baseline
   - 白天由疲劳、互动、心情做轻量偏移
   - 由 `StateManager` 更新

5. `lingering_concern`
   - 主要夜间生成
   - 白天一般不改
   - 重大事件时允许覆盖
   - 由 `StateManager` 更新

### 六、补充约束

为了防止状态再次失控，额外加两条执行约束：

1. `BehaviorEngine` 不应成为情绪真源
   - 它读取状态做决策
   - 但不应长期直接定义“NPC 现在是什么情绪”

2. `对话系统` 不直接写长期状态
   - 对话系统只输出“这次互动带来的短时影响”
   - 最终由状态层吸收

## 计划中的代码改造方向

### Phase 1：让行为计划开始吃“夜间摘要”

目标：

- 不让 `_plan_day()` 只看 personality + routine
- 让它读到“这个 NPC 最近记得什么、对谁在意、明天更可能做什么”

做法：

1. 在夜间生成 `next_day_plan_context`
2. 在 [backend/src/npc/behavior_engine.py](F:/GameProject/unity/AISc/backend/src/npc/behavior_engine.py) 的 `_plan_day()` Prompt 中注入该摘要
3. 允许该摘要由：
   - 图检索结果
   - 当天短期记忆汇总
   - 熟人基准印象
   共同构成

这里性能不敏感，允许使用图辅助。

### Phase 2：让“搭话决策”从看 bond 改为看印象

目标：

- `_llm_decide_talk()` 不再只读 `bond + energy`
- 改为读取“我现在怎么看这个人”

做法：

1. 夜间生成熟人基准印象
2. 白天维护印象微调
3. 在 [backend/src/npc/behavior_engine.py](F:/GameProject/unity/AISc/backend/src/npc/behavior_engine.py) 的主动搭话判断里注入：
   - 基准印象
   - 白天微调
   - 最近一次相关互动摘要

### Phase 3：让对话语气吃到印象

目标：

- Prompt 不只是“角色设定 + 近期记忆”
- 还要能体现“现在对这个人的主观看法”

做法：

1. 在 [backend/src/dialogue/prompt_builder.py](F:/GameProject/unity/AISc/backend/src/dialogue/prompt_builder.py) 中加入印象字段
2. 玩家对话和 NPC ↔ NPC 对话都可读取：
   - 基准印象
   - 白天微调
   - 最近相关短期记忆

### Phase 4：夜间生成熟人印象

目标：

- 把图真正压缩成白天可用的决策信号

做法：

1. 以熟人列表为范围生成印象，不全量遍历所有对象
2. 印象输入来源：
   - 图中与目标人物关联的高权重节点
   - 近期短期记忆
   - 当天最后的白天微调
3. 输出内容建议至少包括：
   - 当前总体看法
   - 亲近 / 回避倾向
   - 说话语气建议

### Phase 5：引入状态调度入口，收口状态写入

目标：

- 不再让行为层、对话层、夜间流程各自随意写状态
- 把状态更新统一收口到状态层

做法：

1. 为 `StateManager` 增加明确入口，例如：
   - 夜间结算入口
   - tick 推进入口
   - 互动结果应用入口
2. 让 `BehaviorEngine` 从“直接改状态”改为“触发状态更新请求”
3. 让对话结束逻辑只提交互动结果，不直接发散修改多个字段

### Phase 6：移除或降级现有 `bond` 使用点

目标：

- 收敛关系语义，避免旧逻辑继续把 `bond` 当主判据

做法：

1. 梳理所有 `npc_bonds` 读写点
2. 先改读路径：由印象优先，`bond` 只做兼容占位
3. 再改写路径：停止把白天互动直接写成长期 `bond`
4. 最终决定：
   - 删除 `bond` 相关主逻辑
   - 或将其降级为临时兼容字段，后续清理

## 这次方案的实现原则

1. **先收敛语义，再写代码**
   - 先明确图、印象、状态各自表达什么

2. **图是唯一真源**
   - 不再引入新的长期关系主字段

3. **白天轻、夜间重**
   - 重计算放夜间
   - 白天只读压缩结果

4. **优先改决策入口**
   - 先让计划 / 搭话 / 对话语气真的吃到记忆
   - 再考虑更复杂的辅助层

5. **不为了目录完整而拆模块**
   - 以能力落地优先

## 我对这个方案的补充判断

这次最关键的，不是“再发明一个关系变量”，而是停止让多个层同时表达关系。

如果后续执行得当，系统会收敛成：

- 图：长期关系事实
- 基准印象：白天决策读取的主入口
- 白天微调：当天即时变化

这是目前最不容易打架、也最符合“午夜重处理、白天轻响应”设计方向的一套方案。
