## 需求理解

本轮目标是修复两个相互放大的记忆检索问题：

1. 冷启动初始人物关系边把 `bond` 当成 `clarity`，导致低亲疏但已知人物的基础认知过低。
2. 玩家发言明确点名人物或地点时，图候选本地排序的语义相关性不足，容易被玩家背景事件、结构性加分或高 clarity 旧边压过，只能依赖向量兜底补回。

修复后应满足：

1. `bond` 仍表达亲疏、态度和 confide 相关语义，不直接决定是否记得对方。
2. `clarity` 表达沿图边联想的清晰程度，初始核心人物关系必须有稳定认知下限。
3. 明确点名人物或地点时，语义相关性应成为候选边排序的强信号，但不能把话题实体加入图起点。
4. 图仍负责联想路径，向量层仍只负责语义内容和兜底，不扩张 ADR-0001 的轻量图边界。

## 边界与约束

1. 不改 Unity 场景、Prefab、SerializeField、UI 结构或资产连线；本方案不触发 Unity MCP。
2. 不新增完整 NLP 基础设施，不引入完整认知本体，不新增全图实体目录服务。
3. 不在玩家正式对话链中恢复 `DialogueEntityResolver` 作为话题实体状态，也不恢复 `topic_entity_ids`、话题人物关系预查或话题实体扩张起点。
4. 可在 `RetrievalEngine` 内部使用确定性实体匹配辅助候选评分；它只影响候选边排序和诊断，不改变 `RetrievalRequest` 对外契约。
5. 若执行中需要新增 3 种以上实体类型并持续增长，必须暂停并回到方案对话评估统一实体目录 interface。
6. 冷启动规则只影响新建图。旧数据库中的历史 relationship 边是否迁移，本案第一版不静默批量改；如执行中确需迁移，必须先补充迁移方案并明确风险。

## 当前代码观察

1. `backend/src/memory/edge_semantics.py` 当前只有 `resolve_edge_clarity()`，`relationship` 分支使用传入 importance 下限 0.18 / 0.15，适合通用边语义，但不足以表达“核心人物基础认知”。
2. `backend/src/npc/state_manager.py` 的 `_init_graph()` 在创建人物节点时把 `max(0.1, bond_val)` 传给 `resolve_edge_clarity("relationship", ...)`，注释也写着“初始 clarity = bond”。这是第一处明确修复点。
3. `backend/src/memory/retrieval.py` 的 `_score_candidate_edge()` 当前以图结构分、节点类型、共享字符、玩家背景关键词和目标名命中为主；共享字符是弱语义，无法稳定支持“你知道千早在哪里吗”这类明确实体问题。
4. `_find_start_nodes()` 当前只返回 self + 当前对话对象 person 节点，符合 Workstream 已确认边界。执行时不得把“千早”之类话题实体加入 `start_node_ids`。
5. `RetrievalResult` 已包含 `start_node_ids`、`selected_edge_ids`、`retrieved_node_ids`、`vector_query_preview`、`fallback_used` 和 `diagnostics`，后端结果层已有验收所需字段。
6. Unity `aisc_debug.conversation_retrieval_snapshot` 已覆盖图起点、选边、检索节点、向量兜底和失败原因。若执行中只调整内部评分且不新增诊断语义字段，不需要改 Unity DTO；若新增 semantic score 诊断字段，必须同步 DTO、MCP action 文档和测试。
7. `backend/src/dialogue/entity_resolver.py` 已有确定性别名解析能力，但 Dialogue Workstream 和逐轮路由纠正案明确要求玩家对话链不再接入它作为话题实体状态。执行时若复用，也只能在 memory retrieval 内部作为纯评分辅助，并避免对外暴露为请求字段。

## 设计方案

### A. 初始人物关系 clarity 规则

在 `backend/src/memory/edge_semantics.py` 增加小型规则函数，建议命名：

```python
initial_relationship_clarity(recognition_importance: float = 0.9, bond: float = 0.0,
                             is_core_person: bool = True) -> tuple[float, float]
```

规则要求：

1. 函数定义前添加简短功能注释。
2. `recognition_importance` 表达“是否值得被稳定记住”，默认 0.9。
3. `bond` 只参与小幅修正，建议影响范围不超过 `±0.08`。
4. `is_core_person=True` 时，self -> person clarity 建议落在 `0.72` 到 `0.80` 的下限区间；person -> self 反向联想建议落在 `0.60` 到 `0.70` 的下限区间。
5. 高 bond 可以略微提高 clarity，但必须 clamp 到合法范围，不能突破 1.0。
6. 低 bond 不得把 clarity 压到遗忘阈值附近。
7. 保留 `resolve_edge_clarity()` 的通用 relationship 语义，避免把特殊冷启动核心人物规则塞进通用分支。

`backend/src/npc/state_manager.py` 只替换 `_init_graph()` 中人物 relationship 边的 clarity 计算：

1. 保留 `npc_bonds`、`npc_impressions` 和 `confide_level` 的 bond 逻辑。
2. 创建人物节点时继续使用 person 节点 importance 0.9。
3. relationship 边调用 `initial_relationship_clarity(recognition_importance=0.9, bond=bond_val, is_core_person=True)`。
4. 更新旧注释“初始 clarity = bond”，改成 bond 只微调基础认知 clarity。
5. 不新增人物专属 `if target_id == ...` 分支。

### B. 图候选语义评分

在 `backend/src/memory/retrieval.py` 内部调整 `_score_candidate_edge()` 的职责边界，建议拆成私有小函数，避免继续加重单函数：

1. `_score_candidate_edge()`：负责组合总分。
2. `_semantic_relevance_score(candidate, route_context)`：根据当前发言、近期对白和候选节点元数据计算语义相关分。
3. `_explicit_entity_match_score(candidate, resolved_entities)`：处理明确人物 / 地点命中。
4. `_structural_context_score(candidate, target_id, route_context)`：保留并限幅 source、edge type、mode、玩家背景关键词等结构性分数。

候选分数按下列口径组织，不要求暴露为公开公式：

```text
candidate_score
= graph_path_score
 + semantic_relevance_score
 + type_and_mode_score
 + target_context_score
```

执行细则：

1. `graph_path_score` 继续由 `_graph_candidate_rank_value()` 提供，但考虑限幅 `source_is_target` 和反向 `involved` 的叠加效果，避免结构分完全淹没当前发言。
2. 明确实体命中应强于共享字符弱命中，足以在 Top-N 竞争中把被点名人物 / 地点对应节点或边推进候选。
3. 玩家背景关键词只在没有明确实体，或候选本身也与当前话题相关时发挥主要作用；有明确人物 / 地点时，不应让“喫茶店、奶奶、从城市回来、樱桥通”自动压过点名实体。
4. 普通文本相关第一版可以使用确定性别名、稳定 ID、显示名、近期对白文本和候选 value 的轻量匹配；不要为了本修复引入新的 embedding 打分路径。
5. 若候选是 relationship person 节点且命中明确人物名，不再因最终 rerank 中 relationship / person 惩罚导致完全掉出最终上下文；可在 `_rerank_node_data()` 中增加同一套话题相关补偿，但不得让无关人物关系泛滥。
6. 向量兜底仍只在图结果不足时触发；修复目标是让图路径正常命中，不是提高兜底优先级。

### C. 实体识别方式

第一版不改 `RetrievalRequest`，不把实体列表从对话层传入检索器。实体匹配在 `RetrievalEngine` 内部从以下文本构造：

1. `route_context["query_text"]`
2. `route_context["recent_turns"]` 最近 4 到 6 轮文本
3. 必要时 `route_context["conversation_summary"]`

执行优先级：

1. 优先复用现有 NPC 名称常量、玩家名称候选和共享地点配置的稳定 ID / display name。
2. 可以复用 `DialogueEntityResolver` 的确定性别名索引逻辑，但不得在玩家正式对话链重新持有“已解析话题实体状态”。
3. 如果直接复用 `DialogueEntityResolver` 会引入路径或配置初始化成本，可在 `RetrievalEngine` 内部先实现小型纯函数，覆盖 NPC 名称、玩家候选、地点 zone label；执行记录必须说明为何没有复用。
4. 地点实体第一版优先匹配 zone 级 ID 和 label；spot 级 display name 若当前共享配置没有稳定 label，不在本轮扩张。

## 风险与控制

1. clarity 提高导致图过度发散：只提高初始核心人物 relationship 边，不统一抬高所有 relationship。
2. 明确实体加分过高造成关键词吸附：用“喫茶店最近怎么样”反例保护玩家背景回忆。
3. 最终 rerank 压掉人物 relationship 节点：执行中检查 `_rerank_node_data()`，必要时只对“已明确命中的人物 / 地点”补偿。
4. 重新引入话题实体起点扩张：测试必须断言 `start_node_ids` 不包含被点名但非参与者的人物节点。
5. 旧库历史边不变导致手动复测看不到冷启动修复：execution 必须说明需要新开局或显式重建初始图，不做静默迁移。
6. Windows 下中文 CLI 参数可能乱码：真实 LLM 或脚本验证如需中文 prompt，优先用测试文件或 UTF-8 文件输入，避免直接把长中文作为命令行参数。
7. 并行 Python 测试可能争写 `__pycache__`：不要并行运行会导入同一模块的 Python 编译和测试命令。

## 验收标准

1. 九条莲以低 bond 初始化后仍清楚认识千早，初始 relationship clarity 不再接近遗忘阈值。
2. 玩家问“你知道千早在哪里吗”时，图路径优先命中千早关系边，而不是先取无关玩家背景。
3. 图中没有千早地点事实时，回答只能表达不知道或有限认知，不能因为检索修复制造位置事实。
4. 玩家问喫茶店或玩家经历时，相关玩家背景仍能正常命中。
5. 现有玩家逐轮对话检索边界不回退：当前发言和近期对白刷新，图起点不扩张。
6. 现有对话记忆、夜间印象和路由 profile 相关测试不回退。
7. `aisc_debug` 或后端结构化结果能说明实际起点、选边、最终节点、是否兜底和失败原因。
