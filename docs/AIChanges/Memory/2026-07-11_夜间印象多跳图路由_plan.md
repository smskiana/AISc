# 夜间印象多跳图路由 — 设计方案

## 需求理解

白天对话已经接入多跳图路由，但夜间基准印象刷新仍然使用旧逻辑：

1. 用目标名字做一次向量搜索
2. 取 top_k 片段作为 `graph_memories`
3. 交给 LLM 生成熟人基准印象

这会让夜间印象没有真正复用“沿图联想”的记忆模型，也会让向量补充在夜间仍偏主路。

本轮目标：

1. 给夜间印象单独做一套更宽松的多跳路由参数
2. 让夜间 `graph_memories` 优先来自多跳图路由
3. 保留现有夜间并发池，避免把路由过程串行化

## 参数设计

新增模式：`nightly_impression`

```python
"nightly_impression": {
    "max_edges_per_hop": 8,
    "edges_per_route": 3,
    "max_hops": 2,
    "short_term_limit": 6,
    "short_term_days": 7,
    "min_graph_nodes": 4,
    "vector_fallback_limit": 3,
    "vector_search_top_k": 8,
    "final_memory_limit": 6,
}
```

## 与白天模式的区别

1. 夜间不需要即时响应，所以每跳候选边更宽
2. 夜间目标是整理和压缩印象，所以短期记忆窗口更长
3. 夜间允许更多向量兜底，但仍以图路由结果为主
4. 夜间不加到 3 跳，避免 `NPC x target` 的 LLM 路由数量膨胀过快

## 实现方案

### 1. 检索层

在 `backend/src/memory/retrieval.py` 增加 `nightly_impression` 参数，并为该模式补充路由偏好：

1. 优先沿与目标人物、近期互动、核心关系相关的边整理
2. 可以比白天更宽松地纳入远一点的旧事
3. 仍避免无关跳转

### 2. 状态层

给 `StateManager` 增加 `set_retrieval()` 注入点。

夜间生成任务中：

1. 构建任务时只收集短期记忆、当前印象、目标信息
2. 每个并发任务内部调用 `_graph_target_memories()`
3. `_graph_target_memories()` 优先调用 `retrieval.retrieve(..., mode="nightly_impression")`
4. 如果检索器不可用或失败，再回退到旧的目标名向量搜索

这样图路由和印象生成仍在同一个夜间并发池内，不会先串行跑完整个图路由阶段。

### 3. 运行时接线

在后端启动和向量服务刷新时，将 `retrieval` 注入 `state_mgr`。

## 验证计划

1. Python 编译检查：
   - `backend/src/memory/retrieval.py`
   - `backend/src/npc/state_manager.py`
   - `backend/src/application/runtime.py`
2. 冷启动后执行一次 `nightly_refresh(1)` smoke test
3. 直接调用 `_graph_target_memories("kujo", "player", ...)`，确认结果来自多跳图路由而不是单纯向量 top_k

## 后续可调项

如果午夜耗时偏高，第一优先调：

1. `edges_per_route: 3 -> 2`
2. `max_edges_per_hop: 8 -> 6`
3. `final_memory_limit: 6 -> 4`

如果印象仍然浅，优先补 NPC 间共享旧事和店铺日常长期记忆，而不是继续放大跳数。
