# 玩家印象单目标并发午夜重构：执行方案

## 1. 需求与依据

完整 Day 4 午夜实测总耗时 `154.89s`：`impression_state=99.72s`（64.4%），`event_extraction=47.05s`（30.4%）。当前午夜为 5 名 NPC 分别刷新玩家和其余 4 名 NPC，共执行 25 次 `memory_direction` 与 25 次 `nightly_impression`。

本方案按用户确认的取舍调整为：

1. 午夜只生成 5 条 `NPC -> player` 基准印象。
2. 停止午夜生成 NPC-NPC 基准印象，接受其叙事精细度下降以换取性能。
3. 先冻结玩家印象需要的路由和上下文，再并发执行玩家印象生成与当日事件提取。
4. 评估并收敛 `memory_direction` 在夜间固定目标场景中的职责。

Workstream：Memory 为主，关联 NpcBehavior、Dialogue、TestingAndDiagnostics。相关 ADR 为 `ADR-0001-memory-v07-light-graph`；本轮不改变图/向量数据边界，不新增 ADR。

## 2. 已确认语义

### 2.1 NPC-NPC 印象不删除，只停止午夜重建

现有 `npc_impressions` 仍被 NPC 社交意愿、NPC-NPC 对话语气和对话记忆上下文读取，因此不得删除 NPC-NPC 行：

1. 冷启动创建的 NPC-NPC `baseline_impression / speech_hint / approach_bias` 继续保留。
2. 白天真实互动产生的 `delta_note / delta_bias` 继续更新和生效。
3. 午夜不再为 NPC-NPC 调用 `memory_direction` 或 `nightly_impression`，也不清空其 delta。
4. 午夜只重建 `owner_id=<npc> / target_id=player`，并仅对该行清空已被吸收的玩家关系 delta。

这是明确的产品取舍：NPC-NPC 关系仍可由冷启动事实和白天即时变化驱动，但不再每天用 LLM 压缩成长线基准印象。

### 2.2 次日基准状态与陈旧 NPC-NPC 印象解耦

当前 `_refresh_baseline_state()` 会平均 owner 的全部 `approach_bias`。停止 NPC-NPC 午夜重建后，这个平均值会混入长期陈旧基线，因此必须调整：

1. `emotion_baseline` 与 `lingering_concern` 保留现有当天状态/事件计算。
2. `sociability_baseline` 由 profile `social_base`、当日 `sociability_delta` 和玩家印象的受限 bias 计算。
3. NPC-NPC 陈旧基线不再参与次日全局 sociability 平均值。
4. 具体权重沿用现有量级并集中在一个命名策略函数中，禁止散落 magic number。

## 3. `memory_direction` 取舍

### 3.1 夜间取消，白天策略不变

本方案不再为 `nightly_impression` 调用 LLM `memory_direction`，理由：

1. 午夜目标固定为玩家，不需要 LLM 解析目标人物。
2. 查询意图固定为“关系、近期互动、共同事件和印象依据”，没有玩家自然语言中的代词、描述性称呼和开放问题。
3. 当前 5 次夜间方向调用只是在不同 owner 上重复理解同一模板。
4. 完整实测表明本地图搜索约 0.2s/次；主要成本来自供应商调用。

夜间改用受控确定性方向：

- `entity_mentions`：当前玩家显示名及稳定 player aliases。
- `themes`：`relationship / recent_activity / past_event / emotion`。
- `relation_facets`：`affinity / shared_event / impression_basis`。
- `time_scope`：`recent`。
- `recall_intent`：固定的玩家关系回忆枚举。
- `retrieval_query`：由本地模板和玩家显示名生成，不接收任意自由文本。

`player_dialogue` 的 `llm_guided_local` 和其他显式选择该策略的白天请求继续使用 `memory_direction`，不改变公开三策略契约。

### 3.2 配置语义

不得把 `nightly_impression.strategy` 静默改名为 `local_only` 后假装业务语义完全相同。执行时应在 typed policy 中增加明确的夜间固定方向来源，或由午夜协调器构造受控 `RetrievalRequest` 执行本地方向检索，并在 trace 中记录：

- `direction_source=nightly_fixed_player`
- `llm_direction_calls=0`
- `target_id=player`

如果实现需要新增枚举，应使用稳定 `snake_case`，并同步配置校验、诊断 DTO 和测试。

### 3.3 夜间图路由敲定参数

夜间固定玩家方向沿用已完成的 Day 4 真实图平衡测试结论，执行时直接采用以下参数，不重新估算默认值：

| 参数 | 敲定值 |
|---|---:|
| `max_depth` | 8 |
| `beam_width` | 12 |
| `max_neighbors_per_node` | 16 |
| `max_expanded_edges` | 48 |
| `max_anchor_count` | 6 |
| `max_answer_candidates` | 20 |
| `min_path_score` | 0.18 |
| `early_stop_margin` | 0.12 |

其中 `max_depth=8` 与 `beam_width=12` 是已敲定的质量边界：25 个真实图组合中，平衡档相对超高预算参考的平均节点保持率为 `98.67%`，23/25 完全一致，最差保持 5/6；将深度降到 7 或宽度降到 10 时，平均保持率均降至 `94%`，低于 95% 验收线。因此执行不得为了进一步压缩本地耗时擅自改成 `7 / 10` 或更低值。

## 4. 新午夜数据流

```text
edge_decay
-> prepare_player_impression_routes
   - 为 5 名 NPC 读取当日短期记忆
   - 使用夜间固定方向完成图/向量路由
   - 冻结 graph_memories / recent_memories / previous player impression
-> concurrently
   A. event_extraction（写新图节点/边）
   B. generate_player_impressions（只生成结果，不写 SQLite）
-> graph_evolution（处理提取后的最新图）
-> commit_player_impressions（主线程顺序写 5 行）
-> refresh_next_day_baselines
-> stm_cleanup
```

重要语义：路由预处理发生在事件提取前，因此当晚新提取的图节点不会进入同一晚的 `graph_memories`；但当日相关短期记忆已经冻结到 `recent_memories`，仍会直接提供给玩家印象 Prompt。这样可以并发两个重阶段，同时不丢失当日事件文本。新图节点从下一次检索开始参与图路由。

## 5. 并发与写入边界

1. 玩家印象分支只做 5 次有界并发 LLM 生成，返回纯结果，不在线程池中写 SQLite。
2. 事件提取保持按 NPC 有界并发，并写入隔离的图/向量存储路径。
3. 两个分支通过 `asyncio.gather(..., return_exceptions=True)` 汇合；每个分支必须有独立成功数、失败数和失败原因。
4. 汇合后由协调线程顺序提交玩家印象，避免 SQLite 多线程写竞争。
5. 图演化必须等待事件提取结束，保证新节点可以在当晚参与衰减后的演化/融合。
6. 单个玩家印象失败沿用启发式 fallback；事件提取局部失败必须显式标记午夜为 `partial_failure`，不能只写日志后仍报告无错误成功。
7. 同步修复已发现的提取边缺少 `from/to` 时 `KeyError`：非法边应计数并跳过，合法节点和合法边继续入图；测试覆盖缺 `from`、缺 `to` 和未知临时 ID。

## 6. 类减重与模块边界

`StateManager` 和 `GameRuntime` 已承担过多职责，本轮禁止继续把预处理、并发编排、结果提交和诊断聚合塞进原类。

执行时新增深模块，例如：

- `backend/src/memory/midnight_coordinator.py`
  - `prepare_player_impression_inputs(...)`
  - `run_parallel_heavy_stages(...)`
  - `commit_player_impressions(...)`
  - `MidnightMaintenanceResult`
- `backend/src/npc/player_impression_refresh.py`
  - 固定玩家目标的输入 DTO、5 次生成、fallback 和次日基准更新策略。

`GameRuntime` 只调用协调器并发送阶段/终态；`StateManager` 保留兼容 facade 和白天状态接口，不再拥有完整午夜编排。

执行前用户必须在新对话中明确给出“按该 plan 减重并执行”。

## 7. 诊断同步

业务语义和执行阶段发生变化，必须同步结构化诊断：

1. 午夜 operation ID、当前阶段和总状态：`running / success / partial_failure / failed`。
2. 路由预处理：目标数固定 5、方向来源、LLM 方向调用数、每 owner 检索 trace ID。
3. 玩家印象：计划数、成功数、fallback 数、失败 owner IDs、耗时。
4. 事件提取：有事件 NPC 数、成功数、失败数、非法节点/边计数、耗时。
5. 并发墙钟与两个分支各自耗时，验证收益来自重叠而不是漏执行。
6. 图演化、提交、状态刷新和清理耗时。
7. `aisc_debug` 增加或扩展午夜快照 action；`aisc_control` 的午夜触发/等待入口若已有则复用，没有稳定入口时增加编辑器专用白名单 action。
8. 后端入口文档、Unity Diagnostics README 和对应 DTO/provider 测试同步更新。

## 9. 测试与验收

### 9.1 功能测试

1. 午夜只创建/更新 5 条 `NPC -> player` 基准印象。
2. NPC-NPC 基准印象行保持不变，白天 delta 不被午夜清空。
3. 玩家印象吸收并清空自身 delta，不影响其他目标。
4. `memory_direction` 夜间调用为 0；玩家对话仍按原策略调用。
5. 夜间路由 trace 明确记录 `max_depth=8`、`beam_width=12`，并覆盖配置加载回归测试。
6. 预处理后的玩家印象包含当日 `recent_memories`，即使相应图节点尚未提取。
7. 提取与印象生成实际重叠执行；图演化严格等待提取结束。
8. 缺 `from/to` 的 LLM edge 不再中断该 NPC 的其余合法提取。
9. 局部失败返回 `partial_failure`，诊断包含 owner、阶段和原因。

### 9.2 性能验收

使用与 `2026-07-17_midnight_full_profile` 同源的隔离 Day 4 快照和真实 LLM 至少复跑 3 次：

1. 每轮 `nightly_impression` 调用数从 25 降为 5。
2. 每轮 `memory_direction` 调用数从 25 降为 0。
3. 玩家印象与提取并发墙钟接近两者较慢者，而不是两者之和。
4. 完整午夜 P50 目标不高于 65s；若供应商单次提取仍出现约 40s 长尾，优先保证总耗时不再叠加旧 99s 印象阶段。
5. 5 条玩家印象均有非空结果；路由节点与现有高预算参考保持率不低于 95%。
6. 正式数据库哈希不变，所有真实 LLM 测试使用复制后再初始化的隔离快照，禁止直接打开权威快照源。

### 9.3 回归范围

1. 后端记忆、状态、行为、对话和世界准备定向测试。
2. 完整后端测试套件。
3. Unity EditMode 诊断 DTO/provider/MCP action 测试。
4. Unity Play Mode 触发一次午夜，用 `aisc_control` 驱动并由 `aisc_debug` 验证阶段、终态和次日可操作状态。

## 10. 风险与回退

1. NPC-NPC 长期关系不再每日压缩，社交表现可能更依赖冷启动基线和最近一次 delta；这是已接受的性能取舍，但必须在 execution 明示。
2. 路由先于提取意味着新图节点晚一夜进入图路由；当日短期记忆直接进入 Prompt 作为补偿。
3. 供应商并发限制可能让“5 次玩家印象 + 提取”相互争抢；全局并发需有上限，并记录 429、超时和 fallback。
4. 取消夜间 `memory_direction` 可能损失自由语义方向；固定玩家目标评估保持率低于 95% 时停止执行并回到本 plan 调整，不得临时恢复 5 次 LLM 而不记录口径变化。
5. 并发阶段不得共享未冻结的可变任务字典，也不得从工作线程直接提交玩家印象。
