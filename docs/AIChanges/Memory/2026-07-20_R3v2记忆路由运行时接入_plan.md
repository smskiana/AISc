# R3 v2 记忆路由运行时接入 - 执行案

## 1. 方案定位

本方案以 `MemoryRouteR3V2RuntimeIntegrationDraft.md` 为需求来源，并以当前代码和 canonical `AISc` codebase-memory 图为依据，作为后续实现的唯一执行案。本会话只创建 plan，不修改运行代码、配置、测试或模型产物，不创建 execution / test。

- Workstream：`MEM` 记忆系统；运行时装配涉及 Application，检索验收涉及 TestingAndDiagnostics。
- Roadmap item：MEM 近期目标“保持轻量图 + 向量数据层，补诊断与质量评估”。本方案不改变路线图排期，由用户决定何时进入执行会话。
- 相关 ADR：`ADR-0001-memory-v07-light-graph`。
- 当前口径变化：只把 R3 v2 设为 `llm_guided_local` 的默认方向 provider；保留 `local_only`、原通用 LLM 定向、本地深搜和 `llm_full_route`。
- 不新建 ADR：不改变每 NPC 独立图、图/向量分工、权限、archived、clarity、图起点、向量兜底或存档边界，也不引入网络服务或通用模型平台。

## 2. 当前代码级结论

### 2.1 已存在且必须保持的 seam

1. `backend/src/memory/retrieval_contracts.py` 已定义同步接口 `DirectionProvider.provide(request, context) -> DirectionResolution`。
2. `backend/src/memory/retrieval_direction.py` 的 `DirectionResolver.resolve()` 已统一执行方向字段清洗、本地语义校准和 mention 解析。
3. `LocalDirectionProvider` 与 `LlmDirectionProvider` 已是正式实现；后者在不可用、超时、非法 JSON 或空输出时返回确定性本地方向和稳定 `llm_*` 原因。
4. `RetrievalPolicyRegistry` 启动时严格读取 `backend/config/memory_retrieval.yaml`，当前只接受 version 1、固定三种 mode 和既有预算字段。
5. `RetrievalEngine.retrieve()/probe()` 是正式 facade；图起点、本地深搜、单次向量查询、最终重排、clarity 恢复和 trace 已形成独立 seam。
6. `GameRuntime.start()/stop()` 是后端生命周期入口；`_refresh_vector_services()` 会在存档恢复后重建 `RetrievalEngine`，因此 provider runtime 不能随 engine 一起重复启动。
7. 训练端 `backend/training/memory_route/common.py` 当前独立保存 `SYSTEM_PROMPT`、方向字段白名单和训练文本模板；`evaluate_route_specialist.py` 直接加载模型并确定性生成，尚未经过正式 worker 或 provider chain。

### 2.2 当前必须修正的装配问题

`RetrievalEngine._retrieve_result()` 目前在请求内根据 strategy 临时构造 `LocalDirectionProvider` 或 `LlmDirectionProvider`。R3 v2 接入不能继续在该方法增加模型专用分支；必须把 provider 注册、链选择和生命周期移到注入的深模块，engine 只读取本次已冻结的 chain。

`DirectionResolver` 当前已包含上下文裁剪、字段清洗、语义校准和 mention 解析，不能再承接模型加载、Prompt 模板、子进程协议、超时、重启或 provider 回退编排。

训练 input DTO 与当前 `DirectionResolver.build_context()` 的字典并不完全相同。运行时 codec 必须显式构造 schema v1 input，不能直接把现有 context 字典原样发送给 R3 v2，否则会形成训练/运行模板漂移。

### 2.3 codebase-memory 影响链

canonical `AISc` 图显示 `init_retrieval()` 的直接调用者为 `GameRuntime.start()` 与 `GameRuntime._refresh_vector_services()`；其上游还覆盖正式对话、协议消息和七日跑测。执行时必须同时处理首次启动和存档恢复后的 engine 重建，不能只修启动路径。

## 3. 执行前置授权与会话边界

本方案预计新增不少于 7 个文件、修改不少于 10 个文件，必须在独立执行会话实施，并在独立测试会话验收。

`RetrievalEngine` 和 `GameRuntime` 都已承担较多职责。后续执行指令必须明确为“按该 plan 减重并执行”：

1. 对 `RetrievalEngine` 的减重是移除请求内 provider 构造，改为依赖注入；不得趁机重写图检索、向量查询或最终重排。
2. 对 `GameRuntime` 只增加 provider runtime 的装配、异步预热和关闭调用；进程管理细节全部留在 Memory 深模块，不继续堆入 runtime。
3. 若执行分析发现仍需把 worker 状态机、协议读取或 provider 分支塞进上述类，立即停止并回到方案会话调整本 plan。

## 4. 冻结的功能决策

### 4.1 三策略和默认 mode

| mode | strategy | 方向来源 | R3 v2 行为 |
|---|---|---|---|
| `player_dialogue` | `llm_guided_local` | 默认 `[r3_v2, local]` | 调用一次专项方向模型，失败立即本地回退 |
| `npc_dialogue` | `local_only` | 本地方向 | 不加载请求、不调用模型 |
| `nightly_impression` | `local_only` | 固定/本地方向 | 不加载请求、不调用模型 |

`llm_full_route` 继续使用现有 `LlmGraphRetriever` 逐跳选择边；R3 v2 不参与逐跳路由。`request.direction_override` 继续绕过 provider chain。

原通用实现保留为稳定 provider ID `general_llm`。配置 `[general_llm, local]` 时，Prompt、一次通用 LLM 调用、本地校准、失败原因、预算和最终检索结果必须与接入前等价。

### 4.2 R3 v2 冻结身份

| 字段 | 值 |
|---|---|
| provider ID | `r3_v2` |
| model | `Qwen/Qwen3-0.6B` |
| model/tokenizer revision | `c1899de289a04d12100db370d81485cdf75e47ca` |
| adapter ID | `route-lora-r3-v2-approved-480` |
| `adapter_model.safetensors` SHA-256 | `cd2676f7f64f28a351fb35b2d2d76fa01b30662a509bf7bbddcced6f9cf92b8d` |
| quantization | NF4、BF16 compute、double quant |
| decoding | `do_sample=false`、thinking 关闭、`max_new_tokens=384` |
| direction/input schema | v1 |

模型路径、Adapter 路径、HF cache 和 venv 都不进入仓库或诊断。Adapter hash 只计算 `adapter_model.safetensors`，与前置独立测试证据保持同一口径。

2026-07-21 方案勘误：原文中的 Adapter SHA-256 在 `509bf7bbdcced` 片段少抄了一个 `d`，形成 63 位值。已对项目外批准产物 `adapter/adapter_model.safetensors` 重新计算 SHA-256，确认上表 64 位值；执行会话仍须在写入配置前独立复算并同时校验长度、字符集和实物一致性。

## 5. 目标模块与职责

### 5.1 `route_specialist_contract.py`

新增生产侧共享契约与 `SpecialistRouteCodec`，职责限定为：

1. 定义唯一 `DIRECTION_FIELDS`、R3 v2 system prompt 和 schema v1 input 字段。
2. 把 `RetrievalRequest + 已裁剪 context` 映射为稳定、紧凑、UTF-8 的 schema v1 input；字段名必须与训练数据一致：`schema_version`、`npc_id`、`query_text`、`conversation_summary`、`recent_turns`、`recent_memories`、`location_id`、`location_display_text`、`game_time_snapshot`、`participant_ids`、`known_entity_aliases`、`mode`。
3. 根据正式人物/地点目录补稳定 alias；只纳入当前问题、近期上下文、参与者或当前位置可证明的别名，不向模型暴露完整私有人物/地点目录。
4. 生成唯一 chat messages；训练和 worker 都调用同一方法，并统一 `enable_thinking=False`。
5. 严格解析单一 JSON object：拒绝 Markdown fence、前后自然语言、未知/缺失字段、错误类型、非法枚举、超限数组/文本、节点 ID、边 ID和推理文本。
6. 对 entity/location mention 做输入来源校验；无法从输入、稳定参与者或当前位置证明的 mention 返回 `specialist_semantic_rejected`，不得交给后续图检索。
7. 只构造 `RetrievalDirection`，不做本地校准、mention ID 绑定、图查询或 provider 回退。

`backend/training/memory_route/common.py` 改为从该生产模块导入/复用 prompt、字段白名单、方向序列化和 messages 构造；训练目录可以依赖生产契约，生产代码禁止反向依赖 `backend/training/`。现有训练 import 可通过窄 re-export 暂时兼容，避免无关脚本一次性重写。

### 5.2 `route_specialist_worker.py`

同一文件包含轻量父进程 adapter 和可执行 worker 入口，但父进程 import 时不得导入 `torch / transformers / peft / bitsandbytes`；重依赖只在 worker main 内延迟导入。

`RouteSpecialistWorkerAdapter` 负责：

1. 从配置给出的环境变量名解析 Python、Adapter 和 HF cache；不接受仓库配置中的机器绝对路径。
2. 后台校验 Python 可执行、`adapter_model.safetensors` SHA-256、Adapter manifest/revision，然后启动 `python -m backend.src.memory.route_specialist_worker`。
3. 使用独立 stdout reader 线程读取 versioned JSONL，避免 Windows pipe 无法可靠 `select` 的问题；stderr 由单独 reader 只输出安全状态日志。
4. 以非阻塞互斥实现单 worker、容量 1；已有在途请求时立即返回 `specialist_busy`，不排无限队列。
5. 每个请求用 request ID 关联；超时、EOF、协议版本错误、错配 ID 或超长行都终止当前 worker，并进入固定冷却后允许一次有界重启。
6. `close()` 幂等：先发 shutdown，短等待后 terminate，最后才 kill；关闭期间和关闭后不再重启。
7. 提供只读健康快照，不暴露 prompt、完整 input、原始模型输出、环境变量值或绝对路径。

worker JSONL 协议 version 固定为 1：

```text
worker stdout ready: {protocol_version,event=ready,model_id,revision,adapter_id,schema_version}
parent stdin infer:  {protocol_version,request_id,operation=infer,input}
worker stdout result:{protocol_version,request_id,status=ok|error,output?,reason?,inference_ms}
parent stdin close:  {protocol_version,operation=shutdown}
```

stdout 只允许协议行，日志只写 stderr。第一版不使用 HTTP、端口、socket、多 worker、批推理或共享模型服务。

### 5.3 `route_specialist_provider.py`

新增三个深 seam：

1. `R3V2DirectionProvider`：调用 codec 和 worker adapter，把成功输出适配为 `DirectionResolution`；失败只返回稳定原因，不执行图搜索或本地校准。
2. `ChainedDirectionProvider`：按冻结的 provider tuple 顺序调用，第一个无失败原因的结果生效；失败原因和耗时追加到有界诊断，最终 local 终点保证可检索。
3. `DirectionProviderRuntime`：根据严格配置用 builder mapping 注册 `r3_v2`、`general_llm`、`local`，提供 `chain(ids)`、`warmup()`、`close()` 和健康快照。

三种及未来更多 provider 通过 `kind -> builder` 映射扩展，不在 `RetrievalEngine` 或调用方增长专项 `if/elif`。provider chain 在请求开始时解析成不可变 tuple；请求中途配置或 worker 状态变化不能替换链顺序。

### 5.4 `RetrievalEngine` 与 `DirectionResolver`

1. `RetrievalEngine.__init__()` 和 `init_retrieval()` 增加可注入的 `DirectionProviderRuntime`。
2. `_retrieve_result()` 只根据 strategy 决定“无需 provider / local / policy chain”，不再构造具体 R3 或通用 LLM provider。
3. 存档恢复后的 `_refresh_vector_services()` 必须复用同一个 provider runtime，只重建依赖新 SQLite/Lance 句柄的 engine。
4. `DirectionResolver.resolve()` 继续只做最终被采用方向的清洗、校准和 mention 解析，并原样传递 provider 诊断；不得对每个失败 attempt 重复校准。
5. 原 `LlmDirectionProvider` 保留现有行为。chain 以 `failure_reason` 判断是否继续，并丢弃其内部生成的 fallback 方向，再调用正式 `local` 终点；因为本地 provider 纯函数无副作用，最终方向与旧行为等价。

### 5.5 Application 生命周期

1. `GameRuntime.start()` 先同步完成配置结构校验和 provider runtime 装配，再把它注入 `init_retrieval()`。
2. R3 worker 预热作为受管后台任务启动，不阻塞后端 READY；在 ready 前的正式请求返回 `specialist_unavailable` 并立即采用 local。
3. 配置结构错误仍使启动失败；外部 Python/Adapter/hash/CUDA/模型加载失败只把 R3 provider 标为 unavailable，不阻止后端 READY。
4. `GameRuntime.stop()` 在取消业务后台任务后关闭 provider runtime；无论 worker 正在加载、推理、冷却还是已退出，关闭都必须有界完成。
5. provider runtime 由 `GameRuntime` 持有，不进入 `AppServices` 的领域服务集合；engine 只持窄引用。

## 6. 配置 version 2

`backend/config/memory_retrieval.yaml` 根级字段升级为严格的 `version`、`direction_providers`、`modes`。现有三 mode 的检索预算和值保持不变，只新增 provider 配置和 `direction_chain`。

```yaml
version: 2
direction_providers:
  default_chain: [r3_v2, local]
  providers:
    r3_v2:
      kind: subprocess_specialist
      model_id: Qwen/Qwen3-0.6B
      revision: c1899de289a04d12100db370d81485cdf75e47ca
      adapter_id: route-lora-r3-v2-approved-480
      adapter_sha256: cd2676f7f64f28a351fb35b2d2d76fa01b30662a509bf7bbddcced6f9cf92b8d
      python_env: MEMORY_ROUTE_PYTHON
      adapter_path_env: MEMORY_ROUTE_R3_V2_ADAPTER
      hf_home_env: HF_HOME
      timeout_ms: 16000
      max_new_tokens: 384
      restart_cooldown_ms: 5000
    general_llm: {kind: general_llm}
    local: {kind: local}
modes:
  player_dialogue:
    strategy: llm_guided_local
    direction_chain: [r3_v2, local]
```

严格校验规则：

1. 使用拒绝重复 mapping key 的 YAML loader；重复 provider、重复字段、未知根/块字段均失败。
2. provider ID、kind、chain 成员必须已注册；chain 非空、无重复且以 `local` 结束。
3. `subprocess_specialist` 必须完整包含冻结字段；hash 为 64 位小写十六进制；env 字段只允许环境变量名，拒绝路径值。
4. timeout、token 和 cooldown 使用显式安全范围；布尔值不能冒充整数。
5. `direction_chain` 可保留在任意 mode 便于切换 strategy，但只在 `llm_guided_local` 生效；`local_only` 和 `llm_full_route` 不解析、不预占 worker 请求。
6. 第一版无热加载。配置 version 和本次有效 chain 进入 trace。

## 7. 诊断与兼容口径

### 7.1 新增不可变诊断契约

在 `retrieval_contracts.py` 增加小型 `DirectionProviderAttempt` / `DirectionProviderDiagnostics`，并由 `DirectionResolution` 持有；至少映射以下安全字段：

- `direction_provider_requested`
- `direction_provider_adopted`
- `direction_provider_chain`
- `direction_model_id`
- `direction_model_revision`
- `direction_adapter_id`
- `direction_schema_version`
- `direction_inference_ms`
- `direction_queue_ms`
- `direction_fallback_reasons`
- `direction_worker_state`
- `direction_model_call_count`

attempt 只保存 provider ID、稳定状态、失败枚举和毫秒数，最大长度受 chain 长度限制。不得保存完整对白、Prompt、原始输出、记忆文本或绝对路径。

### 7.2 旧字段保持

1. `llm_route_calls` 继续只计算 `llm_full_route` 逐跳调用。
2. `llm_direction_calls` 继续表示原通用 `general_llm` 方向来源的既有口径，不把 R3 v2 混入。
3. `direction_model_call_count` 计算 chain 中实际发出的 R3/general 方向模型调用；默认 `[r3_v2, local]` 上限为 1，显式 `[r3_v2, general_llm, local]` 可为 2。
4. `degraded_to_local` 改从 provider diagnostics 判断，而不是依赖 `source.startswith("llm_")`，同时保持 `[general_llm, local]` 失败时的旧 failure reason。
5. 新 R3 成功的 `direction_source` 为 `r3_v2`；失败回退时 primary failure 保留具体原因，`direction_provider_adopted=local` 明确最终来源。

### 7.3 稳定失败原因

第一版固定：

```text
specialist_unavailable
specialist_manifest_mismatch
specialist_load_failed
specialist_busy
specialist_timeout
specialist_worker_exited
specialist_protocol_invalid
specialist_invalid_json
specialist_schema_invalid
specialist_semantic_rejected
specialist_fallback_local
specialist_fallback_general_llm
```

非法输出只拒绝本次 attempt；超时、进程退出和协议损坏同时触发 worker 终止与冷却。第一版不引入 confidence。

## 9. 执行顺序

### 阶段 A：配置和纯契约

1. 扩展不可变 policy/provider DTO 和严格 YAML v2 解析。
2. 新建 codec，把 system prompt、字段白名单和序列化权威迁到生产侧。
3. 改训练 common 复用 codec，先以 golden test 锁定迁移前后 prompt 字节一致。

### 阶段 B：worker 隔离

1. 完成 versioned JSONL worker main 和父进程 adapter。
2. 用 fake worker 覆盖 ready、成功、busy、超时、退出、坏协议、坏 JSON、关闭与冷却。
3. 再用真实训练 venv 做一次模型加载/关闭 smoke；不得在项目默认 Python 导入重依赖。

## 11. 独立测试会话验收

独立测试先读取本 plan、对应 execution 和 TestingAndDiagnostics README，再创建互链 test。最低覆盖：

1. 重新计算 `adapter_model.safetensors` SHA-256，确认 revision、tokenizer revision、schema、thinking 和量化与冻结值一致。
2. 对同一批准的冻结字段集复跑 direct evaluator；schema 合法率、明确实体召回、未知实体数和字段准确率不得低于 R3 v2 已保存基线。
3. 使用带 `expected_node_ids`、明确允许/拒绝权限和隔离 SQLite/LanceDB 的 corpus，分别运行 `r3_v2`、`general_llm`、`local`。
4. 玩家默认成功路径必须证明 `requested=r3_v2`、`adopted=r3_v2`；故障注入必须证明 adopted=local、原因稳定且最终检索不中断。
5. `local_only`、`llm_guided_local + general_llm`、`llm_guided_local + r3_v2`、`llm_full_route` 各自验收；原九组合离线结果不得回退。
6. 证明默认 R3 路径方向模型调用不超过 1、向量查询不超过 1、`llm_route_calls=0`，且 clarity/持久日志在 probe 中无副作用。
7. 在 RTX 4060 Laptop GPU 同等条件下记录预热墙钟、常驻/峰值显存、推理 p50/p95/p99、端到端检索和对话等待；预热后 p95 不得高于冻结 R3 v2 独立 p95 的 120%。
8. 注入缺 Python、缺 Adapter、hash 错误、CUDA 不可用、busy、timeout、worker exit、坏 JSON、schema 错误、语义越权、关闭中请求；后端 READY、回退和有界退出必须符合本 plan。
9. 所有测试只使用隔离数据和项目外模型资产，不修改正式存档、clarity、配置或 Adapter。

## 12. 风险与停止条件

1. R3 v2 当前约 11–13 秒级推理可能不满足对话体验；接入成功不等于体验通过，不能靠无限提高 timeout 掩盖。
2. Windows pipe、加载中终止和 CUDA 进程退出是首要工程风险；只要 timeout 不能可靠杀停或 stop 无界等待，停止默认接入。
3. codec golden test 若不能证明训练与运行 messages、紧凑 JSON 和 thinking 参数完全一致，停止实施，不允许接受“语义大致相同”。
4. `[general_llm, local]` 若在 Prompt、方向、失败原因、调用计数或最终节点上不等价，停止默认切换并先修兼容层。
5. 隔离检索出现越权、重复 ANN、固定起点变化、clarity 副作用或 expected node 命中低于基线，不得认定整体通过。
6. 模型/Adapter/revision/tokenizer/system prompt/schema/量化/generation 参数任一改变都必须使用新 provider revision 并重新冻结，不能继续沿用 `r3_v2`。
7. 若执行时发现需要网络模型服务、多 worker 池、配置热加载、共享 GPU 调度或协议/UI 改动，视为超出本 plan，停止并另开方案。
