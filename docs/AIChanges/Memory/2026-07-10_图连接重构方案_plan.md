# 图连接重构方案 — 计划

## 完成时间
2026-07-10

## 这次方案要解决什么

当前项目的“图记忆”已经能工作，但更准确地说，它现在是：

- `LanceDB` 保存节点正文和向量
- `SQLite` 保存节点 ID 和边
- 午夜把 `short_term_memories` 提取成节点和边

也就是说，它已经具备“图”的外形，但在**新旧节点连接、边方向、融合后的继承关系、检索可达性**上仍然存在明显结构问题。

这次方案不讨论对白风格、印象系统或状态系统，专门只聚焦：

1. 图怎么生成
2. 图怎么更新
3. 新节点如何连接旧节点
4. 图怎么被检索用起来

## 当前实现的核心问题

### 1. 新旧节点连接主要依赖 LLM 自由输出

当前入口是 [backend/src/memory/manager.py](F:/GameProject/unity/AISc/backend/src/memory/manager.py) 的 `extract_and_ingest()`。

现在的实际策略是：

1. LLM 从短期记忆原文里提取 `nodes`
2. LLM 同时提取 `edges`
3. 后端把这些结果直接写入库

问题在于：

- 代码层没有真正掌控“旧节点优先复用”
- 代码层也没有真正掌控“该连哪些结构边”
- 只要 LLM 没复用旧节点，新节点就会越来越碎

这会导致：

1. 同一个人物节点可能重复
2. 同一类事件会形成多个彼此不连的碎片簇
3. 图结构越来越依赖向量召回兜底，而不是边结构本身

### 2. 建边方向与检索方向错位

当前建边里，大量边是：

- `event -> person`
- `event -> self`
- `event -> place`
- `event -> time`

但检索起点又是：

- `"我"` 节点
- `"对方"` 的人物节点

而图召回阶段主要查的是**起点的出边**。

这会造成一个很现实的问题：

- 很多事件节点虽然已经存在
- 但从“我”或“人物节点”并不能自然走到这些事件

结果是：

- 图结构在存
- 但可达性不足
- 真实对白和计划生成更多还是靠向量搜索补救

### 3. 旧节点复用条件不稳定

当前 `_get_today_active_nodes()` 希望把“今天活跃过的旧节点”提供给 LLM 做复用。

但这里有两个问题：

1. “活跃节点”依赖边被 traversal 记录
2. `recover_clarity()` 虽然定义了，但项目里没有真正形成完整的遍历恢复闭环

也就是说：

- 给 LLM 的“可复用旧节点池”本身就不稳定

这会让：

- 同一条人物关系难以持续收敛到同一个节点
- 新事件更容易重新发明一批节点

### 4. 融合节点没有继承旧边

当前 [backend/src/memory/evolution.py](F:/GameProject/unity/AISc/backend/src/memory/evolution.py) 的融合逻辑会：

1. 生成一个新的 merged 节点
2. 把旧节点在 LanceDB 里标记 `archived=1`

但它没有做：

1. 将旧边迁移到新节点
2. 将旧节点的邻接关系合并到新节点
3. 建立 `contains` 或 `merged_from` 对应的结构边

这意味着：

- 融合后的新节点很容易变成孤点
- 下一轮孤点清理时可能直接被删
- 融合在结构上没有真正完成，只完成了文本和向量层

### 5. 代码层没有真正执行“人物节点唯一”

设计文档里明确写过：

- 人物节点应全局唯一

但当前实现没有代码级唯一性约束，只是把“不重复”写进 prompt。

这在原型阶段还凑合，但只要跑得久：

- 人物、地点、常见情绪、重复事件簇都会越来越容易碎片化

## 本次重构目标

### 一、让图重新成为“结构真源”

目标不是让向量层消失，而是让职责重新清晰：

1. `LanceDB`
   - 负责节点正文、向量、语义相似度

2. `SQLite 图边`
   - 负责事实结构、关系传播、可达性

3. 检索
   - 先依赖图结构做自然联想
   - 再用向量层做补充和纠错

### 二、把“新增节点如何连旧节点”改成代码主导

新的原则是：

1. LLM 负责抽取“这段记忆里有什么”
2. 后端负责决定“这些东西怎么接到已有图上”

也就是说：

- LLM 不再是建图结构的唯一裁判
- 后端必须接管节点复用、结构边补全、去重和继承逻辑

### 三、保证从“我 / 人物节点”能稳定走到事件

图不是为了存而存，最终要服务：

1. 对话检索
2. 印象派生
3. 次日计划摘要
4. 长期关系沉淀

所以必须保证：

- 起点节点出发时，能真正走到相关事件节点

## 新的结构口径

### 一、节点类型分成两类

#### 1. 锚点节点

这些节点强调“唯一性”或“强复用”：

1. `self`
2. `person`
3. `place_exact`
4. `place_vague`
5. `time_cycle`
6. 常用 `emotion` 标签节点（可控范围内）

特点：

- 应优先复用已有节点
- 不轻易重复创建

#### 2. 事件/表达节点

这些节点允许增长，但要有聚类和融合机制：

1. `event`
2. `quote`
3. `reflection`
4. `sensory`
5. `item`
6. `time_exact`

特点：

- 可以多实例
- 但要通过相似度、共享参与者、共享地点等逐步连回旧结构

### 二、边也分两类

#### 1. 结构边

这些边是“事实关系”，应主要由后端生成，不依赖 LLM 自由发挥：

1. `involved`
2. `located_at`
3. `happened_at`
4. `felt`
5. `contains`
6. `relationship`
7. `merged_from`

#### 2. 联想边

这些边表达“容易想到”，可以随演化变化：

1. `associated_with`
2. `sequenced`
3. `similar_to` 或其等价替代

## 新的入图流程

### Phase A：先抽取，再规范化，再接图

把现在“LLM 直接产 nodes+edges”改成三段：

1. **抽取**
   - LLM 先只抽：
     - 人物
     - 地点
     - 时间
     - 事件
     - 情绪
     - 物品/话语

2. **规范化**
   - 后端把抽取结果映射到标准节点类型
   - 人物名、地点名、时间表达先归一

3. **接图**
   - 后端决定：
     - 复用哪个旧节点
     - 新建哪个节点
     - 要补哪些结构边

### Phase B：节点复用优先级

#### 1. `person` 节点

规则：

1. 先按标准名字精确匹配
2. 命中则复用旧 `person` 节点
3. 未命中才创建新人物节点

这个规则必须代码化，不能只交给 prompt。

#### 2. `place` 节点

规则：

1. 优先映射到配置里的固定地点 ID
2. 如果是 spot 层地点，可选保留 exact 节点
3. 同时允许挂到 zone/vague 节点上，方便模糊回忆

#### 3. `emotion` 节点

建议先走“半去重”：

1. 常见基础情绪标签优先复用
2. 高自由度复杂情绪允许新建
3. 新节点需与基础情绪团建立连接

#### 4. `event` 节点

规则：

1. 默认新建
2. 但要强制连接到：
   - `self`
   - 参与人物
   - 发生地点
   - 发生时间
   - 相关情绪

这一步是整个图重构的关键。

## 新事件与旧图的连接策略

### 一、最小结构连接集

每个新 `event` 节点入图时，至少补齐：

1. `event -> self` 的 `involved`
2. `event -> 每个参与人物` 的 `involved`
3. `event -> place` 的 `located_at`
4. `event -> time` 的 `happened_at`
5. `event -> emotion` 的 `felt`（如果有）

这意味着：

- 任何事件都必须被挂在锚点结构上
- 不允许出现“只在 LanceDB 里有值，但图上结构几乎没接住”的事件节点

### 二、反向可达性必须可用

为了解决“检索起点走不到事件”的问题，有两个可选方向：

#### 方案 A：保留单条边，但检索按邻边查

也就是：

- 结构边仍记一条
- 检索时同时看 `node_a == sid` 和 `node_b == sid`

优点：

- 改动小

缺点：

- clarity 方向语义会更混

#### 方案 B：显式维护双向边语义

也就是：

- 仍是一条 edge 记录
- 但对 `person -> event`、`self -> event` 的反向清晰度要合理初始化

例如：

- `event -> person` 强
- `person -> event` 也不能低到几乎不可用
- 至少要保证“想到某人时，能回忆起相关事件”

我建议采用 **方案 B**。

原因：

- 你这个项目强调“搭话、印象、熟人关系”
- 这些都天然从人物节点出发
- 所以 `person -> related_event` 的反向可达性必须被设计出来

### 三、新事件与旧事件的连接

现在这部分几乎全靠 LLM 输出，建议改成代码主导：

1. 找共享人物的旧事件
2. 找共享地点的旧事件
3. 找高相似度旧事件
4. 找相邻时间窗口的旧事件

满足条件时补：

1. `sequenced`
2. `associated_with`
3. 必要时弱 `similar_to`

建议规则：

1. 共享参与者 + 同地点 + 同日
   - 优先补 `sequenced`

2. 共享参与者 + 向量高相似
   - 补 `associated_with`

3. 同类型 + 高相似 + 文本足够接近
   - 标记为融合候选，而不是立刻乱连

## 检索层重构

### 一、起点不变，但图召回要改

起点仍保留：

1. `"我"` 节点
2. `"对方人物"` 节点

但召回逻辑要调整：

1. 不只看“出边”
2. 对结构边的反向可达性给明确分值
3. 优先把人物/自我节点通向事件节点的路径打通

### 二、把图召回和向量召回的职责分开

建议职责如下：

1. **图召回**
   - 回答：
     - 我跟谁有关
     - 在哪发生过事
     - 哪件事和谁绑定

2. **向量召回**
   - 回答：
     - 这件事语义上像什么
     - 我想不起精确结构时，近似能搜到什么

目标是：

- 图决定主路径
- 向量负责补洞

而不是像现在这样：

- 图有点像摆设
- 真正能捞出内容更多靠向量

## 融合流程重构

### 一、融合必须继承旧边

新 merged 节点生成后，必须做三件事：

1. 收集旧节点的全部邻边
2. 将这些边合并/重写到新节点
3. 建 `contains` 或 `merged_from` 结构关系

### 二、边继承规则

建议：

1. 如果 A/B 都连向同一个目标节点
   - 新边权重取 `max` 或带轻微增益的合并值

2. 如果只有 A 或只有 B 连某目标
   - 新节点继承该边

3. 如果旧边类型冲突
   - 结构边优先于联想边

### 三、旧节点如何处理

建议：

1. 旧节点先 `archived=1`
2. SQLite 里节点不立刻删
3. 只有在确认 merged 节点已接管全部关键连接后，才允许后续归档清理

否则会出现：

- 新节点刚建好
- 旧节点就被抹掉
- 结果 merged 节点根本没接住结构

## 推荐实施顺序

### Phase 1：先修检索可达性

先做：

1. 明确 `person/self -> event` 的反向权重初始化
2. 修图召回只看出边的问题

原因：

- 这是当前最直接影响“图到底有没有被用起来”的问题

### Phase 2：再修节点复用

先把这些做成代码逻辑：

1. `person` 唯一复用
2. `place` 标准化复用
3. `emotion` 半去重

### Phase 3：把新事件接图改成代码主导

做：

1. 新事件最小结构连接集
2. 新事件与旧事件的规则补边

### Phase 4：最后修融合继承

做：

1. merged 节点接管旧边
2. archived 节点延迟清理

## 风险点

### 1. 改动后旧存档图结构与新逻辑不一致

建议：

- 为旧图提供最小兼容路径
- 或在重构阶段优先对新档验证

### 2. 边数量可能明显增加

因为新方案会更积极补边。

建议：

1. 控制每个新事件的补边上限
2. 对事件-事件连接设置阈值
3. 不让弱关联无限膨胀

### 3. 检索相关行为会改变

这是预期变化，但要注意：

- 不是所有“更多连边”都一定等于“更自然的回忆”

所以后续必须做实际对白和次日计划效果验证。

## 验证计划

### 一、结构验证

重点检查：

1. 新事件入图后，是否稳定连到：
   - self
   - 参与人物
   - 地点
   - 时间

2. 同一人物是否仍反复创建重复节点

3. merged 节点是否继承旧边

### 二、检索验证

重点检查：

1. 从 `"我"` 或 `"对方人物"` 出发，能否走到相关事件
2. 图召回结果是否明显比现在更稳定
3. 向量召回是否从“主力”降回“补充”

### 三、行为/对白验证

重点检查：

1. NPC 提到熟人时是否更容易接上共同经历
2. 夜间印象是否更稳定地从人物邻域派生
3. 次日计划摘要是否更容易体现“最近和谁发生了什么”

## 我的总体判断

这个重构不应该被理解成“优化一下连边”，而应该被理解成：

- **把图从“松散存储层”重新拉回“关系结构真源”**

如果不做这一步，后面即使继续做：

1. 印象系统
2. 搭话决策
3. 夜间压缩
4. 长期关系沉淀

这些能力也会越来越依赖向量语义和 prompt 临时补救，而不是稳定的图结构本身。

所以我认为，这份方案的优先级是高的，而且值得按阶段认真落地。
