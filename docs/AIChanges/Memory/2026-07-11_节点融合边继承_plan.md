# 节点融合边继承 — 设计方案

## 需求理解

当前 `EvolutionEngine` 已经可以：

1. 找到高相似 event 节点。
2. 用 LLM 生成融合后的概括记忆。
3. 新建融合节点并写入向量层。
4. 将旧节点在向量层标记为 archived。

但缺口很关键：旧节点的图边没有继承到融合节点，导致融合节点可能成为孤点，旧节点虽然 archived 但仍可能占据原图路径。

本轮目标是补完当前轻量图口径下的融合闭环，不扩展到旧 v0.5 的完整四阶段退化。

## 方案

### 1. 融合节点继承旧节点邻边

融合 `A + B -> M` 时：

1. 收集所有连接到 A 或 B 的 `memory_edges`。
2. 跳过 A 与 B 之间的内部边。
3. 对每条旧边按方向换算成 `M -> neighbor` / `neighbor -> M` 的 clarity。
4. 相同 `neighbor + edge_type` 的继承边合并，clarity 取较大值，importance 取较大值。
5. 为 M 插入新的继承边。

这样图路由仍能从原来的邻域走到融合节点。

### 2. 旧节点退场

继承完成后：

1. 删除 A / B 相关旧边。
2. 删除 A / B 的 SQLite `memory_nodes`。
3. 保留向量层旧节点，但标记 `archived=1`。

原因：

1. 日常图路由不再走旧节点。
2. 强制回忆仍可通过向量层 include_archived 找到旧文本。
3. 避免旧节点和新融合节点在图里同时争夺路径。

### 3. 记录融合来源

当前 schema 没有 `merged_from` 字段，本轮不做迁移扩表。

采用轻量做法：融合节点 value 中保留概括表达；旧节点保留在 archived 向量层。后续如需 UI 展开来源，再单独补 `memory_merge_sources` 表。

### 4. 可达性验证

融合后立即检查：

1. 融合节点度数是否大于 0。
2. 继承边数量是否大于 0。
3. 旧节点是否已从 SQLite 图节点表删除。

如果没有任何可继承边，则跳过本次融合，避免制造孤立融合节点。

## 风险点

1. 旧边方向不能弄反，否则图联想会变味。
2. `created_day=0` 等合法零值不能用 `or 默认值` 覆盖。
3. 不应继承 A-B 内部边，否则融合节点可能出现自环或无意义边。
4. 当前只处理 event 节点两两融合，不做多节点聚类融合。
