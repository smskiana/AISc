# 多跳图路由对话检索 — 设计方案

## 需求理解

当前对话检索链路仍然是：

1. 从 `我 + 对方` 起点做一轮邻边召回
2. 混合向量补充
3. 对候选节点做一次 LLM 节点筛选
4. 直接把节点重建成相关回忆

这不够像“联想过程”，也无法体现：

1. 时间 / 地点 / 短期记忆对联想路径的影响
2. 从不同起点出发逐跳扩散的回忆过程
3. 不同对话类型需要不同检索深度

用户希望将：

1. NPC ↔ 玩家对话
2. NPC ↔ NPC 对话

都改成多跳图路由，但使用不同参数配比；同时继续讨论并实践“向量补充是否必要”。本轮默认采用保守实现：**白天对话图主导，向量只做兜底补充**。

## 总体方案

### 1. 图与向量的职责划分

保持现有数据层，不推翻结构：

1. 图（SQLite）
   - 决定“如何联想到”
   - 负责节点间的结构、方向、清晰度和遗忘
2. 向量库（LanceDB）
   - 存节点语义正文与向量
   - 负责按 `node_id` 取回记忆内容
   - 只在图路由结果过少时做少量语义兜底

因此白天对话主链将是：

1. 图多跳路由
2. 收集最终节点
3. 必要时少量向量补充
4. 取回这些节点的语义文本
5. 注入对话 Prompt

### 2. 新的多跳边路由流程

统一抽象为：

1. 起点：`self` 节点 + `target person` 节点
2. 当前前沿节点并发取本地候选边
3. 本地算法先筛出每跳 `X` 条候选边
4. LLM 结合上下文选择 `Y` 条边
5. 进入下一批节点
6. 维护 `visited_nodes / visited_edges`，禁止成环
7. 共进行 `Z` 跳
8. 汇总所有命中节点，按配置决定是否做向量兜底

LLM 每跳看到的上下文包括：

1. 当前时间
2. 当前地点
3. 最近短期记忆摘要
4. 当前状态（情绪 / 精力 / 当前需求）
5. 对目标的当前印象
6. 当前前沿节点
7. 候选边清单

### 3. 双对话模式参数

#### NPC → 玩家

- `max_edges_per_hop = 6`
- `edges_per_route = 2`
- `max_hops = 2`
- `min_graph_nodes = 3`
- `vector_fallback_limit = 2`
- `final_memory_limit = 5`

目标：

1. 联想更深一点
2. 优先保证玩家相关背景能被想到
3. 允许少量私人旧事被顺带带出

#### NPC → NPC

- `max_edges_per_hop = 4`
- `edges_per_route = 2`
- `max_hops = 1`
- `min_graph_nodes = 2`
- `vector_fallback_limit = 1`
- `final_memory_limit = 3`

目标：

1. 日常闲聊保持轻量
2. 让场景和熟人关系主导，而不是深挖私人往事

### 4. 向量补充策略

本轮默认实现为：**fallback-only**

也就是：

1. 图路由结果已满足 `min_graph_nodes` 时，不做向量补充
2. 只有当图结果太少时，才补 `1-2` 条向量候选
3. 向量补充不会替代图的遗忘机制，只是避免对白过空

这样更符合“白天靠联想，晚上整理融合”的设计目标。

### 5. 接线改动

1. `PromptBuilder.build()` 调用玩家对话模式检索
2. `PromptBuilder.build_npc_to_npc()` 调用 NPC 对话模式检索
3. `RetrievalEngine.retrieve()` 支持显式传入 `mode`

## 验证计划

1. Python 编译检查通过
2. 冷启动后直接对 `kujo -> player` 做一次检索，确认路径优先经过玩家背景节点
3. 对一组 `npc -> npc` 检索做一次 smoke test，确认结果更轻量
4. 检查日志中每跳候选边、LLM 选边、最终节点与向量补充是否符合预期
