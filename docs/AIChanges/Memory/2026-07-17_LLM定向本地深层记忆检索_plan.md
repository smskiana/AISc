# 三策略可配置记忆路由与本地深层检索 - 设计方案

## 1. 方案定位

本方案以 [`LlmGuidedLocalDeepRetrievalDraft.md`](../../Workstreams/Memory/LlmGuidedLocalDeepRetrievalDraft.md) 为需求来源，作为后续实现的唯一执行案。本轮只创建方案，不修改运行代码、不创建 execution。

- Workstream：`MEM` 记忆系统，关联 `DIA` 玩家逐轮对话与测试诊断。
- Roadmap item：MEM 当前近期目标“保持轻量图 + 向量数据层，补诊断与质量评估”。本方案不改变现有三档排期顺序，由用户另行决定何时执行。
- 相关 ADR：`ADR-0001-memory-v07-light-graph`。
- 当前口径变化：计划让 `player_dialogue`、`npc_dialogue`、`nightly_impression` 都可在三种路由策略间配置，并分别保存参数；不改变每 NPC 独立图、SQLite 图层、LanceDB / SQLite 向量层、边 clarity 与 archived 语义。
- 不适用 ADR：不扩张图模型、不新增共享运行时图、不推翻存档边界，因此不新建 ADR。

## 2. 执行前置授权

当前 `backend/src/memory/retrieval.py` 的 `RetrievalEngine` 已同时承担兼容入口、上下文读取、起点发现、逐跳路由、本地评分、LLM 调用、向量兜底、重排、clarity 恢复、日志和诊断聚合。继续直接加入方向解析与 beam search 会违反类减重硬约束。

后续执行前，用户必须在新对话中明确给出“按该 plan 减重并执行”的指令。执行会话只引用本 plan，不重新创建第二份 plan。若不授权减重，则停止实现，不允许把新逻辑继续塞入 `RetrievalEngine`。

本方案预计新增文件不少于 2 个、修改文件不少于 3 个，必须在独立执行对话中完成，并在结束后创建顶部互链的 execution。

## 3. 已确认决策

### 3.1 固定起点不变

第一阶段只允许以下节点进入图搜索初始前沿：

1. 当前 NPC 的 `self` 节点。
2. 当前对话对象的 `person` 节点。

LLM mention、本地实体目录解析结果和向量命中均只能成为“目标锚点”，用于方向相关性、启发式排序、停止语义和诊断，不得直接进入初始前沿。这样保持 Dialogue / Memory Workstream 已确认的固定起点口径，不修订 ADR。

如果目标锚点存在但在预算内无法从固定起点到达，返回 `no_reachable_path` 或 `budget_exhausted`，不能把锚点偷接成检索结果。只有后续真实样本证明固定起点长期不可用时，才另开方案讨论辅助前沿，并同步 Workstream / 必要 ADR。

### 3.2 三种公开路由策略

每种业务模式都允许从以下三个稳定枚举中选择一项，不能再用阈值组合暗示第四种混合策略：

| 配置值 | 玩家可读名称 | LLM 职责 | 正常 LLM 路由调用 |
|---|---|---|---:|
| `local_only` | 完全本地路由 | 不调用 LLM；本地解析方向并执行深搜 | 0 |
| `llm_guided_local` | LLM 定向 + 本地深搜 | LLM 只生成一次 `RetrievalDirection`，本地决定路径 | 1 |
| `llm_full_route` | 完全 LLM 路由 | 每个存在选择的图路由步骤都由 LLM 选边 | `0..max_llm_route_calls` |

`local_only` 的“完全不使用 LLM”不禁止现有本地 BGE embedding / 向量查询；诊断中 LLM 调用必须为 0，向量查询单独计数。

`llm_full_route` 中，候选数量不大于可选数量时无需调用 LLM，因为不存在路由选择；一旦存在竞争候选，必须调用 LLM，不允许通过本地置信阈值跳过。LLM 失败时仍需确定性本地降级以保持运行可用，但诊断必须标记 `degraded_to_local`，不得把降级结果记作完全 LLM 成功。

“完全 LLM 路由”指预算内所有竞争边的最终选择权属于 LLM，不表示把完整图发送给 LLM。本地仍负责固定起点、权限 / archived 硬过滤、环路排除、稳定候选整理和数据量上限；这些步骤不得根据本地置信度提前替 LLM 选定竞争边。

`local_only` 与 `llm_guided_local` 共用同一个本地深搜实现和路径评分。两者唯一差异是方向来源：前者使用确定性 `LocalDirectionProvider`，后者使用一次 `LlmDirectionProvider`，失败后再降级到本地 provider。

### 3.3 三种业务模式的默认策略

| 业务模式 | 默认策略 | 默认理由 |
|---|---|---|
| `player_dialogue` | `llm_guided_local` | 玩家问题需要理解描述、因果和指代，同时保持路径预算稳定 |
| `npc_dialogue` | `local_only` | 默认优先低延迟、零路由 LLM 成本；保留手动切换能力 |
| `nightly_impression` | `llm_guided_local` | 夜间允许一次方向理解，并使用高于玩家对话的本地深搜预算 |

Day 4 旧样本曾显示 NPC / 夜间本地逐边评分与 LLM 基线不一致。用户已明确选择上述新默认值，因此执行时必须重跑三策略对照评估；默认值可以按用户配置切换，但不得由执行者擅自改回旧建议。

`RetrievalEngine.retrieve(request)` 继续是调用者唯一 seam，旧位置参数入口继续兼容字符串返回。业务模式与路由策略是两个正交维度：业务模式决定默认配置和上下文，路由策略决定使用哪个检索模块；不得复制九套入口，也不得在 Dialogue / NPC / 夜间调用层自行判断策略。

### 3.4 策略选择与参数生效规则

1. 配置来源为提交到项目的 `backend/config/memory_retrieval.yaml`，启动时一次读取并校验；修改后重启后端生效，第一版不做热重载。
2. 每种业务模式都保存 `strategy`、共享结果参数，以及三种策略各自的参数块。切换策略不删除其他参数，便于直接比较和回切。
3. 未知 strategy、未知字段、错误类型、负数、越过安全上限或不满足 `selected_edges <= candidate_edges` 时启动失败，禁止静默回退到另一策略。
4. `RetrievalRequest` 不携带任意策略覆盖，避免对话调用者绕开配置；测试 / 评估通过显式构造 `RetrievalModePolicy` 注入。
5. 本轮“选择权”指后端工程配置，不新增游戏内设置 UI、协议字段或存档字段。若后续要求玩家在游戏内实时切换，需另开方案处理 UI、协议、权限和存档边界。

### 3.5 向量层双重职责但每轮只执行一次查询

`local_only` 与 `llm_guided_local` 在方向形成后构造一次向量 query，并保留同一批结果：

1. `anchor_discovery`：将当前 NPC 自有、未归档、通过权限过滤的 Top-K 节点作为目标锚点，只引导固定起点搜索，不直接返回。
2. `content_fallback`：图证据不足时，从同一批已过滤结果中补最终内容，不重复做 embedding / ANN 查询。

`llm_full_route` 不使用向量结果改变图起点；图结果不足时可按其配置执行一次 `content_fallback`，但不额外做 `anchor_discovery`。

向量命中不得跨 NPC、不得绕过 archived 与权限口径。诊断必须逐项记录用途，区分仅作锚点和最终兜底。

## 4. 目标模块与 seam

采用少量深模块，避免把每个小步骤包装成浅模块。

### 4.1 `RetrievalEngine`：稳定 facade

保留在 `backend/src/memory/retrieval.py`，对外接口不变：

```python
retrieve(request: RetrievalRequest) -> RetrievalResult
```

它只负责：

1. 兼容旧调用并标准化请求。
2. 按已校验的 mode policy 与 strategy registry 选择检索模块。
3. 统一读取最终节点内容、恢复实际采用路径的 clarity、重建上下文和写汇总日志。
4. 将内部 `RetrievalTrace` 映射到稳定 `RetrievalResult`。

不再负责方向 Prompt、逐边候选编排、beam 状态、分数细项或实体目录规则。

### 4.2 `DirectionResolver`：统一方向 seam

新增 `backend/src/memory/retrieval_direction.py`，提供单一接口：

```python
resolve(
    request: RetrievalRequest,
    context: RetrievalContext,
    provider: DirectionProvider,
) -> DirectionResolution
```

模块内部封装 JSON 解析、枚举白名单、mention 本地解析和统一结果契约。`LocalDirectionProvider` 封装零 LLM 的确定性方向；`LlmDirectionProvider` 封装 `memory_direction` Prompt、一次 LLM 调用和失败降级。调用者不需要依次调用 parse / validate / resolve_entity，也不能自行拼接 provider 流程。

实体目录只在 Memory 模块内部构建，来源复用现有 NPC profile、玩家昵称和共享地点语义；不恢复 Dialogue 层的 `DialogueEntityResolver`，也不查询 topic impression 作为答案旁路。

### 4.3 `DirectedDeepRetriever`：本地深搜 seam

新增 `backend/src/memory/deep_retrieval.py`，提供单一接口：

```python
search(request: DeepSearchRequest) -> DeepSearchOutcome
```

内部封装目标锚点、批量邻接扩张、路径评分、预算、权限 / archived 过滤、去重、终止和向量兜底复用。测试只通过该 seam 输入固定图与方向，断言路径和结果，不访问内部 beam 状态。

三种以上节点类型、边类型、intent 和 relation facet 统一使用 scorer / policy 映射表；不得继续向一个大型 `switch` 或 `if-else` 链增加分支。

### 4.4 `LlmGraphRetriever`：完全 LLM 路由 seam

新增 `backend/src/memory/llm_graph_retrieval.py`，承接当前候选收集、逐跳 `memory_route` LLM 选边、调用预算、失败降级和内容兜底逻辑，提供单一接口：

```python
search(request: LlmGraphSearchRequest) -> LlmGraphSearchOutcome
```

三种业务模式选择 `llm_full_route` 时都通过该模块运行。正常竞争分支禁用当前 `local_route_min_score / local_route_margin` 跳过逻辑；本地评分只负责候选预排序和 LLM 失败后的显式降级。共享节点读取、结果重建和日志仍由 facade 统一处理。

### 4.5 `RetrievalPolicyRegistry`：配置 seam

新增 `backend/src/memory/retrieval_policy.py`，对外只提供：

```python
get(mode: RetrievalMode) -> RetrievalModePolicy
```

模块负责读取 `memory_retrieval.yaml`、合并完整默认值、拒绝未知字段、执行跨字段校验并构建不可变 policy。`RetrievalEngine` 只能读取 policy，不能直接访问 YAML 字典；测试可注入内存 registry，避免修改全局配置。

strategy registry 通过映射注册 `local_only`、`llm_guided_local`、`llm_full_route` 三个实现，不增长大型分支。新增第四种策略必须增加独立实现、契约、配置 schema、诊断和三模式测试，不能只加一个字符串 case。

### 4.6 共享契约

新增 `backend/src/memory/retrieval_contracts.py`，集中保存不可变 DTO、受控枚举和诊断结构；避免 `retrieval.py` 继续增长，也避免 Python / Unity 字段各自猜测。

现有 `RetrievalRequest` / `RetrievalResult` 从原模块重导出，保证旧 import 不立即失效。新契约包括：

- `RetrievalDirection`
- `DirectionResolution`
- `ResolvedMention`
- `RetrievalStrategy`
- `RetrievalModePolicy`
- `RetrievalExecutionOptions`
- `LocalSearchPolicy`
- `LlmRoutePolicy`
- `SearchBudget`
- `SearchPathEvidence`
- `DeepSearchOutcome`
- `RetrievalTrace`
- `StopReason`
- `FailureReason`

## 5. 方向契约

### 5.1 `RetrievalDirection`

第一版字段固定为：

```text
entity_mentions: list[str]
location_mentions: list[str]
themes: list[Theme]
relation_facets: list[RelationFacet]
time_scope: TimeScope
source_preferences: list[SourcePreference]
recall_intent: RecallIntent
negative_directions: list[NegativeDirection]
```

枚举第一版白名单：

- `Theme`：`identity`、`current_location`、`recent_activity`、`relationship`、`cause`、`past_event`、`emotion`、`object`、`promise`、`general`。
- `RelationFacet`：`familiarity`、`affinity`、`occupation`、`shared_event`、`impression_basis`、`knowledge_source`。
- `TimeScope`：`current`、`recent`、`past`、`any`。
- `SourcePreference`：`direct`、`heard`、`inferred`。
- `RecallIntent`：`locate_person`、`identify_entity`、`explain_cause`、`compare_relationship`、`recall_event`、`continue_reference`、`general_recall`。
- `NegativeDirection`：`unrelated_player_background`、`unrelated_private_memory`、`stale_location`、`unrelated_person`。

LLM 只能返回原文 mention 和枚举值，不能返回 node ID、edge ID、最终答案或权限判断。非法枚举丢弃并记录校验错误；mention 解析失败时保留为语义文本，不转成数据库 ID。

### 5.2 本地方向 provider 与 LLM 降级

`LocalDirectionProvider` 按同一 `RetrievalDirection` 契约输出，既是 `local_only` 的正式方向来源，也是 `llm_guided_local` 的失败降级：

1. 从当前发言优先匹配稳定人物、玩家昵称和地点别名。
2. 再使用近期对白与摘要补充指代上下文，但来源置信度低于当前发言。
3. 通过疑问词和短语映射 `recall_intent`、`relation_facets` 与 `time_scope`。
4. 无任何稳定信号时使用 `general_recall + any + general`，仍允许固定起点做有限检索。

`local_only` 正常记录 `direction_source=local`，不使用“回退”字样。只有 LLM provider 失败后才记录 `llm_invalid`、`llm_timeout`、`llm_unavailable` 或 `llm_empty`，不能伪装成 LLM 成功。

## 6. 检索广度、数据预算与本地深搜

### 6.1 三模式本地深搜默认预算

`local_only` 与 `llm_guided_local` 使用相同字段，但每种业务模式有独立默认值：

| 配置 | 玩家对话 | NPC-NPC | 夜间印象 | 含义 |
|---|---:|---:|---:|---|
| `max_depth` | 4 | 2 | 6 | 最深路径边数 |
| `beam_width` | 8 | 4 | 12 | 每层保留路径数 |
| `max_neighbors_per_node` | 12 | 8 | 20 | 每个前沿节点最多读取的方向邻边数 |
| `max_expanded_edges` | 64 | 32 | 128 | 单轮最多检查边数 |
| `max_anchor_count` | 4 | 2 | 6 | 目标锚点上限 |
| `max_answer_candidates` | 12 | 6 | 20 | 进入最终重排的证据节点上限 |
| `min_path_score` | 0.25 | 0.30 | 0.18 | 继续扩张最低分 |
| `early_stop_margin` | 0.18 | 0.22 | 0.12 | 充分证据领先阈值 |
| `vector_search_top_k` | 8 | 4 | 12 | 单次向量候选数 |
| `vector_fallback_limit` | 2 | 1 | 3 | 内容兜底上限 |
| `final_memory_limit` | 5 | 3 | 6 | 最终上下文节点上限 |

夜间模式的深度、每节点广度、beam、展开边、锚点、答案候选和向量预算均高于玩家对话，阈值更宽松；这就是“参数高一点”的第一版明确口径。NPC-NPC 默认本地模式使用最小预算，避免每段闲聊放大图查询。

### 6.2 三模式完全 LLM 路由默认预算

`llm_full_route` 使用独立参数，不复用 beam 字段：

| 配置 | 玩家对话 | NPC-NPC | 夜间印象 | 含义 |
|---|---:|---:|---:|---|
| `max_hops` | 4 | 2 | 4 | 最大逐跳层数 |
| `max_frontier_nodes` | 8 | 4 | 12 | 每跳可参与候选收集的前沿节点上限 |
| `max_neighbors_per_node` | 12 | 8 | 20 | 每个前沿节点最多读取的方向邻边数 |
| `max_candidate_edges` | 8 | 6 | 12 | 每跳发送给 LLM 的候选上限 |
| `selected_edges_per_hop` | 3 | 2 | 4 | 每跳最多选择边数 |
| `max_llm_route_calls` | 4 | 2 | 4 | 单轮路由 LLM 调用硬预算 |
| `candidate_summary_chars` | 96 | 72 | 128 | 单条候选提供给 LLM 的摘要字符上限 |
| `max_route_prompt_chars` | 5000 | 3500 | 8000 | 单次路由 Prompt 动态数据字符上限 |
| `vector_search_top_k` | 8 | 4 | 12 | 内容兜底候选数 |
| `vector_fallback_limit` | 2 | 1 | 3 | 内容兜底上限 |
| `final_memory_limit` | 5 | 3 | 6 | 最终上下文节点上限 |

`max_llm_route_calls` 不得大于 `max_hops`，`selected_edges_per_hop` 不得大于 `max_candidate_edges`。候选先按权限、archived 和结构规则过滤，再按本地相关性稳定排序，最后受候选条数与 Prompt 字符总量双重截断；不能只限制条数而忽略每条数据大小。LLM 调用达到预算但仍有竞争候选时返回 `budget_exhausted`；不允许悄悄切到本地继续并记作正常完成。

执行时允许基于固定评估集校准默认数值，但必须保留用户指定的三个默认 strategy；若修改数值，在 execution 记录样本、理由和最终值。

### 6.3 方向与上下文数据预算

`llm_guided_local` 的一次方向 LLM 和 `llm_full_route` 的逐跳 LLM 共用业务模式级上下文预算：

| 配置 | 玩家对话 | NPC-NPC | 夜间印象 | 含义 |
|---|---:|---:|---:|---|
| `recent_turn_limit` | 8 | 4 | 4 | 提供的近期对白条数；夜间无对白时为 0 |
| `recent_memory_limit` | 4 | 2 | 10 | 提供的近期记忆条数 |
| `memory_summary_chars` | 80 | 64 | 120 | 单条近期记忆摘要字符上限 |
| `conversation_summary_chars` | 800 | 400 | 400 | 会话 / 汇总摘要字符上限 |
| `max_direction_context_chars` | 3000 | 2000 | 6000 | 方向 Prompt 动态数据总字符上限 |

数据裁剪顺序固定为：保留当前问题或当前夜间目标 → 保留明确 mention 所在片段 → 保留最近内容 → 按相关性补其余内容。不得直接从字符串尾部机械截断而丢失当前问题。所有截断必须记录原始条数 / 字符数、提供条数 / 字符数和淘汰原因。

这些是输入数据预算，不是 LLM 输出 `max_tokens`；项目“不设置记忆 LLM 输出硬 token 上限”的现有口径保持不变。

### 6.4 本地深搜扩张规则

1. 从固定起点建立零深度路径。
2. 每层收集当前 beam 的节点 ID，通过 SQLite 批量读取方向邻边，并对每节点读取数和全局展开边数同时限流，避免按路径逐条 N+1 查询或高 degree 节点吞掉全部广度。
3. 先应用 NPC 归属、archived、已访问环路和现有权限过滤，再计算分数。
4. 同一节点允许保留不同来源路径，但每个目标最多保留分数最高且证据来源不同的少量路径。
5. 只有路径增加与方向相关的新证据时才进入下一层；重复主题、回环和只增加无关节点的路径被惩罚或淘汰。
6. 达到充分证据、无候选、深度上限或边预算时停止。

若现有 SQLite 方法不能批量返回方向邻边，在 `SQLiteClient` 增加领域化的批量 adapter 接口，不把原始 SQL / query builder 暴露给检索模块。

### 6.5 路径评分

所有正向项先归一化到 `0..1`，第一版使用数据表组合：

```text
score =
  0.24 * direction_relevance
+ 0.18 * entity_alignment
+ 0.12 * relation_facet_alignment
+ 0.10 * edge_clarity
+ 0.08 * time_alignment
+ 0.08 * source_reliability
+ 0.06 * target_context
+ 0.06 * node_importance
+ 0.08 * novel_evidence
- 0.06 * depth_penalty_per_extra_hop
- 0.12 * repeated_topic_penalty
- 0.10 * uncertainty_penalty
```

约束：

1. 实体精确命中不能覆盖权限和来源可信度。
2. 深路径只有带来新证据才可能超过浅路径。
3. scorer 按受控节点 / 边类型注册；未知类型走中性默认值并进入诊断，不因新增类型崩溃。
4. 最终按来源稳定 ID、节点 ID、归一化语义内容和路径证据去重。

## 7. 停止与失败语义

`stop_reason` 表示搜索为何结束，`failure_reason` 表示为何无法形成可用结果，两者不得混用。

`StopReason`：

- `sufficient_evidence`
- `frontier_exhausted`
- `depth_limit_reached`
- `edge_budget_exhausted`
- `early_stop_margin_reached`

`FailureReason`：

- `none`
- `start_nodes_not_found`
- `no_reachable_path`
- `knowledge_absent`
- `knowledge_forbidden`
- `direction_parse_failed`
- `storage_unavailable`
- `budget_exhausted`

判定顺序：

1. 图 / 向量存储不可访问且不能安全降级：`storage_unavailable`。
2. 无固定起点：`start_nodes_not_found`。
3. 有候选但全部被现有明确权限规则过滤：`knowledge_forbidden`。
4. 有方向锚点但固定起点无可达路径：`no_reachable_path`。
5. 仍有高于阈值的候选但预算耗尽：`budget_exhausted`。
6. 图与向量层均无相关事实：`knowledge_absent`。

第一阶段不新增权限 schema；`knowledge_forbidden` 只基于已有明确来源 / scope 元数据，不能凭内容猜测秘密。

## 9. 诊断与控制同步

### 9.1 后端 `RetrievalTrace`

每次检索生成稳定 `retrieval_trace_id`，至少记录：

1. 业务 `mode`、实际 `strategy`、配置版本和本轮有效 policy 摘要。
2. `direction_source`、LLM 安全摘要、校验错误和最终方向；`llm_full_route` 无方向 LLM 时明确记为 `not_applicable`。
3. 方向 LLM、逐跳路由 LLM 和总 LLM 调用次数，不能只记一个合计。
4. mention → 稳定 ID 的解析结果、来源和置信度。
5. 固定起点、目标锚点、锚点来源与用途。
6. 实际深度、每层 frontier / 邻边 / 候选 / beam 数量、展开边总数。
7. 每次提供给 LLM 的候选条数、候选摘要字符数、动态上下文字符数和 Prompt 动态数据总字符数。
8. 最终路径的 node / edge ID 与主要分数组成。
9. 权限、archived、低分、重复、环路、每节点广度、总广度、字符预算和调用预算淘汰计数。
10. 向量查询次数，以及每个命中的 `anchor_discovery` / `content_fallback` 用途。
11. `stop_reason`、`failure_reason`、`degraded_to_local` 和总耗时。

LLM 原始输出只保留截断、去除自由文本的安全摘要，不把完整 Prompt 或可能含隐私的长文本发送到 Unity。

### 9.2 `aisc_debug`

保留并扩展现有 `conversation_retrieval_snapshot`，同时新增通用 `memory_retrieval_snapshot`，使玩家对话、NPC-NPC 和夜间检索都能按 `retrieval_trace_id`、`npc_id`、`mode`、`strategy` 查询。同步修改：

- 后端 `conversation_context.py` / `conversation_memory.py` 的诊断 DTO 与消息载荷。
- Unity `DiagnosticModels.cs`。
- `WebSocketClient` 的反序列化缓存。
- `AiscDiagnosticsTests`。
- `Assets/Scripts/Diagnostics/README.md` 与相关协议入口说明。

通用快照通过后端只读诊断 provider 返回最近固定容量的结构化 trace；Unity 不读取 SQLite / LanceDB 重建路径。保留现有会话字段兼容，新增策略、有效参数、结构化方向、锚点、每层广度、输入数据量、路径和停止原因字段。会话自动断言继续选择同会话最后一轮快照。

### 9.3 `aisc_control`

为可重复 Play 验收补齐编辑器白名单动作：

- `start_dialogue`
- `send_player_choice`
- `end_dialogue`
- `run_memory_retrieval_probe`

前三个动作只调用现有正式对话命令链。`run_memory_retrieval_probe` 使用当前已加载 mode policy 执行一次不恢复 clarity、不写持久记忆的编辑器诊断检索，返回 `retrieval_trace_id`，随后由 `aisc_debug.memory_retrieval_snapshot` 读取；它不允许任意 strategy 覆盖，也不进入 Player 构建。

三种以上 action 通过 action handler 注册表统一分派，不继续扩张单个 switch。诊断 probe 必须复用正式 `RetrievalEngine.retrieve` seam 和显式 `side_effects_disabled` policy，不能复制检索实现或直接查询数据库拼结果。

执行 Unity C# 文件导入、编译、EditMode / PlayMode 验证前，必须先阅读 `docs/DesignDocs/UnityMCPUsageRules.md` 并使用 Unity MCP；若 Unity MCP 不可用，停止 Unity 侧修改并请求用户指示。本方案不涉及场景、Prefab、UI 或 SerializeField 连线。

## 12. 测试与验收

### 12.1 快速自动化

1. 三种业务模式 × 三种 strategy 的 9 个组合都能从同一 `retrieve` seam 执行。
2. 默认值严格为玩家=`llm_guided_local`、NPC-NPC=`local_only`、夜间=`llm_guided_local`，且夜间本地广度 / 数据预算高于玩家。
3. 未知 strategy、未知字段、非法范围和跨字段冲突在启动校验时报错。
4. `local_only` 的方向与路由 LLM 调用均为 0；`llm_guided_local` 方向调用最多 1、路径调用为 0；`llm_full_route` 每个竞争步骤调用 LLM 且不超过 `max_llm_route_calls`。
5. `RetrievalDirection` 正常解析、非法值清洗和确定性降级。
6. 固定起点始终只有 `self + 当前对话对象`。
7. 目标锚点能引导超过两跳的路径，但不会直接成为前沿或无条件结果。
8. 每节点邻边、frontier、beam、候选边、展开边、摘要字符、动态上下文字符、Prompt 字符和最终上下文均不越界。
9. 超过数据预算时按确定顺序截断，当前问题和明确 mention 不丢失，诊断包含截断前后计数。
10. 图路径被截断、图内无事实、权限过滤、LLM 降级和存储失败能返回不同语义。

### 12.2 草案场景

至少覆盖：

1. “千早在哪”命中人物 + 定位意图并寻找近期地点证据。
2. “那个面包店的女孩”通过本地 mention 目录解析到千早。
3. “谁和樱关系最好”按关系维度比较，不被玩家背景吸走。
4. “你为什么觉得龙之介有才能”沿人物 → 印象依据 → 事件路径。

## 13. 完成标准

只有全部满足才可宣称实现完成：

1. 三种业务模式都可配置 `local_only`、`llm_guided_local`、`llm_full_route`，默认 strategy 与本 plan 一致。
2. 三种策略的实际 LLM 行为与名称一致：0 次、一次方向、逐竞争步骤路由；所有调用都有预算和降级证据。

5. 固定起点口径未被 mention 或向量锚点绕过。
6. 本地算法可在统一预算内搜索超过两跳，并输出完整路径证据。
7. 单轮向量查询最多一次，并能区分锚点发现与内容兜底。
8. 描述称呼、关系、因果和近期指代场景较旧实现有明确改进证据。
9. 无事实、不可达、无权限、预算耗尽和存储失败语义可区分。
10. `RetrievalEngine` 已减重为稳定 facade，新职责位于可独立测试的深模块。
11. `aisc_debug`、`aisc_control`、后端测试、Unity EditMode / PlayMode 和文档全部同步。
12. execution 记录实际文件、最终参数、九组合验证、真实样本结果、未完成项以及本轮是否改变工程口径。

## 14. 风险与回退
