# 记忆检索与对话链路修复 — 设计方案

## 背景

本次修复基于代码审查中已确认的高风险问题，目标是恢复以下主链路的可用性与一致性：

1. 玩家对话 Prompt 构建
2. NPC ↔ NPC 轻量对话生成
3. LanceDB 向量层的增量写入与历史记忆保留
4. 对话结束后的玩家记忆提取
5. NPC 社交冷却与跨天行为
6. 调试接口与 Unity 消息发送稳定性

## 问题摘要

### 1. PromptBuilder 时间解析调用错误
- `PromptBuilder.build()` 与 NPC 对话记忆构建处调用了 `self._parse_day(...)`
- 实际只有模块级 `_parse_day(...)`，会导致对话入口直接报错

### 2. LanceDB 批量写入覆盖整表
- `upsert_nodes()` 每次都 `overwrite`
- 午夜提取或冷启动后续写入会覆盖旧向量数据

### 3. `main.py` 使用未声明的 `lancedb`
- `lifespan()` 中是局部变量
- 对话结束、玩家记忆提取和搜索使用了同名全局，但未提前声明

### 4. NPC 社交冷却跨天失效
- 冷却只按“当天分钟数”记录，不带 day
- 跨天后分钟归零，导致旧冷却无法正常过期

### 5. v0.6 调试接口仍使用旧 schema
- `/api/npc/{npc_id}/memories` 仍按 `energy` 排序
- `memory_nodes` 已不再包含该字段

### 6. 图检索起点偏离设计
- 设计要求从“我 + 对方人物节点”出发
- 当前实现只取 `memory_nodes` 的第一条记录，既不稳定也不包含目标人物

### 7. Unity 发送 `PLAYER_CHOICE` 时未做 JSON 转义
- 一旦选项中包含引号、换行等字符，请求体会损坏

## 修复原则

1. **最小修改**：仅修复已确认问题，不重构整套系统
2. **兼容当前 v0.6 结构**：以 SQLite 轻图 + LanceDB 数据层为准
3. **优先修主链路**：先保证“能对话、能记忆、能检索、不会覆盖”
4. **调试接口次级修复**：保留接口能力，避免继续误导联调

## 具体方案

### A. 修复 PromptBuilder 时间解析
- 将类内部所有 `self._parse_day(...)` 调整为模块级 `_parse_day(...)`
- 保持调用范围最小，不引入新的实例方法

### B. 修复 LanceDB 增量写入
- `upsert_nodes()` 改为“读取现有表 → 以 `node_id` 去重合并 → overwrite”
- `upsert_node()` 复用批量逻辑，避免单条/批量实现分叉
- 保留当前表结构，不改外部调用方

### C. 补齐 `lancedb` 全局服务
- 在 `main.py` 全局服务区声明 `lancedb = None`
- `lifespan()` 中显式 `global lancedb`
- 保证对话结束、玩家记忆写入和记忆搜索使用同一实例

### D. 修复 NPC 社交冷却跨天问题
- 将冷却记录改为绝对游戏分钟：`(day - 1) * 1440 + hour * 60 + minute`
- 所有冷却读写统一走绝对分钟

### E. 修复 v0.6 调试接口
- `/api/npc/{npc_id}/memories` 不再查询不存在的 `energy`
- 改为读取 `memory_nodes` 基础信息，并在 LanceDB 可用时补充节点值

### F. 修复图检索起点
- 明确查找：
  - 自我节点：`type == self` 且 `value == 我`
  - 目标人物节点：`type == person` 且 `value == 对方名字`
- 若找不到目标人物节点，则至少回退到自我节点

### G. 修复 Unity JSON 发送
- 为 `choiceText` 做 JSON 安全转义
- 优先用小范围字符串转义方法，避免改动现有消息框架

## 验证计划

1. 后端静态导入检查：确保关键文件语法可过
2. 读取链路检查：
   - 玩家对话 `build()` 不再因 `_parse_day` 报错
   - NPC 对话 Prompt 构建不再因相同问题失败
3. 数据层一致性检查：
   - `upsert_nodes()` 合并而非覆盖
   - `main.py` 中玩家记忆提取路径不再缺少 `lancedb`
4. 行为链检查：
   - 冷却值改为绝对分钟后跨天不锁死
5. Unity 发送检查：
   - `PLAYER_CHOICE` 对引号/换行安全

## 风险点

1. LanceDB 的表重写仍然不是最高性能方案，但比“每次覆盖全表”安全得多
2. 图检索起点修复依赖 LanceDB 中已有 `self/person` 节点值正确写入
3. 当前环境无法代替 Unity Editor 做完整 Play 模式验证，最终仍需联调确认
