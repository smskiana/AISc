## 需求理解

当前冷启动会给每个 NPC 创建 self、所有人物节点、relationship 边、四条共享背景和本人 `core_memories`，但缺少统一的“事实源 → 知识权限 → 观察者视角记忆”链路。人物节点因此常只有名字，其他 NPC 的公开事实、共同经历和传闻无法按观察者权限进入独立图。

本次实现目标：

1. 用稳定事实配置定义公开事实、社区知识、参与者经历、指定知情者和私密事实。
2. 通过统一规则判断观察者能否知道，并生成确定性的观察者视角投影 DTO。
3. 将投影写入观察者自己的 SQLite 图与 LanceDB 向量层，并连接 self 与所有已有人物节点。
4. 记录稳定 `source_fact_id`、权限依据、来源类型、可信度、涉及人物和实际边，支持诊断、去重和审计。
5. 保持 ADR-0001 的“每个 NPC 独立轻量图”边界，不建立全局运行时记忆图。

## 已确认取舍

1. 第一版仅对新开局冷启动生效，不提供旧存档重建工具，也不静默灌入旧档。
2. 第一版使用配置提供的观察者模板或确定性文本组合，不调用 LLM 自由生成投影。
3. `initial_impression_traits` 暂时保留为独立主观配置；它不自动升级为客观事实，也不作为投影事实的唯一来源。
4. 第一版不新增地点图节点。`location_ids` 只保存在投影来源元数据中。
5. 第一版复用 `involved` 与既有 relationship 结构，不新增 `heard_from`、`observed` 等边类型，不修改 ADR。
6. `clarity` 继续表示联想可达性；`confidence` 与 `importance` 保存为投影元数据 / 向量数据，三者不得混用。

## 边界与约束

1. 不改玩家对话检索算法、图搜索深度、候选排序或 Prompt。
2. 不读取或复制其他 NPC 的运行时记忆数据库；所有投影只来自统一初始事实配置。
3. 不新增人物专属 `if / elif`。事实类型、知识范围和来源类型必须走枚举与统一规则表。
4. 第一版知识范围固定为 `public`、`community`、`participants`、`explicit_knowers`、`private`；未来新增范围应扩展规则表，不修改调用方分支。
5. 私密事实默认只进入主体本人图；只有显式知情者才能额外获得投影。
6. 传闻文本必须保留“听说”等不确定口径，主观印象不得升级为事实。
7. 本轮不涉及 Unity 场景、Prefab、SerializeField、UI 结构或资产连线；不触发 Unity MCP 资产编辑规则。若执行中出现资产层需求，立即停止并另行确认。

## 当前代码观察

1. `backend/src/npc/state_manager.py` 的 `cold_start()` 会清空图和向量数据，再逐 NPC 调用 `_init_graph()`；第一版天然适合只对新开局生效。
2. `_init_graph()` 已承担人物节点、共享背景、个人核心记忆和持久化编排。可见性判断、文本投影和事实解释不得继续塞入该类，必须放入独立模块。
3. `_seed_initial_memory_node()` 目前只支持单个 `target_id`，无法连接多个 `subject_ids`，且随机节点 ID 不足以提供投影幂等性。
4. SQLite `memory_nodes` 保持极简，只存节点 ID、所属 NPC 和创建时间；来源、权限与可信度应放入独立投影来源表，避免扩张图节点本体。
5. LanceDB 节点保存内容、类型、importance、created_day 与 archived；来源审计不能只依赖 LanceDB 可选字段。
6. 现有 `aisc_debug` 只有运行态与对话检索快照，不能回答冷启动投影来源、排除原因和实际连边，需要新增只读 action 与 DTO。

## 设计方案

### A. 初始事实配置

新增 `backend/config/initial_knowledge.json`，作为后端冷启动初始知识的稳定配置入口。顶层包含 `schema_version` 与 `facts`。每条事实至少包含：

```text
fact_id
fact_type
subject_ids
location_ids
canonical_summary
knowledge_scope
knower_ids
participant_ids
excluded_observer_ids
source_type
confidence
importance
created_day
projections
```

约束：

1. 所有稳定 ID 使用小写 `snake_case`。
2. `projections` 至少允许 `default`、`subject`、`participant`、`explicit_knower`、`rumor` 五类确定性文本模板；缺少适用模板时配置校验失败，不得由 LLM 补写。
3. `subject_ids`、`participant_ids`、`knower_ids` 和排除列表必须引用已知 NPC / player 稳定 ID。
4. 第一批内容只迁移能支持主要人物基础问答的稳定公开事实和少量关键共同经历；不把全部 `core_memories` 批量改造成共享事实。
5. 现有四条 `INITIAL_SHARED_MEMORIES` 应迁入该配置，避免共享事实继续存在两套入口；迁移后保持原有文本时间口径与玩家 token 渲染能力。

### B. 投影领域模块

新增 `backend/src/memory/initial_knowledge.py`，定义只负责配置解析、校验、可见性判断和投影生成的深模块。建议接口：

```python
load_initial_knowledge(path: Path) -> tuple[InitialKnowledgeFact, ...]
project_initial_knowledge(
    facts: Sequence[InitialKnowledgeFact],
    observer_id: str,
    relationship_context: KnowledgeRelationshipContext,
) -> KnowledgeProjectionResult
```

模块内部 DTO 至少包括：

1. `InitialKnowledgeFact`：稳定事实定义。
2. `KnowledgeVisibilityDecision`：`included`、`rule`、`reason`。
3. `ProjectedMemory`：稳定 `projection_id`、观察者、来源事实、节点类型、文本、人物 / 地点 ID、来源类型、confidence、importance、created_day。
4. `KnowledgeProjectionResult`：已投影项与被排除项，供写入和诊断共用。

规则：

1. `projection_id` 由 `observer_id + fact_id` 确定性构造，例如 `initial_knowledge__{observer_id}__{fact_id}`，不得使用随机 UUID。
2. `public` 对正常世界角色开放；`community` 由统一关系上下文与事实级排除列表判断；`participants` 只看参与者；`explicit_knowers` 只看配置知情者；`private` 只允许主体和显式知情者。
3. 观察者同时匹配多条规则时只生成一个投影，并记录最终采用的最强来源角色。
4. 文本模板只能使用事实显式字段，不得补齐观察者不知道的时间、地点、因果或人物关系。
5. 模块不依赖 SQLite、LanceDB、`StateManager` 或 Unity，保证规则可单测。

### C. 图初始化写入

在 `backend/src/npc/state_manager.py` 中只保留编排职责：加载一次初始事实，按 NPC 调用投影模块，再把 DTO 交给统一写入函数。

1. 将 `_seed_initial_memory_node()` 扩展或拆成能接受多个 `subject_ids` 的通用写入 seam。
2. 每个投影节点连接观察者 self 和所有存在于 `person_node_ids` 的 `subject_ids`；重复人物去重。
3. 边 ID 基于 projection ID、目标人物 ID 确定性构造，重复冷启动不会产生重复边。
4. 投影节点 ID 直接使用稳定 `projection_id` 或从其确定性派生；SQLite 与 LanceDB 使用相同 node ID。
5. 原个人 `core_memories` 继续只进入本人图。若其格式暂不支持多主体，不在本轮批量重构。
6. `StateManager` 不实现 scope 分支、模板选择或配置校验。若执行中仍需把这些职责放入 `StateManager`，必须停止并请求用户确认进一步减重。

### D. 来源元数据与数据库

在 `backend/src/database/sqlite_client.py` 新增 `memory_initial_projections` 表，作为投影来源与权限审计表，建议字段：

```text
projection_id TEXT PRIMARY KEY
node_id TEXT NOT NULL UNIQUE
observer_id TEXT NOT NULL
source_fact_id TEXT NOT NULL
fact_type TEXT NOT NULL
knowledge_scope TEXT NOT NULL
visibility_rule TEXT NOT NULL
visibility_reason TEXT NOT NULL
source_type TEXT NOT NULL
confidence REAL NOT NULL
importance REAL NOT NULL
subject_ids_json TEXT NOT NULL
location_ids_json TEXT NOT NULL
created_day INTEGER NOT NULL
created_at TEXT NOT NULL
```

要求：

1. 建表与索引顺序兼容旧数据库；新表为空不改变旧档内容。
2. `cold_start()` 清空该表，保持新游戏幂等。
3. 提供小型 CRUD 查询：批量写入、按 observer 查询、按 `source_fact_id` 查询。
4. `memory_checkpoint` 的表清单必须加入该表，保证保存 / 恢复事务完整。
5. 被排除的事实不持久化为运行时节点；排除原因由同一事实配置重新计算并通过诊断返回，避免为所有 NPC × facts 保存冗余拒绝记录。

### E. `aisc_debug` 诊断同步

新增只读 action `initial_knowledge_projection_snapshot`，参数支持 `npc_id`、可选 `source_fact_id` 和 `include_excluded`。返回至少包括：

1. 观察者 ID、事实 ID、投影 / 排除状态。
2. `projection_id`、实际 node ID、value 预览。
3. `knowledge_scope`、`visibility_rule`、`visibility_reason`。
4. `source_type`、confidence、importance、subject IDs、location IDs。
5. 实际连接的 edge IDs 与 person node IDs。
6. 配置或图数据缺失时的稳定 failure reason。

执行时需同步：

1. 后端提供读取投影元数据、事实配置和实际边的只读诊断入口；优先复用现有后端 HTTP / WebSocket 诊断传输，不在 Unity 本地猜测数据库状态。
2. `Assets/Scripts/Diagnostics/DiagnosticModels.cs`
3. `Assets/Scripts/Diagnostics/AiscDiagnostics.cs`
4. `Assets/Scripts/Editor/Diagnostics/AiscDebugMcpTool.cs`
5. `Assets/Tests/Editor/AiscDiagnosticsTests.cs`
6. `Assets/Scripts/Diagnostics/README.md`

若现有 Unity 诊断链无法访问后端冷启动数据，执行对话必须先确认最小只读传输 seam；不得退化为 Console 日志或运行时代码自动重建。

## 规模预算

第一版以“支持主要人物基础问答”为准，不追求设定全量投影：

1. 每个 NPC 初始投影建议控制在 8～20 条。
2. 单 NPC 新增投影边按每节点 self 边 + 1～3 条人物边估算，并在 execution 记录实际最大值。
3. 冷启动总构建耗时和 LanceDB 写入量必须与现状对比；若投影导致明显阻塞，优先批量写入和减少事实粒度，不引入冷启动 LLM。

## 风险与控制

1. 配置过量导致图膨胀：限定首批事实与每 NPC 投影预算，禁止直接复制所有 profile 文本。
2. `community` 泄密：支持事实级排除，并为新来者 / 封闭角色写反例测试。
3. 模板制造新事实：模板变量白名单 + 配置校验，传闻必须带不确定口径。
4. SQLite schema 破坏旧档启动：新表独立创建，遵守先迁移后索引规则，并跑旧库 checkpoint 测试。
5. Day 0 被默认值覆盖：读取 `created_day` 时显式区分 `None` 与合法 0，不使用 `value or 1`。
6. `StateManager` 继续增重：所有 scope、模板和权限逻辑留在独立模块；StateManager 只编排和写入。
7. 诊断只剩日志：必须交付结构化 `aisc_debug` action、DTO、测试和文档后才能宣称完成。
8. 与初始图语义路由修复重叠：执行前读取最新 `state_manager.py`，复用已落地的 clarity 规则，不覆盖相关人工或既有修改。

## 验收标准

1. 每个 NPC 对主要人物拥有符合权限的基础可回答事实，而不是只有名字和 bond。
2. 公开事实进入合理观察者图；共同经历只进入参与者 / 指定知情者图；私密事实不泄露。
3. 同一事实能形成不同观察者口径，且不增加事实源没有提供的确定信息。
4. 每个投影能追溯到稳定 fact ID、来源类型、权限规则、confidence、人物 / 地点 ID 和实际图边。
5. 重复冷启动不产生重复节点、来源记录或边。
6. 旧存档不会被静默注入新知识；第一版无重建控制 action。
7. 图模型仍符合 ADR-0001，不新增共享运行时图或未经决策的边类型。
8. `aisc_debug.initial_knowledge_projection_snapshot` 能结构化说明生成与排除原因。
9. 冷启动图规模、耗时和向量写入量有实际数据，并落在执行案预算或在 execution 中说明偏差。
