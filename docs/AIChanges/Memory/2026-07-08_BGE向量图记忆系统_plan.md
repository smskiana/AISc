# BGE 向量图记忆系统 — 设计方案 v0.6

---

## 核心概念

### 清晰度是边的属性

```
节点 = 事实本身（数据在 LanceDB，永久存在）
边   = "从 A 能想起 B 吗？"（有向，独立衰减）
清晰度 = 边的有向权重 = 旧版的 weight
cost   = 1/清晰度 = LLM 路由的检索成本
```

```
                clarity=0.9              clarity=0.05
    "我" ───────────────────→ "千早送面包" ←────────────────── "下雨那天"
         ←──────────────────                          
              clarity=0.7              (从"下雨"几乎想不起"送面包")
```

### 图只做联想，向量存数据

```
SQLite Graph (联想层)
  节点: {id, subject_id}     ← 只存 ID
  边: {a→b, type, clarity}   ← 有向权重
  职责: 结构连接、联想路径

LanceDB (数据层)
  {node_id, vector, value, type, importance}
  职责: 实际内容、语义搜索
  永久不删
```

> 图边 = 神经突触（连接，会衰减）  
> 向量 = 突触里的记忆内容（永久保存）  
> 边断了 ≠ 数据丢了，只是自然想不起来 ← 强制回忆直接搜向量层

## 一、三层架构

| 层 | 存哪里 | 存什么 | 保留 |
|------|--------|------|------|
| **联想层** | SQLite | 节点ID + 边(clarity 有向) | 边衰减即删除 |
| **数据层** | LanceDB | value, vector, importance, type | **永久** |
| **上下文层** | SQLite short_term_memories | 原始对话 | ≤3天 / 玩家1年 |

### Schema

```sql
-- 节点: 极简，只存 ID
CREATE TABLE memory_nodes (
    id TEXT PRIMARY KEY,
    subject_id TEXT NOT NULL,
    created_at_game_time TEXT
);

-- 边: 有向清晰度
CREATE TABLE memory_edges (
    id TEXT PRIMARY KEY,
    node_a TEXT NOT NULL,          -- 源
    node_b TEXT NOT NULL,          -- 目标
    type TEXT NOT NULL,            -- involved/sequenced/relationship
    clarity_ab REAL DEFAULT 0.7,   -- A→B
    clarity_ba REAL DEFAULT 0.7,   -- B→A
    created_at_game_time TEXT
);
```

```python
# LanceDB: vectors_{npc_id}
{
    "node_id": str,
    "vector": [512],        # BGE-small
    "type": str,            # event/emotion/person/reflection
    "value": str,           # 原始文本
    "importance": float,    # 0.0-1.0 不变
    "created_day": int,
    "archived": int,        # 0活跃 1边已断但保留
}
```

## 二、清晰度衰减（二次 U 型）

衰减速度由**目标节点**的 importance 决定：

```
decay_rate(imp) = peak * (1 - (2*|imp - 0.5|)^2)
peak = 0.343

每日: clarity_ab *= (1 - decay_rate(importance_of_B))
```

| importance | decay_rate/天 | 效果 |
|-----------|--------------|------|
| 1.0 | 0 | 永不忘 |
| 0.8 | ~0.055 | ~50天模糊 |
| **0.5** | **0.343** | **7天忘（最快）** |
| 0.3 | ~0.055 | ~50天（对称） |
| 0.1 | ~0.008 | ~365天（背景噪音） |

```
曲线:
0.34┤     ●(0.5)  ← 中等重要，忘得最快
    │    ╱ ╲
    │   ╱   ╲
0.05┤  ●     ●
    │ (0.1)  (0.8)
0──┴──────────────●(1.0)
   不重要  中等   重要
```

## 三、清晰度恢复

边被 LLM 路由实际**选中遍历**时，clarity 微涨：

```
触发: LLM 路由选择了边 A→B
恢复: clarity_ab += importance_B * 0.1 * (1 - clarity_ab)
上限: min(importance_B * 1.5, 0.95)
```

常想起的记得牢，不用的自然忘。

## 四、检索流程

### 完整流程

```
对话触发 (NPC A ←→ 目标 B，位置 L，时间 T)
│
├─ Step 1: 构建查询向量
│     query = "{B的名字} {位置L} {A当前情绪}"
│     query_vec = BGE.encode(query)
│
├─ Step 2: 双路召回（并行）
│     ┌─ 图联想路 ───────────────────┐
│     │ 从"我"+"B的person"节点出发    │
│     │ 查出边, 按 clarity 排序       │
│     │ 取 Top-15 (玩家) / 10 (NPC)   │
│     │ cost = 1/clarity + 跳数×0.5   │
│     └──────────────────────────────┘
│     ┌─ 向量语义路 ─────────────────┐
│     │ query_vec → LanceDB ANN       │
│     │ 过滤 archived=1               │
│     │ 取 Top-15 (玩家) / 10 (NPC)   │
│     │ score = cosine × importance   │
│     └──────────────────────────────┘
│
├─ Step 3: 合并排序
│     两路 node_id 合并去重
│     score = 图得分 + 向量得分
│
│       图得分 = clarity_ab × 2.0      (A→B有边)
│              + clarity_ba × 1.0      (反向也加分)
│              - hop_count × 0.3       (跳数惩罚)
│
│       向量得分 = cosine_sim × 1.5
│                + importance × 1.0
│                - days_ago × 0.02     (轻微近因)
│
│     取 Top-20 候选
│
├─ Step 4: LanceDB 取值
│     node_id 列表 → LanceDB 批量查 value/type/importance
│
├─ Step 5: LLM 路由 (仅玩家对话, NPC跳过省token)
│     候选列表 → LLM 选 1-3 条
│     budget = energy × bond × 5
│     被选中的边 → 触发清晰度恢复
│
└─ Step 6: 扩展 & 组装
       LLM选中的节点 → 查其出边(1跳) → 补充关联
       → 重建自然语言 → 注入 Prompt
```

### 玩家 vs NPC 区别

| | 玩家对话 | NPC 对话 |
|------|---------|---------|
| 双路召回 | 各 Top-15 | 各 Top-10 |
| LLM 路由 | ✅ 选 1-3 条 | ❌ 跳过 |
| 图扩展 | 1 跳 | 0 跳 |
| 清晰度恢复 | ✅ 选中边恢复 | ❌ |
| 最终注入 | ≤5 条 | ≤2 条 |

### 强制回忆

```
BGE(query) → LanceDB 搜 ALL(含archived) → 5-10条 → 自然语言
（不经过图，不走 LLM 路由，向量直达）
```

## 五、午夜流程

```
1. 边衰减: 所有 clarity *= (1-decay_rate) / clarity<0.05 → 删边
2. 孤点清理: 无边连接的节点 → SQLite删除 / LanceDB 标记archived=1
3. 向量持久化: 新节点 → BGE encode → LanceDB
4. 短期记忆清理: >3天删除（玩家1年）
5. 提取: short_term_memories → LLM → 节点(LanceDB) + 边(SQLite)
   初始 clarity = importance
6. 融合: 相似>0.85 → LLM → 融合节点(importance=源均值) / 旧节点archived
   (BGE 天然的语义聚类替代了反思)
```

## 八、补充定义

### 冷启动（初始图）

```
每个 NPC 首次创建:
  LanceDB:
    "我" 节点: {value="我", importance=1.0}
    核心记忆节点: {value=记忆原文, importance=profile.core_memories[].importance}
    人物节点: {value="千早"/"小李"/..., importance=0.9}

  SQLite:
    "我" ↔ 人物 (type=relationship, clarity=initial_bond)
    核心记忆 → "我" (type=involved, clarity=1.0, 永不变)
```

### 边类型区分

| 类型 | 性质 | clarity 衰减 | 说明 |
|------|------|-------------|------|
| `relationship` | 联想边 | ✅ 二次U型 | NPC 对某人的感觉，会变淡 |
| `involved` | 结构边 | ❌ 固定 1.0 | 事件→人物，客观事实不变 |
| `sequenced` | 结构边 | ❌ 固定 0.5 | 事件先后顺序 |
| `located_at` | 结构边 | ❌ 固定 1.0 | 事件→地点 |
| `felt` | 联想边 | ✅ 二次U型 | 事件→情绪，情绪会忘 |
| `mentioned` | 联想边 | ✅ 二次U型 | 提到了什么话 |
| `contains` | 结构边 | ❌ 固定 1.0 | 融合包含事件 |
| `associated_with` | 联想边 | ✅ 二次U型 | 一般关联 |

**规则**: 联想边衰减，结构边不衰减。involved/sequenced/located_at/contains 是"客观事实"不应该忘。

### importance 存储优化

```
问题: 边衰减公式需要目标节点的 importance，但 importance 只在 LanceDB
      每夜衰减 N 条边 = N 次 LanceDB 查询

解决: memory_edges 加一个冗余字段 target_importance
      创建边时从 LanceDB 读取一次写入
      边衰减时直接用，不查 LanceDB
```

### Archived 节点再激活

```
新边指向 archived=1 的节点 → archived 恢复为 0
节点重新进入活跃图（被新的记忆关联"唤醒"了）
```

### 核心记忆

```
在 v0.6 中等价于:
  - 节点 importance = 1.0（LanceDB）
  - 关联边 clarity = 1.0 且不衰减（SQLite）
  - decay_rate(1.0) = 0 → 永不衰减
```

### 节点创建时序

```
不是在午夜批量写 LanceDB——创建时立即写:
  提取/反思/融合生成新节点 → 立即 BGE encode → LanceDB insert → SQLite insert node + edges

午夜只做: 衰减 + 孤点清理 + 短期记忆清理
```

## 九、玩家记忆（简化向量层）

玩家只需 LanceDB 向量搜索，不需要图：

```
vectors_player:
  对话结束 → LLM 提取节点 → BGE encode → LanceDB

检索 (生成对话选项时):
  query = "关于{target_npc}的回忆"
  → BGE → vectors_player ANN Top-5
  → 注入选项生成 Prompt: "你记得上次..."
```

| | NPC | 玩家 |
|------|-----|------|
| LanceDB | ✅ | ✅ |
| SQLite 图 | ✅ | ❌ |
| clarity 衰减 | ✅ | ❌ |
| 检索 | 图+BGE | BGE 直搜 |
