# 记忆路由零 LLM 确定性查询切换 - 执行案

## 1. 方案定位

用户已根据 A/B/C 实验结果选择 C：玩家实时对话不再调用 R3 v2 或 LongCat 生成方向/检索提示，改由现有 `LocalDirectionProvider` 生成本地图方向，并由现有 `RetrievalQueryPlanner` 使用原问题、经校验的本地 retrieval query 和至多一条相关近期对白构造唯一 embedding query。

本文件是后续实现的唯一执行案。本会话只创建 plan，不修改运行代码、正式配置、测试或模型资产，不创建 execution/test。

- 主要功能域：`MEM` 记忆系统。
- Workstream：`docs/Workstreams/Memory/README.md`。
- 需求来源：`MemoryRouteVectorQueryHintExperimentDraft.md` 中的 C 组。
- 相关 ADR：`ADR-0001-memory-v07-light-graph`；本轮不改变每 NPC 独立图、图/向量分工、权限、archived、clarity、图起点或存档边界，不新建 ADR。
- Roadmap：不改变排期，不读取或修改 `docs/Roadmap.md`。

## 3. 当前代码级结论

codebase-memory 项目 `AISc_r3v2_runtime_retest_20260721` 显示：

1. `RetrievalEngine._retrieve_result()` 已对 `local_only` 使用 `direction_provider_runtime.local()`，随后统一调用 `RetrievalQueryPlanner.plan()`、一次 `_vector_search_once()` 和现有本地深搜；C 所需运行路径已经存在，不需要新增分支。
2. `LocalDirectionProvider.provide()` 已是零 LLM 确定性方向 provider，使用当前问题和近期对白识别人物、地点、意图与时间范围。
3. `RetrievalQueryPlanner.plan()` 已把原问题、经本地校验的 retrieval query 和至多一条相关近期对白组装为唯一 embedding query，不混入场景或摘要。
4. `DirectionProviderRuntime` 会构建所有已注册 provider，并在 `warmup()` 中预热所有专项 worker；仅把 player chain 改为 `[local]` 仍会加载未使用的 R3 worker。
5. 因此正式配置必须同时把 `player_dialogue.strategy` 切为 `local_only`、chain 切为 `[local]`，并从生产 provider 注册表移除 `r3_v2`，才能真正消除模型调用、后台预热和 GPU 常驻。
6. `general_llm` 不创建专项进程，可保留为未被 mode 引用的显式兼容 provider；原 `llm_full_route` 实现也继续保留。

## 4. 冻结功能决策

### 4.1 正式默认策略

| mode | strategy | direction chain | 方向模型调用 | 向量查询 |
|---|---|---|---:|---:|
| `player_dialogue` | `local_only` | `[local]` | 0 | 最多 1 |
| `npc_dialogue` | `local_only` | `[local]` | 0 | 最多 1 |
| `nightly_impression` | `local_only` | `[local]` | 0 | 最多 1 |

`direction_override` 继续优先于 mode provider；午夜固定方向、`llm_full_route` 的代码能力和原通用 `general_llm` provider 不删除。

### 4.2 生产 provider 注册表

正式 `memory_retrieval.yaml` 冻结为：

```yaml
direction_providers:
  default_chain: [local]
  providers:
    general_llm: {kind: general_llm}
    local: {kind: local}
```

移除的只是生产配置中的 `r3_v2` 注册块，不删除以下实现和历史证据：

- `route_specialist_contract.py`
- `route_specialist_provider.py`
- `route_specialist_worker.py`
- R3 worker/provider/codec 单元测试
- 训练代码、Adapter 和前置 plan/execution/test

未来若重新启用专项 provider，必须通过新的正式方案和配置恢复，不依赖隐式环境变量或自动发现模型资产。

### 4.3 诊断口径

C 的正式 trace 预期为：

```text
strategy = local_only
direction_source = local
direction_provider_requested = not_applicable
direction_provider_adopted = not_applicable
direction_provider_chain = []
direction_model_call_count = 0
llm_direction_calls = 0
llm_route_calls = 0
vector_query_count = 1（LanceDB 可用时）
degraded_to_local = false
```

`not_applicable` 是现有纯 local trace 口径，不为显示美化修改过重的 `_build_trace()`。

## 5. 实现范围

### 5.1 必须修改

1. `backend/config/memory_retrieval.yaml`
   - `default_chain` 改为 `[local]`。
   - 删除生产 `r3_v2` provider 配置。
   - `player_dialogue.strategy` 改为 `local_only`。
   - `player_dialogue.direction_chain` 改为 `[local]`。
   - 其他检索预算、query、scoring 和 mode 参数保持不变。
2. `backend/tests/test_retrieval_policy.py`
   - 默认 policy 改断言三 mode 均为 `local_only/[local]`。
   - 继续验证配置 version、预算和严格 chain。
   - specialist 环境变量/path/hash 校验改用测试内存 payload，不再假设生产配置注册 R3。
3. `backend/tests/test_conversation_memory_routing.py`
   - 默认玩家对话改断言本地方向、模型调用 0、唯一向量 query 和确定性 query 来源。
   - 需要覆盖显式 R3/general chain 的测试继续使用注入 policy，不把生产默认改回模型链。
4. `backend/tests/test_route_specialist_provider.py`
   - `test_runtime_registers_all_configured_provider_kinds` 使用显式测试 payload 构造三 provider，继续锁定 builder mapping、chain 和 worker health；不得从生产默认配置推断 R3 已注册。
5. `backend/tests/route_runtime_isolated_factory.py`
   - local/general 继续从正式配置复制预算。
   - `provider_id=r3_v2` 时显式注入冻结的测试 provider 配置，保留 R3 隔离回归能力。
   - local provider 的 player strategy 使用 `local_only`；general/R3 显式评估仍使用 `llm_guided_local`。
6. `backend/src/memory/README.md`
   - 当前默认改为玩家、NPC、午夜均本地确定性方向；R3/LongCat 保留为未配置能力。
7. `docs/Workstreams/Memory/README.md`
   - 实现后把当前工程口径更新为 C 已生效，并把 R3 实时延迟从“待优化默认路径”改为历史切换原因。

### 5.2 明确不修改

1. `RetrievalEngine`、`GameRuntime`、`DirectionProviderRuntime` 和 `RetrievalQueryPlanner` 实现。
2. R3 codec/provider/worker 实现与模型资产。
3. SQLite/LanceDB schema、存档和正式数据。
4. Unity 场景、Prefab、SerializeField、UI 或协议 DTO。
5. 三种公开 strategy 枚举；`llm_guided_local` 和 `llm_full_route` 继续作为可配置能力存在。

如果执行分析发现必须向 `RetrievalEngine`、`GameRuntime` 或 `_build_trace()` 增加 C 专用分支，应停止执行并回到方案会话更新本 plan，不允许把职责继续塞入过重类。

## 8. 独立验收标准

1. 正式配置启动后不要求 `MEMORY_ROUTE_PYTHON`、`MEMORY_ROUTE_R3_V2_ADAPTER` 或 `HF_HOME` 才能完成 provider runtime 装配。
2. `DirectionProviderRuntime.health_snapshot()` 为空，warmup 立即返回空结果，不生成 R3 worker 进程，不增加专项模型显存。
3. 玩家真实检索 trace 为 `strategy=local_only`、`direction_source=local`、所有方向模型调用数为 0。
4. LanceDB 可用时每轮向量查询仍为 1；`llm_route_calls=0`。
5. 当前问题、至多一条相关近期对白和确定性 retrieval query 仍按 `RetrievalQueryPlanner` 预算组装；不混入场景或摘要。
6. 隔离 corpus expected 节点全部命中、forbidden 节点为 0，clarity 和持久检索日志在 probe 前后不变。
7. 现有显式 R3/general provider 单元和隔离测试仍可通过测试 payload 运行，证明能力是“未配置”而不是被破坏。
8. `local_only`、`llm_guided_local`、`llm_full_route` 三种公开策略的离线回归不回退。
9. Unity Player 独立真实对话至少记录首轮和后续轮检索墙钟；不得再出现 `specialist_timeout`，检索阶段 p95 目标不高于 3 秒。
10. 文档不再宣称 R3 v2 是玩家默认 provider。

## 9. 风险与回滚

1. **质量风险**：当前实验主要是 BGE proxy 和冷生成 corpus，缺完整真实图证据。通过隔离 factory 和 Unity 实际对话补足，不用正式数据冒充隔离测试。
2. **图方向风险**：LocalDirectionProvider 词表对隐式代词、否定纠正和复杂关系的覆盖弱于模型。保留原问题与近期对白 query，并在独立测试记录未覆盖案例；若失败，回到新的执行会话修正本地方向，不恢复 10 秒级默认模型调用掩盖问题。
3. **测试耦合风险**：部分 R3 测试从生产配置取得 provider block。改为显式测试 payload，避免生产默认与历史能力测试互相绑死。
4. **回滚**：若独立测试不通过，在新的执行会话按本 plan 的前置 R3 配置块恢复 `r3_v2` 注册和 player `[r3_v2, local]`；不得在测试会话直接改配置。

## 10. 会话边界

本方案预计修改不少于 8 个既有文件，并创建 execution/test 记录，触发三会话分离：

1. 本方案会话只创建并索引本 plan，随后停止。
2. 新执行会话必须引用本 plan，先使用 codebase-memory；若索引落后于当前工作树，更新后再验证可行性。执行完成后创建唯一 execution，只能结论为“实现完成，待独立测试”或“实现未完成”。
3. 新测试会话读取本 plan、execution 和测试工具 Workstream，完成独立验收并创建 test。
4. execution/test 必须与本 plan 互链；未通过 test 前不得宣称切换整体完成。
