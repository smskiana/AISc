# LLM 检索想法、单次向量路由与最终原子条目选择修复执行案

## 1. 需求理解

本案修复玩家询问“你知道千早在哪里吗，我在找她”时，结构合法但语义泛化的 LLM 方向、含噪向量 query 和最终类型先验共同导致千早证据退出对白 Prompt 的问题。

本轮目标不是注入 `npc_states.current_location`，也不保证 NPC 得到实时正确位置；只保证已有、可达且有权限的相关记忆证据不会因方向改写、噪声 query、类型先验或字符截断而丢失。没有证据时仍返回既有无知识 / 不可达 / 无权限语义，不允许检索想法制造事实。

需求来源：

1. [LlmRetrievalQueryVectorRoutingFixDraft.md](../../Workstreams/Memory/LlmRetrievalQueryVectorRoutingFixDraft.md)
2. [LLM 方向合法误判与最终重排丢失实体记忆](../../DesignDocs/errors/2026-07-17_llm_direction_and_final_rerank_memory_loss.md)
3. [三策略可配置记忆路由与本地深层检索方案](2026-07-17_LLM定向本地深层记忆检索_plan.md)

## 2. Workstream、Roadmap 与 ADR

- 主要功能域：Memory。
- 关联功能域：Dialogue、TestingAndDiagnostics；只同步既有诊断 DTO 和入口，不新增正式业务协议。
- Roadmap item：MEM“保持轻量图 + 向量数据层，补诊断与质量评估”。本案不改变排期，也不扩张旧 v0.5 图模型。
- 相关 ADR：ADR-0001。保持每个 NPC 独立图、SQLite 图路径、LanceDB 语义内容、固定图起点、权限 / archived 过滤和向量仅作锚点或内容兜底的边界。
- 不新增 ADR：本案是在既有边界内修正检索 query、评分和最终上下文组装。

## 3. 已确认设计决定

### 3.1 三种策略的行为

| strategy | 检索想法来源 | 方向 LLM 调用 | ANN 查询 | 本案变化 |
|---|---|---:|---:|---|
| `local_only` | 当前问题 + 本地方向 | 0 | 最多 1 | 使用统一 query plan、单次向量命中和最终选择 |
| `llm_guided_local` | 扩展现有 `memory_direction` 的 `retrieval_query` | 最多 1 | 最多 1 | 修复主路径；不新增第二次 LLM 调用 |
| `llm_full_route` | 原始问题；不生成额外检索想法 | 既有逐步 route 调用 | 0 | 只接入通用最终原子条目选择，不改变 route 调用语义 |

`llm_full_route` 若未来也要增加一次方向调用或 ANN，属于策略语义和预算变化，必须另行更新本 plan；本轮不得顺手加入。

### 3.2 图路由深度观测期统一增加 3

为让真实图验证能观察到深层路径效果，本轮先将三个业务 mode 的本地深搜深度和完全 LLM 路由跳数统一增加 3。`llm_full_route` 每个竞争步骤需要一次 route LLM，因此 `max_llm_route_calls` 必须同步增加 3，否则新增 hop 会被调用预算提前截断：

| mode | local `max_depth` | LLM `max_hops` | `max_llm_route_calls` |
|---|---:|---:|---:|
| `player_dialogue` | 4 → 7 | 4 → 7 | 4 → 7 |
| `npc_dialogue` | 2 → 5 | 2 → 5 | 2 → 5 |
| `nightly_impression` | 6 → 9 | 4 → 7 | 4 → 7 |

这是观测期参数，不同时放大 `beam_width`、每节点邻边、frontier、候选条数或 `max_expanded_edges`。这样可以单独判断“更深”是否带来有效证据，而不会把深度与广度效果混在一起。若因既有总边预算提前停止，必须保留 `edge_budget_exhausted` / `budget_exhausted` 诊断，不得静默扩大全局广度预算；execution 根据固定图和隔离真实图结果记录实际达到深度，再决定是否保留这些值。

### 3.3 embedding query 组成

第一版坚持每轮最多一次 embedding / ANN。实际 query 按固定标签和固定顺序组成：

```text
当前问题：{query_text}
检索想法：{validated_retrieval_query}
相关近期对白：{optional_selected_turn}
```

规则：

1. 当前问题始终完整保留，是最高优先级输入。
2. `llm_guided_local` 使用校验后的 LLM 检索想法；`local_only` 使用本地确定性检索想法。
3. 最多加入一条近期对白，且只能由本地选择器确定性选出：优先与当前问题的明确人物 / 地点 mention 重叠；若问题含“她 / 他 / 那里 / 那个人 / 上次你说”等指代，再选最近一条能解析到稳定实体的对白。
4. 近期对白按完整条目选择后再做单条安全字符上限；不得拼入会话摘要、当前场景 location、玩家背景或近期记忆列表。
5. `retrieval_query` 自身不得新增当前问题没有的人物、地点、精确时间或确定事实；被选近期对白只作为独立 query 段，不授权 LLM 把其中事实改写成确定答案。
6. query 组装和字符预算是纯函数，输出同时包含选择原因、各段字符数和最终字符数。

第一版预算直接进入各 mode 的 typed policy：

| mode | `retrieval_query_chars` | `selected_recent_turn_limit / chars` | `embedding_query_chars` | `final_context_max_chars` |
|---|---:|---:|---:|---:|
| `player_dialogue` | 180 | 1 / 160 | 640 | 1800 |
| `npc_dialogue` | 160 | 1 / 120 | 480 | 1200 |
| `nightly_impression` | 240 | 0 / 0 | 800 | 2400 |

启动校验必须拒绝负数、`selected_recent_turn_limit > 1`、分段上限大于总 query 上限，以及 `final_context_max_chars < 64`。若总 query 超限，按“去掉近期对白 → 压缩检索想法到完整句边界 → 保留完整当前问题”的固定顺序处理；当前问题本身超过总上限时不得中段截断，允许 query 超出配置并记录 `original_query_exceeds_budget`，由上游输入长度约束另行治理。

### 3.4 方向契约与本地校准

复用现有 `memory_direction` task，不新增 Prompt task。扩展 `RetrievalDirection`：

```text
retrieval_query       短文本；失败时回退到原始问题
query_constraints     受控枚举集合
```

第一版 `query_constraints` 只允许：

```text
person_location
identity
relationship
cause
past_event
recent
```

现有 `entity_mentions`、`location_mentions`、`themes`、`relation_facets`、`time_scope`、`recall_intent` 继续保留。`DirectionResolver` 必须同时计算本地方向并执行校准：

1. 当前问题的明确人物 / 地点 mention 由本地目录提取，是 `explicit_entities` 的权威来源；LLM 不得删除或替换。
2. 明确“在哪 / 哪里 / 位置”等定位问句至少校正为 `locate_person + current_location + person_location`。
3. LLM 合法但与本地方向冲突时，保留本地最低约束并记录 `direction_semantic_calibrated`，不能因 JSON 合法就直接接受。
4. `query_constraints` 与校准后的 intent / theme / time 必须一致；未知枚举丢弃，矛盾项由本地规则收口。
5. 检索想法为空、超长、出现未授权稳定实体或新增精确时间时，回退为原始问题，并分别记录稳定原因：`retrieval_query_empty`、`retrieval_query_too_long`、`retrieval_query_entity_mismatch`、`retrieval_query_fact_risk`。
6. LLM 不可用、超时、非法 JSON 的既有降级语义保持不变。

### 3.5 通用最终评分 profile

不设置人物专用保底槽，不硬编码任何 NPC。先合并图命中与内容兜底并按 `node_id` 去重，再由统一选择模块评分。初始 profile 写入已校验的 `memory_retrieval.yaml`，不开放 UI、运行时协议或存档字段：

| 分量 | 初始权重 | 约束 |
|---|---:|---|
| query semantic similarity | 0.42 | 仅使用本轮单次 ANN 的归一化相似度；无命中为 0 |
| graph path relevance | 0.25 | 使用受限图搜索的方向 / 路径分并归一化到 0..1 |
| explicit entity alignment | 0.12 | 通用原文实体信号，不保证占槽 |
| recency | 0.08 | 根据 `created_day` 与请求日归一化；缺失用中性值 |
| importance | 0.08 | 归一化到 0..1 |
| node type prior | 0.05 | 硬上限 0.05；不得让 `event` 类型单独压过明确相关证据 |

所有分量和总分都必须进入诊断。LanceDB 返回的 `_distance` 必须由存储 adapter 按明确、经测试的度量转换成 0..1 相似度；业务层不得直接猜测 `_distance` 含义。排序 tie-break 固定为总分、图路径分、ANN rank、`node_id`，保证可重复测试。

### 3.6 原子记忆条目与字符预算

新增内部 `RetrievedMemoryEntry` / `RetrievalContextAssembly` 契约。第一版一个节点对应一个不可拆分条目，不引入跨节点证据束：

1. 条目包含 node ID、类型、完整正文、时间摘要、来源用途、分数组成和渲染文本。
2. `final_memory_limit` 计完整条目数。
3. 新增 `final_context_max_chars`；按排序结果贪心装入完整条目，放不下则整条淘汰并继续尝试下一条，禁止从中间截断正文、来源或时间。
4. `person`、`identity`、`place` 等当前 `_rebuild()` 未覆盖的合法节点类型必须有中性渲染，不得把节点名称改写成关系或位置事实。
5. 若所有相关条目都因预算被淘汰，返回明确 `budget_exhausted` 证据，不能输出半条文本。
6. 最终 `retrieved_node_ids`、trace 的 final IDs 和实际进入对白 Prompt 的条目必须完全一致。

## 4. 模块与 seam

### 4.1 外部 seam

调用者继续只使用：

```python
RetrievalEngine.retrieve(RetrievalRequest) -> RetrievalResult
```

Dialogue、NPC-NPC 和夜间调用层不感知 `retrieval_query`、query 组装、ANN 分数或最终评分权重。

### 4.2 内部深模块

新增两个内部深模块，禁止继续向 `RetrievalEngine` 塞入多状态、多分支实现：

1. `RetrievalQueryPlanner.plan(request, calibrated_resolution, recent_turns, policy) -> RetrievalQueryPlan`
   - 封装本地明确实体、检索想法校验、相关对白选择、query 组装、字符预算和降级原因。
2. `RetrievalContextAssembler.assemble(candidates, query_plan, vector_hits, request_time, policy) -> RetrievalContextAssembly`
   - 封装去重、相似度归一化、通用评分、原子条目渲染、条目数 / 字符预算和最终诊断。

`RetrievalEngine` 只编排 provider、query planner、一次向量 adapter、图 retriever 和 context assembler。现有 `_build_vector_query()`、`_rerank_node_data()`、`_rebuild()` 在迁移完成后删除或只保留必要兼容委托，不保留两套正式实现。

这是减重型实施。执行前用户必须明确给出“减重并执行”授权；未取得授权不得把新逻辑直接追加到 facade。

## 5. 实施阶段

### 阶段 A：先写失败测试与契约

1. 扩展 typed contracts、query constraint 枚举、query plan、vector hit、原子条目和 assembly outcome。
2. 用 fake LLM 固化“合法 `general_recall` + 千早定位问句”的失败场景，先证明会校准。
3. 用固定竞争图固化“千早人物节点 vs 高 importance event”最终选择场景。
4. 固化单次 ANN、完整条目预算和 `person` 中性渲染测试。
5. 新增超过旧深度上限但位于新上限内的固定路径，分别证明本地深搜和完全 LLM 路由可实际到达新增的 3 层；同时覆盖总边 / 调用预算先到时的明确停止原因。

### 阶段 B：扩展方向并实现 query planner

1. 更新 `memory_direction` response contract 和 system contract，要求返回短 `retrieval_query` 与白名单 `query_constraints`。
2. `DirectionResolver` 同时获取本地方向并校准 LLM 方向；保留原始安全摘要、校准动作和失败原因。
3. 实现 `RetrievalQueryPlanner` 的明确实体提取、检索想法校验、单条近期对白选择和固定标签组装。
4. 覆盖空、超长、新增实体、精确时间风险、泛化方向、非法 JSON、超时与不可用回退。

### 阶段 C：单次向量 adapter 与本地图路由

1. 让 `local_only` / `llm_guided_local` 都只对 `RetrievalQueryPlan.embedding_query` 编码一次、查询一次。
2. 将 LanceDB 命中转换为 typed `VectorSearchHit(node_id, rank, similarity, usage)`；明确并测试距离度量转换。
3. 同一批命中继续同时服务 `anchor_discovery` 与 `content_fallback`；不得第二次 ANN。
4. 锚点仍不能成为额外起点或直接答案；固定起点断言保持。
5. 将三种 mode 的本地 `max_depth`、LLM `max_hops` 和耦合的 `max_llm_route_calls` 按 3.2 表统一提高 3；不修改其他广度预算。

### 阶段 D：最终选择减重与原子组装

1. 实现 `RetrievalContextAssembler`，迁移 facade 内的最终去重、评分、排序、渲染和预算。
2. 接入配置化六分量 profile 与类型先验上限。
3. 为 `person` / `identity` / `place` 增加中性、无事实扩写的完整条目渲染。
4. 同步 final IDs、上下文字符串和淘汰诊断，删除正式路径上的旧重排 / 重建实现。

### 阶段 E：诊断、Unity DTO 与文档同步

1. 扩展 `RetrievalTrace`、对话诊断 DTO 和 `memory_retrieval_snapshot`，区分原问题、检索想法、embedding query、图命中、最终条目和淘汰原因。
2. 同步 Unity `DiagnosticModels.cs`、反序列化缓存和 `AiscDiagnosticsTests`；不修改场景、Prefab、UI 或 SerializeField。
3. 执行 Unity C# 导入、编译和 EditMode / PlayMode 前，必须先阅读 `UnityMCPUsageRules.md` 并使用 Unity MCP；不可用时停止 Unity 侧修改并请求用户指示。
4. 回写 Memory / Dialogue Workstream、后端 Memory / Dialogue README、Diagnostics README、Prompt / config / tests 入口和 execution。

## 8. 自动化测试与验收

### 8.1 方向和 query

1. LLM 返回合法 `general_recall` 时，“千早在哪里”仍校准为 `locate_person + current_location + person_location`。
2. `llm_guided_local` 正常路径恰好一次方向 LLM、最多一次 ANN；`local_only` 为 0 次方向 LLM、最多一次 ANN；`llm_full_route` 不新增方向调用和 ANN。
3. embedding query 必含原问题和千早，不含当前 `flower_shop.counter`、玩家背景、会话摘要或近期记忆噪声。
4. 只有符合规则时加入一条近期对白；普通明确问题不得带入无关对白。
5. 检索想法漏实体、加入其他人物 / 地点、加入原文没有的精确时间、为空或超长时可校准 / 回退并输出稳定原因。
6. “喫茶店最近怎么样”等无人物问题不获得实体加分；指代问题能按规则选择一条相关对白。
7. 三个 mode 的本地深度分别为 7 / 5 / 9，完全 LLM 路由 hop 与调用预算分别为 7 / 5 / 7；未知值或 hop 大于调用预算时启动校验失败。

### 8.2 图路由、最终选择与原子性

1. 固定起点始终只有 `self + 当前对话对象`；向量锚点只影响评分和路径，不进入初始前沿。
2. 千早人物节点与高 importance 无关 event 竞争时，由六分量相关性决定，不能由 `event` 类型先验单独挤掉。
3. 不要求人物节点强制占槽；低相关人物节点可以正常淘汰。
4. `person` / `identity` / `place` 被选择时实际进入 `rebuilt_context`，且不扩写不存在的关系或位置事实。
5. `final_memory_limit` 按完整条目计数；字符不足时整条淘汰，不出现半句、半条时间或来源。
6. final IDs、trace final entries 和对白 Prompt 条目完全一致。
7. 图内无相关事实、不可达、权限过滤、archived、预算耗尽和向量不可用保持可区分。
8. 位于旧深度之外、新深度之内的证据可以被检索；达到既有总边预算时停止原因准确，不能通过隐式扩大 beam / frontier 伪造深度收益。

### 8.3 回归、真实图与 Unity

1. 全量 `python -m pytest backend/tests -q` 通过。
2. 九组合离线评估脚本继续通过，额外报告 query 来源、ANN 次数、最终分数组成、淘汰原因和条目字符量。
3. 在隔离 SQLite / LanceDB 副本上复测目标 trace 场景，不写正式 `backend/data`；比较修复前后千早相关图命中、最终条目和 P50 / P95。
4. Unity Console 0 Error，`AiscDiagnosticsTests` 全部通过。
5. 使用 `aisc_control.run_memory_retrieval_probe` 和正式对话链验证“千早在哪”、无实体问题、无知识问题；再用 `aisc_debug.memory_retrieval_snapshot` 核对 query、单次向量、图候选、final entries 与淘汰原因。
6. 无实时位置注入时，回答不确定不记为检索系统异常；只有证据在检索链中丢失才算失败。

## 9. 完成标准

只有全部满足才可宣称完成：

1. 四个待确认项已按本 plan 固化：最多一条相关对白、一次 ANN、六分量初始 profile、本地明确实体权威。
2. 三种 mode 的本地深度统一增加 3；完全 LLM 路由 hop 与耦合调用预算也统一增加 3，其他广度预算不变，并有实际到达深层证据。
3. `memory_direction` 在原有一次调用内同时提供检索想法和受控方向，没有第二个 LLM task。
4. 明确定位语义经过本地校准，合法 JSON 不再等同于语义正确。
5. query 不再被场景地点、玩家背景、会话摘要和近期记忆列表污染。
6. 最终选择不含人物硬编码保底，类型先验不超过 0.05。
7. 完整人物 / 身份 / 地点条目可以进入对白 Prompt；条目数和字符预算都不截断语义单元。
8. 固定起点、权限 / archived 过滤和向量用途边界未改变。
9. `RetrievalEngine` 保持稳定 facade，新职责位于两个可独立测试的深模块。
10. 功能本体、诊断钩子、Unity DTO / 测试、真实或隔离样本验证和相关入口文档全部同步。
11. execution 记录实际文件、最终配置值、测试 / Unity MCP / 真实样本证据、实际达到深度、未完成项，以及是否触及方案边界。

## 10. 风险与回退

1. **LLM 检索想法夹带事实**：本地实体 / 时间校验，失败回退原始问题，保留稳定诊断原因。
2. **近期对白重新引入噪声**：最多一条、确定性选择、独立标签、严格字符上限；可通过 policy 将数量降为 0。
3. **ANN 距离误用**：相似度转换收口在 adapter 并用已归一化向量固定样本测试，业务层只接收 0..1。
4. **评分调参掩盖问题**：profile 完整配置化、类型先验硬上限、固定竞争集和逐分量 trace；不在代码里散落魔法数。
5. **人物节点进入 Prompt 后造成事实扩写**：使用中性类型渲染，只呈现节点原文与已有时间，不推断关系或位置。
6. **字符预算导致高相关长条目全部淘汰**：记录 `entry_exceeds_context_budget`，通过隔离真实样本校准 `final_context_max_chars`；禁止恢复中途截断。
7. **facade 再次增重**：实施前必须取得“减重并执行”，并先落 query planner / context assembler 测试 seam。
8. **Unity 诊断不同步**：后端 DTO 改动与 Unity DTO / EditMode 测试同阶段完成；Unity MCP 不可用时停止 Unity 侧工作，不宣称完成。
9. **深度增加放大延迟和 LLM 成本**：只增加深度及完全 LLM 路由必需的调用预算，不增加广度；九组合与隔离真实图必须记录实际深度、P50 / P95、展开边数和 LLM 次数，execution 据此给出保留或回调建议。

## 11. 执行会话要求

本案预计新增文件不少于 2 个、修改文件不少于 3 个，且涉及后端检索、Prompt、诊断与 Unity DTO。必须在新的独立执行对话中引用本文件作为唯一方案来源，并明确授权“减重并执行”。执行对话不得重新创建第二份 plan；若实现边界变化，应停止并回到独立方案对话更新本文件。
