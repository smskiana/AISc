# 双向边语义实现 — 计划

## 完成时间
2026-07-10

## 需求理解

用户已明确选择“方案 B：显式维护双向边语义”，本轮不再停留在预研或风险讨论，直接把图的建边和检索改成方向语义明确、代码层可控的实现。

这轮目标聚焦在两个点：

1. 建边时明确维护 `A -> B` 与 `B -> A` 的不同可回忆强度
2. 检索时按“当前起点真实朝外看的 clarity”取邻边，而不是继续把双向 clarity 混成一个值

## 当前问题

### 1. 检索抹平了方向差异

`backend/src/memory/retrieval.py` 当前只查 `get_out_edges()`，并且最终直接取 `max(clarity_ab, clarity_ba)`。这会让：

1. 很多 `event -> person/self` 结构从人物起点回不去事件
2. 边方向虽然被存下来了，但检索层没有真正使用

### 2. 建边规则没有显式语义

`backend/src/memory/manager.py` 当前对午夜入图边统一使用：

1. `clarity_ab = importance`
2. `clarity_ba = importance * 0.7`

`backend/src/npc/state_manager.py` 里的初始化边也各自散落着硬编码，缺少统一口径。

## 实施思路

### 1. 抽出共享的双向边语义规则

新增一个后端辅助模块，集中定义这些边型的默认双向语义：

1. `relationship`
2. `involved`
3. `located_at`
4. `happened_at`
5. `felt`
6. `mentioned`
7. `sequenced`
8. `contains`
9. 其他默认边

规则里不仅看 `edge_type`，也会参考 `node_a_type` / `node_b_type`，让“事件指向人物”和“人物回忆事件”都落到明确的初始值。

### 2. 初始化图和午夜入图统一走同一套规则

把以下入口统一改成共享规则驱动：

1. `backend/src/npc/state_manager.py`
2. `backend/src/memory/manager.py`

这样初始图和后续新增图不会再各写各的。

### 3. 检索改成方向感知邻边查询

在 `backend/src/database/sqlite_client.py` 增加面向“起点节点”的邻边查询，让 SQL 直接返回：

1. 邻居节点 ID
2. 当前遍历方向 `ab/ba`
3. 当前方向对应的 `directional_clarity`

然后 `backend/src/memory/retrieval.py` 只按这个方向值排序和打分。

## 涉及文件

## 验证计划

1. 对改动文件做 `python -m py_compile`
2. 做一次 `import src.main` 级别的导入检查
3. 如环境允许，再做一次轻量冷启动，确认初始化图和检索链路都不报错

## 本轮不做

这轮先不展开以下更大改造，避免目标飘移：

1. 融合节点继承旧边
2. 人物节点唯一约束的深层重构
3. `recover_clarity()` 的完整遍历恢复闭环
4. 新事件与旧事件的规则补边体系
