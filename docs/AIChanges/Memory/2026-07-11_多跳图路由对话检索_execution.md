## 关键实现说明

### 1. 对话检索正式改成多跳边路由

旧链路是：

1. 图邻边 + 向量召回
2. 合并成候选节点
3. 一次 LLM 直接选节点

现在改成：

1. 从 `self + target person` 双起点出发
2. 当前前沿节点并发收集候选边
3. 本地算法先筛每跳候选边
4. LLM 基于时间 / 地点 / 短期记忆 / 状态 / 印象选择下一跳边
5. 进入下一批节点
6. 维护 `visited_nodes / visited_edges` 防成环
7. 汇总命中节点后再重建为相关回忆

### 2. 玩家对话和 NPC 闲聊使用不同参数

#### 玩家对话

- `max_edges_per_hop = 6`
- `edges_per_route = 2`
- `max_hops = 2`
- `vector_fallback_limit = 2`
- `final_memory_limit = 5`

特点：

1. 更深一点
2. 更容易带出玩家背景与商店街旧事
3. 仍允许顺带捞出 NPC 私人记忆

#### NPC ↔ NPC

- `max_edges_per_hop = 4`
- `edges_per_route = 2`
- `max_hops = 1`
- `vector_fallback_limit = 1`
- `final_memory_limit = 3`

特点：

1. 更轻
2. 更偏日常闲聊
3. 不主动深挖太长的私人回忆链

### 3. 向量补充降级为兜底

本轮没有让向量库继续做白天主检索，而是改成：

1. 图多跳先跑完
2. 只有图结果低于模式阈值时，才补少量向量候选
3. 补的数量严格限制在 `1-2` 条

这样保住了“图清晰度衰减 = 遗忘主机制”的地位。

### 4. 每跳路由已经开始吃上下文

每一跳 LLM 现在能看到：

1. 当前时间
2. 当前地点
3. 当前情绪 / 精力 / 需求
4. 对目标的当前印象
5. 最近 `2-3` 条短期记忆摘要
6. 当前前沿节点
7. 候选边列表

此外对 `npc_dialogue` 还加了模式偏好提示：

1. 优先与对方、眼前场景、商店街日常相关的轻量回忆
2. 不要一上来跳到沉重私人往事

### 已完成验证

1. Python 编译检查通过：

#### `kujo -> player` 检索结果

输出优先为：

1. 小李准备重开奶奶留下的喫茶店
2. 小李小时候在樱桥通长大，后来去了城市又回来
3. 奶奶去世后喫茶店关了很久
4. 樱桥通熟人社会

说明玩家对话模式已经明显优先经过玩家背景长期记忆节点。

#### `sakura -> chihaya` 检索结果

当前 smoke test 能稳定返回轻量回忆条目，但仍不算非常理想；在无白天短期记忆输入的冷启动场景下，仍可能混入 Sakura 的私人核心记忆。

这说明：

1. 多跳路由和双模式参数已经工作
2. 但 NPC ↔ NPC 的“日常轻聊偏好”还可以继续加强

### 当前残留限制

1. 冷启动场景下，NPC ↔ NPC 路由仍可能抓到角色的高重要度私人旧事
2. 这不是流程失效，而是“图里可供日常闲聊的目标相关记忆还不够多”
3. 后续若要继续优化，最直接的是补更多 NPC 之间的共享旧事 / 店铺日常初始长期记忆，而不是继续堆 Prompt 补丁

## 结论

本轮已经把系统从“单次节点筛选”推进到“多跳边路由”。

也就是说，对话检索现在已经开始体现：

1. 记忆是沿边联想出来的
2. 不同对话类型可以走不同深度
3. 白天图主导、向量兜底，而不是反过来
