# 记忆清晰度演化系统 — 设计方案

## 背景

当前实现（v0.5）每夜午夜对所有 NPC 全量执行 5 步演化，其中 `similar_to` 使用 LLM 做 O(n²) 两两语义比较——Day 1 即达 ~1000 次 API 调用，Day 2 起更严重（9 分钟未完成）。同时存在以下问题：

- 节点数量不受控（"今天天气真好"也进图）
- 演化策略不区分记忆新旧/重要度
- 午夜处理串行执行，5 个 NPC 逐个跑
- JSON 提取因 token 截断频繁失败

## 核心设计：清晰度驱动四阶段

### 清晰度公式

```
初始化：  clarity = importance（LLM 评估，0.1 ~ 1.0）
每夜衰减： clarity *= importance

importance = 1.0 → clarity 永远 1.0（核心记忆，永不衰减）
importance = 0.5 → Day0=0.5, Day1=0.25, Day2=0.12, Day3=0.06, Day4≈0.03(遗忘)
importance = 0.3 → Day0=0.3, Day1=0.09, Day2≈0.03(遗忘)，2天遗忘
```

**不使用时间分阶段**——从节点自身的 clarity 值决定它处于哪个阶段。

### 四阶段

| 阶段 | 清晰度 | 含义 | 触发操作 |
|------|--------|------|----------|
| **清晰** | > 0.7 | 细节完整 | sequenced 边 |
| **比较清晰** | 0.4 ~ 0.7 | 大概记得 | sequenced + similar_to (event) |
| **模糊** | 0.15 ~ 0.4 | 只剩印象 | 精度转换 + 全similar + 反思 |
| **非常模糊** | 0.05 ~ 0.15 | 即将遗忘 | 全力融合 + 精度全转 vague |
| **遗忘** | < 0.05 | 从活跃图删除 | 归档到 original_text，节点移除 |

### 每日午夜流程

```
午夜
  │
  ├── 1. 全节点衰减: UPDATE clarity = clarity * importance
  │       clarity<0.05 → 归档 original_text → DELETE 节点+关联边
  │
  ├── 2. 写入过滤: NPC对话 importance≥0.3 才进 short_term_memories
  │
  ├── 3. LLM 提取: 短期记忆 → 图节点（5 NPC 并发）
  │
  └── 4. 按 clarity 分阶段处理（只处理刚跨阈值的节点）:
          │  避免每晚重处理未变化的节点
          │
          清晰节点       → 只做 sequenced
          比较清晰节点    → sequenced + similar_to(event)
          模糊节点       → 精度转换 + 全similar + 反思
          非常模糊节点    → 全精度转换 + 融合
          遗忘节点       → 归档删除
```

## 加速策略

### 1. similar_to 改用本地 BGE Embedding

- 当前：每对节点 1 次 LLM → O(n²) API 调用
- 改为：BGE encode value → 余弦相似度 → 毫秒级，零 API
- 仅对跨越阈值进入"比较清晰"或"模糊"的节点执行
- BGE 模型已在 `requirements.txt`（sentence-transformers），无需额外部署

### 2. 午夜流程并发化

```
提取阶段: 5 个 NPC 同时 asyncio.gather() → 耗时 = 最慢那个 ≈ 10s
演化阶段: 同上
总耗时: ~30s（当前串行 5+ 分钟）
```

### 3. 阈值触发（只处理变化的节点）

```
当前: 每晚遍历所有节点判断做什么
改为: 记录每个节点的 last_processed_clarity
      仅当当前阶段 != 上次处理阶段 时才执行对应操作
```

每晚实际处理的节点量：

| 总节点 | 跨阈值/新增 | 重处理 |
|--------|-----------|--------|
| ~50 | ~3-5 | ~45 跳过 |

### 4. LLM 提取 JSON 鲁棒性

- `max_tokens` 600 → 1200（避免截断）
- 裸换行修复（逐字符扫描替换）
- token 截断修复（找最后 `}` → 补全 `]}`）
- 事件原文截断 1500 字符

## 写入过滤

| 来源 | importance | Day 0 写入？ | 
|------|-----------|-------------|
| NPC↔NPC 闲聊 | 0.1-0.3 | ❌ |
| NPC↔NPC 重要对话 | ≥0.3 | ✅ |
| 玩家↔NPC 对话 | 0.5 | ✅ |
| 核心记忆 | 0.8-1.0 | ✅ |
| 午夜反思生成 | 0.6 | ✅ |

importance 由 LLM 在提取时评估，写在 JSON 的每个事件节点里。低于阈值的节点直接丢弃。

---

## 遗忘处理

1. clarity < 0.05 → 节点 value 写入 `original_text` 字段归档
2. 节点 + 关联边从图中 `DELETE`
3. 不保留残影（energy=0 节点）

### 强制回忆

玩家触发 NPC"回忆某事"时：
- 默认 LLM 不查归档
- 强制回忆时查 `original_text` 中该 NPC 的历史记录
- 不重新激活节点——这次对话本身会创建新节点

### 清晰度提升

**当前不做。** 如果一个新事件与旧记忆高度相关，similar_to 已经链接它们，最终融合成新节点（继承组合后的 clarity）。新事件本身会创建节点承载当前语境。不需要单独"提升旧节点清晰度"的机制——融合就是它的"复活"。

---

## 核心记忆保护

```
importance = 1.0
clarity *= 1.0 = 永远 1.0
→ 永远"清晰"，永不精度转换，永不融合
```

## 风险点

- DeepSeek API 并发限制未知（同时 5 请求是否限流）
- BGE 中文语义相似度精度可能不如 LLM（可后续加 LLM 二次确认）
- clarity 乘性衰减对低 importance 节点可能过快（2-3 天就遗忘）
