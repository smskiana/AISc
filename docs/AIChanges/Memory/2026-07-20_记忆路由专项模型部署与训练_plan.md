# 记忆路由专项模型部署与训练 - 执行案

## 1. 方案定位

本方案用于验证 `Qwen3-0.6B + Route LoRA` 在本机替代 `memory_direction` 通用 LLM 的可行性。本轮执行只部署训练/离线推理资产、构建数据、完成 SFT 和离线评估，不把专项模型接入 `RetrievalEngine`，不修改 `backend/config/memory_retrieval.yaml` 的正式策略，也不启动 shadow 或生产推理。

- Workstream：Memory。
- 需求来源：`docs/Workstreams/Memory/MemoryRoutingSpecialistModelSwitchDraft.md`。
- 相关 ADR：`ADR-0001-memory-v07-light-graph`；本轮不改变轻量图、权限、起点、向量次数或检索 facade，因此不新增 ADR。
- 路由基线：用项目现有 `llm_full_route` 作为最强检索质量基线和教师证据来源；专项模型仍只学习 `RetrievalDirection`，不学习节点 ID、边 ID或逐跳选边。
- 当前状态：本方案只授权后续独立执行会话按此实施；本会话不下载模型、不安装依赖、不启动训练。

## 2. 已确认环境与可行性

### 2.1 本机条件

- GPU：NVIDIA GeForce RTX 4060 Laptop GPU，显存 8188 MiB。
- 当前 Python：3.13.7。
- 已安装：PyTorch 2.12.0、Transformers 5.11.0。
- 当前缺少：`peft`、`datasets`、`accelerate`、`bitsandbytes`、`trl`。

结论：0.6B 模型的 LoRA/QLoRA SFT 和确定性离线推理在 8GB 显存上可行。为规避 Python 3.13 与量化训练依赖的兼容风险，执行时必须创建独立 Python 3.11 或 3.12 环境，并先做 CUDA、量化加载和单步反向传播 smoke；不得直接改造现有后端解释器环境。1.7B 只作为 0.6B 通过后、显存 smoke 成功时的可选对照，不得阻塞首轮结论。

### 2.2 外部依据与锁定要求

1. 官方模型卡：[`Qwen/Qwen3-0.6B`](https://huggingface.co/Qwen/Qwen3-0.6B)，Apache-2.0，0.6B 参数、32,768 上下文；官方要求 Transformers >= 4.51，并支持 `enable_thinking=False`。
2. 官方/上游训练框架：[`modelscope/ms-swift`](https://github.com/modelscope/ms-swift)，支持 Qwen3、RTX、LoRA/QLoRA、BNB 量化和自定义数据集。
3. 执行记录必须冻结并记录：底模完整仓库名、commit/revision、tokenizer revision、许可证快照、训练框架 commit/version、CUDA/PyTorch 版本、量化方式和 Adapter 配置。

本案不预先锁死当前最新版本号。执行会话先完成兼容 smoke，再把实际可复现版本写入训练锁文件和 execution；若官方模型 revision、许可证或依赖兼容性与上述结论不一致，停止执行并回到方案会话更新本 plan。

## 3. 训练目标与“最强路由”口径

### 3.1 教师与标签

每条训练候选必须经过以下链路：

```text
规范化输入 DTO
-> 当前 general_llm memory_direction 原始输出
-> DirectionResolver 本地校准后的方向
-> llm_full_route 检索
-> 期望节点/权限/回答证据复核
-> 人工接受、修正或剔除
-> Route SFT 标签
```

`llm_full_route` 只用于验证方向最终带来的检索质量，不可把它返回的节点或边写进专项模型输出。SFT 标签以人工复核后的、通过现有 `_validate_direction` 与 `_calibrate_direction` 语义的 `RetrievalDirection` 为准，禁止直接蒸馏通用 LLM 原始 JSON。

### 3.2 输入输出冻结

输入按草案的 `schema_version=1` DTO 固定，包括当前问题、摘要、预算内近期对白/记忆、地点、时间、参与者、已知别名和 mode。中文通过 UTF-8 JSONL 文件传递，不通过 PowerShell 中文参数。

输出沿用现有 `RetrievalDirection` 字段和枚举；首轮 `confidence` 只保留在离线评估元数据，不修改正式 DTO。训练和推理都关闭 thinking，使用确定性解码；离线解析后必须继续经过现有本地校准、权限和语义拒绝。

### 3.3 数据切分与覆盖

1. 数据按剧情主题、人物组合或会话来源分组切分，禁止把相邻对话随机拆到 train/validation/test。
2. 覆盖明确人物/地点/事件/时间、跨轮指代、否定纠正、比较、多人物竞争、别名、无相关记忆、无权知道、错误前提和场景噪声。
3. 专门加入历史回归：“在哪里”被通用模型误判为 `general_recall`、场景地点污染目标地点、明确实体在最终重排中丢失。
4. 原始对白、未授权记忆和真实存档副本不得提交 Git；提交的 corpus 只能是脱敏、人工批准的固定夹具。大体积或敏感原始产物放在 Git 忽略的本地 artifact 目录。

## 4. 实施阶段

### 阶段 A：训练环境与模型离线部署

1. 新建项目内声明、项目外缓存的独立训练环境；锁定 Python、CUDA、PyTorch、Transformers、PEFT/训练框架及数据依赖。
2. 下载 `Qwen/Qwen3-0.6B` 到独立模型缓存，记录 revision 与 SHA；禁止放入 `Assets/`、正式后端包体或 Git。
3. 完成 tokenizer、BF16/FP16 推理、4-bit 量化加载、单条非 thinking JSON 生成、单步 LoRA 反向传播 smoke。
4. 任一 smoke 失败时先收敛兼容矩阵，不进入数据生产或正式训练。

### 阶段 B：数据采集与 golden corpus

1. 新增独立离线采集脚本，复用 `RetrievalEngine.retrieve`、正式 `DirectionResolver` 和只读/隔离数据副本，不复制业务校准逻辑。
2. 对固定夹具和经批准的隔离真实会话运行 `general_llm + llm_full_route`，保存原始方向、校准方向、query、候选、最终节点与权限结果。
3. 提供人工复核清单；未复核样本不得进入 golden corpus 或测试集。
4. 生成带 schema/version/source_group 的 UTF-8 JSONL，并输出数据覆盖、拒绝原因和 train/validation/test 分组摘要。

### 阶段 C：Route LoRA SFT

1. 先跑极小数据过拟合，证明模板、loss mask、EOS 和 Adapter 保存/加载正确。
2. 再运行 0.6B Route LoRA/QLoRA SFT；优先保证稳定 JSON 与语义字段，不使用 DPO、GRPO 或多任务训练。
3. 训练配置至少记录 seed、context length、batch/gradient accumulation、learning rate、epoch/step、LoRA target/rank/alpha/dropout、量化和 checkpoint 策略。
4. Adapter、日志和 checkpoint 写入 Git 忽略的 artifact/model 目录；仓库只提交小型 manifest、配置、脚本和脱敏评估摘要。

### 阶段 D：离线评估但不接入

1. 比较 R0 本地 provider、R1 当前通用 LLM、R2 0.6B zero/few-shot、R3 0.6B Route LoRA。
2. 所有候选方向都经过同一校准链，并用 `llm_full_route` 对同一隔离图执行检索级对照；禁止多跑 ANN 或写 clarity/检索日志。
3. 记录字段级、检索级、延迟、显存、冷启动、非法输出与回退模拟指标。
4. 本阶段不新增 `specialist_python` 正式 provider，不修改运行配置，不启动 shadow；评估通过只能得出“具备接入候选资格”。

## 6. 最低门禁与独立验收

### 6.1 执行会话最低门禁

1. 训练环境可从锁文件重建，模型 revision 与 Adapter manifest 可追溯。
2. 量化加载、单步反向、极小集过拟合、Adapter 重载和固定样本推理均成功。
3. 数据 schema 校验、分组无交叉、UTF-8 往返、枚举白名单和隐私扫描通过。
4. 训练脚本 dry-run 和聚焦 Python 测试通过。
5. 0.6B SFT 产物完成一次全量离线评估；失败时保留证据并得出“不可行/需调整”，不得假报完成。

### 6.2 独立测试会话通过线

1. JSON/schema 合法率 >= 99.9%。
2. 明确实体召回率不低于当前通用 LLM + 本地校准基线。
3. 未知实体、检索 query 新事实、知识越权均不得高于当前基线。
4. 在同一 `llm_full_route` 检索条件下，最终相关记忆命中率达到通用 LLM 基线的 95% 以上。
5. 所有失败样本可被现有本地校准/回退模拟收口，不产生空检索或异常中断。
6. 单轮 embedding/ANN 次数不超过当前预算，离线评估不改变正式存档、clarity 或运行配置。
7. 报告冷启动、p50/p95/p99、峰值/常驻显存和单样本吞吐；不预设 1500ms 达标，先以本机实测决定后续 shadow 是否值得进入下一份方案。

独立测试会话先读取本 plan、execution、`docs/Workstreams/TestingAndDiagnostics/README.md`、`backend/tests/README.md` 和 `backend/scripts/README.md`，再创建互链 test record。没有通过 test record 前，不得宣称专项模型整体完成或可替换当前路由。

## 7. 风险与停止条件

1. Python 3.13 或 Windows 量化训练依赖不兼容：切换独立 Python 3.11/3.12 环境；不修改正式后端环境。
2. 8GB 显存不足：依次降低序列长度、micro batch、启用 gradient checkpointing/4-bit；仍失败则记录 0.6B 不可行并停止，不擅自迁移云端或上传项目数据。
3. 教师偏差：保留原始输出、校准输出、人工结论三层，未复核样本不作金标。
4. `llm_full_route` 成本与外部调用：执行前确认现有 LLM 凭据可用，并设置样本数、调用数和费用上限；不得无限采集。
5. 数据泄露：训练集仅使用已授权、脱敏内容；任何上传外部服务或模型仓库的动作都需用户另行明确授权。
6. 正式接入、shadow、C# 迁移和 1.7B 对照均不属于本轮；需要另开方案或更新本 plan 后再执行。
