# 日程提交幂等与重连恢复执行方案

> 执行记录: [2026-07-18_日程提交幂等与重连恢复_execution.md](2026-07-18_日程提交幂等与重连恢复_execution.md)

## 1. 目标与边界

修复同一游戏日重新生成 `revision=1` 导致 `revision_content_conflict` 的缺陷，并明确开局、跨日、读档、重连和互动重规划的计划提交语义。本阶段不改变 planner 选什么，只保证生成、保存、恢复和整体提交具有持久幂等性。

原方案明确允许后端在开局、跨日、有效互动重规划和显式失败恢复后，通过 `NPC_DAILY_SCHEDULE_READY` 整体提交计划。本阶段按该口径实施；若产品决定改为纯查询模式，必须先更新原方案和 Workstream，再执行协议改造。

## 2. 实施步骤

1. 定义日程持久身份：`game_day + npc_id + schedule_revision + payload_fingerprint + planner_version + operation_id`。
2. 将已提交日程及单调 revision 持久化到明确的数据层；`_prepared_days` 只能作为进程内优化，不再承担正确性。
3. `ensure_daily_plans()` 先恢复同日已提交计划；存在有效计划时不得再次调用 LLM，只允许幂等重发相同 revision 和内容。
4. 新计划内容替换时 revision 必须基于持久化 base 递增；提交失败不得提前推进 revision。
5. Unity `GAME_START`、CONTINUE、读档和 WebSocket 重连区分世界初始化与连接恢复；重连不触发重新规划。
6. Unity 恢复同日存档后向后端提交或查询其权威 revision/fingerprint，明确冲突时采用何方恢复，而不是静默覆盖。
7. 不同游戏日必须先完成 Unity 跨日清理，再接受新日 revision；旧日迟到结果稳定拒绝。
8. 为 `NPC_DAILY_SCHEDULE_READY` 增加 request/operation 关联和明确 status；失败或 skipped 结果不得被当成新完整计划替换。

## 4. 诊断与验证

1. `aisc_debug.daily_schedule_snapshot` 增加 payload fingerprint、最后提交 operation、接收来源和拒绝原因。
2. 后端诊断显示 `generated / restored / idempotent_replay / replaced / stale_rejected`。
3. 自动化覆盖：同进程重复 GAME_START、后端重启、Unity 重连、同日读档、跨日、迟到结果和同 revision 不同内容。
4. 反馈循环必须证明同一日重连不增加 LLM 调用次数，且 Unity 不再产生 `revision_content_conflict`。

## 5. 验收标准

- 同一游戏日、相同上下文只生成一次计划。
- 后端重启后恢复原计划，不重新生成随机内容。
- 内容变化必然使用更高 revision；相同 revision 内容恒定。
- Unity 存档和后端托管状态可在重连时明确对账。

## 6. 风险与停止条件

- 若现有 SQLite schema 无法表达日程完整 DTO，先提出 schema 迁移方案，不以 `_plans` 内存缓存继续兜底。
- 若纯查询模式成为新产品口径，停止本方案并先更新原总方案的 3.3.1、4.2 和协议验收项。
