# NPC 任务语义与节点检测设计方案

## 需求理解

将 NPC 正式行为从“移动 action 或原地 action”收敛为统一任务：任务声明要做什么、在哪里做、如何移动过去；Unity 负责移动、执行和终态确认，后端只接受 Unity 返回的成功，并通过阶段节点询问、停滞检测和较长硬超时处理失联兜底。

## Workstream / Roadmap

- Workstream: `NpcBehavior`、`FrontendArchitecture`、`ProtocolAndIntegration`
- Roadmap item: 正在进行 / NPC 行为语义收敛
- 相关 ADR: `ADR-0002`；导航 typed path 边界保持不变
- 工程口径变化: `walk_to` / `run_to` 不再作为正式任务，移动方式改为任务参数；任务成功只能由 Unity 回报

## 方案

1. 拆出后端任务目录与任务追踪职责，`BehaviorEngine` 只负责编排。
2. 共享配置建立 action affordance 与 spot tags，统一校验 action-location-role 组合。
3. `NPC_BEHAVIOR` 增加 `movement_mode`、`expected_duration_sec` 与 `plan_source`。
4. Unity 新增任务执行器，统一执行“必要时移动 -> 动作阶段 -> 前端终态回报”。
5. 新增 `NPC_TASK_STATUS_QUERY / NPC_TASK_STATUS`，后端按真实单调时间做节点询问、停滞检测和硬超时。
6. 日计划移除 `walk_to` / `run_to`，迁移为 `patrol`、`read`、`rest`、`drink`、`work_tend` 等实际任务。
7. 行为记忆延后到 Unity 成功回报之后。

## 风险

1. 旧 `walk_to` routine 必须按真实意图迁移，禁止机械替换成另一种移动命令。
2. Unity `JsonUtility` 需要明确 DTO，不能依赖动态字典。
3. 任务存活时间必须使用真实单调时间，不能受游戏暂停、倍速或跳时影响。
4. 状态询问不能让无进展任务无限续命，必须比较 phase 与 progress revision。
5. 新 Unity 脚本可能在编辑器刷新前未进入生成的 csproj，需记录命令行验证边界。

## 验证

1. Python 单元测试覆盖 affordance、终态提交、状态询问、停滞和硬超时。
2. 规范检查覆盖 routine 的 action-location 组合并禁止正式 `walk_to` / `run_to`。
3. Python compileall。
4. Unity C# 命令行编译；若生成工程未收录新文件，补充静态引用检查并记录 Play 回归项。
