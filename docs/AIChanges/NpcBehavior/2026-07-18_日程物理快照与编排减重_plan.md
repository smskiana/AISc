# 日程物理快照与编排减重执行方案

> 执行记录: [2026-07-18_日程物理快照与编排减重_execution.md](2026-07-18_日程物理快照与编排减重_execution.md)

## 1. 目标与边界

为“日程三层候选与记忆证据深化”提供两个实施前置：由 Unity 提供可版本化、同批次冻结的日程物理快照；将日程请求组装、批次编排和持久化从过重的 `BehaviorEngine` 中迁至独立 facade。完成后，`BehaviorEngine` 只保留兼容委托和既有社交语义入口。

本案不实现记忆证据、候选评分、LLM 选择、傍晚社交机会求解或 Unity 日程执行压缩；这些仍由后续既有执行案完成。不得通过 Python 猜测、扫描场景或写入 Unity 运行时状态来填充物理字段。

## 2. 已验证的现状与决定

1. `DailySchedulePlanner` 已接收单 owner `physical_state`，但 `BehaviorEngine._plan_all_npcs()` 在开局批次传入 `{}`；互动后请求虽可携带该字段，却没有版本化 DTO 和批次共享契约。
2. `runtime._apply_world_snapshot()` 只把基础 NPC 状态写入 Python 推理缓存，未保存地点营业、spot、可达性、社交锁或日程快照版本。
3. Unity 是位置、移动状态、spot 可用性和运行时任务的权威；Python 只消费冻结事实并生成计划，不得倒置裁决权。
4. `BehaviorEngine` 已约 587 行，继续加入快照聚合、retrieval 注入和跨 NPC 编排会违反类减重约束。因此新建 `DailyScheduleService` 作为深模块边界，而非继续向该类添加职责。

## 3. 接口与职责设计

### 3.1 冻结快照 DTO

新增后端纯 DTO（建议 `backend/src/npc/schedule_world_snapshot.py`）：

- `ScheduleWorldSnapshot`：`game_time`、`time_revision`、`world_revision`、`weather`、`snapshot_id`、地点/spot/NPC 状态映射。
- `ScheduleLocationState`：稳定 `location_id`、`open_state`（`open / closed / unknown`）、`reachable_state`（`reachable / unreachable / unknown`）、营业/有效时间窗和来源版本。
- `ScheduleSpotState`：稳定 `spot_id`、所属 location、`availability`（`available / unavailable / unknown`）与原因码。
- `ScheduleNpcPhysicalState`：稳定 NPC ID、实际 location、movement status、当前任务、社交锁、角色营业职责和可接待窗口。

DTO 仅接受稳定 ID 和有限枚举；缺失信息保留为 `unknown`，不可转换为 `open`、`reachable` 或 `available`。所有 owner 必须引用同一 `snapshot_id + time_revision + world_revision`。

### 3.2 Unity 协议与快照缓存

1. 扩展现有 Unity `world_snapshot` 载荷，新增版本化 `npc_schedule_physical_state`；先按 [UnityMCPUsageRules.md](/F:/GameProject/unity/AISc/docs/DesignDocs/UnityMCPUsageRules.md:1) 用 Unity MCP 修改 DTO、序列化和资产侧连线。
2. 后端新增只读 `ScheduleWorldSnapshotStore`，按 `snapshot_id` 保存有界最新快照并拒绝倒退的 `time_revision / world_revision`；不写入 `npc_states` 替代 Unity 权威源。
3. `GAME_START` 日程批次与 `NPC_SCHEDULE_REPLAN_REQUEST` 必须显式引用完整快照；缺失、过期或版本不匹配时返回稳定拒绝原因，不以空字典降级生成“物理合法”计划。

### 3.3 日程编排深模块

新增 `DailyScheduleService`（建议 `backend/src/npc/daily_schedule_service.py`）：

1. 接收 `DailySchedulePlanner`、快照 store、routine/profile provider、持久化 gateway、发送 gateway 与日程诊断 provider。
2. 在同一冻结快照下构建全部 `NpcScheduleRequest`，为每个 owner 投影自己的 `physical_state`，再调用 planner 的批量 seam。
3. 统一负责日程结果序列化、SQLite snapshot 保存、幂等重放、广播和互动后重规划；它不做候选细节、LLM Prompt、物理裁决或记忆检索。
4. 将 `BehaviorEngine.ensure_daily_plans()`、`_plan_all_npcs()`、`replan_from_unity_request()` 收缩为 facade 委托；保留对外调用名，避免协议路由一次性大迁移。
5. `application/runtime.py` 在装配期构造快照 store 和 service；将 retrieval 通过明确的构造参数注入后续三层日程模块，不让 `BehaviorEngine` 隐式抓取运行时单例。

## 4. 实施步骤

1. 先为快照 DTO、store、倒退版本拒绝、owner 投影和 service facade 写后端测试；覆盖 unknown、closed、unavailable、unreachable、缺失、陈旧和五 owner 同快照。
2. 使用 Unity MCP 检查并扩展现有 Unity 协议 DTO 与 `world_snapshot` 发送端；生成 `npc_schedule_physical_state` 时只读取 Unity 权威运行时数据，并为不存在的动态事实输出 `unknown`。
3. 实现后端 DTO 与 store，并在 runtime 协议接收处验证快照 ID、有限枚举和版本单调性；同步 Python codec、协议 README、诊断 DTO 与测试。
4. 实现 `DailyScheduleService`，迁移 `BehaviorEngine` 中日程组装、批次、持久化和重规划代码；每次迁移后保持原协议字段与幂等语义不变。
5. 接线 runtime composition root 与 `BehaviorEngine` 兼容 facade；移除已迁移私有状态和旧 Prompt/候选辅助代码，防止双路径调用。
6. 扩展 `aisc_debug.daily_schedule_snapshot`，暴露快照 ID、time/world revision、每 owner 物理字段覆盖率、拒绝原因和 service trace；必要时新增只读 `aisc_control` 快照注入探针，必须复用正式校验 seam。
7. 在 Unity 编辑器中用 Unity MCP 验证快照发送、后端接收、五 owner 共享版本和互动后重规划；每次 C# 修改后检查编译 Console。

## 6. 风险与停止条件

1. 若 Unity 无法在同一世界 revision 下给出全部 NPC 的物理快照，停止后续三层日程实施；不得用 Python 缓存拼接不同 revision 的事实。
2. 若跨 NPC 协调需要通用资源占用/预约求解器，停止并另行评估 ADR；本案只提供快照与编排边界。
3. 若迁移无法将 `BehaviorEngine` 的日程职责显著收缩，停止继续添加功能并重新做减重设计。
4. 旧客户端缺少新字段时，按显式协议版本拒绝或维持旧日程路径；不得让其静默走“物理合法”的新 planner。

## 7. 验收标准

1. 开局和重规划均使用同一份、版本可验证的 Unity 权威快照，任何缺失或陈旧快照都有稳定失败原因。
2. `BehaviorEngine` 不再持有日程批次组装、快照解析、持久化和广播细节，仅保留薄 facade。
3. `DailyScheduleService` 是日程编排的唯一 owner；同日幂等重放、120 秒 owner 隔离和原有协议字段保持可用。
4. `aisc_debug` 能关联日程 operation 与快照 ID/revision，不输出 Prompt、原始 LLM 内容或 embedding。
5. Unity Play Mode 验证 5 名 NPC 共用快照；关闭地点、不可用 spot、不可达地点和 stale revision 不进入后续候选构建路径。
