# NPC 运行时权威与社交协议收口执行方案

> 执行记录: [2026-07-18_NPC运行时权威与社交协议收口_execution.md](2026-07-18_NPC运行时权威与社交协议收口_execution.md)

## 1. 目标与边界

本阶段落实原方案 3.6、7.2、8.3 和完成标准中的运行时权威边界：Python 只处理社交语义与内容生成，Unity 持有候选、reservation、会合、播放、超时、冷却和终态。同步移除仍可绕过 Unity controller 的旧主动行为协议与后端运行时状态写入。

本阶段不修改日程候选算法、记忆证据、17:00 压缩或场景资产。

- 主方案：`2026-07-17_外界感知日程与互动重规划_plan.md`

## 3. 实施步骤

1. 将 `NPC_SOCIAL_DECISION_RESULT` 收敛为纯语义响应；后端不得在 decision handler 中创建 session 或启动会合。
2. 建立 `NPC_SOCIAL_CONTENT_REQUEST / NPC_SOCIAL_CONTENT_RESULT`，由 Unity 在双方完成会合并重新校验后请求内容。
3. Unity 增加独立社交 session/reservation 状态机，负责双方原子占用、移动、READY、播放、COMPLETE、失败和超时释放；候选扫描器只负责候选与 pending 去重。
4. 删除后端主动 `NPC_SOCIAL_PREPARE / NPC_SOCIAL_CANCEL` 会合监督路径；记忆仅在 Unity 提交有效 COMPLETE 后写入。
5. 将 `NPC_BEHAVIOR` 从网络正式入口移除。Unity 内部任务启动改用独立命令 DTO 或任务接口，禁止复用后端消息类型。
6. 删除 `WAKE_UP / GAME_TIME_UPDATE / SLEEP_TRIGGER` 的 Unity 路由和 callback；删除旧 `NPC_ACTION_RESULT / NPC_TASK_STATUS` sender、DTO 与拒绝 ACK，确认无兼容消费者后再删除后端分支。
7. 删除 Python `StateManager.apply_tick()` 等运行时世界写入入口；跨日恢复只返回结构化 effect 或由 Unity 自行处理，不直接写 energy、need、asleep 等权威字段。
8. 以冻结 `GameTimeSnapshot + time_revision + world_revision` 替换社交、对话和午夜 operation 对全局可写 `game_clock` 的业务依赖；保留只读诊断缓存时不得触发业务。

## 5. 诊断与验证

1. `aisc_debug` 显示候选、decision、reservation、会合、内容、播放和终态阶段及稳定失败原因。
2. 供应商超时返回 `want_to_talk=false` 或内容失败，不冻结双方、不留下 reservation。
3. 后端断线、迟到 decision/content 和 world revision 变化均由 Unity 丢弃，不启动会合。
4. 全仓确认不存在网络 `NPC_BEHAVIOR` 正式执行入口、后端会合超时 task 和 Python 运行时状态 tick。
5. Python 测试覆盖纯 decision/content handler；Unity 测试覆盖 reservation 原子性、移动失败、超时、玩家抢占、迟到结果和存读档冷却口径。

## 6. 验收标准

- Python 不主动启动、取消或监督 NPC-NPC 会合。
- Unity 是社交运行时状态机与高频 NPC 状态的唯一权威。
- 旧时间、任务监督和行为命令协议不再可达正式执行链。
- 诊断 DTO、`aisc_debug`、测试和入口文档同步完成。

## 7. 风险与停止条件

- 若正式删除旧协议会影响尚未迁移的玩家对话或剧情调用方，停止删除并列出调用方，不保留静默兼容执行。
- 若需要修改 Prefab、SerializeField 或场景连线，停止代码修改并转入 Unity MCP 资产流程。
