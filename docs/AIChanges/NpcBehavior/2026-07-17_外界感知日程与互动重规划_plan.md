> 执行记录: [2026-07-17_外界感知日程与互动重规划_execution.md](2026-07-17_外界感知日程与互动重规划_execution.md)

## 1. 需求理解

本轮把现有 NPC 日计划从 `BehaviorEngine` 内部的字符串目录、字符截断、宽松 LLM 输出和零散重规划，收敛为一个具有结构化候选、三层决策、严格校验、确定性 fallback、互动后局部重规划和完整诊断的日程深模块。

最终目标不是让 LLM 自由创造行为，而是让本地系统先确定世界中合法且有意义的候选，再由每名 NPC 每日一次 LLM 调用完成语义筛选和日程组合。供应商不可用或单次调用超过 120 秒时，本地根据世界模型基础日程、权威必须度和相关度生成可复现计划。后端完成生成与最终校验后，把当日计划整体交给 Unity 托管；Unity 成为剩余计划、到点判断、切换、消费状态和正式存档的运行时权威。

计划时间只是 Unity 的开始与切换评估依据，不是后端强制完成命令；Unity 默认允许切换，当前处于 NPC-NPC 对话等特殊锁定状态时延迟切换，并通过可扩展的裁决接口为后续特殊状态保留能力。后端只在开局 / 跨日生成、互动后重规划和显式失败恢复时提交完整替换结果，不逐项驱动日程执行。

本轮同时完成运行时权威收口：删除 Python 自主推进的 `GameClock`、速度、暂停、午夜、wake 和行为 tick，并把 P0 生存、need / 状态触发、NPC-NPC 物理候选扫描、冷却和会合调度迁入 Unity。Python 只作为“脑”处理结构化规划、记忆检索与写入、LLM 社交意愿 / 对白、互动重规划和午夜记忆维护；每次调用显式接收 Unity 冻结的时间与世界上下文，不维护第二套运行时循环。

## 3. 已确认的产品与执行契约

### 3.1 日计划规模

1. 每名 NPC 每日计划默认目标为 8 条，常规允许范围为 6-10 条。
2. 条目数是生成目标，不以截断方式强制达成。合法候选不足、剧情或强制任务导致偏离时，保留可执行计划并记录稳定原因。
3. 计划项只保存计划开始时间，不预估或保存固定任务持续时间。排序后每项的“日程槽位”由本项 `planned_start_time` 到下一项 `planned_start_time` 动态派生，最后一项到午夜睡眠清理点；槽位只用于计划冲突、社交意图机会窗口和下一次切换评估，不代表行为必须持续到槽位结束，也不作为行为完成条件，无需额外持久化结束时间。
4. planner 只接受当前游戏日内的 `HH:MM` 开始时间，不把 Unity 表现层的 `expected_duration_sec` 用于推导跨午夜。午夜由现有睡眠 / 跨日流程强制取消当日任务、清空当日剩余计划并回去睡觉；次日重新生成计划，因此日程模块不另外预测行为是否持续跨日。

### 3.2 时间与营业语义

1. 默认营业参考窗口为 `10:00-17:00`。
2. 该窗口是职业任务、顾客访问、傍晚收尾和社交安排的高权重参考，不是绝对关闭边界；允许提前准备、弹性营业和延迟收尾。
3. 世界快照明确给出的地点关闭、spot 缺失或不可用仍是硬约束，不能被营业时间的弹性语义覆盖。
4. 每个计划开始时间默认具有 `-30` 至 `+30` 分钟的执行弹性。窗口内可启动或延迟切换，窗口外按必须度、任务条件和 miss policy 决定跳过、压缩或重规划。
5. 启动弹性窗口与日程槽位是不同概念：前者决定该项何时仍可被提出执行，后者由相邻开始时间派生并用于计划级冲突判断。二者都不推导 Unity 行为持续时间。
6. 等待正式计划项期间允许 Unity 播放可抢占的空闲表现；空闲表现不进入正式日程、不回写后端行为状态或长期记忆。
7. 社交直接作为现有 `need` 候选的一种基础需求语义，不新增独立日程类别、独立社交计划系统或额外 LLM 调用。人物关系、近期互动和记忆证据用于计算社交 need 的相关度；拜访只表达单名 NPC 前往某地点的意图，不承诺目标人物届时在场，也不要求双方日程形成原子配对。
8. 约 `16:30-18:30` 作为傍晚社交软窗口，尽量让 NPC 在 `17:00` 左右结束或暂停职业任务，为相互拜访、接待、散步、聚餐或偶遇留出时间。该目标是高权重软偏好，不是每名 NPC 每天必须完成的任务；不得覆盖 `required`、吃饭与休息等其他基础需求、剧情任务、Unity 当前不可切换状态或世界物理约束。
9. 傍晚安排需要避免所有承担营业职责的 NPC 同时离店，但不建立 visitor / host 双边预约。拜访候选进入计划前只校验发起 NPC 的日程槽位、目标地点当时开放且可用、路程可达，以及该 NPC 自己的计划不存在硬冲突；人物是否实际在场由到达后的世界状态决定。
10. 没有合适拜访意图、合法地点或可行窗口时，允许社交 need 自然回退为散步、吃饭、休息或个人兴趣活动；到达后目标人物不在场也不视为计划校验失败，可按普通个人行为或空闲表现收口，不创造未知 action 或强行修改世界状态。
11. 物理事实以 Unity 为运行时权威：游戏时间、天气、NPC 实际位置、移动阶段、社交锁和 Unity 已知的 location / spot 可用状态通过世界快照、任务开始和任务终态同步给后端。共享 action-location-role 配置继续提供静态 affordance；快照未提供的动态开放、spot 占用或可达性必须标为 `unknown`，不得由 planner 猜成可用事实。

### 3.3 Unity 切换裁决

1. 后端生成成功后通过 `NPC_DAILY_SCHEDULE_READY` 一次提交单 NPC 的完整已验证计划，携带稳定 `operation_id / npc_id / game_day / schedule_revision / planner_version`；Unity 按 game day 和 revision 幂等接收并整体替换，不接受同 revision 的不同内容。
2. 默认允许切换；当前至少将 NPC-NPC 对话视为不可切换的特殊锁定状态。
3. Unity 的 `NpcDailyScheduleController` 持有剩余计划和当前 pending candidate，在游戏时间推进、当前任务终态或锁解除时调用统一切换策略；策略结果为 `switch / defer / skip_next / request_replan`，并记录稳定原因，不需要逐项等待后端 proposal / decision 往返。
4. 当裁决为 `switch` 时，Unity 原子地把可抢占旧任务回报为 `cancelled + schedule_preempted`，再通过现有 `NpcTaskExecutor` 启动计划项；若旧任务已自然终态则直接启动。不得先启动新任务再补旧终态。
5. 当裁决为 `defer` 时，Unity 不改变当前任务终态，在本地保留 candidate 为 pending，并在后续游戏分钟或当前任务终态后重新评估，直到 `+30` 分钟窗口结束。
6. 当裁决为 `skip_next` 时，Unity 以稳定原因消费该 candidate 并继续评估后续计划；当裁决为 `request_replan` 时，Unity 保留当前不可切换任务并向后端发送 `NPC_SCHEDULE_REPLAN_REQUEST`。窗口结束仍未执行时，由 Unity 按 `miss_policy` 统一决定 skip、本地压缩或 request_replan。
7. P0 生存、显式 need、状态触发和其他非日程运行时行为由 Unity 基于权威状态检测并通过同一切换策略与 `NpcTaskExecutor` 执行；需要 LLM / 记忆语义选择时，Unity 先向 Python brain seam 提交结构化请求，Python 只返回已校验意图，不直接按分钟推送 `NPC_BEHAVIOR` 覆盖活动任务。
8. Unity 本地决定 `switch` 后直接通过权威 `NpcTaskExecutor` 启动任务，不等待 Python 批准。Unity 以非阻塞 `NPC_RUNTIME_EVENT` 同步 started / phase / terminal 世界事实，携带 `event_id / world_revision / request_id / candidate_id / schedule_revision / npc_id / action_id / origin_location_id / target_location_id / actual_location_id / target_person_id / movement_status / result`；Python 只幂等更新推理缓存并按重要终态写记忆，不查询、超时、重发或裁决 Unity 任务。
9. 当前任务真实完成、失败、取消、被日程抢占和因特殊状态延迟保持不同结果语义，禁止把“时间到了”当作成功。
10. 后续玩家对话、剧情演出或不可中断动画是否加入锁定集合，不在本轮逐项设计；本轮必须预留注册或策略 seam，避免继续向执行器堆叠分支。

### 3.4 供应商超时与 fallback

1. 单名 NPC 的单次日计划或互动重规划供应商调用硬边界为 120 秒，包含实际等待供应商响应的时间。
2. 超时后立即进入本地 fallback，不无限阻塞同一批次；其他 NPC 的成功结果继续保留。
3. 5 名 NPC 继续有界并发，批次墙钟接近最慢的有界 owner，而不是串行总和。
4. 每次调用必须绑定 operation / owner token；超时后将该 token 终态化并丢弃迟到结果，后台线程或供应商迟到响应不得覆盖已经提交的 fallback。优先在供应商 adapter 设置真实网络超时；若底层调用无法取消，也必须隔离其迟到副作用并在诊断中标记 `provider_call_not_cancelled`。
5. fallback 以世界模型中的 profile routine、职业规则、基础需求、当前世界快照和合法的单方拜访意图构成基础日程，不复制未经验证的旧计划字符串。
6. 必须度是权威约束，分为：
   - `required`：必须优先安置，加权随机无权淘汰；只有世界物理不合法或硬冲突才能拒绝，并记录原因。
   - `important`：在不与 required 冲突时优先保留。
   - `optional`：在类别覆盖满足后按相关度进行加权选择。
7. 相关度综合记忆证据、人物关系、天气、当前位置与路程、营业参考窗口、近期重复惩罚和即时状态。相关度只能影响同一必须度层级内的排序与抽样，不能覆盖更高必须度。
8. fallback 使用由 `game_day + npc_id + fallback_attempt` 派生的稳定种子；诊断记录种子、权重分解、候选淘汰原因和最终选择，保证测试与问题复现。

### 3.5 下班时本地压缩

1. 任务延迟造成后续日程连锁挤压时，在 `17:00` 参考下班时间附近触发一次本地压缩评估，不增加 LLM 调用。
2. 压缩先保留仍合法的 `required`，再保留时间可行的 `important`，最后从 `optional` 中删除重复、低相关度或已经失去条件的任务。
3. 压缩只能调整尚未开始的计划；已完成任务和 Unity 当前不可切换任务不重写。
4. 压缩后的计划重新执行时间排序、`+/-30` 分钟窗口、候选归属、世界合法性、单方拜访意图和当日时间校验。
5. 相邻窗口重叠时按原计划顺序、必须度和当前 Unity 裁决选择，不允许在两个任务之间反复抢占。

### 3.6 Unity 运行时权威与 Python brain 请求

1. Unity `GameTimeController` 是唯一推进游戏日、小时、分钟、天气、暂停和跳时的模块。Python 删除现实秒累加、`TimeSpeed`、`start / stop / _tick_loop / _advance_one_minute`、pause reason、`on_midnight / on_wake / on_tick` 等自主时钟能力，也不接收每分钟业务 tick。
2. P0 生存、need / 状态触发、NPC-NPC 物理候选扫描、同 zone / 玩家可见范围判断、社交冷却、占用、锁和会合超时全部由 Unity 基于权威物理状态执行。后端删除 `_tick_npc()`、`_check_social_opportunities()` 和相关分钟累加 / 冷却运行态。
3. Unity 发现满足物理条件的 NPC-NPC 社交候选后发送 `NPC_SOCIAL_DECISION_REQUEST`，携带稳定 request ID、双方 ID、冻结 game time、实际位置 / zone、当前任务与锁状态。Python 使用人物关系、记忆和 LLM 返回 `want_to_talk / reason / opening_intent`；Unity 收到后重新校验双方仍在场且可切换，再由 Unity 自己完成 reservation、会合、READY、播放和 COMPLETE 状态机。
4. Python 不维护全局可变时间镜像作为业务依赖。日计划、对话、记忆事件、社交决策、互动重规划和午夜维护的请求 DTO 都显式携带不可变 `GameTimeSnapshot(day / hour / minute / weather / time_revision)`，同一 operation 始终使用开始时的冻结快照。
5. 开局、读档和重连世界快照只更新 Python 推理缓存与诊断，不触发历史分钟补跑、行为 tick 或社交扫描。miss policy、日程压缩、跨日清理和 NPC 行为恢复全部由 Unity 处理。
6. 午夜只由 Unity `MIDNIGHT_SETTLEMENT_REQUEST` 触发。Python 使用请求携带的冻结 game time 执行记忆维护和次日 planner，不再自行检测午夜、发送 `SLEEP_TRIGGER` 或等待本地 wake。
7. 后端旧 `GAME_TIME_SYNC / FAST_FORWARD / GAME_PAUSE_STATE` 时钟控制入口移除；快进、17:00 调试、暂停和午夜调试只修改 Unity `GameTimeController`。如诊断需要观察 Python 最近一次请求时间，只记录 operation 输入，不形成会推进或触发业务的镜像时钟。
8. `NPC_SOCIAL_DECISION_RESULT` 必须回传 request / social candidate ID 和输入 world revision。Unity 在等待 brain 时不锁定或停止 NPC；结果返回后候选已过期、revision 不符或任一方状态变化时直接丢弃。供应商不可用 / 超时默认 `want_to_talk = false`，不为促成社交强行冻结双方；冷却只在 Unity 实际接受并开始会合后记账。

### 3.7 NPC 运行时状态权威

1. `energy / sociability / current_need / is_asleep / current_action / current_location / movement_status` 等高频运行时字段由 Unity 更新并进入 Unity 主存档。Python SQLite 中同名字段只作为推理缓存，重连和读档后由 Unity 快照覆盖。
2. 当前 `StateManager.apply_tick()` 的分钟衰减、need 生成和清理迁入 Unity 独立运行时状态模块；暂停、跳时和跨日使用 Unity 时间语义，不在 Python 补跑分钟。
3. Python 仍可根据记忆、关系和互动摘要计算 `NpcStateEffect`，但只返回稳定字段、delta、上下限依据、source operation ID 和原因；Unity 校验后应用。Python 不直接把即时 emotion / energy / sociability / need 写成世界权威。
4. NPC-NPC / 玩家互动、午夜印象刷新和日计划 brain 结果如需改变运行时字段，必须通过结构化 effect 协议返回 Unity；记忆节点、关系图、向量和检索 trace 仍由 Python 权威保存。
5. Unity 每次正式状态变化按 revision 同步最小世界事实给 Python 推理缓存；Python 的迟到 effect 只有 base world revision 仍匹配时可应用，否则返回 `stale_world_revision` 并丢弃。

## 4. 模块方案

### 4.1 后端深模块

新增 `DailySchedulePlanner`，公开接口只保留：

```python
class DailySchedulePlanner:
    async def prepare_day(
        self,
        request: DailyScheduleBatchRequest,
    ) -> DailyScheduleBatchResult:
        ...

    async def replan_after_interaction(
        self,
        request: InteractionReplanRequest,
    ) -> NpcScheduleResult:
        ...
```

建议按职责拆分：

- `backend/src/npc/daily_schedule.py`：公开 DTO、planner 编排、最终结果契约。
- `backend/src/npc/schedule_candidates.py`：候选构建、物理过滤、类别覆盖、必须度与相关度和 fallback；不保存或推进 Unity 权威的运行时剩余计划。
- `backend/src/npc/schedule_memory_evidence.py`：按 NPC / group 批量检索图与向量证据，并复用到候选。
- `backend/src/npc/schedule_prompt_adapter.py`：DTO 到紧凑标签的单向渲染、受控 JSON 输出解析。
- `backend/src/npc/schedule_diagnostics.py`：批次、owner、候选、证据、超时、fallback 和重规划快照；Unity 本地 pending、切换与压缩由 Unity 诊断 provider 输出并在 `aisc_debug` 聚合。

如果执行时发现上述拆分产生空壳文件，可在保持两个公开 seam 和职责边界的前提下合并相邻实现，但不得把职责重新塞回 `BehaviorEngine`。

### 4.2 BehaviorEngine 边界

`BehaviorEngine` 保留：

1. `ensure_daily_plans(game_day, ...)` 幂等入口。
2. 调用 planner，并把已验证计划整体发布给 Unity；仅保留生成批次幂等状态和诊断关联，不保存权威剩余计划，也不再按分钟匹配 P4 日程项。
3. 接收社交或玩家互动终态并构造稳定的重规划上下文；需要重规划时先请求 Unity 回传该 NPC 的权威剩余计划。
4. 作为迁移期兼容 facade 转发日计划、社交语义决策和重规划请求；若这些接口已被 `DailySchedulePlanner` 和独立 brain handler 完整承接，执行中应删除或进一步减重该 facade，不保留运行时 tick。

从 `BehaviorEngine` 移出：

1. 目录字符串拼接和 `[:420] / [:1800] / [:800] / [:600]` 日程语义截断。
2. 日计划与社交后重规划 Prompt 构造和 JSON 收口。
3. 候选筛选、生成期时间冲突修复、fallback 和后端生成诊断细节；下班运行时压缩由 Unity 日程 controller 负责。
4. `_match_plan_step()`、发送前从 `_plans` 删除步骤和后端 P4 分钟匹配；这些运行时职责迁入 Unity `NpcDailyScheduleController`。
5. P0-P6 分钟优先级循环、随机 / need 触发、NPC 物理访问追踪、社交候选扫描、同 zone / 玩家附近判断和社交冷却；这些运行时职责迁入 Unity，Python 只保留收到明确请求后的语义决策。

### 4.3 三层决策

1. 第一层确定性过滤 action、location、role、天气硬限制、地点真实开放状态、spot 可用性、路程可达性和开始时间属于当前游戏日。
2. 第二层按 NPC / group 批量获取长期记忆证据。缺少证据不得删除 `occupation / routine / need`；关系、探索和个人兴趣可以升降权或淘汰。
3. 在 LLM 调用前从现有 `need` 候选中生成单 NPC 的拜访、散步、聚餐等社交意图。候选可引用目标人物用于相关度与理由，但只承诺发起 NPC 的地点和开始窗口合法，不要求 visitor / host / peer 成对进入日程；营业职责错峰通过批次级软配额避免所有店主同时离店。
4. 第三层每名 NPC 只调用一次 `npc_daily_plan`，同时完成语义筛选和日程组合，不新增第二次筛选调用。
5. LLM 只返回本次输入中的 `candidate_id` 和可选 `evidence_ids`；本地恢复稳定 action、location、target 和来源，未知 ID 一律拒绝。
6. 日初候选过滤使用最近一次 Unity 权威物理快照。静态配置明确非法时硬拒绝；动态状态为 `closed / unavailable / unreachable` 时硬拒绝；动态状态缺失或 `unknown` 时只降权并记录原因，最终由 Unity 在执行前按实时物理状态重新裁决，失败时 `skip_next` 或 `request_replan`。

### 4.4 互动后重规划

1. NPC-NPC 社交正常完成和具有有效内容的玩家正常对话共用 `replan_after_interaction()`。
2. 后端互动终态先生成稳定 `interaction_id` 和摘要，通过 `NPC_SCHEDULE_REPLAN_CONTEXT` 发送参与者 ID、互动类型、结束原因、互动摘要、当前时间和地点；Unity 收到后以 `NPC_SCHEDULE_REPLAN_REQUEST` 回传该 NPC 当前任务、权威剩余计划、当前 schedule revision 和 Unity 世界快照。后端不得使用生成时旧副本猜测剩余计划。
3. 已完成计划不重写；Unity 当前不可切换任务保持；新计划开始时间不得回到过去，且只属于当前游戏日。
4. 新旧步骤按 candidate、目标人物和时间窗口去重；本轮互动重规划只产生单 NPC 后续计划，不创建需要双方同步提交的双人任务。
5. 互动重规划同样受 120 秒单调用上限保护，超时使用相同候选、必须度和相关度 fallback；结果通过新的 `NPC_DAILY_SCHEDULE_READY` revision 整体替换 Unity 中尚未终态消费的剩余计划，Unity 保留当前不可切换任务。

### 4.5 Unity 执行 seam

在现有 `NpcTaskExecutor` 附近新增独立 `NpcDailyScheduleController` 和切换策略，但不把后端候选、记忆检索或 LLM 规则移入 Unity：

1. `NpcDailyScheduleController` 托管计划接收、revision 幂等、槽位派生、游戏时间推进、pending、miss policy、本地压缩、终态消费、重规划请求和存档导入导出；时间推进直接订阅 Unity 权威 `GameTimeController.TimeChanged` 分钟事件，不依赖后端低频 `GAME_TIME_UPDATE`。
2. 当前正式任务通过只读 seam 暴露执行阶段、是否完成和是否允许日程切换；NPC-NPC 对话锁接入 `NpcScheduleSwitchPolicy`。
3. 日程 controller、Unity 运行时行为 controller 和 brain 意图结果 adapter 共同调用切换策略；只有 `switch` 才调用 `NpcTaskExecutor.Execute()`，不得形成两个互不知情的抢占入口。旧 `NPC_BEHAVIOR` 只作隔离兼容并退出正式新流程。
4. Unity 本地下班压缩只操作后端已验证计划的尚未开始项，使用持久化的 necessity、相关度顺序、分组和 miss policy，不在 Unity 重做候选生成、记忆检索或 LLM 选择；遇到世界条件需要重新选候选时请求后端 replan。
5. Unity 接受 `NPC_SOCIAL_DECISION_RESULT` 后，通过同一切换策略为双方原子获取“社交会合 reservation”并开始会合；任一方不可切换、已有 reservation 或真实物理状态不允许时保持原任务并本地记录稳定失败原因。reservation 只阻止新的日程 / 正式任务抢占，不得复用会阻止 `MoveToLocation()` 的 motion lock；双方到达后 Unity 发送 `NPC_SOCIAL_CONTENT_REQUEST`，Python 只返回对白内容，Unity 播放时才使用现有对话 motion lock。COMPLETE / FAILED / CANCEL 必须同时释放 reservation 和 motion lock，再让双方日程重新评估。
6. Unity NPC-NPC 社交候选扫描只使用非 transit 实际位置、睡眠状态、冷却和玩家可见范围；拜访计划的 `target_person_id` 只提高目标人物在场时提交 brain 请求的倾向，不绕过双方在场、同 zone、未锁定和 Unity 会合成功条件。Python 返回社交意愿后，Unity 必须重新校验物理条件；目标人物不在场时不触发失败或强制召回。
7. 空闲表现仅在没有正式活动或等待下一计划时运行，并继续保持可抢占。
8. 不修改场景、Prefab、SerializeField 或 UI。若实际接线需要资产变更，执行会话必须先阅读 `UnityMCPUsageRules.md` 并通过 Unity MCP 完成。

### 4.6 Python brain 请求 seam

Python 业务入口统一接收显式 `BrainOperationContext`，不再注入可变 `GameClock`：

```python
@dataclass(frozen=True)
class BrainOperationContext:
    operation_id: str
    game_time: GameTimeSnapshot
    world_revision: int

class NpcBrain:
    async def prepare_day(self, request: DailyScheduleBatchRequest) -> DailyScheduleBatchResult: ...
    async def decide_social(self, request: SocialDecisionRequest) -> SocialDecisionResult: ...
    async def replan_after_interaction(self, request: InteractionReplanRequest) -> NpcScheduleResult: ...
```

1. `GameTimeSnapshot` 是不可变 operation 输入；Dialogue、Memory、WorldPreparation、planner 和诊断从当前请求获取时间，不读取全局可变 clock。
2. `decide_social()` 只回答关系 / 记忆 / 性格层面的语义意愿，不扫描场景、不维护冷却、不移动 NPC、不获取 Unity 锁。
3. 午夜维护和次日准备显式接收 Unity 请求中的冻结时间 DTO，不从全局状态推断目标日。
4. 所有旧 `game_clock.day / hour / minute / weather` 直接读写、callback 注册、pause 和 speed 入口必须迁移或删除；禁止为了少改保留可写属性兼容层。
5. Python 可保留最近 operation 输入的只读诊断副本，但该副本不被业务读取、不触发行为，也不参与时间推进。

## 5. 数据与协议契约

计划 DTO 至少需要：

- `candidate_id / action_id / location_id / target_person_id`
- `planned_start_time`
- 默认 `execution_window_before_minutes = 30`
- 默认 `execution_window_after_minutes = 30`
- `necessity`：`required / important / optional`
- `primary_group / groups`
- `evidence_ids / retrieval_trace_ids`
- `source`：LLM、fallback、互动重规划或下班压缩
- `miss_policy`

当日日计划进入 Unity 权威正式存档，但只保存“当前游戏日仍未终态消费的剩余计划”，不保存已完成 / 已跳过历史、LLM 原文、诊断评分明细、Unity 活动任务、in-flight request 或 defer 重试记录。

存档契约固定为：

1. `GameSaveData.schema_version` 升级，并为每名 NPC 增加 `schedule_day`、`schedule_planner_version` 和 `remaining_daily_schedule`。
2. 每个剩余计划项必须持久化 `candidate_id / action_id / location_id / target_person_id / planned_start_time / execution_window_before_minutes / execution_window_after_minutes / necessity / primary_group / groups / evidence_ids / source / miss_policy`。
3. `retrieval_trace_ids`、相关度权重分解和随机种子只属于当次诊断，不持久化；读档后可通过 candidate 与 evidence ID 解释计划来源，但不假装旧 trace 仍可查询。
4. Unity 在计划项收到最终 `succeeded / failed / cancelled`、`skip_next` 或明确 miss policy 终态前都将其保留在剩余计划中；运行时以 `candidate_id + request_id` 标记 in-flight，避免每分钟重复执行。in-flight 标记不持久化，读档后将该项重新视为待评估。
5. 旧存档缺少日计划字段时使用显式兼容路径：空剩余计划、`schedule_day = 0`，加载当前游戏日后调用正式 `ensure_daily_plans()` 生成一次；不得依赖反序列化器的隐式字段默认。
6. 若存档中的 `schedule_day` 不等于当前游戏日，丢弃该剩余计划并按当前日重新生成。午夜睡眠流程同样清空旧日计划。
7. `NPC_DAILY_SCHEDULE_READY / NPC_RUNTIME_EVENT / NPC_SOCIAL_DECISION_REQUEST / NPC_SOCIAL_DECISION_RESULT / NPC_SOCIAL_CONTENT_REQUEST / NPC_SOCIAL_CONTENT_RESULT / NPC_SOCIAL_COMPLETE / NPC_SCHEDULE_REPLAN_CONTEXT / NPC_SCHEDULE_REPLAN_REQUEST` 的枚举与字段全部使用稳定 `snake_case` 值，并同步 Python codec、Unity DTO、协议 README、存档兼容和测试。
8. `schedule_revision` 在同一 NPC、同一 game day 内单调增加。Unity 是已接收 revision 的运行时权威；后端重规划必须基于 Unity 请求携带的 base revision，Unity 拒绝应用 base revision 已过期的返回结果，并以稳定 `stale_schedule_revision` 请求重新规划或结束该 operation。
9. Unity 世界快照增加版本化 `npc_schedule_physical_state`，至少携带快照 game time、weather、NPC 实际 location、movement status、当前社交锁，以及 Unity 已有数据源能确认的 location / spot `open / closed / available / unavailable / unknown` 状态。不得为了填满字段扫描或猜测不存在的场景事实；执行时实时裁决仍高于日初快照。
10. 所有 Python brain 请求统一携带完整 `game_time / time_revision / world_revision`；`NPC_SOCIAL_DECISION_REQUEST` 额外携带双方权威物理快照和 Unity social candidate ID。后端不接收每分钟 `GAME_TIME_TICK`，旧 `GAME_TIME_SYNC / FAST_FORWARD / GAME_PAUSE_STATE` 时钟控制协议在正式调用迁移后删除。
11. Python 返回的 `NpcStateEffect` 至少包含 `operation_id / npc_id / base_world_revision / effect_type / field_deltas / field_values / clamp_reasons / source / reason`；Unity 应用后用新的 world revision ACK。effect 不得包含位置、移动终态或任务成功等只能由 Unity 产生的事实。

12. 将 NPC-NPC 社交与玩家正常对话接入统一的单 NPC 互动重规划，移除旧的专用重规划分支。
13. 在 Unity 增加 NPC 运行时行为 / 社交候选 controller，承接 P0、need / 状态触发、物理候选扫描、冷却和 `NPC_SOCIAL_DECISION_REQUEST`；Python 只返回语义意愿，Unity 返回后重新校验物理条件。
14. 实现 `NPC_DAILY_SCHEDULE_READY`、Unity `NpcDailyScheduleController`、本地 pending / miss / 压缩状态机，以及默认可切换、NPC-NPC 对话锁定和可扩展切换策略；保持空闲表现与正式任务边界。
15. 把任务节点询问、停滞、超时和重发监督迁入 Unity；实现非阻塞 `NPC_RUNTIME_EVENT` 同步，使 Python 只更新推理缓存和写重要终态记忆，不参与任务启动或恢复裁决。

18. 将当前日未终态消费的剩余计划和版本化物理状态接入 Unity 权威存档 / 世界快照，补 schema 升级、旧存档兼容、读档重新评估和午夜清理。

20. 运行后端定向测试与完整测试、Unity EditMode 测试、隔离 Play Mode 跨日验证；最后回写相关 README、Workstream 和 execution。

## 9. 测试与验收

### 9.1 后端单元与集成测试

1. 字符截断全部移除，候选增长不产生字母排序偏置。
2. `required` 永远优先于高相关度 `optional`；物理非法 required 被拒绝时具有稳定原因。
3. 相同 game day、NPC 和 fallback attempt 生成相同 fallback；不同种子仍满足相同硬约束。
4. 正常计划目标为 8 条并通常位于 6-10 条；不足或超出时有原因而非截断。
5. `10:00-17:00` 影响权重和默认安排，但允许准备、弹性营业和收尾；世界真实关闭仍剔除候选。
6. 每项只有开始时间，并正确派生 `+/-30` 分钟窗口；乱序、重复、未知 candidate、非法 `HH:MM` 和非当前日条目全部拒绝，不根据 Unity 表现持续时间推导跨日。
7. 120 秒超时进入 fallback，不阻塞其他 owner，诊断指出最慢 owner 和超时阶段。测试使用可控短超时注入，不真实等待 120 秒。
8. 后端输出的 necessity、相关度顺序、分组和 miss policy 足以让 Unity 确定性压缩，不要求 Unity 重新检索候选或调用 LLM。
9. `16:30-18:30` 候选尽量让 NPC 在 `17:00` 左右结束或暂停工作，并为拜访、接待、散步、聚餐或偶遇提供高权重软机会；不把该倾向升级为 required。
10. 拜访作为单方意图成立：发起 NPC 的窗口、地点开放与可用性、路程均合法；不要求目标人物同时拥有接待计划。批次软配额应尽量避免所有承担营业职责的 NPC 同时离店，但不足时不得把软目标误判为硬校验失败。
11. NPC 非工作时段存在合法关系、可行窗口和可用地点时，现有 `need` 候选与最终计划对拜访等社交意图表现出可诊断的高权重倾向；目标人物最终不在场、条件不足或更高必须度任务占用时，可回退为散步、吃饭、休息、个人兴趣或普通到访行为。
12. 玩家正常对话和 NPC-NPC 社交进入统一重规划；异常结束和空对话不触发。
13. 每名 NPC 仍只有一次日计划 LLM 调用，记忆证据按 NPC / group 批量检索。
14. Unity 日程任务不等待 Python ACK；started 立即形成 Unity transit 权威，终态成功才提交实际 location。`NPC_RUNTIME_EVENT` 断线可按 event ID 补送，但不阻塞或回滚本地任务，Python 对重复 / 乱序 world revision 幂等收口。
15. Python 不再扫描或持有 NPC 社交物理候选 / 冷却；社交 brain handler 对同一 request 幂等，只基于请求快照、关系和记忆返回语义意愿。
16. Python 不启动任何时间或行为推进 task，不存在 `GAME_TIME_TICK / tick_non_schedule`；大跨度跳时不在后端补跑历史分钟。
17. 午夜维护只响应 Unity 请求并使用请求冻结时间；Dialogue / Memory / planner / social decision 长 operation 的时间戳在 operation 内稳定。
18. Python 即时状态计算只返回 `NpcStateEffect`，不直接提交 Unity 世界字段；base revision 过期时 effect 稳定拒绝，记忆写入仍按 Python 权威完成。

### 9.2 Unity EditMode / Play Mode

1. 普通移动或动作状态默认允许日程切换。
2. `GameTimeController.TimeChanged` 每个游戏分钟驱动一次日程评估；后端低频 `GAME_TIME_UPDATE` 缺失或延迟不会造成日程漏触发，暂停和跳时继续服从 Unity 时钟现有语义。
3. NPC-NPC 对话期间返回 `defer`，对话结束后在弹性窗口内重新评估。
4. 延迟切换不把当前任务伪报为成功，也不丢失待执行任务。
5. `switch / defer / skip_next / request_replan` 与稳定原因可通过 `aisc_debug` 读取。
6. 等待下一项时空闲表现可以播放，并能被正式任务抢占。
7. 同一 `NPC_DAILY_SCHEDULE_READY` revision 重放不会重复替换或执行；不同内容复用 revision 被拒绝。`switch` 先回报旧任务 `cancelled + schedule_preempted` 再启动新任务，`defer` 由 Unity 在窗口内本地重评估。
8. Unity 运行时行为、日程到点和 brain 意图结果使用同一切换策略，不会出现 Python 命令绕过 controller 直接覆盖活动任务；计划项根据真实终态继续 pending、被抢占或消费，旧 `NPC_BEHAVIOR` 不进入正式新流程。
9. 互动后 replan request 携带 Unity 权威剩余计划和 base revision；迟到结果不能覆盖更高 revision，冲突返回稳定 `stale_schedule_revision`。
10. 保存只包含当前日未终态消费的剩余计划；读档不恢复旧 in-flight，旧 schema 显式生成当前日计划，午夜清空旧日计划并由现有流程强制睡眠。
11. 连锁延迟在下班附近由 Unity 触发本地压缩，结果只包含当前日合法开始时间；需要新候选时才请求后端 replan。
12. Unity 只有在接收有效 social decision 且双方原子取得不会阻止移动的逻辑 reservation 后才开始会合；任一方不可切换时不取消其原任务；READY 后才向 Python 请求对白，会合 / 内容失败时本地释放 reservation，播放阶段 motion lock 与 reservation 分阶段生效和释放。
13. 日初物理快照中的明确 closed / unavailable 会过滤候选，unknown 只降权；执行时 Unity 实时不可达或 spot 不可用会稳定 skip / request_replan，不伪造任务成功。
14. 隔离 Play Mode 完成存读档、跨日生成、供应商超时 fallback、物理状态同步、NPC-NPC 对话锁和次日日程重新开始验证。
15. Unity 暂停、恢复、跳到 17:00、午夜和读档后，Python 不接收分钟 tick、不补跑行为且不能改写 Unity 时间；不存在本地分钟漂移或第二次午夜触发。
16. Unity 社交扫描排除 transit、睡眠、已有 reservation / 对话锁和超出玩家可见范围的 NPC；冷却和每日上限在存读档 / 跨日后符合明确持久化口径。
17. Python 返回 `want_to_talk` 后任一 NPC 已移动、被锁定或进入其他任务时，Unity 拒绝过期决定且不启动会合；拜访目标不在场不制造失败或修改对方日程。

### 9.3 完成标准

功能本体、诊断钩子、控制钩子、测试和入口文档全部同步后才可宣称完成。完成时 Python 不得存在自主时间 / 行为推进 task、物理社交扫描、任务监督或世界状态权威写入；Unity 必须独立托管时间、NPC 运行时状态、日程、任务、社交候选、会合与存档。若某项钩子不适用，execution 必须记录具体原因、替代验证和剩余风险。

## 10. 风险与停止条件

1. 当前营业状态和动态 spot 可能没有统一权威数据源；实施时只能使用明确世界快照和默认参考窗口，不得让 planner 猜测场景事实。
2. 必须度标注错误会比相关度错误影响更大。required 来源必须限定为职业规则、基础需求、固定约定或明确系统/剧情规则，并在诊断中显示来源。
3. `+/-30` 分钟窗口可能重叠；必须通过顺序、必须度和 Unity 当前裁决一次性选择，防止反复抢占。
4. 120 秒是硬上限，不代表目标耗时。正常批次仍应保持现有约 7-12 秒级别，并诊断长尾。
5. 若 `BehaviorEngine`、`NpcTaskExecutor` 或社交控制器在实施中将继续承载多个新状态和多分支，触发项目类减重硬约束时必须停止并向用户请求减重指令，不得先堆入现有类。
6. 若需要新增场景 spot、Prefab、SerializeField 连线或 UI，立即停止资产修改，转入 Unity MCP 资产优先流程。
7. 午夜边界必须复用现有睡眠 / 跨日取消流程；若该流程没有清理剩余计划或仍允许旧日程 task / in-flight 在次日恢复，先补统一清理和回归测试，不得另建持续时间推导逻辑。
8. 移除 Python 自主时钟和行为循环会影响 Behavior、Dialogue、Memory、WorldPreparation、社交、旧协议兼容和诊断；在所有直接可写 `game_clock` 访问、callback 注册、pause / speed / fast-forward、P0-P6 tick 和后端物理社交扫描入口清零前不得删除旧文件或宣称迁移完成。
9. Unity 新运行时行为 / 社交候选 controller 若同时承载日程、需求、冷却、社交会合和协议细节会立即过重；必须保持候选扫描、日程托管、切换策略和会合协议的独立 seam，触发类减重规则时先请求减重指令。
