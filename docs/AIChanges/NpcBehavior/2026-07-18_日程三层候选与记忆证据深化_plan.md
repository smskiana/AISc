# 日程三层候选与记忆证据深化执行方案

> 执行记录: [2026-07-18_日程三层候选与记忆证据深化_execution.md](2026-07-18_日程三层候选与记忆证据深化_execution.md)

## 1. 目标与边界

完成原方案的日程深模块：确定性物理过滤、图/向量记忆证据增强、单 owner 单次 LLM 组合、完整最终校验和确定性 fallback。本阶段只输出已验证计划，不修改 Unity 执行状态机。

## 2. 实施步骤

1. 扩展 `ScheduleCandidate` DTO，覆盖 action/location/role、合法时间窗、天气、营业状态、spot 状态、可达性、位置成本、目标人物、来源、必要度、相关度分解和证据引用。
2. 候选构建按 `occupation / routine / need / relationship / weather / nearby / exploration` 保证席位；职业、routine、吃饭和休息不能因无记忆证据被删除。
3. 新增 `schedule_memory_evidence.py`，按 NPC/group 批量调用现有图与向量检索 seam；保存 evidence ID、相似度、图路径分和 retrieval trace ID，不保存 embedding。
4. 增加傍晚 `16:30-18:30` 软窗口和跨 NPC 单方拜访协调：目标地点开放可达、时间窗可行、营业职责错峰；不建立 visitor/host 原子预约。
5. 相关度综合关系、近期互动、天气、当前位置、路程、营业窗口、重复惩罚和即时状态；必要度优先级不得被相关度覆盖。
6. Prompt adapter 仅渲染结构化 DTO，输出仍为受控 JSON；每名 NPC 每日只调用一次 LLM。
7. 最终校验统一检查候选归属、6-10 目标、required 保留、类别覆盖、严格递增 HH:MM、重复/冲突、当日范围和物理合法性；任何硬失败整体 fallback。
8. fallback 按稳定种子与 required → important → optional 生成，记录每个保留和淘汰原因；供应商调用使用真实 adapter 超时或 owner token 隔离迟到结果。

## 3. 预计涉及文件

- `backend/src/npc/schedule_candidates.py`
- `backend/src/npc/schedule_memory_evidence.py`
- `backend/src/npc/schedule_prompt_adapter.py`
- `backend/src/npc/schedule_diagnostics.py`
- `backend/src/npc/daily_schedule.py`
- `backend/src/npc/behavior_engine.py`
- 现有记忆 retrieval facade 与相关测试

若跨 NPC 协调需要通用多人资源占用求解，停止实施并先评估 ADR；不得把协调逻辑堆回 `BehaviorEngine`。

## 4. 诊断与验证

1. owner trace 记录候选分组、物理过滤、必要度/相关度分解、证据、社交机会、最终校验、fallback seed、耗时和迟到隔离状态。
2. 测试覆盖晴雨、关闭地点、unknown spot、不可达地点、职业身份、基础 need、长期记忆有/无、傍晚错峰和无合适社交对象。
3. 验证 5 owner 有界并发，单 owner 120 秒超时不阻塞其他 owner；迟到供应商结果不能覆盖 fallback。
4. 验证个性化/社交条目均可追溯到合法 candidate 与真实 evidence ID。

## 5. 验收标准

- 三层决策真实进入生产链，而非空字段或 Prompt 文案占位。
- LLM 无法创造未知 ID、漏掉 required 或绕过物理硬约束。
- fallback 可复现并提供完整淘汰依据。
- `BehaviorEngine` 仅保留 facade 和批次编排，不重新承载候选细节。
