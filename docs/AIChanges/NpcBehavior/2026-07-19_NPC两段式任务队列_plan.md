# NPC 两段式任务队列分阶段执行方案

> 设计来源：[TwoSegmentNpcTaskQueueDraft.md](../../Workstreams/NpcBehavior/TwoSegmentNpcTaskQueueDraft.md)
>
> 计划中的执行记录：[2026-07-19_NPC两段式任务队列_execution.md](2026-07-19_NPC两段式任务队列_execution.md)
>
> 计划中的独立测试记录：[2026-07-19_NPC两段式任务队列_test.md](2026-07-19_NPC两段式任务队列_test.md)

## 1. 需求理解与实施结论

本案把 NPC 日程从“普通任务精确时间点 + 统一执行窗口 + 错过后整表 replan”迁移为“08:00-17:00 工作队列 + 17:00-午夜休息队列”。后端仍负责合法候选的语义选择和排序；Unity 继续权威裁决任务执行、阶段边界、抢占、失败、取消、强制终态和存档恢复。

本案采用一个正式 plan、多个有界执行阶段。每个阶段在独立执行会话中引用本 plan，只实现该阶段，并把实际变更追加到同主题 execution；阶段门禁通过后才能进入下一执行会话。全部实现结束后另开独立测试会话创建同主题 test。任何阶段不得另建第二份 plan；若领域边界或协议方向改变，应停止执行并回到方案会话更新本文件。

关键迁移策略：允许新旧 DTO 在迁移期双读，但生产运行时永远只有一个 owner。先让 Unity 能解析、验证和隔离演练新队列，再让后端生产新契约，最后通过单一切换点启用新 owner；不得让旧分钟日程和新队列同时驱动 `NpcTaskExecutor`。

## 2. 工程口径与范围

- Workstream：`NpcBehavior` 为主要功能域，同时影响 `Dialogue`、`ProtocolAndSave`、`TestingAndDiagnostics`。
- Roadmap item：NPC 行为语义收敛、夜间日常闭环、NPC 行为 affordance 第二阶段。
- ADR：遵守 `ADR-0003` 的 Unity 资产优先和 `ADR-0006` 的 Unity 世界/存档权威；本案不改变权威边界。
- 保留口径：结构化候选、物理合法性过滤、图/向量记忆证据、每 NPC 每日单次 LLM、统一最终硬校验、deterministic fallback、冻结世界快照、`NPC_RUNTIME_EVENT` 非阻塞终态同步。
- 退役口径：普通任务 `planned_start_time`、统一 `execution_window_*`、窗口错过整表 replan、NPC-NPC COMPLETE 后整表 replan、普通玩家聊天后整表 replan。
- 非目标：真实动画/交易/经营玩法、替代 spot DTO 扩展、后端自动重连、自由文本直接生成任务、场景 spot/anchor 细化。

## 3. 执行前必须确认的领域决策

以下采用草案推荐值；在阶段 1 执行前，用户必须逐项确认或修改：

1. 活动工作任务异常跨到午夜时，以稳定 `forced_terminal_at_day_boundary` 强制终态，再进入午夜结算。
2. 每次玩家对话最多产生一个 `NpcTaskMutation`。
3. `work_open / work_close` 改为工作 segment enter/exit 的确定性生命周期动作，不再进入 LLM 队列。
4. 默认 `duration` 使用 Gameplay 时间域；初始持续量必须来自 task catalog，不沿用无语义的固定 1 秒。
5. 玩家抢占后旧任务默认退役；只有 interrupt policy 明确声明时才允许恢复或重新入队。
6. 因 `NpcDailyScheduleRuntimeCoordinator` 已持有接收、分钟裁决、边界、消费、跨日和存读档职责，用户必须在阶段 1 前明确下达“允许减重”指令，授权把队列状态与裁决迁入 `NpcDayPlanRuntime` 深模块。

任一项未确认时，只能继续做只读可行性检查，不能进入实现。

## 4. 阶段 1：共享契约与纯逻辑运行时核心

### 4.1 目标

先固定跨端语言和 Unity 状态机 seam，不接管生产日程，不改后端 LLM 输出。

### 4.2 实施内容

1. 定义 `NpcDayPlan`、`NpcPlanSegmentDefinition`、`NpcPlannedTask`、`NpcTaskMutation`、稳定 segment/boundary/terminal/mutation reason。
2. 为 task catalog 增加 `segment_id`、`completion_policy_id`、`interrupt_policy` 和初始 duration 元数据；`both` 必须在候选投影时落到一个实际 segment。
3. 新建纯 C# `NpcDayPlanRuntime`，封装 `AcceptPlan / OnTaskTerminal / OnSegmentBoundary / ApplyMutation / CreateSnapshot`；不得依赖场景对象、网络或 `MonoBehaviour`。
4. 固定队列消费、17:00 当前工作续跑、剩余工作退役、午夜强制终态、跨日原子换代、迟到 mutation 和 revision 冲突规则。
5. 新 DTO 只进入隔离测试和兼容解析，不连接正式 `NpcTaskExecutor`。

### 4.3 预计文件

- `backend/src/npc/task_catalog.py` 及共享 catalog/config 投影入口
- `Assets/Scripts/NPC/NpcDailyScheduleModels.cs`
- 新增 `Assets/Scripts/NPC/NpcDayPlanRuntime.cs`
- `Assets/Tests/Editor/NpcDailyScheduleTests.cs` 或按既有测试边界新增对应 EditMode 文件
- 对应 Python/C# 契约测试

### 4.4 门禁与回滚

- Python catalog 聚焦测试通过；Unity EditMode 覆盖队列、边界、抢占、revision 和换代，不依赖场景。
- `NpcDailyScheduleRuntimeCoordinator` 的生产行为保持不变。
- 回滚只需移除未接线的新 DTO/runtime 与 catalog 新字段；旧日程仍是唯一 owner。

## 5. 阶段 2：后端两段式生成与协议 V2

### 5.1 目标

让 planner 在固定输入下稳定生成两个候选队列，但生产发送仍受单一 feature/capability 切换点控制。

### 5.2 实施内容

1. 候选构建按 catalog 元数据投影为 `work / rest`，保留物理过滤、记忆证据、required group 和来源 trace。
2. Prompt adapter 以每 NPC 单次 LLM 返回 `work_tasks[] / rest_tasks[]`，只接收 candidate ID；不接收时间、duration、完成条件、移动方式或失败策略。
3. parser、最终校验和 fallback 共用同一结构化结果契约；校验 segment 兼容、重复、required group、基础 need 和 `work_open/work_close` 禁入。
4. 新日计划协议携带 operation、day、revision、planner version、status/failure reason 和 segments；Unity 先具备 V2 reader，后端 V2 producer 再启用。
5. 保持同日幂等、5 NPC 有界并发和单 owner 超时；fingerprint 必须覆盖两个有序队列及计划身份。

### 5.3 预计文件

- `backend/src/npc/daily_schedule.py`
- `backend/src/npc/schedule_candidates.py`
- `backend/src/npc/schedule_prompt_adapter.py`
- `backend/src/npc/schedule_validation.py`
- `backend/src/npc/schedule_diagnostics.py`
- `backend/src/npc/behavior_engine.py` 或实际唯一日程 service facade
- `backend/src/application/runtime.py`、协议 DTO/发送入口
- `backend/tests/` 中现有日程与协议测试
- Unity 协议 DTO/接收适配与解析测试

### 5.4 门禁与回滚

- 聚焦 pytest 覆盖合法输出、跨段/重复/未知 candidate、required group、timeout、fallback、幂等和 fingerprint。
- 固定 provider 的完整计划不出现普通任务 `HH:MM`；真实 provider 只做实现冒烟，不替代独立测试。
- V2 未切换为生产 owner 前，可关闭唯一 capability 开关回到 V1；禁止按 NPC 或按消息随机混用 owner。

## 6. 阶段 3：Unity 生产接管、存档与阶段边界

### 6.1 目标

以一次原子切换让 `NpcDayPlanRuntime` 成为唯一日程 owner，并接入现有任务执行、存读档、跨日和诊断。

### 6.2 实施内容

1. `NpcDailyScheduleRuntimeCoordinator` 减重为协议/生命周期适配器，队列、segment、活动计划任务、边界和 mutation 状态迁入深模块。
2. 通过唯一 seam 把队首交给 `NpcTaskExecutor`；只有匹配终态才能消费当前计划任务并推进下一项。
3. `succeeded / failed / cancelled / forced_terminal` 均消费当前项并记录稳定原因；失败/取消不触发整表 replan。
4. 实现 17:00 工作收口、休息启动、午夜强制终态/退役，以及 `work_open/work_close` 生命周期动作。
5. 主存档保存 plan identity/fingerprint、active segment、两个剩余队列、已消费/退役 task ID 和 revision；不保存 coroutine、移动、动画、reservation 或 mutation in-flight。
6. `day_plan_snapshot` 提供 segment、活动任务、剩余数量、边界、mutation 和最近终态；旧诊断字段仅在兼容期标记 legacy。
7. 切换顺序固定为：暂停旧 owner 新启动 -> 接收/恢复 V2 plan -> 原子退役旧 pending/关联 -> 启用新 owner -> 恢复任务推进。任一步失败都保持 Gameplay 阻塞或稳定回退，不能产生双 owner。

### 6.3 预计文件

- `Assets/Scripts/NPC/NpcDailyScheduleRuntimeCoordinator.cs`
- `Assets/Scripts/NPC/NpcDailyScheduleController.cs`（减重或退役）
- `Assets/Scripts/NPC/NpcDayPlanRuntime.cs`
- `Assets/Scripts/NPC/NpcTaskExecutor.cs`
- `Assets/Scripts/NPC/NpcBehaviorApplier.cs`
- `Assets/Scripts/Save/GameSaveData.cs`、migration/restore 入口
- `Assets/Scripts/Diagnostics/AiscDiagnostics.cs`
- `Assets/Scripts/Diagnostics/DiagnosticModels.cs`
- 相关 EditMode 与隔离 PlayMode probe

### 6.4 门禁与回滚

- Unity EditMode 覆盖 owner 切换、顺序推进、失败/取消、17:00、午夜、存读档和跨日。
- 隔离 probe 证明新 owner 可完整推进两段队列，且旧 owner 没有向 executor 发起任务。
- 若发现双 owner、跨日混合状态或 coordinator 继续显著增重，立即停止；回滚到 V1 唯一 owner，不继续做对话接入。

## 7. 阶段 4：对话任务变更统一入口

### 7.1 目标

在新队列稳定后，移除“对话完成即整表重规划”，让玩家明确安排通过有界 mutation 进入同一运行时。

### 7.2 实施内容

1. NPC-NPC COMPLETE 不再触发日程 replacement；社交摘要和关系变化仍进入记忆并影响次日计划。
2. NPC-NPC 社交只允许 runtime 按 interrupt policy 暂停/恢复或终态化当前任务；对话结束后继续原队列。
3. 普通 Player-NPC 聊天不改计划；只有明确、合法且被 NPC 接受的安排才返回至多一个 mutation。
4. mutation 使用独立 operation/request/terminal，校验 base revision、candidate、segment、玩法状态和 interrupt policy；实现 append/preempt/defer/reject。
5. `preempt_current` 不可抢占时只允许稳定降级为 defer 或 reject；不得绕过 runtime 直接调用 executor。
6. 为未来实时 LLM 保留相同 interface，但 source 权限和冷却独立配置，本阶段不启用新的实时决策 owner。

### 7.3 门禁与回滚

- NPC-NPC 对话前后 plan revision 和队列内容不变；暂停/恢复或继续有稳定证据。
- 玩家四类 mutation 均有 operation、revision、终态和旧任务裁决证据；每次对话不得产生第二个 mutation。
- mutation producer 可独立关闭；关闭后普通聊天仍正常，但不得恢复旧整表 replan。

## 8. 阶段 5：旧时间点/replan 清零与资产时间档

### 8.1 目标

在新生产链稳定后删除旧 owner 和生产依赖，并通过 Unity MCP 将 Test 一天调整为 1200 秒。

### 8.2 实施内容

1. 删除窗口错过触发的 `NPC_SCHEDULE_REPLAN_REQUEST` 生产入口和每分钟窗口裁决。
2. 删除普通任务对 `planned_start_time / execution_window_* / miss_policy` 的生产依赖。
3. 删除 NPC-NPC/普通玩家对话整表 replan context；迁移或删除 `NpcScheduleReplanTracker`，确认无第二 owner。
4. 对协议、存档和诊断做最终 migration；兼容 DTO 若保留，必须无生产 caller、明确删除条件，并不得进入新存档。
5. 先读取 `UnityMCPUsageRules.md`，定位 `_flowProfile / _testDayRealSeconds` 的真实序列化 owner，再通过 Unity MCP 把 Test 档改为 `1200` 秒；不得只改 C# 默认值。
6. 更新受影响代码目录 README；Workstream 和草案仍待独立测试通过后才改为“已上线”。

### 8.3 门禁与回滚

- codebase-memory 调用图与 `rg` 静态审计均确认无旧生产消息、旧窗口裁决、旧对话整表 replan 或双 owner caller。
- Unity MCP 资产检查确认序列化值为 1200；MCP 不可用时停止资产修改，本阶段不得宣称完成。
- 清零阶段完成后不再以恢复旧 owner 作为普通回滚；失败应修复新 owner，除非出现存档破坏或世界无法进入等阻断级问题。

## 9. 阶段 6：独立测试与口径切换

独立测试会话先读取本 plan、累计 execution 和 `TestingAndDiagnostics/README.md`，再使用最小充分组合：

1. Python 聚焦 pytest：双队列生成、catalog 分段、最终校验、fallback、幂等、超时、mutation 协议和旧生产入口缺失。
2. Unity EditMode：纯 runtime、边界、抢占、revision、存档 migration、跨日和 owner 唯一性。
3. `aisc_control` 隔离 probe：固定计划、17:00、午夜、失败/取消、NPC-NPC 暂停恢复、玩家四类 mutation。
4. 真实 PlayMode：新游戏完整 day 1、同日读档、跨日 day 2、移动/动作、NPC-NPC 对话、玩家安排和午夜结算；用 `aisc_debug.day_plan_snapshot`、任务快照、protocol trace 和 runtime snapshot 取证，Console 只补异常。
5. 真实 provider：5 NPC 每人单次 LLM 同时返回两队列，记录 success/fallback、耗时、最慢 owner 和稳定失败原因，不保存 Prompt/模型原文。
6. 通过后创建同主题 test，并更新 `NpcBehavior`、`Dialogue`、`ProtocolAndSave` Workstream、相关目录 README、旧草案状态和 Roadmap；失败则回到新的执行会话修复，在原 test 追加有界复测。

### 9.1 阶段 6 失败修复：Unity 存档 schema 版本收口

独立测试已证明跨日链在 Unity 本地存档 prepare 阶段失败：`GameSaveData` 默认版本和迁移链已升至 schema 3，但 `UnitySaveRepository` 的当前版本仍停留在 2，导致新建 schema 3 快照被 `ValidateSaveData` 拒绝，并使事务停在 `PreparingUnity`。

本次修复只统一 Unity 存档仓储的当前 schema 为 3，不改变 Unity 世界权威、Python 记忆权威、checkpoint 协议或两段式任务运行时。实现范围为 `UnitySaveRepository.cs`、既有 `NpcDailyScheduleTests.cs`、相关目录说明、累计 execution 和错误预防记录；不涉及场景、Prefab 或 SerializeField。

验收标准：

1. 默认 schema 3 的新存档可通过 `UnitySaveRepository.Prepare` 并生成 schema 3 manifest。
2. schema 2 存档可通过既有迁移链升级到 schema 3，日计划字段按既定口径清空。
3. Unity 编译与聚焦 EditMode 测试通过，Console 无新增 error，`git diff --check` 通过。
4. 本执行会话只得出“修复实现完成，待原 test 记录追加独立复测”；真实跨日解除暂停、提交完成、day 2 推进和读档仍由后续测试会话裁决。

## 10. 总体验收标准

1. 每 NPC 每日一次 LLM 返回工作/休息两个有序队列，普通任务没有精确时间点。
2. Unity 只在当前任务匹配终态后启动下一任务；失败或取消继续队列，不触发 revision/replan 循环。
3. 17:00 当前工作可继续、其余工作退役，终态后进入休息；午夜活动任务稳定强制终态，结算不悬挂。
4. `work_open/work_close` 由 segment 生命周期确定性执行。
5. NPC-NPC 对话不改变 plan revision/队列且不整表 replan；玩家对话最多产生一个合法 mutation。
6. append/preempt/defer/reject 均由同一 Unity runtime 裁决并可诊断，不存在绕过 executor owner 的路径。
7. 新游戏、同日读档和跨日不恢复旧任务 owner、旧 mutation pending、旧 segment 或旧 V1 日程。
8. Test 一天为现实 1200 秒；完整 PlayMode 日内无 `schedule_window_expired`、重复 replan 或双 owner 证据。
9. 新旧协议兼容期结束后，旧字段和消息无生产 caller；若保留 reader，只用于明确版本迁移。
10. 所有验收项有结构化证据并写入 test；在此之前只能声明“实现完成，待独立测试”。

## 11. 全局风险与停止条件

1. catalog 不能稳定声明 segment/completion/interrupt policy 时，停止阶段 1，不允许 Python 和 Unity 分别按 action 名猜测。
2. 午夜强制终态或玩家抢占语义未确认时，停止相关状态机实现。
3. 任意迁移时点出现 V1/V2 双 owner，立即停止后续功能，先收口 owner。
4. 若协议调整改变 Unity 世界权威、Python 记忆权威或 checkpoint 边界，停止并先更新 ADR。
5. 若玩家安排需要未实现玩法或未知 candidate，稳定拒绝，不执行自由文本。
6. 若实现需要新增场景 spot、Prefab 连线或 UI，超出本 plan；另行方案并遵守 Unity MCP 资产优先规则。
7. 若后端重启导致新队列恢复失败，先区分日程存档问题与既有重连状态机问题；不得把后端自动重连扩入本案。

## 12. 每阶段实施会话最低门禁

1. 执行前确认 codebase-memory 索引新鲜度；过期则先更新索引，再做代码级可行性分析。
2. 编辑前读取最新文件并检查工作树，保留人工和前序会话修改。
3. 运行与该阶段匹配的 `compileall/pytest`、Unity EditMode、Unity MCP 编译/Console、结构化 probe 和 `git diff --check`；不机械运行无关工具。
4. 更新 codebase-memory 向量库，并同步 `docs/AIChanges/codebase-memory-mcp_更新.md` 的最后更新时间。
5. 将实际文件、门禁结果、文档回写、问题和未完成项追加到同主题 execution；完成当前阶段后停止，要求用户在新对话进入下一阶段。

## 13. 本方案会话边界

本文件是后续实现的唯一方案来源。本轮只创建 plan 和索引，不修改运行代码、协议、测试资产、场景、Prefab 或 SerializeField，也不把草案口径提前写成已上线状态。
