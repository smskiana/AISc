# NPC 空闲表现前端托管 — 设计方案

## 需求理解

后端脚本当前会通过 P6 随机微动作和随机需求，把发呆、喝水、休息、闲逛、移动等行为作为正式 `NPC_BEHAVIOR` 推给 Unity。该行为不是 AI 日程，也缺少互斥语义，导致 NPC 可能在移动、对话或主日程执行时出现不可信的小动作。

本轮目标是把这类“随机小行为”收敛为前端空闲表现资源：

1. 后端明确行为继续拥有最高优先级。
2. 前端只在 NPC 空闲表现槽位可用时播放小动作或同区域踱步。
3. 空闲表现不回写后端、不进入记忆、不改变权威 `current_location/current_action`。
4. 每个 NPC 至少间隔游戏内 30 分钟才进行一次空闲表现判断。
5. 玩家搭话时，前端把玩家看到的空闲表现作为视觉上下文附给后端对话入口。

## Workstream / Roadmap

- Workstream：`docs/Workstreams/NpcBehavior/README.md`
- Workstream：`docs/Workstreams/FrontendArchitecture/README.md`
- 相关口径：不再用随机可达点或后端随机微动作替代行为语义；前端负责把行为与表现落地。

## 相关 ADR

本轮不改变导航 typed path、记忆架构或 Unity MCP 资产优先级，不新增 ADR。

## 方案思路

1. 后端 `BehaviorEngine`
   - 关闭 P6 随机微动作发送。
   - 关闭随机需求生成入口，保留显式 `current_need` 的 P3 执行能力。
   - 避免后端再把脚本随机行为伪装成正式 NPC 行为。

2. Unity 前端
   - 新增 `NpcAmbientBehaviorController`，维护每个 NPC 的空闲表现判断冷却和表现池。
   - 扩展 `NpcEntity`，增加可抢占的空闲表现槽位：
     - 后端行为到达时取消空闲表现。
     - 支持短时原地表现与同区域踱步。
     - 空闲表现不覆盖权威 `CurrentAction`。
   - `NpcSpawner` 只做薄转接：创建控制器、转发时间 tick / 行为消息 / Update，不承接具体表现逻辑。

3. 玩家视觉上下文
   - 新增前端 `NpcVisualContext`。
   - 玩家开始对话时，从被点击 NPC 读取当前视觉上下文并附加到 `DIALOGUE_START`。
   - 后端对话服务把该上下文作为临场用户消息，不写入计划或记忆。

## 风险点

1. 前端未进入 Play 模式时无法确认空闲表现的实际观感。
2. 空闲踱步复用现有移动 provider，若 A* 对某些同区域点不可达，会静默取消或很快被后端抢占。
3. 视觉上下文只在玩家开始对话时进入 prompt，不应被误当作长期事实。
