# 跑测问题修复 — 计划

## 完成时间
2026-07-10

## 这次计划解决什么

基于最近一轮长跑压测和记忆诊断日志，当前系统已经证明“能持续跑、能入图、能回忆到”，但还没有达到“记忆结果稳定、结构收敛、日志可读”的状态。

这次计划聚焦修下面 5 类问题：

1. 熟人检索能召回，但排序不够贴目标人物
2. 图增长不均衡，说明节点复用和低连接节点控制还不够
3. `relationship` 边衰减后不会因为被经常想起而恢复
4. `felt` 情绪边明显偏少
5. 日志链路编码混乱，Unity 回退日志可读性差

## 当前问题归纳

### 1. 记忆排序偏“高重要旧事件”，不够“当前熟人相关”

现状：

1. [backend/src/memory/retrieval.py](F:/GameProject/unity/AISc/backend/src/memory/retrieval.py) 已经能从人物节点反查事件
2. 但合并排序还是主要靠 `clarity` 和向量分数
3. 与当前目标人物直接相关的新事件，经常被旧核心事件压在后面

结果：

1. “能想起来”
2. 但“不一定先想到最该想到的那件事”

### 2. 图增长不均衡，部分 NPC 可能在堆低连接节点

现状：

1. 某些 NPC 节点增长很快，但边增长没有同步
2. 这说明部分新节点被创建了，但没有被充分接回已有图
3. 当前 [backend/src/memory/manager.py](F:/GameProject/unity/AISc/backend/src/memory/manager.py) 仍然比较依赖 LLM 输出复用已有节点

结果：

1. 节点数上升
2. 图未必更“紧”
3. 之后检索会越来越靠向量兜底

### 3. `relationship` 边在衰减，但没有真正回血

现状：

1. [backend/src/memory/manager.py](F:/GameProject/unity/AISc/backend/src/memory/manager.py) 已有 `recover_clarity()`
2. 但 [backend/src/memory/retrieval.py](F:/GameProject/unity/AISc/backend/src/memory/retrieval.py) 检索遍历并没有接上它
3. 导致关系边被频繁使用，也不会被强化

结果：

1. 熟人关系边越来越弱
2. 图检索里“人物 → 人物 / 人物 → 事件”的熟悉感不会沉淀

### 4. 情绪边太少

现状：

1. 长跑后 `felt` 边数量明显低于 `involved / located_at / happened_at / mentioned`
2. 这说明事件里的情绪没有稳定入图

结果：

1. 印象和状态系统仍缺少“图里的情绪支撑”
2. 语气、状态、夜间印象压缩会偏事实，偏弱感受

### 5. 日志编码链路不统一

现状：

1. `MessageBus` 已经能在无 Unity 接收时回退记录 [backend/src/application/message_bus.py](F:/GameProject/unity/AISc/backend/src/application/message_bus.py)
2. 但本轮长跑日志已经出现 GBK / UTF-8 混用

结果：

1. 控制台难读
2. 长跑后很难直接看日志复盘对白和行为链

## 修复目标

### 目标一：让熟人检索优先回忆“和这个人有关”的事

不是只要召回事件，而是要让：

1. 当前目标人物直接参与的事件优先
2. 最近发生、地点相关、情绪相关的事件也能合理上浮

### 目标二：让图结构增长更收敛

目标不是减少增长，而是让增长变得“连接更充分、复用更稳定”。

### 目标三：让关系边能形成“经常想起就更稳”的闭环

这对熟人感、搭话倾向和长期关系都很关键。

### 目标四：让日志变成可靠调试工具

后续还会继续跑测，所以日志链本身必须先干净。

## 分阶段实施

## Phase 1：修检索排序

### 要做什么

调整 [backend/src/memory/retrieval.py](F:/GameProject/unity/AISc/backend/src/memory/retrieval.py) 的图召回和合并排序逻辑。

### 具体策略

1. 图候选里增加“是否直接来自目标人物起点”的额外权重
2. 如果候选节点是事件，且它与目标人物存在直接 `involved` 关系，额外加分
3. 最近时间、当前地点命中的候选再加轻量上下文分
4. 降低纯 `relationship -> person` 节点在最终文本候选里的占位权重

### 预期效果

1. 从 `sakura -> chihaya` 检索时，更容易先看到“和千早发生的事”
2. 旧核心事件仍能回忆到，但不再总是抢第一位

## Phase 2：接上 clarity 恢复闭环

### 要做什么

让检索遍历真正调用 `recover_clarity()`。

### 具体策略

1. 在图召回阶段记录被采用的边
2. 对进入最终候选的高优先边，按方向调用 `recover_clarity()`
3. 恢复幅度先保守，只给真正被检索使用到的边回血

### 预期效果

1. 常想起的人和事会更稳
2. `relationship` 不会只掉不涨

## Phase 3：补节点复用和低连接控制

### 要做什么

增强 [backend/src/memory/manager.py](F:/GameProject/unity/AISc/backend/src/memory/manager.py) 的节点复用和入图后检查。

### 具体策略

1. 人物节点改成代码级优先复用，不只靠 prompt
2. 地点节点优先映射到固定 location / zone
3. 对新写入的事件节点，检查是否至少连到：
   - `self`
   - 参与人物
   - 地点或时间
4. 对“节点增长但边没增长”的情况补统计日志

### 预期效果

1. 图增长更收敛
2. 孤立或弱连接节点更少

## Phase 4：补情绪边覆盖

### 要做什么

提高 `felt` 边进入图的稳定性。

### 具体策略

1. 提取 prompt 里强调“有明显情绪就必须抽”
2. 入图时如果事件文本带明显情绪词，但 LLM 没抽到，启发式补一个基础情绪节点
3. 先只补常见基础情绪，避免情绪节点失控膨胀

### 预期效果

1. `felt` 边不再过少
2. 夜间印象和状态系统更容易从图里读出“情绪色彩”

## Phase 5：统一日志编码和 Unity 回退日志

### 要做什么

统一后端日志写入为 UTF-8，并把 Unity 消息单独落一份易读日志。

### 具体策略

1. 后端入口显式统一 UTF-8 输出
2. `MessageBus` 的 `NPC_BUBBLE / NPC_SOCIAL_ACTION` 单独写文件
3. 保留当前总日志，同时新增一份“纯 Unity 消息日志”

### 预期效果

1. 后续跑测可以直接读对白
2. 不再因为混码影响复盘

## 推荐实施顺序

建议按下面顺序做，而不是一起改：

1. `Phase 5` 日志编码
2. `Phase 1` 检索排序
3. `Phase 2` clarity 恢复闭环
4. `Phase 3` 节点复用和低连接控制
5. `Phase 4` 情绪边补强

原因：

1. 先把日志修好，后续验证成本最低
2. 排序和恢复闭环会直接影响“现在是否更像有记忆”
3. 节点复用和情绪边补强属于第二层结构优化

## 验证计划

### 第一轮验证

1. 编译和导入检查
2. 定向 smoke：
   - 人物节点回忆事件
   - 检索后边清晰度是否回血
3. Unity 消息日志文件是否 UTF-8 可读

### 第二轮验证

1. 重新跑 1 天定向测试
2. 对比：
   - 熟人检索排序
   - `relationship` clarity 变化
   - `felt` 边数量

### 第三轮验证

1. 再跑一轮完整长跑
2. 重点看：
   - 图增长是否更均衡
   - 午夜耗时是否没有明显恶化
   - 日志是否全程可读

## 当前判断

这轮长跑说明系统已经过了“能不能跑起来”这一关。

接下来的关键不是继续堆新设计，而是先把这三件事做实：

1. 检索更像“真的在想这个人”
2. 图增长更像“在收敛结构”
3. 日志更像“能支撑持续迭代”
