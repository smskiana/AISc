# 第三天真实运行数据采集 - 设计方案

## 目标

通过 Unity Play 模式真实运行新游戏，不使用跳时间，让 NPC 日程、移动、对话、午夜结算和记忆入图完整发生；在 Day 3 抓取 SQLite、LanceDB 和运行快照，作为后续记忆路由广度/深度测试来源。

## 测试条件

1. Unity 场景：`Town_Main`。
2. 新游戏从 Day 1 08:00 开始。
3. 玩家主要在街道活动，允许人工移动和对话以增加真实数据。
4. 不调用 `FAST_FORWARD`，不直接设置游戏日期或时间。
5. 使用 `aisc_debug` 持续检查 `runtime_snapshot` 与 `health_report`。

## 采集方式

1. 在 Day 3 且游戏因对话暂停的稳定窗口，用 SQLite Backup API 生成一致数据库副本。
2. 同步复制当前 LanceDB 目录。
3. 写入结构化 manifest，记录时间、天气、玩家位置、健康状态、表统计和文件哈希。
4. 产物保存到 `docs/AIChanges/artifacts/memory_route_day3_source/`。
5. 若用户继续运行后要求重抓，保留原始 Day 3 副本，并按实际日期创建独立 `memory_route_dayN_current_source/`，复用相同一致性与验证流程。

## 风险

1. NPC 主动移动和社交频率较低，数据量可能仍不足；manifest 必须披露实际节点、边、短期记忆和检索日志数量。
2. 玩家对话会暂停时间；本轮保留正常对话流程，不通过跳时绕过。
3. LanceDB 在运行中复制可能遇到文件变动；选择对话暂停窗口，并在复制后验证可打开和表数量。

3. `docs/AIChanges/artifacts/memory_route_day3_source/manifest.json`
4. 本 plan 与对应 execution。
