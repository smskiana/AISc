# 导航随机候选与传送检测日志方案

## Workstream

Navigation / FrontendArchitecture

## Roadmap item

NAV / FEA

## 相关 ADR

1. `docs/DecisionRecords/ADR-0002-navigation-typed-path.md`

## 需求理解

当前 typed path 仍未完全 Play 验收，需要补检测日志，重点确认：

1. `MoveToLocation()` 的目标候选点是否真的在随机变化。
2. A* 路径中传送段是否稳定由 pathfinder 产出，并在移动执行时确实触发。

本轮只补诊断日志，不改变 NPC 移动选择、寻路、传送、SceneAnchor 采样或资产配置规则。

## 方案思路

1. 在 `NavigationDebugLog` 中新增候选点批次日志：
   - 输出 location、候选点数量、近似唯一点数量、坐标范围、前若干个 raw 候选点。
   - 让 Play 模式连续触发同一 location 移动时，可以直接对比候选 raw 坐标是否变化。

2. 在 `AStarMovementProvider.MoveToLocation()` 中新增逐候选评估日志：
   - 每个候选点输出 raw 坐标。
   - 输出 raw cell、修正后的可走 cell、修正后的目标 world。
   - 输出寻路成功 / 失败原因。
   - 选中候选时输出最终 raw 与 resolved target。

3. 增强传送日志：
   - `LogPathSummary()` 的 `TELEPORT` 行输出入口、出口、解析后出口、入口距离、出口吸附距离和 reverse。
   - `MoveAlongPath()` 执行传送时输出实际当前位置到入口的距离，便于确认不是普通长距离行走伪装成传送。

## 风险点

1. 日志会变多，可能刷屏；但 `NavigationDebugLog.Enabled` 已存在总开关。
2. 不应为了检测而改变随机采样逻辑，否则验收对象会被污染。
3. 不把传送触发判断重新塞回移动器，只记录 path segment 与执行时的位置关系。

## 验证方式

1. `dotnet build AISc.sln --no-restore`。
2. 静态搜索确认新增日志入口存在。
3. Play 模式观察同一 NPC 多次 `MoveToLocation()`：
   - `候选点批次` 中 raw 坐标是否变化。
   - `候选评估` 中 raw -> resolved 是否合理。
   - 跨店路径是否出现 `TELEPORT` 与 `执行传送段`。
