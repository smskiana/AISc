# 午夜性能优化 — 方案

## 完成时间
2026-07-10

## 这次方案要解决什么

当前后端已经证明“有记忆”，但午夜阶段存在明显性能瓶颈：

1. `SentenceTransformer` 被频繁重复加载
2. 图提取阶段对每个新节点单独做一次 embedding
3. 夜间印象重建按 `owner -> target` 串行调用 LLM
4. 午夜同步重活直接堵在运行时主链路上

这些问题叠加后，会带来两个直接后果：

1. `NEW_GAME` 冷启动耗时过长
2. 午夜处理速度慢，且容易让服务在重处理阶段显得“卡死”

## 本次优化目标

本次先不做功能语义调整，只做性能层优化，目标是：

1. 减少本地 embedding 的重复初始化和重复调用
2. 缩短午夜“短期记忆 -> 图 -> 印象”整条链路的耗时
3. 为后续继续压测和分阶段观测保留足够日志

## 已确认的主要热点

### 1. `MemoryManager._encode()` 重复加载模型

当前 [backend/src/memory/manager.py](F:/GameProject/unity/AISc/backend/src/memory/manager.py) 内部仍然直接 new `SentenceTransformer`。

这会绕过现有的 [backend/src/memory/embedding.py](F:/GameProject/unity/AISc/backend/src/memory/embedding.py) 全局缓存，导致：

1. 每次节点入图都可能重复初始化模型
2. 午夜提取阶段节点一多就会出现大量重复开销

### 2. 图提取按节点逐个 embedding

当前 `extract_and_ingest()` 会对每个新节点单独编码，而不是整批一次性编码。

这会放大：

1. Python 层函数调用开销
2. embedding 前后处理开销
3. 模型侧批处理优势损失

### 3. 夜间印象重建完全串行

当前 [backend/src/npc/state_manager.py](F:/GameProject/unity/AISc/backend/src/npc/state_manager.py) 的 `nightly_refresh()` 会依次执行：

1. 取近期短期记忆
2. 图中找相关片段
3. 调 LLM 生成印象
4. 写回数据库

项目规模虽然只有 5 个 NPC，但夜里仍要处理大约 25 组熟人印象，串行调用 LLM 会明显拖慢总耗时。

### 4. 目标人物查询向量重复编码

`_graph_target_memories()` 里每次都对目标人物名字单独 `encode_batch([target_name])`，属于可缓存但尚未缓存的重复工作。

## 本次执行策略

### 一、统一 embedding 入口并加固全局缓存

在 [backend/src/memory/embedding.py](F:/GameProject/unity/AISc/backend/src/memory/embedding.py) 中：

1. 继续保留全局模型单例
2. 增加初始化锁，避免并发首次加载时重复建模
3. 保持其它模块统一走 `encode_batch()`

原则：

1. 不再允许业务模块私自 new `SentenceTransformer`
2. embedding 模型生命周期只由 `embedding.py` 管理

### 二、图提取改为批量编码

在 [backend/src/memory/manager.py](F:/GameProject/unity/AISc/backend/src/memory/manager.py) 中：

1. 先收集本轮所有需要写入 LanceDB 的节点文本
2. 一次性批量 `encode_batch(texts)`
3. 再回填向量并统一写入 LanceDB

这样提取阶段会从：

- “每个节点一次编码”

变成：

- “每个 NPC 本轮提取一次批量编码”

### 三、夜间印象改为“预处理 + 有界并发 LLM”

在 [backend/src/npc/state_manager.py](F:/GameProject/unity/AISc/backend/src/npc/state_manager.py) 中：

1. 午夜先顺序准备所有印象任务输入
2. 预先缓存所有目标人物名字的查询向量
3. 预先收集：
   - `recent_memories`
   - `graph_memories`
   - `previous impression`
4. 再使用有上限的线程池并发生成基准印象
5. 全部结果返回后统一写库

并发策略：

1. 并发数做成常量，先取保守值
2. 优先保证稳定，再追求更高并发

这样做的原因是：

1. 项目 NPC 数量小，并发收益很直接
2. LLM 调用本身是最适合并发化的一段
3. 统一写库比“边生成边写”更容易排查问题

### 四、缓存目标查询向量

在夜间刷新开始时，一次性为：

1. `player`
2. `sakura`
3. `chihaya`
4. `kazuha`
5. `tatsunosuke`
6. `kujo`

生成目标查询向量，然后 `_graph_target_memories()` 直接复用，避免重复编码同一批名字。

### 五、补充午夜阶段耗时日志

在 [backend/src/application/runtime.py](F:/GameProject/unity/AISc/backend/src/application/runtime.py) 中补充阶段耗时日志，至少覆盖：

1. 边衰减
2. 提取阶段
3. 演化阶段
4. 夜间印象/状态结算
5. 短期记忆清理
6. 午夜总耗时

这样后续即使暂时不压测，也能在正常运行日志里看出最慢的是哪一段。

## 风险与约束

### 1. 并发 LLM 可能触发限流或超时

应对方式：

1. 使用有界并发，不做无限放大
2. 单条失败时回退到启发式印象
3. 不让单个目标失败拖垮整轮午夜刷新

### 2. 并发阶段不直接共享可变写库状态

应对方式：

1. 线程池阶段只负责生成结果
2. 数据库写入回到主线程顺序提交

### 3. 保持功能语义不变

这次优化不改变：

1. 印象生成的输入来源
2. 夜间结算的状态语义
3. 白天行为和对话读取印象的方式
