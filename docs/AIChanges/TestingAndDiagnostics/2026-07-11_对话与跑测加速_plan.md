# 对话与跑测加速 — 设计方案

## 目标

本轮目标是为下一轮跑测前做一组低风险提速：

1. 可并发的 LLM 请求不要串行等待
2. 主通信链尽量走 WebSocket，不再依赖轮询队列
3. 增加跑测可读日志，重点标记性能和 NPC 记忆是否可用

## 改动方案

### 1. 玩家建议回复改为后台推送

当前玩家对话每轮是：

1. 流式生成 NPC 回复
2. 等 3 条玩家建议回复 LLM 完成
3. 才发送 `DIALOGUE_COMPLETE`

这会让 NPC 回复结束后继续卡一段时间。

改为：

1. NPC 回复流式结束后立刻发送 `DIALOGUE_COMPLETE`
2. 同时先给前端兜底建议
3. 后台异步生成真实建议
4. 完成后通过 WebSocket 推送 `DIALOGUE_CHOICES_UPDATE`

Unity 侧新增对 `DIALOGUE_CHOICES_UPDATE` 的处理，收到后刷新选项。

### 2. 对话结束摘要并发/合并

当前 `end_dialogue()` 会先生成 NPC 视角短期记忆，再生成玩家向量记忆，两次 LLM 串行。

改为一次 LLM 结构化输出：

```json
{
  "npc_memory": "...",
  "player_memory": "..."
}
```

这样直接减少一次请求；失败时仍回退旧启发式摘要。

### 3. 消息总线默认不再为轮询双写

当前 `MessageBus.broadcast()` 会同时：

1. 写入 HTTP 轮询队列
2. 尝试 WebSocket 直推

考虑前端主链已经是原生 WebSocket，本轮默认禁用轮询入队，只保留 `/api/poll` 接口作为兼容调试入口。

### 4. 跑测日志标记

新增统一日志标签：

1. `[PERF]` 性能耗时
2. `[MEMORY]` 记忆检索/写入可行性
3. `[WS]` WebSocket 推送与队列行为

重点标记：

1. NPC 首轮回复耗时
2. 玩家建议后台生成耗时
3. 对话结束摘要耗时
4. 玩家记忆向量写入是否成功
5. 多跳图检索最终返回条数
