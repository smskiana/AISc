# UTF-8 长跑日志脚本 — 计划

## 完成时间
2026-07-10

## 需求理解

当前 7 天长跑的核心问题不是后端内部数据写错，而是“原始长跑日志”仍然经过 GBK 控制台再落盘，导致：

1. `summary.json` 干净
2. `tee` 出来的 `.log` 仍可能乱码

用户已经明确要求按最简单的办法解决，所以这轮不去继续折腾宿主终端环境，而是直接绕开它。

## 方案

新增一个可复用的后端长跑脚本，特点是：

1. Python 直接写 UTF-8 日志文件
2. Python 直接写 UTF-8 摘要 JSON
3. 自动使用隔离测试库和隔离 LanceDB 目录
4. 不再依赖 `Tee-Object` 抓控制台输出

## 实施范围

1. 新增 `backend/scripts/run_7day_benchmark.py`
2. 新增本轮执行记录文档

## 脚本目标

脚本至少要支持：

1. 默认按 7 天口径推进
2. 可通过参数改测试分钟数，方便 smoke
3. 可通过参数指定输出 tag
4. 自动产出：
   - `docs/AIChanges/artifacts/<tag>/<tag>.log`
   - `docs/AIChanges/artifacts/<tag>/<tag>_summary.json`
   - `docs/AIChanges/artifacts/<tag>/data` 与 `SaveData`

## 设计取舍

这轮优先选“最少侵入”的做法：

1. 不改现有运行时核心逻辑
2. 不改现有 `run.py`
3. 不改项目主配置结构
4. 只是把已经验证过的长跑流程沉淀成一个正式脚本

## 验证计划

1. 先跑一轮短分钟 smoke
2. 检查：
   - `.log` 是否为 UTF-8 可读
   - `_summary.json` 是否正常生成
   - 隔离 artifacts 目录是否正常创建
3. 如果 smoke 通过，后续完整长跑直接复用这个脚本
