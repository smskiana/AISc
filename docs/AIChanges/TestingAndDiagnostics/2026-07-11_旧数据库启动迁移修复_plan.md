# 旧数据库启动迁移修复 — 设计方案

## 需求理解

Unity 开始界面无法启动后端。实际检查发现 `backend/data/game.db` 仍是旧图记忆 schema，后端启动时在 `SCHEMA_SQL` 的索引创建阶段访问新列 `clarity_ab / clarity_ba`，导致迁移函数还没执行就抛出 SQLite 错误。

## 方案

1. 在 `SQLiteClient._init_db()` 执行完整 `SCHEMA_SQL` 前，先做一段旧库预迁移。
2. 预迁移只补会阻断 `SCHEMA_SQL` 的旧表列：
   - `memory_edges.clarity_ab`
   - `memory_edges.clarity_ba`
   - `memory_edges.target_importance`
3. 对旧 `memory_nodes` 表保留兼容，不强制重建表；`insert_node()` 根据真实表列补写旧 schema 的必填字段，避免冷启动插入失败。
4. 保持现有正式迁移 `_apply_migrations()` 继续补齐 `npc_states`、诊断表和印象表。
5. `SaveManager.load()` 恢复旧存档数据库后，立即对恢复后的 `game.db` 执行同一套 SQLite 迁移，避免读档把旧 schema 覆盖回来。

## 验证

1. 直接实例化 `SQLiteClient("backend/data/game.db")`，确认旧库可迁移。
2. 检查 `npc_states` 和 `memory_edges` 新列存在。
3. 运行 Python 编译检查。
4. 启动后端，确认 `/api/health` 可返回。
