## 暴露的问题

### 1. LLM 日计划仍会产生非法空地点

现象：

```text
LLM 非法计划 kazuha: walk_to@, 已过滤
```

本次共 `97` 条 warning，全部是 `LLM 非法计划`。后端会过滤，不导致崩溃，但会造成当天计划缺块，例如一叶第 10 天早晨出现过一组空 location 计划。

建议后续处理：

1. 强化日计划 prompt，要求 location 必须来自固定 ID。
2. 在解析层对缺 location 的计划做 profile routine 兜底，而不是只过滤。
3. 在 summary 里单独统计非法计划数量，方便之后压测比较。

### 2. 个别计划动作重复触发

巡检中观察到部分重复行为日志，例如：

```text
行为: chihaya → work_close @ bakery.doorway (P4:计划)
行为: chihaya → work_close @ bakery.doorway (P4:计划)
```

这不影响长测完成，但说明行为引擎在某些计划边界可能重复应用同一计划项。

建议后续处理：

1. 给当前计划项增加已触发标记或时间窗口去重。
2. 对同一 NPC 同一分钟同一 action/location 只写一次短期记忆。

### 3. 后台启动命令仍有 PowerShell 引号噪声

本次后台 runner 的 `runner.err` 出现一条：

```text
utf-8: The term 'utf-8' is not recognized...
```

原因是外层 PowerShell / pwsh 的引号转义噪声。脚本本体仍正常运行，主日志和 summary 都是 UTF-8，且主日志 `ERROR_COUNT=0`。

建议后续处理：

1. 后续直接在当前 shell 设置环境变量后运行 Python，减少嵌套 pwsh。
2. 或新增专门的 `.ps1` 跑测入口，统一处理 PowerShell 7 编码和参数。

## 结论

后端可以脱离前端独自运行，并且本次 30 分钟真实时间长测已完成。当前主要问题不是后端独立性，而是日计划 LLM 输出仍需更强约束，以及行为计划触发可以继续做去重。
