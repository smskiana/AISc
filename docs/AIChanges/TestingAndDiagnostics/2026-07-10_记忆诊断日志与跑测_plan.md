# 记忆诊断日志与跑测 — 计划

## 完成时间
2026-07-10

## 需求理解

用户希望先补一轮日志，方便我们再次跑测“记忆是否真的生效”，重点不是大改语义，而是把关键链路变得可观测。

本轮目标：

1. 给图初始化、午夜入图、图召回补充诊断日志
2. 日志要能直接看出“边是不是按方向写进去了”“人物起点能不能回忆到事件”
3. 跑一次定向 smoke test，验证日志和功能结果一致

## 实施思路

### 1. 诊断日志使用开关控制

不把所有日志永久开成高噪音，而是用环境变量开关控制，例如：

1. `SAKURA_MEMORY_TRACE=1`

开启后输出记忆诊断日志；不开启时保持原先日志量。

### 2. 日志只加在三处

#### A. 初始图创建

`backend/src/npc/state_manager.py`

输出：

1. 当前 NPC 初始人物节点数
2. 核心记忆节点数
3. 初始边数量摘要

#### B. 午夜提取入图

`backend/src/memory/manager.py`

输出：

1. 原始提取出的节点/边数量
2. 节点类型分布
3. 实际写入的若干条边样本
4. 每条样本边的 `edge_type / node_a_type / node_b_type / clarity_ab / clarity_ba`

#### C. 图检索

`backend/src/memory/retrieval.py`

输出：

1. 检索起点节点
2. 图召回候选摘要
3. 合并后的候选节点
4. 最终自然语言上下文预览

## 跑测方式

本轮先做定向 smoke，不直接启动长时间 7 天压测。

计划：

1. 临时 SQLite + 临时向量层
2. 执行 `StateManager.cold_start()`
3. 用 Dummy LLM 模拟一次“事件提取入图”，并显式复用熟人人物节点
4. 从熟人人物起点执行一次 `retrieve()`
5. 检查：
   - 是否出现诊断日志
   - 是否能从人物回忆到事件
