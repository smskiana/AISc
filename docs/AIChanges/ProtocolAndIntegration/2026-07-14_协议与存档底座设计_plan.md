# 协议与存档底座设计方案

> 执行记录: [execution.md](2026-07-14_协议与存档底座设计_execution.md)

## 需求理解

本轮建立协议与存档的长期工程口径：Unity 权威保存单机游戏世界数据，Python 只权威保存 AI 记忆及必要索引；两端物理存储分离，通过同一个 `checkpoint_id` 形成一致的逻辑存档。

## Workstream / Roadmap

- Workstream: `ProtocolAndSave`
- Roadmap item: 正在进行 / 协议与存档底座设计
- 相关 ADR: 新增 ADR-0006
- 工程口径变化: Python 不再作为完整游戏存档权威，现有世界状态表进入兼容迁移期

## 七批实施顺序

1. 定稿状态所有权、协议与存档设计、ADR 和工作流入口。
2. 实现双端协议公共层、兼容解析与握手试点。
3. 将 Python 完整存档收缩为原子化记忆检查点。
4. 实现 Unity 主存档 DTO、仓储、迁移与协调器。
5. 接通双端保存 / 读档事务协议。
6. 实现重连握手、世界快照与临时状态清理。
7. 迁移现有后端世界状态权威并完成验证与文档回写。

## 设计原则

1. Unity 保存玩家、时间、天气、NPC 世界事实、背包、任务和经营状态。
2. Python 保存记忆节点、关系边、向量索引及记忆检索所需元数据。
3. Python 可缓存 Unity 世界事实用于推理，但缓存不能反向覆盖 Unity 权威状态。
4. 对话流、移动过程、LLM 请求、UI、动画和临时锁不进入长期存档。
5. 保存与加载均使用 prepare / commit / abort，任何失败保留上一份完整检查点。
6. 新协议先兼容旧扁平 JSON，再按消息族逐步迁移。

## 风险点

1. 现有 Python SQLite 同时包含记忆和世界状态，不能直接按文件整体复制后宣称为纯记忆存档。
2. SQLite 与 LanceDB 必须形成同一记忆检查点，不能只保证单个文件成功。
3. Unity `JsonUtility` 对动态对象支持有限，新 envelope 需要明确 DTO 或受控 JSON 解析边界。
4. 旧存档迁移必须遵循“预迁移 -> schema/index -> 普通迁移”的既有错误预防规则。
5. `ApplicationRuntime` 和 Unity 消息路由已过重，协议与存档逻辑必须拆出独立职责。

## 验证方式

1. 文档和 ADR 互相引用且功能索引可达。
2. Python 单元测试与编译检查。
3. Unity MCP 编译、Console 和测试检查。
4. 双端保存、加载、失败回滚、重复请求、断线重连端到端验证。
