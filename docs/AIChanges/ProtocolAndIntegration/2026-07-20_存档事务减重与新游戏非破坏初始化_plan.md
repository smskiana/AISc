## 需求理解

1. 移除“开始新游戏会永久删除全部 Unity 存档和 Python 记忆检查点”的临时方案及其测试口径。
2. 开始新游戏必须保留所有历史槽位，且这些槽位之后仍可按原 `slot_id + checkpoint_id` 加载。
3. 新游戏仍须获得干净的当前运行态：不能继承上一局已加载的记忆表、LanceDB 工作副本、日程快照或同日幂等缓存。
4. 同时修正当前存读档事务的高风险失败语义，并对已过重的 `UnitySaveService` 减重，为后续玩法状态扩展保留稳定 seam。
5. 本轮不改变 ADR-0006 的权威边界：Unity 仍权威保存世界事实，Python 仍权威保存 AI 记忆，两端仍以 `checkpoint_id` 关联逻辑存档。

## 当前问题

### 1. 新游戏入口具有破坏性副作用

`StartMenuController.BeginNewGame` 当前调用 `UnitySaveService.PurgeAllForNewGame`。后端 `NewGameBackendPurgeService` 删除全部记忆 checkpoint，Unity 收到成功响应后删除整个 `SaveData`。该行为把“初始化一局新游戏”和“永久删除玩家全部存档”错误地绑定在同一个接口中。

### 2. 读档失败错误复用了保存 abort

Unity 的统一 `Abort` 在读档失败时也发送 `memory_checkpoint_abort`。Python `MemoryCheckpointService.abort` 会在 checkpoint ID 匹配时删除正式 checkpoint 并尝试恢复 `.previous`；已 finalize 的槽通常没有 `.previous`，因此读档应用失败存在删除正式记忆 checkpoint 的风险。

### 3. 读档不是跨端原子切换

Python 在 `memory_checkpoint_load` 响应前已经替换运行中的 SQLite 记忆表和 LanceDB，Unity 随后才校验并应用世界存档。Unity 失败时无法把 Python 运行态恢复到读档前状态。

### 4. 协调 facade 过重且共享可变字段可互相覆盖

`UnitySaveService` 同时承担保存、读档、删除、新游戏清理、世界快照、协议分发和 UI 事件通知。Save/Load 受 `SaveCoordinator` 约束，但 Delete 和 Purge 可覆盖共享 `_requestId`、`_slotId` 及 pending 字段。

### 5. 目录提交缺少进程重启恢复

Unity 与 Python 都采用 `final -> .previous`、`pending -> final` 的目录移动。进程在两次移动之间退出时，初始化过程不会自动恢复 `.previous` 或处理残留 `.pending`。

## 目标接口与模块边界

### Unity facade

保留 `UnitySaveService` 作为调用方唯一入口，但接口只表达玩家意图：

- `Save(request)`
- `Load(slotId)`
- `Delete(slotId)`
- `BeginNewGame()`
- `ListSaves()` / `Rename(...)`

调用方不得再编排 prepare、commit、abort、运行态清理或协议消息阶段。

### Unity 内部模块

1. `SaveOperationController`：持有唯一 active operation context，统一拒绝 Save、Load、Delete、NewGame 的并发冲突。
2. `SaveTransactionHandler`：封装保存 prepare / commit / finalize / rollback。
3. `LoadTransactionHandler`：封装读档校验、跨端 prepare、Unity 应用、commit / rollback。
4. `SaveProtocolRouter`：按 envelope type 路由到当前 operation；未知、迟到或 request ID 不匹配的响应不得改变状态。
5. `WorldSnapshotPublisher`：独立负责握手和读档后的 Unity 权威世界快照。
6. `UnitySaveRepository`：继续作为文件系统 adapter，并增加原子元数据写入与启动恢复。

以上是职责边界，不要求为每一项机械增加浅层包装类。实现时优先保持调用接口小、内部行为深，并以可替换 adapter 支撑故障注入测试。

### Python 内部模块

1. 将 `NewGameBackendPurgeService` 替换为非破坏性的 `NewGameRuntimeInitializer`。
2. 初始化新游戏只重建当前工作数据，不调用 `MemoryCheckpointService.purge_all()`，不删除任何 `slot_*` checkpoint。
3. `MemoryCheckpointService` 增加读档运行态 prepare / commit / rollback 能力；正式 checkpoint 在整个读档过程中只读。
4. 协议路由从 `GameRuntime` 的长分支中提取存档 handler 映射，返回稳定失败码。
5. `purge_all()` 若仍有产品需求，只能留给显式“删除所有存档”维护命令；没有明确入口时从生产协议移除，但不可复用于新游戏。

## 新游戏语义

1. 玩家点击新游戏后，Unity 发送新的非破坏性请求，例如 `new_game_runtime_initialize`。
2. Python 先生成可回滚的当前运行态备份，然后清空当前记忆业务表、重建空 LanceDB 工作目录、删除可重建日程快照并清空同日幂等缓存。
3. Python 初始化成功后进入既有新世界准备流程；失败则恢复初始化前运行态并留在开始界面。
4. Unity 不调用 `UnitySaveRepository.PurgeAll()`；历史槽位、截图、manifest、`.previous` 和 Python 正式 checkpoint 均不因新游戏而删除。
5. 新游戏尚未保存时，当前 checkpoint ID 为空；首次保存时生成新的 checkpoint ID，并按正常双端保存事务写入玩家选择的槽位。
6. 新游戏初始化完成后重新读取存档列表，结果必须与初始化前一致。

## 读档事务语义

1. Unity 在请求 Python 前先完整读取、摘要校验和迁移目标世界 DTO，但暂不修改当前世界。
2. Python `load_prepare` 校验正式 checkpoint，并构造隔离的候选 SQLite/LanceDB 运行副本，不替换当前运行态。
3. Unity 收到 prepared 后，在可回滚的世界应用边界内切换世界和背包状态。
4. Unity 应用成功后请求 Python `load_commit`，Python 原子切换候选运行副本；双方成功后再发布世界快照和 LoadFinished。
5. 任一步失败均执行对应 load rollback。读档失败绝不调用保存用 checkpoint abort，绝不修改或删除正式 checkpoint。
6. 若 Unity 当前状态无法提供可靠的内存回滚，则读档失败必须进入受控致命退出或重新加载场景，不能带着半应用世界继续游戏；具体策略在执行前依据现有 `GameStateStore` 能力确认。

## 磁盘一致性与恢复

1. Unity 和 Python adapter 在初始化时检查 `final`、`.previous`、`.pending`/`.loading` 的组合并执行确定性恢复。
2. manifest 等正式元数据使用同目录临时文件写入、flush、replace；不得直接截断覆盖正式文件。
3. 每个 operation context 持久化 operation ID、slot ID、checkpoint ID 和阶段，响应处理必须同时验证 request ID 与 operation identity。
4. 恢复策略必须幂等：重复启动或重复响应不能再次删除、再次回滚或跨槽操作。

## 分阶段执行顺序

### 阶段 A：先锁定非破坏性契约

1. 先写失败测试：开始新游戏前后 Unity 槽位列表和 Python checkpoint 列表完全一致。
2. 把旧 `NewGameBackendPurgeTests` 替换为运行态初始化测试；移除“应永久删除全部 checkpoint”的新游戏测试口径。
3. 保留 `MemoryCheckpointService.purge_all` 自身的纯 adapter 测试，仅当该显式能力仍被保留；它不得再标注为新游戏行为。

### 阶段 B：修复读档数据破坏与跨端原子性

1. 立即分离 save abort 与 load rollback，保证任何 load failure 都不会调用 checkpoint abort。
2. 引入 load prepare / commit / rollback 协议和 Python 隔离候选运行副本。
3. 在 Unity 应用失败、Python commit 失败、重复响应和超时路径下验证双方仍保持读档前运行态。

### 阶段 C：减重与统一操作上下文

1. 让 `UnitySaveService` 退回 facade 定位。
2. 以单一 operation context 串行化 Save、Load、Delete、NewGame。
3. 拆出协议路由和世界快照发布；删除共享 pending 字段之间的隐式耦合。
4. 提取 Python 存档协议 handler，降低 `GameRuntime` 分支复杂度。

### 阶段 D：仓储崩溃恢复

1. 增加 Unity/Python 启动恢复规则和原子元数据写入。
2. 用故障注入覆盖每个目录移动及文件 replace 前后的中断点。
3. 确认历史 schema migration、截图和无限手动槽行为不回归。

### 阶段 E：文档与索引回写

1. 删除所有“新游戏永久清空全部存档”的当前工程口径。
2. 文档明确区分“运行态初始化”“删除单槽”“显式删除全部存档”。
3. execution 记录最终模块和协议名称；更新 codebase-memory 索引及 `docs/AIChanges/codebase-memory-mcp_更新.md` 的更新时间。

## 风险与约束

1. SQLite 记忆表与 LanceDB 必须作为同一运行态切换；只回滚其中一端仍会产生语义不一致。
2. 新游戏清空当前记忆业务表时不得误清 Unity 权威世界表之外的后端必要配置或 schema。
3. 旧 `memory_checkpoints_purge_all` 属于危险兼容入口，执行时应优先移除；若外部调用尚未迁移，必须显式标记 deprecated，不能继续从新游戏触发。
4. 本次不顺带解决通用后端自动重连；但事务中断必须落到可恢复或明确失败状态。
5. 若执行中发现 `GameStateStore` 无法提供可靠世界回滚，必须停止并在本 plan 中补充受控场景重载方案，不得用局部字段补偿伪装原子性。
6. 不涉及场景、Prefab、SerializeField 或 UI 资产连线；若实际实现发现必须修改资产层，执行会话应先阅读 Unity MCP 规则并使用 Unity MCP。

## 验收标准

1. 点击开始新游戏不会删除或改写任何历史 Unity 槽位、截图、manifest 或 Python 正式 checkpoint。
2. 新游戏获得干净运行态，并能首次保存为新 checkpoint；之后仍能加载新游戏前的任意旧槽。
3. 任意读档失败路径都不会调用保存 abort，不会删除或改变目标正式 checkpoint。
4. 读档完成事件只在 Unity 世界和 Python 记忆运行态均成功切换后发出。
5. 任一时刻只允许一个存档相关 operation；所有响应按 operation identity 关联。
6. `UnitySaveService` 不再直接实现所有协议类型和事务细节，新增持久化域时无需继续扩展单个大型消息分支。
7. Unity 与 Python 在模拟崩溃后的目录组合上能够确定、幂等恢复。
8. 聚焦自动化、Unity 编译和双进程冒烟达到最低门禁；最终结论由独立 test record 给出。
