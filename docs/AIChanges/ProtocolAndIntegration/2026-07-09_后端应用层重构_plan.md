# 后端应用层重构 — 设计方案

## 背景

当前后端核心问题不是“功能缺一个两个”，而是**主流程的架构承载方式已经接近上限**：

1. `main.py` 同时承担：
   - 生命周期初始化
   - 全局依赖容器
   - 午夜编排
   - 玩家对话会话管理
   - WebSocket / HTTP 桥接消息路由
   - 玩家记忆提取
2. `PromptBuilder` 通过 `sys.modules` 反向读取 `main.behavior` 内部状态
3. 广播队列和连接状态以模块级全局变量存在
4. 业务编排层与传输层、基础设施层混在一起

这类结构短期能跑，但后续继续加：
- 玩家选项生成
- 强制回忆
- 多会话
- 后台调试接口
- 存档恢复完整回放

都会继续向 `main.py` 堆积。

## 重构目标

本次不推翻玩法设计，不重写记忆系统模型，而是把后端整理成更清晰的分层：

### 目标分层

1. **Application 层**
   - 生命周期编排
   - 消息路由
   - 玩家对话会话
   - 午夜流程
   - 广播与轮询桥接

2. **Domain / Existing Services 层**
   - `BehaviorEngine`
   - `MemoryManager`
   - `RetrievalEngine`
   - `NpcDialogueManager`
   - `StateManager`

3. **Infrastructure 层**
   - `SQLiteClient`
   - `LanceDBClient` / `VectorStore`
   - `LLMClient`
   - `SaveManager`

## 重构原则

1. **保留已有行为逻辑**：不重写 NPC 决策规则，不重做记忆模型
2. **薄入口**：`main.py` 只保留 FastAPI endpoint 和最薄委托
3. **消除模块级全局耦合**：把业务状态收进运行时对象
4. **显式依赖注入**：`PromptBuilder`、`BehaviorEngine` 等通过 setter / provider 注入，而不是偷全局
5. **兼容现有 Unity 协议**：不改消息 type，不要求前端同步重做

## 计划新增模块

### `backend/src/application/services.py`
- 定义 `AppServices` 数据类
- 显式持有 sqlite / vector store / managers / behavior / dialogue manager

### `backend/src/application/message_bus.py`
- 管理：
  - 当前活跃 WebSocket
  - HTTP 轮询队列
  - 广播双写
- 替代 `main.py` 中 `_message_queue` + `active_ws`

### `backend/src/application/dialogue_service.py`
- 管理玩家与 NPC 的对话会话状态：
  - `start_dialogue`
  - `handle_player_choice`
  - `end_dialogue`
  - 玩家记忆提取
- 替代 `main.py` 中 `_active_dialogues` 和对话相关函数

### `backend/src/application/runtime.py`
- 运行时主编排器 `GameRuntime`
- 职责：
  - 初始化全部服务
  - 绑定时钟回调
  - 处理午夜流程 / 苏醒流程
  - 处理 `GAME_START` / `SAVE_REQUEST` / `LOAD_REQUEST` / `FAST_FORWARD`
  - 对外提供统一 `handle_message()`

## 具体方案

### A. `main.py` 退化为薄入口

保留内容：
- FastAPI app 创建
- REST 路由定义
- WebSocket endpoint

移除/下放内容：
- 全局服务初始化
- 午夜逻辑
- 玩家对话逻辑
- GAME_START / SAVE / LOAD / FAST_FORWARD / PLAYER_MOVE 的业务处理

这些统一委托给 `GameRuntime`

### B. 把运行时状态从模块级变量收进 `GameRuntime`

包括：
- sqlite
- vector store
- state manager
- prompt builder
- memory manager
- retrieval
- evolution
- save manager
- behavior engine
- npc dialogue manager

外部只通过 `runtime.services` 访问

### C. 对话状态独立成 `PlayerDialogueService`

好处：
- 会话状态不再散落在入口模块
- 后续更容易加“玩家选项生成”“强制回忆”“中断恢复”
- 记忆提取逻辑和对话终止逻辑归一处

### D. 消息广播独立成 `MessageBus`

好处：
- WebSocket 直推和 HTTP 轮询队列逻辑不再到处复制
- 后续可扩到多客户端 / 调试观察端

### E. 消除 `PromptBuilder` 对 `main` 的反向依赖

改为：
- `PromptBuilder` 增加 `set_plan_provider(...)`
- 由 `GameRuntime` 在初始化完成后注入 `behavior.get_remaining_plan_summary`

### F. `BehaviorEngine` 提供公开只读计划摘要接口

新增轻量公共方法：
- `get_remaining_plan_summary(npc_id, limit=3) -> str`

`PromptBuilder` 只依赖这个接口，不读取 `_plans` 私有字段

## 验证计划

1. Python 语法编译通过
2. `backend/run.py` 仍可正常导入 `src.main:app`
3. `main.py` 中 endpoint 仍存在
4. `GAME_START`、`DIALOGUE_START`、`PLAYER_CHOICE`、`DIALOGUE_END` 路由都委托到新运行时
5. `PromptBuilder` 不再通过 `sys.modules.get("src.main")` 读取计划

## 预期结果

重构后效果应为：

1. `main.py` 变薄，入口职责明确
2. 运行时状态集中到 `GameRuntime`
3. 玩家对话流程成为独立服务
4. 广播与轮询桥接成为独立基础设施对象
5. `PromptBuilder` 和 `BehaviorEngine` 之间从“偷全局”改为“显式注入”

## 风险说明

用户已明确表示不需要优先考虑风险，因此本次按“最佳方向”推进。
但仍保持以下克制：

1. 不改 Unity 协议
2. 不改数据库 schema
3. 不改核心行为规则
4. 不重写记忆模型，只重构编排层和耦合关系
