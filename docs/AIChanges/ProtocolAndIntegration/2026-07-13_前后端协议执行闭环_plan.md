## 需求理解

当前后端能够向 Unity 推送 NPC 行为和对话内容，但缺少执行结果回报，导致后端把“已经发出”误当成“已经完成”。本轮先建立移动与玩家对话的最小双向闭环，并解除对话生成期间 WebSocket 无法继续接收消息的问题。

- Workstream: `Navigation`、`NpcBehavior`、`Dialogue`、`FrontendArchitecture`
- Roadmap item: 正在进行 / 协议与存档底座设计
- 相关 ADR: 无；本轮不改变 ADR-0002 typed path 口径
- 工程口径变化: 后端不再在发送移动行为时提前确认 NPC 当前位置

## 本轮方案

### 1. 消息关联

- 后端发出的 `NPC_BEHAVIOR` 增加 `request_id`。
- Unity 回传 `NPC_ACTION_RESULT`，携带同一 `request_id`。
- 结果状态固定为 `succeeded`、`failed`、`cancelled`、`timeout`、`superseded`。

### 2. NPC 行为结果闭环

- Unity 移动行为完成后回传实际结果、目标 location 和实际 location。
- NPC 不存在、行为体缺失或目标非法时也必须回传失败，不能只写本地日志。
- 后端只在收到成功结果后更新 `npc_states.current_location`。
- 后端保存待执行请求，用于拒绝已经被新行为覆盖的旧结果。

### 3. 玩家对话准备握手

- Unity 发送 `DIALOGUE_START`，语义调整为“请求准备对话”。
- 后端登记准备态并返回 `DIALOGUE_PREPARED`，不在此时调用 LLM。
- Unity 确认本地 NPC、距离和现场快照后发送 `DIALOGUE_READY`。
- 后端收到 READY 后才建立正式会话、暂停时间并异步生成对话内容。
- `DIALOGUE_READY` 的耗时处理通过后台任务执行，WebSocket 接收循环继续监听后续消息。

### 4. 兼容与边界

- 保留现有 `DIALOGUE_START` 字段，新增字段采用可选解析。
- 本轮不实现自动走近 NPC；当前玩家自由移动和距离校验规则保持不变。
- NPC-NPC 社交沿用现有前端等待显示逻辑，本轮只给其后续接入预留 request/result 结构，不同时重写社交状态机。
- 重连全量快照、协议版本协商、消息 sequence 和持久化幂等放到后续阶段。

## 风险点

1. 同一 NPC 的旧移动结果晚于新行为到达，必须用 `request_id` 判断是否过期。
2. 对话准备后玩家取消或 NPC 状态变化，需要允许 `DIALOGUE_END` 清理准备态。
3. 后台对话任务仍会向 WebSocket 发送流式消息，断线异常必须在服务内收口。
4. Unity `JsonUtility` 不支持动态字典，协议 DTO 继续使用明确字段。

## 验证方式

1. Python 单元测试覆盖行为成功、失败、旧结果和对话 PREPARED/READY。
2. Python 编译检查。
3. Unity C# 项目命令行编译；如受 Unity 生成工程或插件环境限制，记录具体阻塞。
4. 静态检查所有新增或实质修改函数的功能注释。
