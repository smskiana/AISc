# 玩家传送与场景入口设计方案

> 执行记录: [execution.md](2026-07-14_玩家传送与场景入口_execution.md)

## 需求理解

玩家靠近现有 AI 导航传送入口时显示“按 F 前往”，按下 F 后传送到配对出口。玩家与 NPC 必须复用同一套 `NavigationTeleportPoint`、`SceneAnchor` 和 `location_id` 语义，但玩家不经过 NPC 的自动寻路执行器。

## 当前工程口径

1. Workstream: `docs/Workstreams/Navigation/README.md` 的“玩家传送排期口径”。
2. ADR: `docs/DecisionRecords/ADR-0002-navigation-typed-path.md`，传送点规则归 `NavigationTeleportPoint` 管理，不能把玩家传送塞入 `AStarMovementProvider`。
3. 资产事实: `Town_Main` 的 `go_tranport_st` 下已有 8 组双端传送点，NPC 导航已使用这些组件。
4. Unity 资产修改遵守 `docs/DesignDocs/UnityMCPUsageRules.md`，通过 Unity MCP 完成挂载、UI 创建和引用连线。

## 方案

1. 新增 `PlayerTeleportController`：选择玩家触发半径内最近的可用 `NavigationTeleportPoint`，驱动提示显示，响应 F 键并执行刚体位置传送；传送后增加短冷却，避免落点反向触发。
2. 新增 `PlayerLocationResolver`：统一从 `SceneAnchorRegistry` 解析玩家地点，并在传送完成后立即通过 `GameManager.SendPlayerMove` 同步。
3. 新增 `PlayerTransitionView`：只负责现有 TMP 提示节点的文本和显隐。
4. `PlayerController` 保留移动和对话职责，改为委托 `PlayerLocationResolver` 做周期地点同步，不接收传送分支。
5. 将 `GameManager` 的 F 快进测试键改为 F8，释放 F 作为场景交互键。
6. 使用 Unity MCP 在 `Town_Main` 创建提示 Canvas，挂载并连线三个玩家侧组件，复用现有传送点，不复制触发区域。

## 风险与控制

1. F 键现有测试快捷键冲突：迁移至 F8。
2. 出口仍处于传送半径：用输入释放门闩和短冷却共同防止连续反向传送。
3. 多入口半径重叠：按距离选择最近入口，避免依赖查找顺序。
4. 游戏未就绪或对话中：隐藏提示并禁止传送。
5. 场景引用丢失：编译完成后再通过 Unity MCP 挂载和连线，并检查 Console。

## 验证计划

1. Unity 编译与 Console 无新增错误。
2. EditMode/PlayMode 可用测试回归。
3. Play Mode 验证靠近/离开提示、F 传送、双向入口、防连跳及地点同步。
