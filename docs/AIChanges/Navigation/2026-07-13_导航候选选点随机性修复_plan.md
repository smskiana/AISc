# 导航候选选点随机性修复方案

## Workstream

Navigation / FrontendArchitecture

## Roadmap item

NAV / FEA

## 相关 ADR

1. `docs/DecisionRecords/ADR-0002-navigation-typed-path.md`

## 需求理解

检测日志显示 `SceneAnchor` 每批候选点本身是随机的，但 `AStarMovementProvider.MoveToLocation()` 总是选中 `candidate#0`，导致最终移动目标固定。根因是 `SceneAnchor.SampleCandidatePoints()` 先把区域中心或 Anchor 中心写入候选列表，再补随机点；移动器按顺序选择第一条可达路径，因此固定中心点长期抢占第一优先级。

## 方案思路

1. 只调整候选列表生成顺序，不改 A*、移动器或传送逻辑。
2. `SampleCandidatePoints()` 先生成随机采样点，让 `candidate#0` 进入随机分布。
3. 把区域中心或 Anchor 中心作为兜底点追加到列表末尾，保留稳定可达兜底。
4. 若没有自定义区域，仍使用旧圆形偏移采样；半径为 0 时会自然退化为中心点。

## 风险点

1. 随机候选可能落在不可走区域，但移动器已有逐候选可达性检查。
2. 如果某个 location 的随机区域配置过宽，可能增加候选失败日志；中心点兜底可以降低完全失败概率。
3. 本轮不改变 `AStarMovementProvider` 第一条可达即选中的策略，避免继续加重移动执行层。

## 验证方式

1. `dotnet build AISc.sln --no-restore`。
2. Unity MCP 刷新脚本并校验 `SceneAnchor.cs`。
3. Play 模式观察：
   - `候选点批次 samples=#0` 不再长期等于固定中心点。
   - `选中候选 previousDelta` 应经常大于 `0.000`。
