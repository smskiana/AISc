# NPC 移动途中位置语义设计方案

> 执行记录: [execution.md](2026-07-13_NPC移动途中位置语义_execution.md)

## 需求理解

NPC 接到移动指令后，不能继续被系统视为仍在出发地区，也不能提前进入目标地区。移动期间应进入独立的特殊位置，并让 Prompt 明确表达“从 A 到 B 的途中”。

## Workstream / Roadmap

- Workstream: `Navigation`、`NpcBehavior`、`Dialogue`
- Roadmap item: 导航与移动闭环验证 / 协议与存档底座设计
- 相关 ADR: ADR-0002；不改变 typed path 执行口径
- 工程口径变化: `current_location` 在移动期间使用内部 transit 标记，不再保留出发地

## 方案

1. 为 `npc_states` 增加 `movement_origin`、`movement_target`、`movement_status`，兼容旧数据库迁移。
2. 移动行为发出时：
   - 保存出发地与目标地。
   - `movement_status = moving`。
   - `current_location = __transit__:<npc_id>:<request_id>`。
3. 移动成功时提交实际地点并清空移动字段；失败或取消时恢复前端实际位置或原出发地。
4. proximity 层将任何 transit 位置判定为：不属于任何 zone、不与任何位置相同或邻近。
5. 玩家对话 Prompt 将 transit 状态渲染为“从 A 到 B 的途中（不属于任何地区）”。
6. transit 状态不加载出发地或目标地的现场物件、气味和局部感知，避免把途中误写成某个店内现场。

## 风险

1. 新移动覆盖旧移动时，不能把旧 transit 标记当成真实出发地。
2. 失败结果必须退出 transit，避免 NPC 永久不属于任何地区。
3. 存档发生在移动途中时会保存 transit 状态；后续重连恢复策略仍需单独设计。
4. Unity 物理移动继续使用行为消息中的目标 location，不使用 transit ID 寻路。

## 验证

1. 测试 transit 与自身、其他 transit、正常地点均不相同且不邻近。
2. 测试移动发出、成功、失败的数据库状态变化。
3. 测试 Prompt 位置文字和途中现场感知。
4. Python 测试、编译、Unity C# 编译和项目规范检查。
