# 导航传送点组件化方案

## 背景

上一轮已把传送判断从 `AStarMovementProvider` 移走，但实际权威仍落在 `NavigationGridAsset.NavigationTeleportLink` 数据行上。这个边界仍不够直观：场景里的 `go_tranport_st` / 具体传送点 GameObject 没有组件承载规则，策划调参时仍要进资产列表改坐标和半径。

## 目标

1. 场景传送点 GameObject 通过组件拥有自身规则。
2. `NavigationGridAsset` 只保存从组件同步来的运行时副本。
3. `NavigationGridSpriteBaker` 负责从 Root 收集传送点组件，并写入目标导航资产。
4. A* 和移动执行层保持不变：A* 读资产，移动器只执行 typed path。

## 涉及文件

1. 新增 `Assets/Scripts/Navigation/NavigationTeleportPoint.cs`
2. 修改 `Assets/Scripts/Navigation/NavigationGridAsset.cs`
3. 修改 `Assets/Scripts/Navigation/NavigationGridSpriteBaker.cs`
4. 更新 `Assets/Scripts/Index.md`
5. 更新本轮执行记录和 handoff

## 设计

### `NavigationTeleportPoint`

挂在场景中的传送点 GameObject 上，字段包括：

1. `id`
2. `enabled`
3. `pairedPoint`
4. `bidirectional`
5. `triggerRadius`
6. `entranceMatchRadius`
7. `exitMatchRadius`
8. `cost`

组件通过 `TryBuildLink()` 生成 `NavigationTeleportLink`。

### `NavigationGridSpriteBaker`

新增：

1. `_teleportPointsRoot`
2. `_teleportPoints`
3. `CollectTeleportPointsFromRoot()`
4. `SyncTeleportLinksToAsset()`

`BakeToAsset()` 完成网格烘焙后同步传送 links。

## 风险点

1. 旧 `NavGrid_Main.asset` 里已有手工配置的 links；同步组件前不能无提示覆盖，除非烘焙器配置了传送点 root 或显式列表。
2. 双向 link 只需要一个起点组件生成，成对两个组件如果都 bidirectional 可能生成重复 link；第一版用 id 去重并 warning。
3. 需要 Unity Play 模式验证现有 `go_tranport_st` 是否已经挂组件并正确配对。
