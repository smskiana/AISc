# 前端职责框架整改方案

## 背景

本轮复审由 NPC 导航虚空移动触发。当前前端已经能跑通 Unity + 后端联调，但近期补丁暴露出职责边界开始失衡：

1. `AStarMovementProvider` 同时负责 location 候选点选择、起点修正、路径执行、传送段反推、日志输出和移动失败兜底。
2. `AStarPathfinder` 在搜索时知道传送边存在，但输出时只返回 `Vector2` waypoint，导致传送语义丢失。
3. `NavigationTeleportLink` 保存传送配置，但入口匹配容差、出口匹配容差等规则被写在移动执行层。
4. `NpcEntity.CurrentLocation` 在移动开始时就改成目标 location，移动失败或中途卡住时会造成语义位置和物理位置分离。
5. `NpcBubbleManager` 为了解决隔空聊天，直接驱动双方 `MoveToLocation()`，与普通行为移动共用通道，容易互相打断。
6. `GameManager` 同时承担状态仓库、协议发送、消息回调、对话 UI 编排和事件分发，后续继续加功能会变重。

## 复审结论

### 当前高风险耦合点

1. **传送语义丢失**
   - `NavigationGridAsset.CollectTeleportExits()` 会把传送出口作为邻居加入 A*。
   - `AStarPathfinder` 搜索后只输出世界坐标列表。
   - `AStarMovementProvider` 只能靠 `from/to` 坐标反猜“是不是传送段”。
   - 结果：容差被迫写在移动器里，且传送点自己无法决定匹配规则。

2. **移动请求没有结果语义**
   - `IMovementProvider.MoveToLocation()` 只有 `onComplete`，没有 `success/failure/cancelled`。
   - `NpcEntity` 无法区分“走到了”“取消了”“找不到路”“被新请求打断”。
   - `CurrentLocation` 过早更新，让后续社交和玩家点击对话拿到不可靠 location。

3. **社交靠近是气泡层临时接管移动**
   - `NpcBubbleManager` 收到 `NPC_SOCIAL_ACTION` 后让双方移动。
   - 气泡队列又根据距离等待。
   - 该逻辑没有明确“会合任务 / 普通行为移动”的优先级，容易反复下发移动。

4. **前端总线和 UI 绑定偏紧**
   - `GameManager` 直接拼 JSON、处理消息、更新全局状态、调用对话 UI。
   - 短期可用，但不利于后续引入更多前端状态或本地验证。

5. **资产配置和运行规则混杂**
   - `NavigationGridAsset` 既是数据资产，又提供较多运行时查询和调试图生成。
   - `NavigationGridSpriteBaker` 只按 bounds 烘焙，当前可用，但未来 alpha 采样 / 显式可走区会继续加重。

## 目标框架

### 分层原则

1. **协议层只负责收发**
   - `WebSocketClient`：连接、收包、发包。
   - `MessageRouter`：消息 type 到强类型事件的分发。
   - 后续可新增 `GameCommandSender`，统一生成 JSON，减少 `GameManager` 手写字符串。

2. **应用状态层只负责状态和事件**
   - `GameManager` 暂时保留为前端 Facade，但逐步减重。
   - 后续拆出：
     - `GameStateStore`：时间、玩家位置、NPC 状态。
     - `DialogueSessionController`：玩家正式对话生命周期。

3. **NPC 层只负责实体和行为落地**
   - `NpcEntity`：只维护实体身份、显示状态、移动状态和气泡引用。
   - `NpcSpawner`：只负责生成、注册、查找 NPC。
   - 后续拆出 `NpcBehaviorApplier`：把 `NpcBehaviorMsg` 转成移动 / 动画 / 状态更新。

4. **导航层必须保留语义路径**
   - `NavigationTeleportLink`：决定自己的触发半径、入口匹配半径、出口匹配半径、成本和双向规则。
   - `AStarPathfinder`：输出 typed path，不再只输出裸 `Vector2`。
   - `AStarMovementProvider`：只执行 typed path，不反推传送。
   - `NavigationDebugLog`：集中输出路径摘要和异常段。

5. **社交层只发会合意图，不直接抢移动细节**
   - `NpcBubbleManager` 保留气泡队列。
   - 后续拆出 `NpcSocialRendezvousController`，负责双方去同一个 meeting location、等待靠近、超时取消。

## 第一阶段整改范围

第一阶段只收敛导航链路，目标是修掉当前真实问题，同时避免继续扩大改动面。

### 新增文件

1. `Assets/Scripts/Navigation/NavigationPath.cs`
   - 定义 `NavigationPath` / `NavigationPathSegment` / `NavigationPathSegmentType`。
   - Segment 类型至少包括：
     - `Walk`
     - `Teleport`
   - Teleport segment 携带：
     - `LinkId`
     - `FromWorld`
     - `ToWorld`
     - `ResolvedExitWorld`

2. `Assets/Scripts/Navigation/NavigationDebugLog.cs`
   - 集中输出 `[AStarMove:npc/name]` 日志。
   - `AStarMovementProvider` 不再拼大段日志。

### 修改文件

1. `Assets/Scripts/Navigation/NavigationGridAsset.cs`
   - `NavigationTeleportLink` 增加 Inspector 参数：
     - `_entranceMatchRadius`
     - `_exitMatchRadius`
   - `NavigationTeleportLink` 提供规则方法：
     - `GetEntranceMatchRadius(float cellSize)`
     - `GetExitMatchRadius(float cellSize)`
   - `NavigationTeleportExit` 扩展为带 link 语义的边信息：
     - `LinkId`
     - `EntranceWorld`
     - `ExitWorld`
     - `ResolvedExitCell`
     - `ResolvedExitWorld`
     - `IsReverse`

2. `Assets/Scripts/Navigation/AStarPathfinder.cs`
   - A* 节点记录保存 `CameFromTeleport` 或 `CameFromEdge`。
   - 新增 `TryFindNavigationPath(...)`，输出 `NavigationPath`。
   - 旧 `TryFindPath(...)` 保留兼容，内部可从 typed path 展平成 waypoint。
   - 路径简化时保留 teleport segment 边界，不再靠 `IsGraphJump()` 猜测。

3. `Assets/Scripts/Navigation/AStarMovementProvider.cs`
   - `MoveAlongPath()` 改为执行 `NavigationPathSegment`。
   - 遇到 `Teleport` 直接使用 segment 的 `ResolvedExitWorld`。
   - 删除 `IsTeleportSegment()` / `MatchesTeleportDirection()` 这类反推逻辑。
   - location 候选点筛选仍暂留在本类，第二阶段再拆到 planner。

4. `Assets/Scripts/Navigation/NavigationGridSpriteBaker.cs`
   - 第一阶段不改烘焙算法。
   - 若传送 link 新增字段需要默认值，不由 baker 写。

### 不在第一阶段做的事

1. 不重写 `GameManager`。
2. 不重写 `PortraitDialogueUI`。
3. 不改后端协议。
4. 不做 alpha 采样烘焙。
5. 不大规模迁移场景 / Prefab。

## 第二阶段建议

### NPC 移动状态收口

1. `IMovementProvider` 增加结果回调：
   - `MovementResult.Success`
   - `MovementResult.Failed`
   - `MovementResult.Cancelled`
2. `NpcEntity` 拆分：
   - `CurrentLocation`：最后确认到达的位置。
   - `TargetLocation`：正在前往的位置。
   - `LastMovementFailureReason`：最近失败原因。
3. `NpcEntity` 只有移动成功时更新 `CurrentLocation`。

### 行为落地拆分

1. `NpcSpawner` 保留 NPC 注册表职责。
2. 新增 `NpcBehaviorApplier`：
   - 判断 action 是否移动类。
   - 调用 NPC 移动。
   - 播放动作视觉。
   - 输出行为日志。

### 社交会合收口

1. 新增 `NpcSocialRendezvousController`。
2. `NpcBubbleManager` 只排气泡，不直接决定如何移动。
3. 社交动作先创建会合任务，双方到位后才允许气泡队列播放。

## 风险点

1. typed path 会改动 A* 回溯逻辑，必须保留旧 `TryFindPath()` 兼容调试工具和未来调用。
2. 传送 link 字段新增后，旧资产会使用默认值；需要确认默认值与当前 `triggerRadius=0.35` 兼容。
3. 如果当前 Play 模式仍有 NPC 卡在不可走区，需重启 Play 或增加第二阶段的“大范围救援吸附”。
4. 日志会变多，第一阶段保留开关；后续可改成 Inspector 配置。

## 验收标准

1. `dotnet build AISc.sln --no-restore` 通过，0 error。
2. Unity Console 编译 error 为 0。
3. 从 `police_box.desk -> street.crossroad` 日志出现 typed `Teleport` segment，不再出现入口到出口的普通长段。
4. 从 `bakery` / `flower_shop` 跨店社交到同一地点时，路径摘要中传送段由 pathfinder 输出，不由 movement provider 反推。
5. `AStarMovementProvider` 中不再存在 `MatchesTeleportDirection()`。
6. `NavigationTeleportLink` Inspector 暴露传送匹配半径，传送点配置自己决定容差。
