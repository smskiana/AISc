# 前端托管后端生命周期 — 设计方案

## 需求理解

当前开始界面启动后只检查后端，不会默认拉起 Python 后端；玩家需要手动点击“启动后端并连接”。同时由前端拉起的后端进程没有被 Unity 保存所有权，退出前端后 Python 后端会继续在后台运行。

本轮目标：

1. Unity 前端进入开始界面时默认尝试启动后端。
2. 如果后端原本已经在线，只连接，不接管也不关闭外部后端。
3. 如果后端由本次 Unity 前端拉起，则 Unity 退出 / 停止 Play 时关闭该后端进程。

## 修改范围

### Unity 脚本

- `Assets/Scripts/Core/StartMenuController.cs`
  - 默认启动链从“只检查”改为“检查并按需拉起”。
  - 应用退出时通知 launcher 关闭本轮拥有的后端进程。

- `Assets/Scripts/Core/StartMenuBackendLauncher.cs`
  - 保存由 Unity 拉起的 Python 进程引用。
  - 重复启动时复用已有 owned 进程。
  - 新增关闭 owned 后端进程的方法。

## 方案原则

1. 不改后端接口，不新增协议。
2. 不杀已有外部后端，只杀本次 `Process.Start()` 成功后记录下来的进程。
3. 不把进程管理塞回 `StartMenuController`，继续保留 `StartMenuBackendLauncher` 的职责边界。

## 风险点

1. 如果 Python 再自行派生独立子进程，普通 `Process.Kill()` 只能保证关闭记录到的主进程；当前 `backend/run.py` 直接运行 `uvicorn.run()`，按现状主进程即服务进程。
2. Unity 编辑器停止 Play 时 `OnApplicationQuit` / `OnDestroy` 的触发顺序可能不同，因此需要在两个入口都做幂等关闭。

## 验证计划

1. C# 脚本静态编译检查。
2. 确认 `StartMenuController.Start()` 默认调用自动启动。
3. 确认 launcher 不再 `using` 立即释放进程对象，并只关闭 owned 进程。
