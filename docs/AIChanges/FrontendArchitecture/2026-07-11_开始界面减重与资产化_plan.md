# 开始界面减重与资产化 — 设计方案

## 需求理解

现有 `StartMenuController.cs` 已膨胀到接近 700 行，并同时承担：

1. 视图创建
2. 视图显隐和列表渲染
3. 后端健康检查
4. Python 进程拉起
5. GameManager 事件编排

这已经同时违反：

1. 类减重与扩展性规则
2. Unity MCP 资产层优先规则
3. 用户对“View 必须真实存在于场景/资产层，不应在 Controller 里生成”的明确要求

本轮目标是在保持现有开始流程能力的前提下，完成两件事：

1. 把开始界面拆成清晰的控制层 / 视图层 / 编辑器构建层
2. 用 Unity MCP + 编辑器资产构建流程，把 `street_arcade_pixel_v1` 作为真实背景接入场景

## 已确认现状

1. Unity MCP 当前已恢复，实例为 `AISc@9db8baa7dcf9cfcd`
2. 当前场景 `Town_Main.scene` 中只有对话 Canvas `cvs_dialogue_dy`，没有开始界面资产
3. 当前 `StartMenuController.cs` 共 694 行，确实明显失衡
4. 背景图 `Assets/Art/Generated/PixelScenes/street_arcade_pixel_v1.png` 已导入为 Sprite，可直接绑定到 `UnityEngine.UI.Image`

## 方案原则

1. 不再在运行时创建开始界面 View 层级
2. `StartMenuController` 只负责流程编排、按钮事件和 GameManager 交互
3. 视图引用由 `StartMenuView` 持有，列表项由独立 View 预制体承载
4. 场景 UI 的创建、命名、组件挂载和引用连线交给 Editor Builder，再由 Unity MCP 执行

## 具体方案

### 1. 控制层减重

`StartMenuController` 收口为：

1. 订阅 / 解绑 `GameManager` 事件
2. 启动检查后端 → 连接 WS → 请求存档的流程状态机
3. 处理按钮点击后的业务动作
4. 调用 `StartMenuView` 刷新显示

不再负责：

1. 创建 Canvas / Button / InputField
2. 直接读图文件做背景 Sprite
3. 维护每个 UI 节点的构造细节

### 2. 视图层拆分

新增 `StartMenuView`：

1. 持有开始界面所有 SerializeField 引用
2. 负责按钮监听注册
3. 负责状态文案、显隐、交互开关
4. 负责基于 `StartMenuSaveButtonView` 预制体渲染存档列表

新增 `StartMenuSaveButtonView`：

1. 持有单个存档按钮与文案引用
2. 暴露 `Bind(...)` 接口供控制层调用

### 3. 后端启动职责拆出

新增 `StartMenuBackendLauncher`：

1. 提供后端健康检查协程
2. 提供 Python 进程启动方法
3. 保留当前 `python` / `py -3` 候选命令策略

这样后端启动依赖不再继续堆在 Controller 内部。

### 4. 资产层开始界面构建

新增 `StartMenuBuilder` 编辑器工具：

1. 菜单入口：`Tools/Build Start Menu`
2. 创建或更新：
   - `cvs_startMenu_dy`
   - `go_startMenuController_dy`
   - `UI_StartMenuSaveButton.prefab`
3. 给开始界面背景 `img_startBackground_dy` 绑定：
   - `Assets/Art/Generated/PixelScenes/street_arcade_pixel_v1.png`
4. 给 `StartMenuView` 与 `StartMenuController` 做 SerializedField 连线
5. 保留场景已有对话 Canvas，不覆盖现有对话 UI

### 5. Unity MCP 执行方式

脚本落地后通过 Unity MCP：

1. 刷新编译
2. 执行 `Tools/Build Start Menu`
3. 检查 Console
4. 确认场景中开始界面对象和引用已建立

## 风险点

1. `StartMenuBuilder` 若重复执行，需要保证幂等，不重复堆子节点
2. 开始界面与已有对话 Canvas 叠层时要明确 sorting order
3. 场景已有 `EventSystem`，Builder 只能复用，不能重复创建激活实例
4. 若后端自动启动失败，开始界面仍需保留清晰错误提示
