# 开局与午夜世界准备加载门执行方案

## 1. 需求与边界

### 需求

1. 玩家点击新游戏后立即显示带背景、加载文字和视觉假进度条的全屏加载遮罩；加载期间不得自由移动、传送、对话或进入世界。
2. 新游戏必须等初始知识、当天 NPC 日计划及首批 NPC 世界状态均完成后，才对 Unity 发送 `GAME_READY` 并解除遮罩。
3. 午夜结算从开始到次日准备完成（含次日 NPC 日计划）始终保持“午夜处理中”；不得在记忆结算结束、日计划尚未完成时让玩家恢复操作。
4. 加载文字必须说明当前正在进行的阶段；进度条是体验反馈，不宣称为真实精确百分比，未收到完成信号前不能达到 100%。
5. 失败时不得解锁世界：新游戏返回开始界面可重试；午夜结算沿用致命错误收口，且保持世界锁定。

### 非目标

1. 不改变 Unity 世界存档权威和 Python 记忆检查点权威。
2. 不改变玩家正式对话的记忆检索策略或 NPC 日计划内容规则。
3. 不把 UI 静态结构、字体或引用改为运行时代码创建；该部分必须经 Unity MCP 写入场景 / Prefab。

## 2. 当前问题与依据

1. `GameClock.start()` 会在 `GAME_READY` 前后立即投递首个行为 tick；首个 tick 对五名 NPC 生成日计划。
2. `_plan_day()` 内部仍调用同步 `llm_client.chat()`。即便外层使用 `asyncio.gather`，同步请求仍会占住事件循环，导致首个玩家对话协议包排队。
3. `GAME_READY` 目前在初始 tick 的日计划完成前发送，前端随即隐藏开始菜单；因此玩家看不到“世界仍在准备”且可误以为已进入可操作状态。
4. 现有 `SystemMessageController.SetLoading()` 已是加载遮罩与 Gameplay 暂停的统一入口，`PlayerController` 已遵守 Gameplay 域暂停；应扩展该深模块，不另建第二套输入锁。
5. 午夜显式结算目前在 `on_midnight()` 完成后就发送 `MIDNIGHT_SETTLEMENT_COMPLETE`，尚未把次日的日计划准备定义为同一完成条件。

## 3. 设计

### 3.1 世界准备模块与稳定接口

新增后端 `WorldPreparationCoordinator` 模块，作为“开局世界准备”和“跨日世界准备”的唯一编排 seam；`Runtime` 与 `BehaviorEngine` 不再由调用方手工拼接冷启动、结算、日计划和完成消息。

模块接口只暴露两项意图：

1. `prepare_initial_world(mode, report_stage)`：执行新游戏 / 续玩所需的世界准备；新游戏含冷启动，二者都保证目标游戏日的计划已准备好。
2. `prepare_next_day(game_time, report_stage)`：执行午夜记忆结算、次日状态刷新、天气决定和目标游戏日计划准备。

返回稳定结果（`operation_id`、目标游戏日、天气、NPC 快照）；失败以稳定原因向调用方抛出。内部负责调用 `BehaviorEngine` 的“为指定日期确保日计划已完成”能力，并维护幂等的已准备日期，避免 `GameClock` 的首个 / 后续 tick 再次重复规划。

`BehaviorEngine` 将日计划调用改用现有异步 LLM 封装或受控工作线程，确保准备期间仍能处理协议、取消和诊断消息；不把同步供应商请求留在事件循环中。

### 3.2 协议与完成条件

新增服务端到 Unity 的 `WORLD_PREPARATION_PROGRESS` 消息，字段使用稳定英文 snake_case：

```json
{
  "type": "WORLD_PREPARATION_PROGRESS",
  "operation_id": "world_prepare_xxx",
  "flow": "initial_world",
  "phase": "daily_plans",
  "message": "正在为居民安排今日行程…",
  "progress_floor": 0.55
}
```

`flow` 取值为 `initial_world` 或 `next_day`；`phase` 为有限白名单。`progress_floor` 只表达后端已完成阶段下限，Unity 负责视觉平滑和 95% 上限。

1. Unity 在发送 `GAME_START` 前立即打开加载遮罩、申请既有 `LoadingOverlay` 暂停来源，并将 `GameStateStore` 标为 not-ready。
2. 后端只在 `prepare_initial_world` 成功后发送 `GAME_READY`。该消息仍是进入可玩世界的唯一成功终态。
3. `MIDNIGHT_SETTLEMENT_REQUEST` 改由 `prepare_next_day` 完整处理；仅在所有午夜阶段和次日计划完成后发送既有 `MIDNIGHT_SETTLEMENT_COMPLETE`。
4. 准备失败发送现有失败终态或 `GAME_ERROR`，并带 `operation_id` / 稳定失败原因；不得发送成功终态。
5. 不为加载遮罩新增暂停来源，避免和 `SleepFlow`、`LoadingOverlay` 的释放顺序互相覆盖。新游戏使用 `LoadingOverlay`；午夜同时保留 `SleepFlow` 与 `LoadingOverlay`，任一仍存在时 Gameplay 都保持暂停。

### 3.3 Unity 加载视图

扩展现有 `SystemMessageController` / `SystemMessageView` 的加载视图接口，使调用方只提交“显示、阶段文案、进度下限、是否完成”这一小接口；假进度、文本切换、超时保持与 100% 收口由该模块封装，开始菜单和睡眠流程不自行管理 Slider。

Unity 资产层通过 Unity MCP 改造现有系统消息 Canvas / Prefab：

1. 全屏背景遮罩节点、标题文本、阶段文本、进度轨道和填充条使用既有 UI 命名标签规范；中文 TMP 文本绑定 `Assets/Fonts/MSYH SDF.asset`。
2. 进度条默认从低值开始，收到阶段后缓动到阶段区间；未收到完成终态时最高停在 95%，成功终态才播放到 100% 并淡出。
3. 新游戏阶段文案至少覆盖“整理初始记忆”“为居民安排今日行程”“正在进入街区”；午夜阶段至少覆盖“整理今日见闻”“沉淀记忆”“安排明日行程”。
4. `StartMenuController` 在 `GAME_READY` 前保持开始界面被加载遮罩覆盖；完成后才隐藏。`SleepController` 在午夜成功且自动存档 / 醒来流程完成后才关闭遮罩。

### 3.4 诊断与可观测性

将世界准备状态加入 `aisc_debug.runtime_snapshot`：`operation_id`、`flow`、`phase`、`is_active`、`progress_floor`、`failure_reason`、`target_game_day`。诊断 DTO 只读，不携带 UI 文案或 LLM 原文。

现有 `aisc_control.start_new_game` 继续作为新游戏验证入口；无需扩展为任意阶段控制。为其补充“等待世界准备完成 / 失败”的测试辅助，而不是开放新的业务绕过入口。

## 4. 涉及文件（预期）

## 5. 实施步骤

1. 先补后端世界准备协调器、阶段枚举 / 结果契约和单元测试；将首日、跨日都收敛到该 seam。
2. 将日计划生成迁离事件循环，并加“指定日期已准备”的幂等保护；确保开局与跨日只各执行一次。
3. 接通 `WORLD_PREPARATION_PROGRESS`、延后 `GAME_READY` 和 `MIDNIGHT_SETTLEMENT_COMPLETE`，补协议测试与失败路径。
4. 在 Unity 运行时接收进度、立即锁定 Gameplay，并扩展现有加载模块的假进度状态机。
5. 使用 Unity MCP 修改 SystemMessages 资产、接好 SerializeField、设置中文字体；编译后检查 Console。
6. 补 `aisc_debug` 快照、控制测试辅助、入口文档和 Workstream 回写。
7. 运行后端定向测试、Unity EditMode / 编译检查，并用 `aisc_control.start_new_game` 做 Play 验证；再验证一次午夜结算从锁定到可玩的完整闭环。

## 6. 验收标准

1. 点击新游戏的一帧内显示加载遮罩并暂停 Gameplay；移动、点击 NPC、传送和位置同步均不生效。
2. `GAME_READY` 之前，所有 NPC 的当日日计划均已准备完成；后端可在准备中响应诊断和连接控制。
3. 加载界面能按服务端阶段更新中文文本，假进度平滑推进且不在完成前显示 100%。
4. 午夜从请求到 `MIDNIGHT_SETTLEMENT_COMPLETE` 前，`SleepFlow` / `LoadingOverlay` 至少一个持续存在；次日日计划完成后才允许醒来和移动。
5. 后端或 LLM 失败不会解除 Gameplay 锁；错误路径可回到明确的重试 / 退出收口。
6. `aisc_debug.runtime_snapshot` 可读出活跃准备任务的阶段、关联 ID、进度下限和失败原因；自动化测试覆盖成功、重复请求和失败场景。

## 7. 风险与约束

1. 新游戏存档清理具有不可逆性；本方案不改变现有“用户点击确认后才清理”的授权边界。
2. LLM 供应商耗时不可预测，因此进度条只能是阶段驱动的体验反馈；不能用超时强行解锁世界。
3. 午夜涉及 Unity 权威时间、自动存档和 Python 记忆结算；必须保持现有完成顺序，不可让后端擅自写入 Unity 世界时间。
4. SystemMessages 已是暂停拥有者；避免让 StartMenu、Sleep 或 PlayerController 各自维护重复的锁定布尔值。
