# 暂停系统三时间域执行方案

> 执行记录: [2026-07-17_暂停三时间域_execution.md](2026-07-17_暂停三时间域_execution.md)

## 1. 需求理解

当前暂停系统以来源集合计算单一 `IsPaused`，并通过 `Time.timeScale = 0` 同时冻结所有使用缩放时间的逻辑。目标不是简单让 UI 或 FX 永不暂停，而是建立三个可独立判断、可由暂停来源组合控制的时间域：

1. `Gameplay`：游戏主逻辑，包括游戏内时钟、玩家/NPC 移动、任务动作和依赖 Unity 缩放时间的世界行为。
2. `UI`：UI 动效，包括对话逐字、加载点动画、气泡淡入淡出、菜单过渡和系统消息计时。
3. `FX`：视觉特效，包括受管 ParticleSystem、特效 Animator 和后续统一接入的世界/界面特效。

“暂停来源”与“受影响时间域”必须分离：业务模块仍只申请或释放自己的暂停来源，暂停模块根据集中策略计算三个时间域的最终状态，不允许调用方自行修改 `Time.timeScale`。

## 3. 模块与接口方案

### 3.1 时间域模型

新增 `[Flags]` 枚举 `PauseTimeDomain`，固定包含：

- `None`
- `Gameplay`
- `UI`
- `FX`
- `All`

暂停来源策略集中定义为 `source -> PauseTimeDomain` 映射。`PauseModel` 保存来源及其时间域，并提供稳定快照；同一时间域只在影响它的最后一个来源释放后恢复。

### 3.2 PauseController seam

保留现有 `RequestPause(string source)` / `ReleasePause(string source)` 调用形态，避免业务模块了解实现细节。新增只读查询接口：

- `IsDomainPaused(PauseTimeDomain domain)`
- 游戏主逻辑、UI、FX 三域的稳定状态快照
- 时间域状态变化事件，事件参数包含域状态和活动来源

现有 `IsPaused` 暂时保留为兼容聚合值，但新增代码不得用它代替具体时间域判断。

`Time.timeScale` 只跟随 `Gameplay` 域：

- `Gameplay` 首次暂停时保存原缩放值并设为 `0`。
- `Gameplay` 最后一个来源释放时恢复原缩放值。
- UI/FX 域变化不得直接修改 `Time.timeScale`。

### 3.3 初始来源策略

执行前以以下矩阵作为基线，并在 Play 验证中确认：

| 暂停来源 | Gameplay | UI | FX | 说明 |
|---|---:|---:|---:|---|
| `dialogue` | 是 | 否 | 是 | 世界和世界特效暂停，对话 UI 动效继续 |
| `inventory` | 是 | 否 | 是 | 世界暂停，背包 UI 动效继续 |
| `pause_menu` | 是 | 否 | 是 | 世界暂停，菜单 UI 动效继续 |
| `blocking_message` | 是 | 否 | 是 | 世界暂停，消息 UI 自身继续 |
| `loading_overlay` | 是 | 否 | 是 | 世界暂停，遮罩/加载 UI 继续 |
| `sleep_flow` | 是 | 否 | 是 | 世界暂停，睡眠 UI 过渡继续 |
| `save_management` | 是 | 否 | 是 | 世界暂停，存档 UI 继续 |

该矩阵的重点是建立可扩展分类，不代表 UI/FX 永不暂停。未来需要冻结 UI 或 FX 的流程应新增稳定暂停来源并在集中策略中声明相应域，禁止在消费者内散落来源判断。

### 3.4 时间消费者

1. 游戏主逻辑继续使用缩放时间，并在已有显式暂停判断处改查 `Gameplay` 域。
2. UI 动效统一通过一个窄接口取得 UI 域 delta time；UI 域运行时使用 `Time.unscaledDeltaTime`，暂停时返回 `0`。首轮至少迁移 `PortraitDialogueUI`、`BubbleUI` 和 `SystemMessageController` 的动效/显示计时。
3. FX 通过独立适配器监听 FX 域变化。受管 ParticleSystem/Animator 使用非缩放更新，FX 域暂停时由适配器暂停，恢复时仅恢复由该适配器暂停的实例，避免错误启动原本未播放的特效。
4. 协议超时、后端启动、睡眠结算等待等真实时间流程继续使用 `WaitForSecondsRealtime` / 单调时间，不归入 UI 或 FX。

## 5. 诊断钩子同步

扩展 `aisc_debug` 暂停快照，至少包含：

- 聚合暂停状态
- 三个时间域各自的暂停状态
- 每个时间域的活动来源
- 当前 `Time.timeScale`
- 受管 FX 数量及由时间域暂停的数量

若执行中需要自动控制回归，再为 `aisc_control` 增加白名单式“按稳定测试来源申请/释放指定时间域暂停”的编辑器专用 action；不得开放任意来源、任意字段写入或反射调用。新增控制 action 时必须同步工具文档与测试。

## 6. 实施步骤

1. 建立快速 EditMode 回归：验证来源策略、同域多来源叠加、跨域独立性和最后来源释放语义。
2. 引入时间域模型和集中来源策略，迁移 `PauseModel` / `PauseController`，保持旧调用方可编译。
3. 将 `Time.timeScale` 收口为只响应 `Gameplay` 域，并迁移已有显式 `IsPaused` 判断。
4. 建立 UI clock seam，迁移首轮 UI 动效消费者；不得直接把所有 `Time.deltaTime` 机械替换成 `unscaledDeltaTime`。
5. 阅读 `docs/DesignDocs/UnityMCPUsageRules.md`，通过 Unity MCP 盘点实际 FX 资产、添加适配器、配置非缩放更新和 SerializeField 连线。Unity MCP 不可用时立即停止，不以运行时代码扫描全场景兜底。
6. 扩展 `aisc_debug` DTO/provider/action 和相关测试；按需要增加最小白名单 `aisc_control`。
7. 运行 EditMode 测试、编译检查和 Play 验证；用 `aisc_debug` 记录三个域及 FX 计数的结构化证据。
8. 检查并同步 Workstream、代码目录入口和诊断文档，创建 execution 记录实际资产、验证与未完成项。

## 7. 风险与控制

1. `Time.timeScale = 0` 会冻结所有历史缩放时间协程；仅迁移 UI 动效，不能误把 NPC 任务、世界气泡队列或协议业务超时改成 UI 时间。
2. ParticleSystem/Animator 的更新模式是资产事实，必须逐个通过 Unity MCP 验证；禁止用 `FindObjectsOfType` 每帧扫描兜底。
3. FX 恢复必须记录“由适配器暂停”的实例，避免恢复进入暂停前已停止的特效。
4. Python 只镜像游戏主逻辑暂停，不应因单独 UI/FX 域暂停而停止后端游戏时钟。
5. `PauseChanged` 兼容事件和新分域事件的调用顺序必须稳定，避免 UI 在状态切换中看到半更新快照。
6. 如首轮盘点发现 FX 数量或类型显著超出 ParticleSystem/Animator 两类，应停止扩张适配器范围并更新本方案，不用多分支硬塞进单类。

## 8. 验收标准

1. 对话、背包、暂停菜单、阻塞消息、加载、睡眠和存档管理均暂停 `Gameplay`，且多来源叠加不会提前恢复。
2. 上述暂停期间，对话逐字、加载点、气泡淡入淡出、菜单/消息 UI 动效按 UI 域策略继续运行。
3. 受管世界 FX 在 `FX` 域暂停时停止，释放最后一个 FX 暂停来源后恢复；原本未播放的 FX 不被误启动。
4. 单独暂停 UI 或 FX 域不会修改 `Time.timeScale`，也不会同步 Python 游戏暂停；暂停 Gameplay 会正确同步 Python。
5. `aisc_debug` 能结构化报告三域状态、来源、`Time.timeScale` 和受管 FX 状态。
6. EditMode 回归、Unity 编译和针对对话/暂停菜单/FX 的 Play 验证通过。
7. 相关 Workstream、代码入口、诊断入口与实际行为一致，并创建互相索引的 execution。
