# 对话加载态补全 — 设计方案

## 需求理解

当前玩家与 NPC 正式对话时，AI 生成回复和快捷回复建议都存在一定等待时间，但前端没有明确的“正在生成中”反馈，容易让玩家误以为卡住了。

本次目标是在 Unity 前端补齐两类加载态：

1. NPC / AI 回复生成中：对话框中显示 `.` / `..` / `...` 这类动态等待效果
2. 玩家快捷回复建议生成中：选项区显示“生成中”的动态等待效果

## 已确认现状

- `PortraitDialogueUI` 当前只有逐字机和最终回复展示，没有加载状态
- `Open()` 和 `SubmitReply()` 后，对话框会短暂空白，直到收到 `DIALOGUE_TOKEN`
- `GameManager` 当前把：
  - `DIALOGUE_COMPLETE`
  - `DIALOGUE_CHOICES_UPDATE`
  都路由到 `PortraitDialogueUI.OnNpcReplyComplete(...)`
- 这会导致“NPC 回复完成”和“快捷回复后续更新”在前端被混成同一个阶段

## 方案原则

1. 不改后端协议，只补前端体验
2. 将“NPC 回复生成”和“快捷回复生成”拆成两个独立 UI 状态
3. 保留自由输入，加载态只做反馈，不锁死输入框

## 具体方案

### 1. 为 NPC 回复增加点点点加载态

在 `PortraitDialogueUI` 中加入：

- NPC 回复加载标记
- 加载计时器
- 文案基底，例如 `思考中`

在以下时机进入加载态：

- `Open()` 后等待首句 NPC 回复
- 玩家提交 `PLAYER_CHOICE` 后，等待下一轮 NPC 回复

在以下时机退出加载态：

- 收到首个 `DIALOGUE_TOKEN`
- 对话关闭

### 2. 为快捷回复选项增加加载态

在 `PortraitDialogueUI` 中加入：

- 快捷回复加载标记
- 使用现有 `ChoiceButton` 生成若干个禁用态占位按钮
- 同步更新 `_replyHintLabel` 为“快捷回复生成中...，也可直接输入”

在以下时机进入加载态：

- NPC 回复结束，但当前还没有可用 `choices`

在以下时机退出加载态：

- 收到 `DIALOGUE_CHOICES_UPDATE`
- 本轮进入下一次提交
- 对话关闭

### 3. 拆分 `DIALOGUE_COMPLETE` 与 `DIALOGUE_CHOICES_UPDATE` 的前端入口

`GameManager` 改为：

- `OnDialogueComplete` → `PortraitDialogueUI.OnNpcReplyComplete(msg.choices)`
- `OnDialogueChoicesUpdate` → `PortraitDialogueUI.UpdateSuggestionChoices(msg.choices)`

避免快捷回复更新时重复走“整轮 NPC 回复结束”的初始化逻辑。

## 风险点

1. 当前后端 `DIALOGUE_COMPLETE` 可能已附带 fallback choices，这时快捷回复加载态可能只是一闪而过或直接跳过
2. 如果后端异常情况下既没有 token，也没有 complete，前端加载态会持续到用户结束对话；这比空白等待更容易理解，但仍属于异常路径

## 验证计划

1. Unity 脚本编译通过，无新增错误
2. 点击 NPC 开聊后，对话框会出现动态 `.` / `..` / `...`
3. 玩家提交下一句后，下一轮 NPC 回复前也会显示加载态
4. 快捷回复未准备好时，会出现动态占位提示；收到更新后替换成真实选项
