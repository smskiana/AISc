# 对话感知标签接入 — 设计方案

## 需求理解

当前 LLM 对话主要知道“NPC 是谁、在哪里、正在做什么”，但缺少对现场真实存在物的感知：

1. 地点缺少统一的视觉、声音、气味、可见物、话题钩子字段。
2. 物品只有 ID / 名称 / 分类 / 来源，无法让 LLM 感受到质感与用途。
3. 人物缺少统一的视觉与存在感标签，导致快捷回复和对白容易靠名字幻觉性别或关系。

本轮目标是补齐地点、物品、人物的感知标签，并接入玩家对话、NPC 闲聊和快捷回复建议 prompt。

## 方案思路

1. 复用现有 `backend/config/location_profiles/`，为 9 个地点补：
   - `perception_tags`
   - `sensory`
   - `visible_objects`
   - `conversation_hooks`
   - `spot_perception`
2. 扩展 `shared/items.json`，为 10 个初始物品补：
   - `perception_tags`
   - `sensory`
   - `presence`
   - `conversation_hooks`
3. 扩展 5 个 NPC profile，并新增玩家 profile：
   - `visual_tags`
   - `presence_tags`
   - `speech_perception`
   - `address_hint`
4. 新增独立 `PerceptionContextBuilder`，由它读取地点 / 物品 / 人物配置并生成短文本。
5. `PromptBuilder` 只接入生成后的感知文本，避免继续膨胀职责。

## 涉及文件

- `backend/src/dialogue/perception_context.py`
- `backend/src/dialogue/prompt_builder.py`
- `backend/config/player_profile.json`
- `backend/config/location_profiles/*.json`
- `backend/config/npc_profiles/*.json`
- `shared/items.json`

## 风险点

1. Prompt 过长：只注入当前地点、当前 spot、当前对话人物和少量相关物品。
2. 配置字段变多：新增字段均为可选字段，旧逻辑不依赖，不改数据库 schema。
3. 人物称呼过死：`address_hint` 只作为边界提示，不强制每句都使用固定称呼。
