# 对话 UI 预制体配置 — 设计方案

## 需求理解

用户希望把此前补上的对话输入区、发送按钮、结束按钮等内容，正式落到 Unity 资产配置层，而不是继续依赖运行时代码临时生成，方便后续在 Inspector 中直接调样式、位置和引用。

## 当前问题

经检查，当前状态是：

1. `DialogueCanvas` 仍是场景对象，不是可复用 prefab
2. 场景中的 `PortraitDialogueUI` 已有回复相关序列化字段，但实际引用仍为空
3. `PortraitDialogueUI` 当前通过运行时创建 `ReplyPanel/TMP_InputField/Button` 兜底，属于本应由场景或 prefab 承担的配置

这会带来两个问题：

1. 美术/UI 调整不直观，必须改代码才知道最终结构
2. 运行时生成逻辑污染前端 MVC 分层，不利于长期维护

## 修复原则

1. **Unity MCP 优先**：场景层级、UI 结构、预制体保存尽量在编辑器资产层完成
2. **最小迁移**：保留当前对话逻辑，只把“应该由配置承担”的部分挪回 prefab / scene
3. **双保险**：
   - 现有 `SampleScene` 可直接使用
   - 同时产出一个可复用的 `DialogueCanvas` prefab，方便后续别的场景拖拽

## 计划修改范围

### 代码

- `Assets/Scripts/Dialogue/PortraitDialogueUI.cs`
- `Assets/Scripts/Editor/DialogueUIBuilder.cs`

### Unity 资产 / 场景

- `Assets/Scenes/SampleScene.scene`
- `Assets/Prefabs/UI/DialogueCanvas.prefab`（新增）

### 文档

- `docs/AIChanges/Dialogue/2026-07-10_对话UI预制体配置_plan.md`
- `docs/AIChanges/Dialogue/2026-07-10_对话UI预制体配置_execution.md`

## 具体方案

### A. 更新 `DialogueUIBuilder`

扩展编辑器构建器，让它正式创建并连线：

- `ReplyPanel`
- `ReplyHintLabel`
- `ReplyInputField`
- `SendButton`
- `EndDialogueButton`

并把这些对象写入 `PortraitDialogueUI` 的 SerializeField。

### B. 用 Unity MCP 修改现有 `DialogueCanvas`

直接在当前场景的 `DialogueCanvas` 下补齐回复区层级和组件，确保：

- 当前场景立即可配、可用
- 不再依赖运行时创建 UI

### C. 产出对话 UI prefab

将整理好的 `DialogueCanvas` 保存为：

- `Assets/Prefabs/UI/DialogueCanvas.prefab`

便于后续：

- 新场景直接复用
- 统一调样式
- 在 prefab 模式下集中配置

### D. 收敛 `PortraitDialogueUI` 的运行时兜底

把当前“创建 UI 物体”的逻辑降级为“仅绑定已有引用 / 子物体”，不再负责创建布局和控件本身。

保留的运行时职责仅包括：

- 订阅事件
- 控制显隐
- 处理输入提交
- 清空文本与打字机状态

## 风险点

1. `TMP_InputField` 的子层级需要正确绑定 `TextViewport / Text / Placeholder`
2. 若 prefab 中保留了对场景对象的非法引用，需要重新检查 SerializeField 是否都指向 prefab 内部对象
3. 当前 `DialogueCanvas` 若被保存成 prefab 后再修改场景实例，需要确认 `GameManager._dialogueUI` 仍指向场景实例

## 验证计划

1. 场景中 `DialogueCanvas` 下可见完整回复 UI 层级
2. `PortraitDialogueUI` 的回复字段在 Inspector 中已不为空
3. `Assets/Prefabs/UI/DialogueCanvas.prefab` 成功生成
4. Unity 编译无新增错误
5. 点击 NPC 后：
   - 对话 UI 正常打开
   - NPC 回复后输入框出现
   - `Enter / 发送 / 结束对话` 正常工作
