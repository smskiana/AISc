# 玩家快捷回复主体契约修复执行案

## 状态

- Status: Implemented
- 日期: 2026-07-17
- 主要功能域: 对话系统
- 关联功能域: 测试与诊断、项目治理
- 问题来源: 玩家快捷回复建议在 Prompt 数据层迁移后丢失明确的玩家发言主体约束，并把玩家名误标为当前 NPC，真实 LLM 因此生成了 NPC 口吻和 NPC 动作描写。
- 前置执行案: [后端 Prompt 数据层化执行案](2026-07-17_后端Prompt数据层化_plan.md)

## 一、需求理解

玩家快捷回复按钮必须表达“玩家接下来可以对当前 NPC 说的话”。当前运行链虽然仍调用 `player_reply_suggestions` task，但实际 Prompt 同时存在以下冲突：

1. `target_name` 在调用方表示玩家显示名，配置却把它标成“当前 NPC”。
2. Prompt 注入的是 NPC 对玩家的印象和 NPC 的说话建议，却未明确这些信息只能帮助理解关系、不能决定玩家口吻。
3. 旧兼容常量保留了“站在玩家口吻说话，不要替 NPC 说话”，但运行入口已经改走 `PromptAssembler`，该约束没有迁入正式 contract。
4. `build_player_reply_suggestions()` 仍构造一份未使用的 `user_prompt`，其中主体说明比正式运行 Prompt 更清楚，形成误导性死代码。
5. 现有解析器只验证 JSON、数量、去重和长度，无法识别 NPC 自述、替 NPC 发言或舞台动作，因此错误输出会被当成合法 choice 下发。

本轮目标是恢复并强化玩家快捷回复的主体契约，以可复用测试和安全诊断防止同类迁移回归；不重新设计对话系统、LLM 供应商或 Unity UI。

## 二、现行口径与边界

1. 保持 `PromptAssembler.build(task_id, context)` 作为唯一 Prompt 组装 seam；不得恢复第二套业务内大段 Prompt。
2. 业务调用方负责提供结构化事实，Prompt 数据层负责表达任务主体、硬规则和响应契约。
3. 玩家昵称继续通过 `player_name.py` 与 `render_player_tokens()` 处理，不硬编码默认昵称。
4. 快捷回复继续注入现场感知、NPC 标签、最近对白和玩家相关记忆，但这些内容不能改变发言主体。
5. 继续返回 `{"choices": [...]}`，恰好 3 条；不修改 WebSocket 消息类型、Unity 按钮结构或自由输入流程。
6. 不设置 `max_tokens`；长度继续由 Prompt 软约束和既有解析收口。
7. 不涉及场景、Prefab、SerializeField、UI 层级或资产连线，因此实现本身不触发 Unity MCP 资产编辑；Play Mode 验证仍应使用现有 `aisc_control`，并用新增的 `aisc_debug` 快照读取结构化证据。
8. 本轮不新增新的 `aisc_control` action：现有 `start_dialogue`、`send_player_choice`、`end_dialogue` 已覆盖最小复现。

## 三、目标主体契约

`player_reply_suggestions` 必须建立以下不变量：

1. 发言者固定为玩家，接收者固定为当前 NPC。
2. 每条 choice 都是玩家可直接发送的一句话；不得替 NPC 回答、不得继续 NPC 自述、不得把 NPC 的动作或心理写成玩家台词。
3. 禁止用括号舞台动作伪装玩家回复，例如“（微微歪头）”“（抬手指路）”；玩家自身极短动作是否允许不在本轮扩展，第一版统一要求纯台词以减少歧义。
4. NPC profile、现场感知、关系印象和 NPC speech hint 只作为“玩家正在面对谁、如何自然回应”的背景，不得作为玩家人格或玩家语气。
5. 三条建议继续保持顺接、追问/关心、轻松/换角度的差异。
6. Prompt 中必须分别出现语义明确的 `npc_name` 与 `player_name`；不得继续复用含义模糊的 `target_name` 表达双方身份。

## 四、方案设计

### 4.1 收敛结构化上下文字段

修改 `PromptBuilder.build_player_reply_suggestions()` 向 assembler 提交的 context：

```text
npc_name
player_name
npc_tags
dialogue
npc_reply
perception_context
relationship_from_npc
npc_speech_hint
player_memory_text
game_time
location
```

约束：

1. `npc_name` 取当前 `npc_id` 对应 profile 显示名。
2. `player_name` 取当前 `target_id` 对应玩家显示名。
3. `relationship_from_npc`、`npc_speech_hint` 在字段名和渲染文案中明确方向，防止被误读为玩家口吻。
4. 不向 Prompt 数据层暴露额外数据库、状态管理或感知读取 interface；数据收集仍封装在 `PromptBuilder` 内。
5. 删除未使用的局部 `user_prompt`。旧未使用 Prompt 常量是否删除应以全局 `rg` 引用结果为准；若删除，不修改或删除周边原有注释。

### 4.2 修正正式 Prompt contract

修改 `backend/config/prompt/system_contracts.yaml` 的 `player_reply_suggestions`：

1. 明确“你不是 NPC，你只为玩家生成下一句候选”。
2. 分别渲染玩家名和 NPC 名，不再把玩家标成当前 NPC。
3. 明确 NPC 关系、NPC 语气和现场感知只是理解背景。
4. 明确只输出玩家可直接发送的纯台词，禁止 NPC 视角、NPC 动作、心理旁白、舞台说明、标题和解释。
5. 保留现有最短 JSON response contract，不复制冗长 JSON 示例。

`response_contracts.yaml` 的 `choices` 继续只负责机器格式，不承载玩家/NPC 业务主体规则；主体规则属于该 task 的 system contract。

### 4.3 增加输出主体防线

Prompt 是第一道约束，测试和诊断是回归防线。本轮不在解析器中加入大规模自然语言主体判定器，原因是仅靠关键词无法可靠判断所有中文人称和动作，容易误杀正常玩家台词。

仅允许增加窄而确定的硬校验：

1. 拒绝明显的角色名前缀，如 `鹿岛樱：...` 或当前 NPC 显示名加冒号。
2. 拒绝以成对全角/半角括号开头的舞台动作 choice。
3. 被拒条目按既有 `_merge_with_fallback_choices()` 补足，不能阻断自由输入或整轮对话。
4. 硬校验应封装为一个私有谓词，避免把主体规则散落进 JSON 解析分支。

若执行时发现该校验需要大量中文语义规则、三个以上不断增长的特例类别，停止扩写，回到方案阶段设计独立的回复候选校验 module；不得在 `PlayerDialogueService` 内堆叠 switch/if 规则。

### 4.4 诊断 seam

新增后端有界快捷回复 trace，记录最近若干次生成结果的安全结构化摘要，不保存完整 Prompt、完整对话、玩家记忆或原始 LLM 输出。

建议字段：

```text
reply_trace_id
npc_id
player_id
task_id
speaker_role_expected
recipient_role_expected
context_keys
choice_count
choice_previews
rejected_choice_previews
rejection_reasons
fallback_used
failure_reason
elapsed_ms
```

其中 preview 必须定长裁剪；如实现时无法证明玩家隐私边界，宁可只记录长度、布尔值和拒绝原因，不记录正文。

提供后端只读 HTTP 入口，并在 Unity `aisc_debug` 增加 `player_reply_suggestion_snapshot` action。Unity 只做代理和 DTO 映射，不重建后端判断。筛选参数至少支持 `reply_trace_id`、`npc_id` 和 `limit`。

现有 `aisc_control` 对话控制 action 足够复现，不新增控制能力。

## 五、实施步骤与涉及文件

### 阶段 A：建立失败反馈环

先新增小于 30 秒的回归测试，确认修改前失败：

1. 通过 `PromptBuilder.build_player_reply_suggestions()` 的公开 seam 渲染 Prompt。
2. 断言 Prompt 明确包含玩家发言者与 NPC 接收者。
3. 断言玩家名不再出现在“当前 NPC”语义下。
4. 断言 NPC speech hint 被标记为 NPC 背景，而非玩家说话建议。
5. 断言当前 NPC 名前缀和括号舞台动作会被拒绝并用 fallback 补足。

## 六、测试与验证

1. Prompt registry/assembler 既有测试通过。
2. 新增快捷回复主体契约测试修改前红、修改后绿。
3. 解析器覆盖：合法玩家台词保留；NPC 名前缀和括号舞台动作拒绝；不足三条用 fallback 补齐。
4. 诊断 trace 覆盖容量限制、筛选、裁剪、拒绝原因和不保存完整 Prompt/原始输出。
5. 运行对话、Prompt 和诊断相关测试，再运行后端全量测试。
6. 按错误预防索引要求，Python 编译和测试不得并行争写同一 `__pycache__`。

### 6.2 Unity 验证

实现涉及 Unity 诊断脚本时，执行前先读取 `UnityMCPUsageRules.md`，使用 Unity MCP 导入/刷新、检查 Console 并运行相关 EditMode 测试；不得只用命令行构建代替编辑器验证。

Play Mode 最小复现：

1. 用 `aisc_control.start_new_game` 进入可对话状态。
2. 用 `aisc_control.start_dialogue` 与鹿岛樱开始对话。
3. 等待首轮三条建议，再发送一条玩家回复触发第二轮。
4. 用 `aisc_debug.player_reply_suggestion_snapshot` 读取最近 trace。
5. 断言期望发言主体为玩家、接收者为 NPC、三条 choice 无 NPC 名前缀和括号动作、无拒绝未补足情况。
6. 实际观察 Unity 三个按钮文字均可由玩家直接发送。

### 6.3 真实 LLM 样本

至少覆盖 5 个 NPC，每个 NPC 首轮和续轮各一次，检查：

1. 发言主体始终是玩家。
2. 不出现 NPC 自述或 NPC 动作描写。
3. 称呼符合 NPC profile 与当前关系。
4. 三条建议有差异且不虚构玩家已经做过的行为。
5. LLM 偶发违规时确定性校验和 fallback 能稳定收口。

真实 LLM 调用不是单元测试替代品；若供应商不可用，应记录为未完成项，不得宣称完整验收。

## 七、风险与处理

1. 风险：仅改字段名但 contract 仍让 NPC 语气主导输出。
   - 处理：测试最终渲染 messages，不只测试 context 字典或 YAML 可加载。
2. 风险：用关键词判断主体造成误杀。
   - 处理：只拦确定性的 NPC 名前缀和开头舞台动作；复杂语义不进入解析器。
3. 风险：诊断泄露完整对白、记忆或 Prompt。
   - 处理：有界 trace、定长 preview、稳定字段白名单；默认不保存完整输入和原始输出。
4. 风险：`PlayerDialogueService` 继续增重。
   - 处理：trace 存储和筛选封装在独立深 module；业务调用点只提交生成结果和结构化元数据。
5. 风险：修改 Unity 诊断工具时误触资产层。
   - 处理：本任务不改场景/Prefab/连线；若编译后发现需要资产连线，立即停止并按 Unity MCP 规则处理，不用运行时代码兜底。
6. 风险：删除旧 Prompt 常量时误删原有注释。
   - 处理：执行前回看错误预防条目“兼容标记误改原有注释”，只删除确认无引用的常量正文，保留非该常量所有原注释。

## 八、验收标准

1. `player_reply_suggestions` 最终渲染 Prompt 明确玩家是唯一发言者、NPC 是接收者。
2. Prompt 分别使用 `player_name` 和 `npc_name`，不再把玩家误标为当前 NPC。
3. NPC 关系、语气、profile 和现场感知被明确标为背景，不再定义玩家口吻。
4. 三条输出仍为可解析 JSON 字符串数组，并保持顺接、追问、轻松三类差异。
5. NPC 名前缀和开头舞台动作被最小确定性校验拒绝，fallback 能补满三条。
6. 回归测试从公开 PromptBuilder/assembler seam 验证最终 messages，能稳定捕获本次错误。
7. `aisc_debug.player_reply_suggestion_snapshot` 可读取期望主体、结果数量、拒绝原因、fallback 和失败原因，不暴露完整 Prompt或未裁剪隐私文本。
8. 现有 `aisc_control` 能完成真实对话复现，不新增重复控制入口。
9. 后端相关测试、Unity 诊断测试、Console 检查和至少一轮真实 Play Mode 验证通过。
10. Dialogue Workstream、Prompt/Dialogue/Diagnostics README 与实际行为一致。
11. 创建独立错误预防明细并回写索引。
12. execution 记录实际修改、测试结果、真实 LLM 样本、诊断证据和所有未完成项。
