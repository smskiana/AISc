# 对话时停 — 设计方案

## 需求理解

目标是把“玩家与 NPC 对话”改成真正的时停状态：

1. 对话期间游戏内时间停止推进
2. 对话期间后端不再继续跑 NPC 行为 tick
3. 对话开始时若有 NPC 正在前端移动，要在画面上冻结住
4. 对话结束后，时间与 NPC 活动从冻结点继续

## 已确认现状

### 前端

当前前端只做了：

1. 玩家移动锁定
2. 再次点击 NPC 锁定

也就是“玩家被锁住”，但不是“世界时停”。

NPC 若正在执行前端 Lerp 移动，仍会继续走完。

### 后端

当前 `GameClock` 虽然有 `TimeSpeed.DIALOGUE`，但其实现是：

1. 1 游戏分钟 = 30 现实秒

这只是“几乎暂停”，不是完全停表。

`BehaviorEngine.tick()` 仍然由时钟分钟推进驱动，因此只要时钟还在走，NPC 行为就还会继续推进。

## 方案原则

1. **最小修改**：不扩协议，不重做对话主链
2. **职责清晰**：
   - 后端负责真正停表，停止 NPC 行为推进
   - 前端负责冻结已经开始的 NPC 表现层移动
3. **可恢复**：对话结束后从原状态继续，不重置 NPC

## 涉及文件

1. `backend/src/world/clock.py`
2. `backend/src/application/dialogue_service.py`
3. `Assets/Scripts/NPC/IMovementProvider.cs`
4. `docs/AIChanges/Dialogue/2026-07-11_对话时停_plan.md`
5. `docs/AIChanges/Dialogue/2026-07-11_对话时停_execution.md`

## 具体方案

### 1. 后端时钟增加真正的暂停机制

在 `GameClock` 中新增显式暂停状态，而不是继续依赖 `TimeSpeed.DIALOGUE` 的“超慢速”语义。

设计为：

1. 支持 `push_pause(reason)` / `pop_pause(reason)`
2. 内部用集合记录暂停源，避免未来重复进入/提前恢复
3. `_tick_loop()` 在暂停期间不推进分钟、不触发 `on_tick`

这样对话期间：

1. 游戏内时间完全不走
2. `BehaviorEngine.tick()` 也不会再被触发
3. 不会再产生新的 NPC 行为更新

### 2. 玩家对话生命周期接入停表

在 `PlayerDialogueService` 中接管对话暂停源：

1. `start_dialogue()` 成功建立活跃对话后 `push_pause`
2. `end_dialogue()` 收尾时 `pop_pause`
3. 若开始对话中途失败，也要保证回收 pause，避免时钟卡死

暂停 key 使用带 `npc_id` 的稳定标识，避免未来多来源暂停互相覆盖。

### 3. 前端冻结正在进行中的 NPC 移动

在 `LerpMovementProvider.MoveRoutine()` 中增加对 `GameManager.Instance.IsDialogueActive` 的检查：

1. 对话进行中时只 `yield return null`
2. 不累计 `elapsed`
3. 对话结束后继续按原路径走完

这样能冻结：

1. 正在移动的 NPC 位置
2. 不会出现“后端停表了，但屏幕上 NPC 还继续走”的割裂感

### 4. 本轮不处理的范围

这轮先不额外改：

1. NPC 气泡显示计时
2. 玩家台词回显
3. 非对话原因的全局暂停系统

原因是当前“活动”主要由：

1. 后端时钟推进
2. NPC 行为下发
3. 前端移动协程

先把这三层停住，就能达到这轮目标。

## 风险点

1. 对话开始时若 LLM 立即失败，必须正确释放 pause，不能把世界永久冻住
2. 若未来还有其他暂停来源，简单布尔值不够，因此本轮直接做“按 reason 计数/集合”的暂停机制
3. 前端当前只冻结移动协程，不会主动冻结所有可能的 UI 特效；但对现阶段 NPC 活动表现已足够

## 验证计划

1. 后端脚本编译通过
2. 玩家开始对话后，`GameClock` 不再推进分钟
3. 对话期间 `BehaviorEngine.tick()` 不再继续驱动 NPC 行为
4. 对话开始时若某 NPC 正在移动，画面中应停在原地
5. 对话结束后，NPC 移动和时间推进继续恢复
