# 项目运行架构治理方案

## 需求理解

当前项目已经从单一功能开发进入多系统并行阶段。`docs/AIChanges/` 中 plan / execution / handoff 数量快速增长，日期文件可以还原历史，但不能稳定回答：

1. 当前项目按哪些长期目标推进。
2. 某个系统当前工程口径是什么。
3. 哪些设计决策已经收敛，不应被反复推翻。
4. 单次执行记录属于哪个路线图条目。
5. 是否需要引入向量库管理项目文档与执行记录。

本轮目标不是修改运行时代码，而是重建协作与文档运行方式，让后续 AI 和人工都能从“日志驱动”转为“路线图 + 决策记录 + 执行证据”驱动。

## 方案思路

### 1. 新增路线图入口

新增 `docs/Roadmap.md`，作为长期目标、近期阶段和冻结事项的总入口。

路线图只登记高层方向，不承载执行细节。

### 2. 新增工作流索引层

新增 `docs/Workstreams/`，每个高频系统一份 README：

1. 项目治理。
2. 记忆系统。
3. 导航系统。
4. 对话系统。
5. 前端职责拆分。

每份工作流 README 记录：

1. 当前目标。
2. 当前工程口径。
3. 已完成阶段。
4. 下一阶段建议。
5. 暂缓 / 不做事项。
6. 相关设计文档与执行记录。

### 3. 新增 ADR 决策记录层

新增 `docs/DecisionRecords/`，记录已经收敛的架构决定，避免后续重复争论或误回滚：

1. 轻量图记忆 v0.7。
2. 导航 typed path。
3. Unity MCP 资产层优先。
4. 项目文档治理与向量库判断。

### 4. 新增变更索引

新增 `docs/AIChanges/ChangeIndex.md`，把历史 plan / execution 按系统归类。

它不替代执行记录，只作为“从系统问题找到相关历史证据”的入口。

### 5. 回写现有入口

更新：

1. `docs/ProjectIndex.md`
2. `docs/DesignDocs/Index.md`
3. `docs/AIChanges/README.md`
4. `AGENTS.md`
5. `docs/AIChanges/Archive/Handoffs/HANDOFF_2026-07-12.md`
6. `docs/DesignDocs/CodebaseBigPicture.md`

让新会话启动链路优先读取 Roadmap / Workstreams / ADR，再进入具体 handoff 和 execution。

## 向量库初步判断

本轮先不建议立即为项目文档补向量库。

理由：

1. 当前主要问题是结构索引缺失，不是语义检索能力不足。
2. 向量库会增加同步、切分、失效、来源追溯和误命中成本。
3. 现阶段 `rg` + Roadmap + Workstream + ADR + ChangeIndex 已能覆盖大多数查询。
4. 如果未来 `docs/AIChanges/` 超过数百份、需要自然语言跨历史问答，再补离线文档向量索引更合适。

因此本轮会把“何时需要补向量库”写入 ADR，而不是马上实现。

## 风险点

1. 文档治理不能变成新的大文档泥潭，因此新增索引只记录入口、状态和边界，不复制完整执行细节。
2. ChangeIndex 若试图全量精确维护历史，会消耗过多维护成本；本轮只做高频系统的初始索引。
3. ADR 一旦写入，需要后续修改者尊重；如果未来推翻，应新增 ADR supersede，而不是直接改历史结论。

## 验证方式

1. 静态检查新增文档是否存在。
2. 检查关键互链是否可读。
3. 确认 `ProjectIndex.md` / `DesignDocs/Index.md` 不再指向过期 7 月 11 日 handoff。
4. 本轮不涉及代码编译。
