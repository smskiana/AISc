# 项目命名与索引规则补充 — 设计方案

## 完成时间

2026-07-10

## 需求理解

用户希望在现有 `Unity MCP优先` 与 `GameObject命名规范` 之外，再补一层更广义的项目规则，用来统一：

1. 跨端 ID 命名
2. 代码文件与代码符号命名
3. Unity 资源文件命名
4. 文档 / 核心文件索引回写规则

目标不是一次性重命名现有全部资产，而是先建立“以后新增内容怎么命名、哪些文件必须进索引”的统一口径。

## 当前情况

经检查，当前项目已有：

1. `GameObject` 命名规则
2. `UnityNamingTags.md` 作为场景层级标签参考

但仍缺少：

1. 跨端稳定 ID 的统一规范
2. “显示名”与“程序 ID”分离规则
3. Unity 资源文件名与运行时 GameObject 名称的边界说明
4. 核心文件索引的统一回写规则
5. 一份总览式的命名 / 索引文档入口

## 具体方案

### A. 在 AI 规则中新增“项目命名与索引规范”

写入一条高层规则，覆盖以下内容：

1. 跨端共享 ID 一律使用 `snake_case`
2. 显示名与程序 ID 必须分离
3. C# / Python 代码命名遵循各自语言惯例
4. Unity 资源文件名与 GameObject 命名分开管理
5. 新增核心文件、核心配置、核心文档后，必须回写对应索引文档

### B. 新增总规范文档

建立 `ProjectNamingAndIndexing.md`，作为：

1. 当前命名规范总入口
2. 各子类规范的汇总
3. 文件索引回写规则说明

文档内容计划包括：

1. 跨端 ID 规则
2. 显示名 / 程序 ID 分离
3. Python 命名规则
4. C# 命名规则
5. Unity 资源文件命名规则
6. GameObject 命名规则索引到 `UnityNamingTags.md`
7. 核心文件索引规则
8. 推荐的索引文档位置

### C. 不在本轮做历史批量迁移

本轮不扩展到：

- 批量重命名已有 JSON key
- 批量改现有 Prefab / Scene 文件名
- 批量补全部目录的索引文档
- 大规模修正文档内历史旧称

本次只建立规则与入口文档。

## 风险点

1. 旧资产命名会与新规范并存一段时间
2. 若“核心文件”的定义过宽，索引维护成本会过高
3. 若没有统一入口文档，规则容易继续散落在多个 AIChanges 记录里

## 处理策略

1. 规则优先约束新增内容
2. 明确“核心文件”是会被高频协作、跨模块引用、作为入口或配置源的文件，而不是所有文件
3. 用一份总规范文档承接细则，避免把 `AGENTS.md` 写得过长
