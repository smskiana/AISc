# 低级错误索引化 — 设计方案

## 完成时间

2026-07-11

## 需求理解

用户认为当前 `AGENTS.md` / `CLAUDE.md` 中的“低级错误记录”过于简单，直接把错误细节堆在规则文件里会有两个问题：

1. 规则文件会越来越长
2. 为了控长度，错误细节容易被压缩失真

用户希望改成“文件 + 索引”的形式：

1. 规则文件里只保留机制和入口
2. 单个错误用独立文档完整记录
3. 用一个索引文件汇总这些错误，方便后续会话先查目录再深入

## 当前问题

目前结构是：

- `AGENTS.md`
- `CLAUDE.md`

中的“低级错误记录”直接承载细节。

这个结构在错误数量少时还行，但随着项目推进会带来：

1. 规则文件被历史错误污染得越来越长
2. 后续新增错误时，不容易保留完整上下文
3. 单个错误无法补充“影响范围 / 示例 / 适用检查点”等更长内容

## 本次修改范围

### 规则文件

- `AGENTS.md`

### 索引 / 明细文档

- `docs/DesignDocs/ErrorPreventionIndex.md`（新增）

### 索引更新

- `docs/DesignDocs/Index.md`

- `docs/AIChanges/ProjectGovernance/2026-07-11_低级错误索引化_plan.md`

## 具体方案

### A. 规则文件改成入口式表述

把“低级错误预防”中的：

- 直接内嵌错误明细

改成：

1. 说明记录机制
2. 指向统一索引文档
3. 规定后续新增错误时：
   - 先建独立明细文档
   - 再回写索引
   - 最后只在规则里保留入口，不堆细节

### B. 新增错误索引文件

创建：

- `docs/DesignDocs/ErrorPreventionIndex.md`

承载：

1. 这个索引的用途
2. 推荐阅读方式
3. 错误列表
4. 每条错误的简述、影响范围、应优先回看的场景

### C. 为现有错误创建独立明细

把当前已有的：

- `Day 0 被 or 1 误覆盖`

迁移为单独文件，例如：

- `docs/DesignDocs/errors/2026-07-11_day0_or1_override.md`

完整记录：

1. 现象
2. 根因
3. 错误代码模式
4. 正确写法
5. 适用范围
6. 修改前自查点

### D. 更新入口索引

把这个新的错误索引入口同步登记到：

- `docs/DesignDocs/Index.md`
- `docs/ProjectIndex.md`

方便后续新会话和规范治理直接发现它。

## 风险点

1. 如果只建明细文件，不建索引，后续仍会找不到
2. 如果只在索引里保留一句话，仍然可能失真

## 处理策略

1. 同时建立“总索引 + 单条明细”两层结构
2. 规则文件不再承载错误细节，只承载机制与入口

## 验证计划

1. `AGENTS.md` 与 `CLAUDE.md` 的“低级错误预防”已改为索引入口式写法
2. `ErrorPreventionIndex.md` 已创建并登记现有错误
3. 现有 `Day 0 / or 1` 错误已有独立明细文件
4. `docs/DesignDocs/Index.md` 与 `docs/ProjectIndex.md` 已能指向该索引
