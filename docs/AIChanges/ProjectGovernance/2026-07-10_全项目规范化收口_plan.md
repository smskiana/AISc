# 全项目规范化收口 — 设计方案

## 完成时间

2026-07-10

## 需求理解

用户希望“趁项目还来得及”按既定规则对整个项目做一次规范化收口。

这不是单点 bug 修复，而是一次面向长期协作的工程治理，目标应当是：

1. 让规则文档形成稳定入口
2. 让核心目录具备清晰索引
3. 让新增内容有可执行的命名与放置准则
4. 尽早控制会继续扩散的结构债
5. 避免在没有边界的情况下直接做全量重命名或大规模业务重构

## 当前现状

经检查，当前项目已经具备一部分规范基础：

1. `AGENTS.md` 与 `CLAUDE.md` 已同步 AI 开发规则
2. `docs/DesignDocs/ProjectNamingAndIndexing.md` 已建立总命名规范
3. `docs/DesignDocs/UnityNamingTags.md` 已建立 Unity 标签参考
4. `docs/AIChanges/` 已开始记录 plan / execution

但仍存在明显缺口：

1. `docs/ProjectIndex.md` 缺失
2. `docs/DesignDocs/Index.md` 缺失
3. 交接文档仍停留在 `2026-07-08`，与今日大量变更不一致
4. `docs/AIChanges/` 中已混入跑测产物、临时目录和日志摘要，记录层与产物层边界不清
5. `.git` 目录存在但当前不是可用仓库，版本控制状态异常
6. 项目尚缺一套“可复查”的轻量规范检查入口，后续仍容易继续漂移

## 本轮建议目标

本轮先做“规范基线工程”，不直接做大规模历史迁移。

建议把执行范围控制在以下四类：

1. 索引补齐
2. 文档入口收口
3. 规范检查基线
4. 项目卫生边界说明

## 分批执行方案

### A 批：文档与索引基线

目标：先把人能快速找到入口这件事解决。

计划处理：

1. 新建 `docs/ProjectIndex.md`
2. 新建 `docs/DesignDocs/Index.md`
3. 视实际需要补充 `Assets/Scripts/Index.md`
4. 为核心目录建立“先看什么、谁会改、何时优先查看”的轻量索引
5. 更新 `docs/AIChanges/Archive/Handoffs/HANDOFF_2026-07-08.md`，或补一份新的 handoff，避免会话感知链继续失真

### B 批：规范落地与检查

目标：不是只写规则，而是让规则有最低限度的落地检查能力。

计划处理：

1. 盘点跨端稳定 ID 的来源文件
2. 增加一个轻量检查脚本，验证共享 ID 是否满足 `snake_case`
3. 检查共享配置、后端 profile、Unity 资源配置之间的主键一致性
4. 将检查入口登记到索引文档，作为后续会话的固定步骤

### C 批：目录与产物边界收口

目标：避免运行产物继续污染文档层。

计划处理：

1. 梳理 `docs/AIChanges/` 下哪些属于记录，哪些属于测试产物
2. 对测试日志、摘要、临时目录给出统一收纳位置建议
3. 如改动成本可控，则把后续产物目录规范到固定子目录
4. 同步更新索引说明，避免新会话继续把产物直接堆在根层

### D 批：暂不在本轮直接执行的高风险事项

以下事项收益大，但不建议在本轮和规范基线混做：

1. 全量重命名历史 GameObject / Prefab / 场景节点
2. 全量重命名历史代码符号或 JSON 主键
3. 调整运行时架构边界的大型重构
4. 修复或重建当前异常的 `.git` 仓库状态

这些事项一旦动手，影响范围会明显扩大，应单独立项并先评估回滚与引用风险。

## 本轮拟涉及文件

- `docs/ProjectIndex.md`（新增）
- `docs/DesignDocs/Index.md`（新增）
- `Assets/Scripts/Index.md`（待定，视扫描结果）
- `docs/AIChanges/Archive/Handoffs/HANDOFF_2026-07-08.md` 或新的 handoff 文档（待定）
- 轻量规范检查脚本（路径待定，优先放在 `backend/scripts/`）
- `docs/AIChanges/ProjectGovernance/2026-07-10_全项目规范化收口_execution.md`（执行后补）

## 风险点

1. 如果把“规范化”理解为一次性全量迁移，改动会快速失控
2. 如果直接改历史命名，Unity 引用和跨端 ID 可能联动破坏
3. 如果在没有可用版本控制的前提下大改，回退成本过高
4. 如果只补文档不补检查入口，规范很快会再次漂移

## 执行原则

1. 先建立入口，再做迁移
2. 先控制新增内容，再清理历史内容
3. 先做低风险高收益项，再触碰高耦合资产
4. 任何涉及历史大规模重命名或 `.git` 修复的动作，必须单独确认

## 验证计划

1. 核心目录都有明确索引入口
2. 新会话能仅凭交接链和索引快速定位项目核心文件
3. 至少有一个可运行的轻量规则检查入口
4. 规范文档、索引文档、交接文档之间不再互相脱节
