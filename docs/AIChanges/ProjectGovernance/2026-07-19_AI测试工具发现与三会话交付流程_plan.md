# AI 测试工具发现与三会话交付流程执行案

## 1. 需求理解

当前规则能提醒 AI 使用 `aisc_debug` / `aisc_control`，但没有提供完整、易发现且按需展开的测试工具入口。测试能力分散在 Unity Test Runner、Python 测试、后端跑测脚本和项目专用 Unity MCP 诊断/控制工具中，AI 容易只使用眼前看到的一类工具，或为了避免遗漏而一次性读取大量说明。

同时，现有治理只正式定义 plan / execution 两类记录。跑测、复测和验收仍常被写入 `_execution.md`，不能稳定约束“执行案、执行记录、测试记录”分别由三个独立会话承担。

本轮目标是：

1. 建立测试工具的分级发现路径，规则只提示何时查找，不复制全部工具说明。
2. 把复杂实际修改固定为 plan、execution、test 三步三个会话。
3. 让 execution 不再以实现会话内的测试结果代替独立验收；第三会话通过 `_test.md` 给出最终测试结论。
4. 保留小修的轻量例外，但任何需要独立测试或已经触发 plan 的修改都必须遵守三会话流程。

## 2. 所属功能与既有口径

- 主要功能域：项目治理。
- 关联功能域：测试与诊断。
- Workstream：`docs/Workstreams/ProjectGovernance/README.md`。
- 相关索引：`docs/ProjectIndex.md`、`docs/AIChanges/ChangeIndex.md`。
- 相关 ADR：ADR-0004、ADR-0005；本轮不改变功能目录优先的架构决定，无需新增 ADR。
- 命名依据：`docs/DesignDocs/ProjectNamingAndIndexing.md`。
- 现有测试入口：`backend/tests/README.md`、`backend/scripts/README.md`、`Assets/Scripts/Diagnostics/README.md`。

## 3. 目标信息架构

采用三级披露，不在 `AGENTS.md` / `CLAUDE.md` 堆放工具清单和参数：

1. 第一级，项目规则：只规定测试前必须先查看测试与诊断入口、按风险选工具、不得无脑加载全部说明，并定义三会话交付状态。
2. 第二级，测试与诊断 Workstream：提供工具类别选择表，按任务类型路由到 Unity 自动化测试、Python 自动化测试、后端跑测脚本、`aisc_debug` / `aisc_control`、Console 和 Unity MCP。
3. 第三级，工具所属目录 README：记录具体适用场景、入口、限制和最小验证方式；AI 只读取本轮选中的叶子入口。

计划新增 `docs/Workstreams/TestingAndDiagnostics/README.md` 作为第二级权威入口，并新增 `Assets/Tests/README.md` 补齐 Unity 自动化测试的第三级入口。现有后端与诊断 README 只补必要的选择说明，不把同一内容复制到多个位置。

## 4. 三会话交付模型

### 4.1 方案会话

- 创建唯一 `YYYY-MM-DD_<主题>_plan.md`。
- 定义实现边界、风险、诊断覆盖、测试工具候选和验收标准。
- 不进入实现；要求新会话引用该 plan。

### 4.2 执行会话

- 以唯一 plan 为方案来源，完成实现和实现期最低限度门禁。
- 创建 `YYYY-MM-DD_<主题>_execution.md`，同时链接 plan 与预定 test record。
- 实现期允许运行编译、静态检查、聚焦单测或诊断冒烟，以防提交明显损坏；这些结果只属于实现门禁，不替代独立验收。
- execution 的最终状态只能写“实现完成，待独立测试”或“实现未完成”，不得在没有第三会话 test record 时宣称整体交付完成。

### 4.3 测试会话

- 读取 plan 的验收标准、execution 的实际变更和第二级测试工具入口。
- 只按任务风险读取并使用相关叶子工具说明，不要求遍历所有工具。
- 创建 `YYYY-MM-DD_<主题>_test.md`，链接 plan 与 execution，记录环境、工具选择理由、命令或 MCP action、结构化证据、结果、失败与未覆盖项。
- 只有 test record 给出通过结论后，任务才可宣称完成；失败则回到新的执行会话修复，同一 test record 追加复测或创建清晰互链的复测记录，具体在执行时以最少歧义的规则收口。

## 6. 兼容与迁移原则

1. 新规则只强制新建记录使用 `_test.md`；历史跑测 `_execution.md` 不批量改名。
2. 已进入执行阶段的旧 plan 可按原约定完成，但若尚未独立验收，应补 test record 并互链。
3. 测试记录仍放在主要功能域；只有跨系统纯跑测/诊断任务才进入 `TestingAndDiagnostics/`。
4. 不在规则文件复制 action 参数、脚本参数或全部测试类清单。
5. 不读取或改写本轮无关的用户未提交业务改动。

## 7. 风险与收口

1. `AGENTS.md` 与 `CLAUDE.md` 可能继续漂移：执行时以同一权威 Workstream 为链接目标，并将两者的核心交付状态用一致措辞表达。
2. 三会话规则可能让小修成本过高：仅对触发 plan、需要独立验收或用户明确要求测试记录的修改强制；无需测试的纯探索不产生三类记录，单文件几行小修仍按规则明确例外。
3. execution 与 test 的边界可能误解为执行期禁止测试：明确“实现门禁允许且必要，独立验收必须另会话”。
4. 工具目录可能持续扩张：第二级入口只维护类别和路由，具体新增工具由所属目录 README 维护。

## 8. 验收标准

1. 新会话仅阅读 `AGENTS.md` 和 `docs/ProjectIndex.md` 即能发现测试与诊断 Workstream。
2. AI 能从第二级入口按 Unity 代码、后端代码、真实运行链、结构化状态或长跑性能选择对应工具，无需读取所有工具文档。
3. 新规则明确 plan、execution、test 必须在三个独立会话完成，并区分实现门禁与独立验收。
4. plan、execution、test 三类文件有稳定命名和互链规则；历史记录无需迁移。
5. 没有 test record 通过结论时，execution 和 AI 最终回复不得宣称复杂任务整体完成。
6. Markdown 本地链接检查通过，项目约定检查通过（若脚本覆盖相关规则）。
7. 本次治理本身在后续执行会话创建 execution，并在第三个独立测试会话创建 test record 后才算完成。

## 9. 后续会话入口

下一会话请明确引用本文件，并要求“按该 plan 执行”。执行会话不得创建第二份 plan；完成实现后必须停止并要求第三个独立会话按同主题 test record 验收。
