# 全面整改 — 设计方案

## 完成时间

2026-07-10

## 需求理解

用户明确要求对整个项目进行一次“不遗余力”的全面整改。

结合当前仓库状态，这里的“全面整改”不应理解为无边界地重写功能，而应理解为：

1. 解决已经明确暴露的工程基础问题
2. 收口历史产物、缓存、入口和规则链路
3. 建立可持续的仓库与协作基线
4. 在不破坏现有运行链的前提下，尽可能多消除结构性脏点

用户已明确同意做大范围整理，因此本轮可以超出“最小修改”原则，但仍要控制在可回查、可验证、可继续开发的边界内。

## 本轮审计得到的主要问题

### 1. 版本控制基础失效

- 根目录存在 `.git/`
- 但目录为空，当前不是可用仓库
- `git status` 直接报错

### 2. 跑测产物规范只建立了一半

- 新规则已要求产物进入 `docs/AIChanges/artifacts/`
- 但历史日志、summary、隔离目录仍堆在 `docs/AIChanges/` 根层
- 多份 execution / plan 文档还在引用旧路径

### 3. Python 缓存已污染源码树

- `backend/`、`backend/src/`、`backend/scripts/` 及子目录存在 `__pycache__/`
- 这些内容不属于项目源资产

### 4. 工程元文件缺失

- 当前没有 `.editorconfig`
- 当前没有 `.gitattributes`
- `.gitignore` 还未覆盖新的 `docs/AIChanges/artifacts/`

### 5. 会话感知链虽然已更新，但还缺更强的工程闭环

- 需要把这轮大整改本身也写入交接链
- 需要给后续“先检查什么、先清什么”建立更清晰的默认动作

## 本轮整改范围

### A. 仓库基线修复

1. 初始化可用的 Git 仓库
2. 验证 `git status` 可正常工作
3. 补充 `.editorconfig`
4. 补充 `.gitattributes`
5. 扩充 `.gitignore`

### B. 历史产物收口

1. 在 `docs/AIChanges/artifacts/` 下为旧跑测产物建立标准归档目录
2. 将历史 `.log`、`_summary.json`、`*_artifacts/`、`_tmp_*` 迁入新结构
3. 更新相关文档中的路径引用

### C. 源码树卫生清理

1. 删除 `backend/**/__pycache__/`
2. 删除本轮执行产生的 `.pyc`
3. 确认不误删源码或运行必需资产

### D. 文档与交接链闭环

1. 补本轮 execution 文档
2. 补新的 handoff 或更新现有 handoff
3. 将“本轮全面整改后的默认入口”写清楚

1. 历史产物迁移后，若文档路径没改全，会留下断链
2. 初始化 Git 后，状态会把当前全部工作区纳入版本视角，输出会很大
3. 删除 `__pycache__` 时需要确保只删缓存，不碰源码
