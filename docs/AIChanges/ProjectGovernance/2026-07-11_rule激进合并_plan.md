# Rule 激进合并 — 设计方案

## 完成时间

2026-07-11

## 需求理解

用户希望在上一轮“精简收口”的基础上继续推进，采用**激进版**整理方式：

- 不只是压缩措辞
- 而是把重复规则真正合并成 `5-6` 条总规则
- 让 `AGENTS.md` / `CLAUDE.md` 的“核心行为底线”读起来更清爽

本次目标是：

1. 继续保留当前已经确认的硬约束
2. 重新组织规则分组，减少“同一个意思拆成 2-3 条”的情况
3. 让执行时先抓大原则，再看细节文档

## 当前问题

虽然上一轮已经做过精简，但当前第 1 节仍然存在：

1. `最小修改` 与 `侵入控制` 高度相关
2. `先读后改`、`尊重人工`、`复用优先` 可以合并为同一类协作原则
3. `GameObject命名规范` 与 `项目命名与索引规范` 在入口层也有重复
4. `中文输出` 与 `执行闭环` 更像交付要求，也可并到一条

因此仍有继续合并空间。

## 计划合并方向

### 1. 变更策略

把以下内容合并成一条：

- 最小修改
- 侵入控制
- 用户明确同意的大改边界

核心口径：

- 修 bug / 小功能 → 最小修改强制
- 新系统 / 结构治理 → 不为了少改而继续沿用失衡结构
- 大改仍需用户明确同意

### 2. 协作编辑原则

把以下内容合并成一条：

- 先读后改
- 尊重人工
- 复用优先

核心口径：

- 先读取最新内容
- 不盲目覆盖人工修改
- 优先复用现有符号和结构

### 3. 类减重与扩展性

保留为独立硬约束，因为这是一个清晰的触发型规则：

- 类过重时必须请求“减重”
- 不允许“先凑合”

### 4. Unity MCP优先

保留为独立硬约束，因为它有明确的资产层 / 运行时代码边界和例外三条件。

### 5. 命名规范入口

把以下内容合并为一条规范入口：

- GameObject命名规范
- 项目命名与索引规范

核心口径：

- 命名与索引统一以 `ProjectNamingAndIndexing.md`
- Unity 层级标签同时参考 `UnityNamingTags.md`

### 6. 交付要求

把以下内容合并为一条：

- 中文输出
- 执行闭环

核心口径：

- 始终中文
- 复杂需求先方案后实现
- 完成后输出修改点清单

## 风险点

1. 合并过度可能让规则失去可执行性
2. 若删掉过多细节，后续会话可能重新走样

## 处理策略

1. 只合并，不改方向
2. 保留关键触发条件和例外边界
3. 细节仍放在现有设计文档和执行记录中承接

## 验证计划

1. `AGENTS.md` 与 `CLAUDE.md` 的“核心行为底线”收敛到 5-6 条
2. 关键硬约束仍保留：
   - 类减重请求
   - Unity MCP 例外三条件
   - 修 bug / 小功能最小修改
3. 文本整体明显比当前更短、更顺
