# AI 诊断钩子强制同步规则 - 设计方案

## 需求理解

将项目专用 AI 诊断钩子纳入项目级开发硬约束：AI 调试 Unity 功能时优先使用现有钩子；新增或修改功能时同步评估、补齐和验证对应钩子，不能只完成业务实现而遗留不可观测状态。

## 所属功能域

- 主要功能域：项目治理
- 关联功能域：测试与诊断、前端架构

## 方案

1. 在 `AGENTS.md` 核心行为底线中新增硬约束。
2. 明确 Unity 运行态调试优先调用 `aisc_debug`，并结合 Unity Console、测试和现有 MCP 工具验证。
3. 功能状态、阶段、失败原因或关键关联 ID 发生新增或变化时，必须同步更新对应诊断 DTO、provider、MCP action 或测试。
4. 若本轮确实不适用诊断钩子，必须在 execution 中说明原因；不得默认省略。
5. 业务功能与对应诊断钩子、测试、文档未同步完成时，不得宣称任务完成。

## 涉及文件

- `AGENTS.md`
- `docs/Workstreams/ProjectGovernance/README.md`
- `docs/AIChanges/TestingAndDiagnostics/README.md`
- 本 plan / execution

## 风险

1. 规则过宽可能迫使纯文案或静态资源修改补无意义钩子，因此保留“明确不适用并记录”的出口。
2. 诊断钩子不能替代业务测试、Console 和 Play 验证，规则中明确它只是强制验证链的一部分。
3. 不要求每个功能都新增 MCP action；优先扩展现有深模块和稳定 action。
