# 阶段路线图排期方案

## Workstream

ProjectGovernance / FrontendArchitecture / Navigation

## Roadmap item

GOV / FEA / NAV / DIA

## 相关 ADR

1. `docs/DecisionRecords/ADR-0002-navigation-typed-path.md`
2. `docs/DecisionRecords/ADR-0003-unity-mcp-asset-first.md`

## 需求理解

需要把当前排期写入项目文档，并分成三档：

1. 正在进行。
2. 正在计划。
3. 未来要做。

本次排期需要纳入以下方向：

1. 前端架构失衡修复。
2. 前端落后能力补齐。
3. 前后端协议优化。
4. 玩家基础 UI，要求严格 MVC。
5. 存档可靠性。
6. 玩家和 NPC 夜间回家睡觉、超时强制晕倒。
7. 玩家传送 / 场景入口能力。
8. NPC 见闻、主动搭话、玩家插入 NPC 对话。
9. 模拟经营店铺功能与后续送礼、交易、耕种、街机等系统。

## 方案思路

### 1. Roadmap 作为主排期入口

在 `docs/Roadmap.md` 增加“三档排期”章节，按当前优先级写清：

1. 正在进行：架构止血、导航验证、协议与存档设计前置。
2. 正在计划：玩家传送、MVC 基础 UI、夜间睡眠闭环、NPC 见闻 / 主动搭话 / 插话。
3. 未来要做：店铺经营核心循环、送礼、交易、耕种、街机。

### 2. 工作流同步

更新：

1. `docs/Workstreams/FrontendArchitecture/README.md`
2. `docs/Workstreams/Navigation/README.md`

让前端职责和导航工作流知道：

1. 玩家传送需要进入排期。
2. 玩家传送不应直接复用 NPC 自动寻路执行器，而应复用 location / SceneAnchor / NavigationTeleportPoint 语义。
3. 前端基础 UI 与经营玩法要建立在架构止血和协议 / 存档可靠性之后。

### 3. 执行证据回写

更新 `docs/AIChanges/ChangeIndex.md`，并新增本轮 execution。

## 风险点

1. 排期不能写成过细任务清单，否则 Roadmap 会退化成执行记录。
2. 玩家传送要和 NPC 导航共享地点语义，但不能直接把玩家交互塞进 `AStarMovementProvider`。
3. 店铺经营虽然以前端为主，但会影响 NPC 记忆的关键事实仍需要后端事件通道。

## 验证方式

1. 静态检查新增章节和关键链接。
2. 确认 Roadmap 中存在“正在进行 / 正在计划 / 未来要做”三档。
3. 本轮不涉及代码编译。
