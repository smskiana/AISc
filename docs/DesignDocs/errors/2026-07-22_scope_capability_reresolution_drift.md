# Scope capability 签发后二次解析导致范围漂移

## 错误现象

IDE 在多个不同文件执行局部更新时都显示固定预算上限数量的可变更项，范围不随当前文件或光标符号变化。resolve 与 preview 表面使用同一个 `scope_id`，但 preview 实际重新搜索，无法证明展示和应用的是签发时同一组实体。

## 根本原因

1. IDE 上下文被压成一个自然语言 query，当前文件与光标符号没有作为结构化约束传递。
2. 文件路径进入 BM25 后产生大量弱相关候选，结果被硬预算裁到固定上限。
3. capability 只保存 symbol ID，服务端没有保留签发时的结构化符号集合；preview 再次用不同 limit 搜索，造成 TOCTOU 式范围漂移。
4. UI 只展示计数，使扩大范围难以被用户发现和审计。

## 正确做法

1. 当前文件、光标符号、选区符号和功能域必须使用独立字段传递，不能拼成自由文本后交给模糊搜索。
2. 已知文件与符号使用 analyzer 的精确 path/name filter；自然语言搜索只用于未知责任或候选发现。
3. scope 签发时冻结具体实体和哈希；preview / apply 只能消费冻结集合，不得重新发现或追加实体。
4. 确认界面必须展示具体包含项、只读证据、排除项、域边界与预计成本，不能只显示总数。

## 本次修复

Project Cognition 增加 `filePath/symbolName` 精确图查询、签发符号冻结、`scope-view` 明细以及真实 stdio `smoke:context`。源码位置同步纳入 CodeSymbol，避免“符号查询正确但跳转仍错误”。
