# 关系表被误认为 UML 主视图

## 错误现象

实现提供了功能域树、关系表、证据展开和源码跳转，但 execution 将阶段 C 描述为已经完成；实际没有分层类 UML 节点、关系箭头、图布局、功能域图内下钻或类中心一跳图。

## 根本原因

验收时只核对了“数据和命令是否存在”，没有逐项核对 plan 指定的产品展示形态；关系查询闭环被错误地当成 UML 交互闭环。

## 正确做法

1. UI plan 明确指定主展示形态时，execution 必须逐项核对节点、边、布局、导航和交互，不能用数据等价替代界面等价。
2. Webview 产物门禁至少验证目标控制器和关系 marker 进入 bundle；自动化验证图模型、节点预算和关系类型映射。
3. Extension Host 中的真实布局、窄窗口、长名称、点击和双击交互必须留给独立测试并保存截图证据。
4. plan 或用户指定入口位置和上下文语义时，必须从该入口的真实用户动作验收；不能用左侧树选择替代“当前编辑文件”入口，也不能用光标附近方法替代文件首个类。
5. IDE document symbol API 必须同时覆盖扁平 `SymbolInformation` 与嵌套 `DocumentSymbol`；旧快照只含方法时，分类器视图必须通过受限文件查询投影 Class，不能在过滤方法后交付空图。
6. Language provider 可能完全不可用，代码图工具不能把它作为硬依赖；Webview 门禁必须检查最终生成脚本的语法敏感转义，不能只搜索控制器或 marker 字样是否进入 bundle。
7. “当前文件所属功能域”同时包含 focus 与导航上下文：当前类应作为首屏 focus，所属域通过 membership 和 breadcrumb 提供返回路径，不能用 domain ID 覆盖当前类入口。
8. Class 搜索结果不保证携带行号或直接 `CALLS`；源码定位应通过公开 snippet 补齐，类关系应沿 `DEFINES_METHOD/CALLS` 聚合，不能因 Class `trace_path` 为空就宣称没有关系。
9. SVG 关系不能只让细线承担点击；可见线、宽命中区和标签应属于统一交互组，并提供真实可见 tooltip，CSS 高亮不等于解释或可操作性。

## 适用范围

IDE 扩展、Webview、图形化工具，以及任何 plan 对信息架构或交互形态有明确要求的前端实现。
