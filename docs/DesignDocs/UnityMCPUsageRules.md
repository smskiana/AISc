# Unity MCP 使用规范

## 作用

本规范专门约束 Unity 资产层问题应如何优先通过 `Unity MCP` 处理，避免把本应在编辑器内完成的配置退回运行时代码硬写，持续污染 MVC 边界。

本文件也是 Unity 侧高频“默认约定”与“补充建议”的统一落点。后续若继续追加 Unity 场景、Prefab、UI、字体、引用连线等要求，优先写到本文件，而不是继续膨胀 `AGENTS.md`。

## 1. 适用范围

以下问题默认视为 Unity 资产层问题：

1. 场景层级组织
2. GameObject 创建、命名与摆放
3. Prefab 创建、拆分、挂载与结构调整
4. UI 面板层级、控件摆放与默认显示结构
5. `SerializeField` 引用连线
6. 编辑器内可提前确定的组件配置
7. TMP 文本、图片、按钮、布局组等可直接在编辑器资产层完成的配置

## 2. 核心硬约束

1. 只要问题本质属于资产层配置，默认必须先用 `Unity MCP` 在 Unity 编辑器资产层解决。
2. 不得把“先跑起来”“先代码兜底”“先运行时动态创建 UI / 节点 / 控件”当作默认方案。
3. 运行时代码应主要承担真正的运行时职责，例如事件订阅、状态切换、输入处理、动态内容刷新、数据驱动显隐，而不是替代编辑器做静态资产配置。

## 3. 标准执行方式

1. 先判断问题是否属于资产层，而不是先写运行时代码。
2. 若属于资产层，优先使用 `Unity MCP` 完成场景、Prefab、UI、组件、引用和默认资源配置。
3. 修改脚本后若要依赖新组件或新结构，先确认 Unity 编译完成，再继续做资产层挂载和连线。
4. 完成 Unity 资产修改后，应检查 Console / 编译状态，避免把错误配置继续向后传递。
5. 若本轮涉及新增的稳定默认约定，应优先回写本文件，而不是只写在聊天结论里。
6. 当 Unity MCP 不可用，必须向用户请求下一步指示。

## 4. 允许例外的条件

只有同时满足以下条件，才允许例外进入代码实现：

1. 需求本质属于运行时逻辑，而不是资产配置
2. 已在本轮 `plan / execution` 中明确记录：
   - 不能使用 `Unity MCP` 的原因
   - 例外范围
   - 后续回收到资产层的计划

若未记录上述内容，则视为未遵守本规范。

## 5. MVC 边界要求

1. 视图层结构尽量固化在场景、Prefab 和编辑器资产中。
2. 控制逻辑只驱动现有资产，不把静态界面结构和默认引用生成逻辑塞进运行时代码。
3. 模型、状态和数据流可以在代码中动态变化，但不应借此合理化资产层偷懒。

## 6. 默认约定

### 6.1 TMP 中文字体

1. Unity TMP 默认字体通常不能稳定覆盖中文。
2. 项目内凡是需要显示中文的 TMP 文本，默认使用 `Assets/Fonts/MSYH SDF.asset`。
3. 若新建 TMP 文本、复制 TMP 控件或新做中文 UI，默认应在资产层把字体替换为该字体资产，而不是留着默认字体等运行时报错或乱码后再补。
4. 只有在该界面已经明确指定另一套可稳定支持中文的字体资产，且有清晰设计理由时，才允许不使用 `MSYH SDF`。

### 6.2 Layout Group 与子物体尺寸

1. 当使用 Unity 自带 `Horizontal Layout Group`、`Vertical Layout Group`、`Grid Layout Group` 等布局组件时，若希望子物体保持预期宽高、比例或视觉形状，默认要给对应子物体补 `Layout Element`。
2. 不要假设子物体原本的 `RectTransform` 尺寸一定会被 `Layout Group` 正确保留；缺少布局信息时，子物体可能被压缩、拉伸，甚至挤成一团。
3. 尤其是按钮、头像框、名字牌、对话项、图标容器这类需要稳定视觉尺寸的 UI 子节点，优先显式配置 `Layout Element` 的首选宽高或弹性参数。
4. 若布局效果异常，不要先用运行时代码硬调尺寸兜底，应先回到资产层检查 `Layout Group`、`Content Size Fitter`、`Layout Element` 三者是否匹配。
### 6.3 UI追加
1. 游戏的canvas默认应该唯一，如果没有特殊需求应该在主要的canvas追加UI，而不是建立新canvas导致渲染管线不清晰。
## 7. 后续追加规则的写法

后续若继续补 Unity MCP 相关建议，优先按以下格式追加到本文件：

1. 适用对象
2. 默认做法
3. 不推荐做法
4. 允许例外
5. 如有必要，补充引用资产路径或文档入口

这样可以保证主规则保持短入口，而细节继续累积也不失控。
