# 记忆图模型（Graph Memory Model）

> 当前执行口径更新：2026-07-11。  
> 本文早期章节保留 v0.5 的完整联想图设想；近期工程实现以“轻量图联想索引 + 向量节点数据层”为准。

## 0. 当前实现口径（v0.7，优先级高于旧章节）

### 0.1 一句话结论

当前项目不做“所有概念都完整图化”的重认知模拟。

近期采用：

```text
SQLite 图层：节点 ID + 双向 clarity 边 + 边类型
LanceDB / SQLite 向量层：节点内容 + embedding + importance + created_day + archived
```

图负责“怎么联想到”，向量层负责“记忆内容是什么”。

### 0.2 当前图层职责

SQLite 图层当前是联想索引层：

1. 每个 NPC 仍然有独立图。
2. `memory_nodes` 只存节点 ID、subject_id、created_at_game_time。
3. `memory_edges` 存 `node_a / node_b / type / clarity_ab / clarity_ba / target_importance`。
4. `clarity_ab` 和 `clarity_ba` 表达方向性联想强弱。
5. 午夜主要衰减边 clarity；被实际检索走过的边会少量恢复 clarity。

这意味着：记忆文本可以还在向量库中，但如果通往它的边弱了，NPC 白天就不容易自然想起。

### 0.3 当前向量层职责

LanceDB 优先，SQLite 向量表作为降级方案。

向量层存：

1. `node_id`
2. `vector`
3. `type`
4. `value`
5. `importance`
6. `created_day`
7. `archived`

用途：

1. 按 node_id 取节点内容，供图路由重建上下文。
2. 图结果不足时做少量语义兜底。
3. 强制回忆时可包含 archived 节点。
4. 融合候选或相似度判断时提供 embedding。

### 0.4 当前检索模式

当前检索不是统一一套参数，而是按场景分模式：

| 模式 | 场景 | 当前参数倾向 |
|------|------|------|
| `player_dialogue` | 玩家和 NPC 对话 | 最多 2 跳，优先玩家背景、喫茶店、奶奶、樱桥通旧事 |
| `npc_dialogue` | NPC 与 NPC 日常闲聊 | 最多 1 跳，偏轻量日常，不主动深挖沉重私人旧事 |
| `nightly_impression` | 午夜基准印象刷新 | 最多 2 跳，比白天更宽，用于整理目标人物长期看法 |

共同原则：

1. 从 `我` 节点和目标人物节点出发。
2. 先查方向性邻边。
3. 本地按 clarity、边类型、目标、地点、关键词排序。
4. 必要时让 LLM 在候选边中选路。
5. 图结果不足时才补少量向量召回。

### 0.5 当前已落地的时间语义

玩家背景和商店街背景已经是 Day 0 长期记忆，不再是 prompt 硬编码。

检索输出需要保留时间标签，例如：

```text
- [Day 0] 听说: 小李准备重新开奶奶留下的喫茶店...
- [第5天] 记得: 小李今天把喫茶店招牌挂回去了...
```

规则：

1. Day 0 “听说 / 准备”只代表当时知道的事。
2. 后续 Day N 真实事件可以更新旧语义。
3. NPC 不应把旧传闻说成亲眼所见或已经完成。

### 0.6 暂缓项

以下内容在本文后续章节仍会出现，但近期不作为必须追齐的实现目标：

| 旧设计项 | 当前处理 |
|------|------|
| 节点 `energy / precision` 完整生命周期 | 暂由边 clarity 衰减与 archived 归档替代 |
| 大规模 `similar_to` 边网络 | 暂不建图边；相似度主要用于向量兜底或融合候选 |
| 四阶段退化状态机 | 保留远期设计语义 |
| 完整四策略选择器与思考预算 | 先用检索 mode 参数和本地排序近似表达 |
| 反思独立模块 | 不强拆，可并入图演化或夜间印象 |
| 节点融合作为核心演化 | 已补基础边继承闭环；仍不作为完整四阶段退化主依赖 |

## 核心决策

1. **每个 NPC 一个独立的记忆图** — 节点 + 边 + 独立衰减。NPC 之间记忆不共享
2. **原文存档保留但尽量不使用** — 短期记忆期（7天内）用原文，长期记忆全走图
3. **印象融入图节点** — 印象是图中人物节点的派生属性（周围簇的统计摘要），不作为独立文本
4. **一切能结构化的都不留提示词** — 能进入图的概念全部节点化
5. **LLM 不做检索，LLM 做路由** — 每次只看起点邻边（10-20条），由 LLM 根据权重和状态选择走哪些路径，后端按选择去库里查

## 1. 图结构

### 节点（Tag Node）

每个节点是一个带向量嵌入的概念单元：

```
{
  id, type, value, vector(1024维), energy(存活度 0~1),
  decay_rate, precision(exact|vague|gone),
  merged_from, created_at_game_time
}
```

### 节点类型与衰减速度

| 类型 | 示例 | 衰减 | 认知依据 |
|------|------|------|----------|
| 自我 | "我" | **不衰减（energy 恒为 1.0）** | 每个 NPC 图中有且仅有一个"我"节点，是所有自我相关记忆的引力中心 |
| 人物 | 千早, 玩家 | 几乎不衰减 | 社交记忆是人类进化核心 |
| 时间-精确 | 2025.8.15, 10:30 | 最快 | 海马体时间细胞精度随年龄急剧下降 |
| 时间-模糊 | 2025年夏天 | 较稳定 | 模糊后不再衰减 |
| 时间-周期 | 秋天, 傍晚 | 很稳定 | 循环事件反复强化 |
| 地点-精确 | 花店门口 | 慢 | 空间记忆有专门的海马位置细胞 |
| 地点-模糊 | 花店, 街上 | 稳定 | 模糊后稳固 |
| 事件 | 吃火锅, 搬花盆 | 中等 | 情节记忆自然衰减 |
| 情绪 | 开心, 感激 | 极慢 | 杏仁核标记的情绪记忆几乎终生保留 |
| 感官 | 辣, 很重, 雨声 | 快 | 感官细节在海马旁回最先衰减 |
| 物品 | 毛肚, 花盆 | 中等 | 依附于事件 |
| 话语 | 某句具体的话 | 最快 | 逐字记忆极不可靠 |
| 反思 | "小李这周帮了我好几次" | 慢 | 高层抽象，比原始事件更稳定 |

**"我"节点的特殊性：**

```
鹿岛樱的图里，人物节点不是"鹿岛樱"而是"我"：
  "我" ──→ 千早（最好的朋友）
  "我" ──→ 玩家（回来的邻居）
  "我" ──→ 搬花盆事件（自己经历的事）
  
NPC 不会在脑子里叫自己名字。
"我"节点是天然的最强引力源——所有自我经历、反思、情绪
最终都直接或间接指向"我"。energy = 1.0，永不衰减。
```

### 边（Association）

节点之间的指向关系。**每条边存两个方向的权重——联想是不对称的。**

```
{
  id, node_a, node_b,
  type,
  weight_ab(0~1),    // A → B 的联想强度
  weight_ba(0~1),    // B → A 的联想强度
  decay_rate_ab,     // 各自独立衰减
  decay_rate_ba,
  created_at_game_time
}
```

**为什么不对称：**

```
吃火锅 ──→ 千早   weight=0.9  "吃火锅 → 立刻想到千早"
千早   ──→ 吃火锅 weight=0.3  "千早 → 很多事，吃火锅只是其中一件"

桂花香 ──→ 奶奶的院子 weight=0.85  "闻到桂花 → 瞬间回到院子"
奶奶的院子 ──→ 桂花香 weight=0.2   "想起院子 → 不一定会想起桂花"
```

触发线索（感官、地点、时间）到事件的联想通常更强——因为线索少，指向明确。反过来，人物或事件节点连接的边太多，任一条反向权重都被稀释。

### 边类型

| 类型 | 适用范围 | 含义 | 典型不对称 |
|------|---------|------|-----------|
| **relationship** | 人物 ↔ 人物 | 直接人际关系 | 双向但不对等（千早→樱 0.9 / 樱→千早 0.7） |
| **similar_to** | **任意同类型 ↔ 同类型** | 模糊记忆中同类型节点互相渗透 | 双向，由向量相似度决定初始权重 |
| **sequenced** | 事件 ↔ 事件 | 同一时空内的先后顺序 | 前→后 强 / 后→前 较弱 |
| happened_at | 事件 ↔ 时间 | 事件发生的时间 | 事件→时间 强 / 时间→事件 弱 |
| involved | 事件 ↔ 人物 | 事件的参与者 | 事件→人物 强 / 人物→事件 弱 |
| located_at | 事件 ↔ 地点 | 事件发生的地点 | 事件→地点 强 / 地点→事件 中等 |
| felt | 事件 ↔ 情绪 | 事件伴随的情绪 | 双向均强 |
| contains | 事件 ↔ 子事件/物品 | 组成关系 | 父→子 强 / 子→父 中等 |
| associated_with | 任意 ↔ 任意 | 通用弱连接 | 取决于共现 |
| caused | 事件 ↔ 事件 | 明确因果 | 因→果 强 / 果→因 弱 |
| mentioned | 话语 ↔ 话题 | 话语涉及的话题 | 话语→话题 强 / 话题→话语 弱 |

**similar_to — 模糊记忆的核心机制**

`similar_to` 是唯一可以连接**任意同类型节点**的边。它不是事件专属——当记忆变模糊时，所有同类型节点都会互相渗透。

```
新鲜记忆: 每个节点各自独立，靠跨类型边（happened_at/located_at/involved）
          连接到时间/地点/人物锚点

模糊记忆: 时间/地点/话语等锚点衰减消失
          → 同类型节点之间建立 similar_to 边
          → 节点不再独立，互相指向、互相渗透
          → 这就是"记忆变模糊"在图里的物理表现
```

**各类型的 similar_to 示例：**

```
时间 ↔ 时间:
  2025.8.15 ──similar_to 0.6──→ 2025.9.12
  "好像是八月...也可能是九月。那阵子的事。"
  → 两个日期挨得近、共享事件，互相渗透
  → 最终融合为 "2025年夏秋"

地点 ↔ 地点:
  花店 ──similar_to 0.5──→ 面包店
  ──similar_to 0.5──→ 喫茶店
  "在花店还是面包店？反正就是那几家店之间..."
  → 同一条街上的店铺互相渗透
  → 具体哪家店记不清了

情绪 ↔ 情绪:
  感激 ──similar_to 0.8──→ 温暖
  ──similar_to 0.6──→ 安心
  "对他有一种...感激吧，也不对，更像是温暖的感觉。说不上来。"
  → 正面情绪模糊后互相渗透
  → 形成"正面"这个大类的模糊情绪团

感官 ↔ 感官:
  辣 ──similar_to 0.5──→ 烫
  ──similar_to 0.4──→ 麻
  "火锅吃了什么...反正很刺激的那种。辣的还是烫的来着。"
  → 强烈的感官体验互相渗透

物品 ↔ 物品:
  毛肚 ──similar_to 0.6──→ 牛肉
  ──similar_to 0.5──→ 豆腐
  "点了好多菜...毛肚牛肉什么的。每次都那几样。"
  → 同场景下的物品互相渗透

事件 ↔ 事件:
  搬花盆 ──similar_to 0.7──→ 修招牌
  ──similar_to 0.5──→ 帮忙浇水
  "小李帮了我很多忙。搬东西、修东西..."
  → 同质事件聚类（之前已描述）

人物 ↔ 人物:
  和叶 ──similar_to 0.4──→ 龙之介
  "他们有点像...都喜欢安静。和叶经常提起他。"
  → 人物之间除了 relationship（直接关系），
    还可以有 similar_to（"他们是一类人"）
  → 注意：relationship 和 similar_to 是不同的边！
     relationship = "我跟他有交集"  similar_to = "他让我想起另一个人"

话语 ↔ 话语（通常最先建 similar_to 边，因为衰减最快）:
  "她笑得有点勉强" ──similar_to 0.4──→ "她说了句'没事'但眼神不对"
  "她说了好几句话...都不太对劲。具体忘了。"
  → 同一主题的多句话语互相渗透
```

**夜间演化中 similar_to 的建立规则（通用于所有类型）：**

```
nightly_evolution() 中的 similar_to 建立:

for each 节点类型:
    same_type_nodes = graph.nodes.filter(type == T)
    
    for each pair (A, B) in same_type_nodes:
        similarity = cosine(A.vector, B.vector)
        
        if similarity > 0.85:
            // 强相似 → 可能触发融合（合并为一个节点）
            if can_merge(A, B):
                merge(A, B)
        
        elif similarity > 0.65:
            // 中等相似 → 建立/强化 similar_to 边
            edge = get_or_create_edge(A, B, type="similar_to")
            edge.weight_ab += 0.05
            edge.weight_ba += 0.05
            
            // 边的权重上限由相似度决定
            max_weight = similarity * 0.9
            edge.weight_ab = min(edge.weight_ab, max_weight)
```

**similar_to 边的效果：同类型节点互相可达**

```
日常检索时不用 similar_to（精确记忆期）:
  花店 → located_at 边 → 搬花盆事件 → 找到了
  不会走到 花店 ──similar_to──→ 面包店

模糊检索时启用 similar_to（久远记忆）:
  花店 → 搬花盆（1跳）
       → similar_to → 面包店（2跳）→ "那天千早好像也..."
       → "具体在哪家店记不清了，反正在街上"
  → similar_to 让检索变成"模糊区域搜索"
  → 不再是精确的一跳直达，而是在一个模糊的节点团里扩散
```

**relationship 边详解：**

```
这是图中最重要的社交骨架。人物之间不通过事件中介，直接连接。

千早的图:
  "我" ←──→ 鹿岛樱    weight_ab=0.9  weight_ba=0.7
  "我" ←──→ 和叶       weight_ab=0.5  weight_ba=0.3
  "我" ←──→ 龙之介     weight_ab=0.6  weight_ba=0.2
  "我" ←──→ 九条       weight_ab=0.4  weight_ba=0.15
  "我" ←──→ 玩家       weight_ab=0.55 weight_ba=0.5

不对称原因：
  - 千早→樱 0.9：千早时刻想着樱（她最好且可能是唯一真正关心的朋友）
  - 樱→千早 0.7：樱也重视千早，但樱的内心世界更安静，不会时刻挂念
  - 千早→龙之介 0.6：想跟他做朋友但不知道怎么接近
  - 龙之介→千早 0.2：太耀眼了不敢看

关系的演化：
  - relationship 边权重随互动更新（每次重要互动 ±0.01~0.05）
  - 长期不互动自然衰减（极慢，约 0.005/天）
  - 重大事件可大幅改变（吵架 -0.3，被救了 +0.3）
```

**为什么事件中介不够——例子：**

```
没有 relationship 边：
  千早想"樱最近怎么样"
  → 千早 → 面包 → 送面包 → 樱（绕三跳）
  → 或者 千早 → 某个共同事件 → 樱（绕两跳）
  → 每次想到樱都要经过一个事件，这不对

有 relationship 边：
  千早想"樱最近怎么样"
  → 千早 ──relationship 0.9──→ 樱（直达）
  → 然后从樱节点出发扩散：樱最近 → 心事 → 看起来很累 → ...
  → 自然、直接
```

### 边的遍历代价

```
traversal_cost = 1 / weight（取当前方向的权重）

A → B: cost = 1 / weight_ab
B → A: cost = 1 / weight_ba

强方向 weight 0.9 → cost ≈ 1.1  几乎不费力
中方向 weight 0.5 → cost = 2.0   需要集中
弱方向 weight 0.3 → cost ≈ 3.3   很费劲
极弱   weight 0.1 → cost = 10    需要高度集中/特殊状态
```
**弱边不是不能走，是贵。** 同样预算下，沿强方向走 5 跳比沿弱方向走 2 跳更轻松——人脑的"省力优先"。

## 2. 思考能量预算（Think Budget）

检索深度不由跳数决定，由预算决定。

```
budget = base_budget × energy_factor × bond_factor × emotion_factor

base_budget（性格基线）:
  千早=3, 龙之介=4, 九条=5, 鹿岛樱=6, 和叶=8

energy_factor:
  > 70 → 1.2,  30-70 → 1.0,  < 30 → 0.4

bond_factor:
  > 0.6 → 1.5,  0.2-0.6 → 1.0,  < 0.2 → 0.6

emotion_factor:
  平静=1.2, 低落=1.3, 开心=0.9, 焦虑=0.5, 疲惫=0.3
```

**检索过程：预算逐边消耗，耗尽即停。强边路径消耗少走得远，弱边路径消耗多走不动。**

### 情绪对边代价的瞬时调节

情绪不只影响预算总量，还动态调节**特定类型边**的代价：

```
低落时:
  指向负面情绪的边 cost × 0.5（更容易想起难过的事）
  指向正面情绪的边 cost × 1.5（快乐的事走不动）

开心时:
  指向正面情绪的边 cost × 0.5
  指向负面情绪的边 cost × 1.5

焦虑时:
  指向担忧对象的边 cost × 0.3（钻牛角尖——极强的路径锁定）
  指向其他事物的边 cost × 2.0（无法分心）
```

### 每次遍历的边被强化

```
edge.weight += 0.01（每次被成功遍历时）
→ "思维习惯"或"叙事模板"的形成
→ 常走的路越来越容易走
→ 但也意味着新联想路径更难以被发现
```

## 3. 四种检索策略

NPC 不只有一种思考方式。在进入图之前，先选择策略。

| 策略 | 走法 | 代价/跳 | 典型产物 | 对应认知模式 |
|------|------|---------|----------|-------------|
| **广度优先** | 邻边各走一步，均匀扩散 | 中等 | 自由联想、发散 | 头脑风暴、"说起来..." |
| **深度优先** | 选最强边一路走到底 | 高 | 追忆、沉思 | 专注回忆、钻牛角尖 |
| **最佳优先** | 永远走 weight 最高的边 | 最低 | 安全、不出错 | 日常寒暄、自动回复 |
| **最轻松优先** | 永远走 cost 最低的边 | 接近零 | 敷衍、节能 | 疲惫/不感兴趣 |

**同一个起点，四种策略走出四条不同的路：**

```
起点: 吃火锅

广度（千早发散时）:
  千早 → 毛肚 → 辣 → 和叶 → 那天很晚了
  不深追任何一条，覆盖面广
  产物: "火锅！毛肚！超辣的！和叶好像也来了吧？啊还有另一家也去过！"

深度（和叶沉思时）:
  和叶 → 那天很安静 → 他在看乱步 → 《人间椅子》→ 
  关于孤独 → 和叶自己也在写类似的东西 → 他从没给人看过
  一条路走到底
  产物: "那天的火锅...和叶一直在看书。我觉得他有很多话没说。"

最佳（九条日常）:
  吃火锅 → 吃火锅最强的边是 → 搭档（weight 0.9，但指向禁区）
  → 绕过 → 第二强：一个人吃（weight 0.7）
  产物: "...还行。一个人吃惯了。"

轻松（龙之介被问时）:
  吃火锅 → cost 最低的边 → 自己（cost 1.1）
  停了。预算还有，但策略不允许走更贵的边。
  产物: "......不太吃。"
```

## 4. 策略选择器

策略由性格 + 状态 + 上下文共同决定。**这个选择本身最能体现"这个人是谁"。**

### 性格基线（概率分布）

| NPC | 广度 | 深度 | 最佳 | 轻松 | 说明 |
|-----|------|------|------|------|------|
| 千早 | 40% | 10% | 30% | 20% | 发散型，精力消耗大时掉到轻松 |
| 鹿岛樱 | 20% | 40% | 30% | 10% | 习惯深想，但输出时过滤为最佳 |
| 和叶 | 30% | 50% | 15% | 5% | 天生沉思者，几乎不走轻松 |
| 龙之介 | 5% | 15% | 30% | 50% | 社交焦虑让轻松成默认（独处时可用深度） |
| 九条 | 5% | 20% | 45% | 30% | 图里有禁区，高度依赖走安全边 |

### 状态修正

```
精力:
  > 70:  策略偏好不变（按性格）
  30-70: 深度↓ 轻松↑
  < 30:  轻松主导，深度几乎不可能

情绪:
  开心: 广度↑（更发散、更愿意尝试弱边）
  平静: 深度↑（最适合沉思）
  低落: 深度↑但路径被情绪过滤（只沿负面路径深入）
  焦虑: 深度↑但路径锁定（钻牛角尖——不是深思，是循环）
  疲惫: 轻松↑↑

社交开放度（影响输出，不影响内部检索）:
  > 70:  想到什么说什么
  30-70: 正常过滤
  < 30:  想到但不说——内部检索照常，输出极简

羁绊度:
  > 0.6: 愿意走远（预算放大 ×1.5）→ 深度和广度都可能
  < 0.2: 最小投入（预算缩小 ×0.6）→ 最佳或轻松

对话深度:
  对方问浅问题 → 轻松/最佳概率上升
  对方问深问题 → 深度概率强制上升，预算放大
  对方分享脆弱 → NPC 可能切换到深度以回应
```

### 策略可在一轮对话中动态切换

人有淺層對話突然變得認真的時刻，也有說完深話又切回輕鬆的時刻。策略在每次 LLM 路由決策時重新計算。

## 5. LLM 路由架构（实际实现）

**核心原则：LLM 不做检索，LLM 只看邻边做路由决策。图遍历的代价算法控制预算，LLM 控制方向。**

### 完整调用链条

```
对话触发

Step 1: 图查询（SQLite，非 LLM）
  输入: 起点节点ID
  输出: 直接邻边列表 [{target_summary, weight, cost, edge_type}]
  成本: 1 次 SQL 查询

Step 2: LLM 路由决策（轻量 Prompt）
  输入:
    - 邻边列表（带权重和摘要）
    - NPC 当前状态（精力/情绪/羁绊/预算）
    - 策略概率分布（从策略选择器计算）
    - 当前场景上下文
  输出:
    - 选择的策略类型
    - 选中哪些路径继续
    - 是否继续深入下一跳
  成本: 1 次轻量 LLM 调用（结构化输出，不需流式）

Step 3: 图查询（SQLite，非 LLM）
  输入: LLM 选中的节点 ID 列表
  输出: 节点的完整内容
  成本: 1 次 SQL batch 查询

Step 4: 如果 LLM 决定继续深入
  → 以新节点为起点，回到 Step 1
  → 直到 LLM 决定停止或预算耗尽
  → 日常对话通常 1-2 轮路由，深度反思 3-5 轮

Step 5: 记忆重建（一次 LLM 调用）
  输入: 所有选中的节点 + 策略类型 + NPC 性格 + 原文存档（仅必要时）
  输出: 自然语言回忆/对话
  成本: 1 次完整 LLM 调用（流式输出）

总成本: 1-3 次轻量路由 + 1 次对话生成
```

### Step 2 路由 Prompt 示例

```
[系统]
你是鹿岛樱的记忆路由。根据当前状态从联想路径中选择回忆方向。

当前状态:
  精力: 65（预算约 5 单位）
  情绪: 平静
  对方: 玩家（羁绊度 0.55）
  策略概率: 深度 38% | 最佳 28% | 广度 18% | 轻松 16%

联想路径（起点: 吃火锅）:
  1. [0.9 cost≈1.1] 千早 — 她每次都来
  2. [0.7 cost≈1.4] 毛肚 — 千早必点
  3. [0.6 cost≈1.7] 辣 — 辣得很开心
  4. [0.3 cost≈3.3] 和叶也在 — 他那天说了句有趣的话
  5. [0.1 cost≈10] 那天你应该去扫墓 — 你没去

规则:
  - 每条路径消耗其 cost 对应的预算
  - 预算不足时不能选该路径
  - 性格倾向影响你的选择但不由它唯一决定
  - 路径 5 是禁区，日常不会选（除非极特殊状态）

输出 JSON:
{
  "strategy": "depth",
  "selected_paths": [1, 2, 3],
  "continue": true,
  "reasoning_brief": "他问起火锅的事。先从千早和毛肚开始回忆..."
}
```

## 6. 图演化（每日睡眠时触发）

```
nightly_evolution(npc_id):
    graph = load_graph(npc_id)
    
    // 1. 衰减
    for node in graph.nodes:
        node.energy -= node.decay_rate × (1 - importance_modifier)
        if node.energy < 0.1:
            node.absorb_edges_to_nearest_neighbor()  // 消失前转移边
            graph.remove(node)
    for edge in graph.edges:
        edge.weight -= edge.decay_rate
        if edge.weight < 0.1:
            graph.remove(edge)
    
    // 2. 融合检测
    candidates = find_merge_candidates(graph)
    // 条件: 同类型 + 向量余弦相似度 > 阈值 + 共享邻边
    for (a, b) in candidates:
        merged = merge(a, b)  // 新节点继承两节点的所有边，权重叠加
        graph.add(merged)
        graph.remove(a).remove(b)
    
    // 3. 反思生成
    clusters = detect_dense_clusters(graph)
    // 一组节点形成紧密簇（相互之间边很多）→ 生成反思节点
    for cluster in clusters:
        reflection = llm_generate_reflection(cluster)
        graph.add(reflection)  // 反思也是节点，type=reflection
    
    // 4. 强化激活路径
    for node in graph.recently_activated_nodes:
        node.energy += 0.02 × node.recent_recall_count
        for edge in node.connected_edges:
            edge.weight += 0.01
    
    save_graph(npc_id, graph)
```

### 节点融合示例

```
初始:
  2025.8.15 ──→ 吃火锅 ←── 2025.9.12
               吃火锅 ←── 2025.10.3

融合后:
  2025年 ──→ 吃火锅 ×3
             ├── 千早在场 ×2
             ├── 毛肚 ×1
             ├── 一个人吃 ×1
             └── 辣 ×2

重建时的产物:
  "2025年好像吃了好几次火锅。千早也在的那几次...
   具体记不清了。有次她点了毛肚，辣得我喝水喝到饱。"
  → 同样的事实，现在跨时间聚合了
```

## 7. 记忆重建

被选中的节点最终要变成 NPC 的自然语言回忆。重建质量取决于：

### 节点的精度决定了说法的确定程度

```
exact → NPC 确定地说出
  "那天千早点了毛肚。"

vague → NPC 模糊地带过
  "好像点了毛肚...不太确定了。"

gone → 节点不注入（LLM 看不到）
  （毛肚这个标签完全丢失，NPC 不会提到它）
```

### 不同 NPC 重建同一组节点

同样的节点 `{who:玩家, where:花店, what:帮忙搬花盆, emotion:感激}`：

| NPC | 重建版本 |
|-----|---------|
| **鹿岛樱** | "小李帮过我搬花盆。我记得那时候花盆很重，他一次就搬起来了。这个人做事很安静，但很稳。" |
| **千早** | "啊！那次！小李帮樱搬花盆！我就说他是个好人嘛！虽然话少了点但手从来不闲着！" |
| **九条** | "...嗯。那小子帮过忙。搬花盆还是什么来着。...挺好。" |

**同一组节点，三个不同性格重建出三个版本。这就是人脑——每次回忆是一次重建，性格决定重建的色调。**

### 印象影响重建

印象是人物节点的派生属性（周围簇的统计摘要）。重建时，当前印象会染色回忆：

```
鹿岛樱 → 玩家印象: "可靠、细心、最近有点疏远"

回忆"帮忙搬花盆"节点时:
  如果印象正面: "小李那时候很热心。搬完还帮我把门口扫了。"
  如果印象变差:  "小李以前帮过我搬东西。那时候还挺热心的。
                 现在...算了，人都会变吧。"
```

## 8. 双向联想与线索触发

图的另一个天然能力：

```
正向: 时间 → 事件（"那天发生了什么"）
反向: 事件 → 时间（"吃火锅是什么时候"）

新事件触发旧记忆:
  "今天又吃火锅了"
  → 激活图中所有"吃火锅"节点
  → 各自指向各自的时间和人物
  → "上次也是跟千早一起的..."

场景线索触发:
  花店 + 下雨 + 玩家站在门口
  → 三个起点节点的激活扩散重叠在一条旧记忆上
  → NPC: "上次下雨的时候，你也是站在这里。"
  → 不是主动检索——是场景让记忆自己浮上来的
```

## 9. 数据库设计

### SQLite

```sql
-- 节点表
CREATE TABLE memory_nodes (
    id TEXT PRIMARY KEY,
    subject_id TEXT NOT NULL,         -- 哪个 NPC 的图
    type TEXT NOT NULL,               -- time|person|event|place|emotion|sensory|item|quote|reflection
    value TEXT NOT NULL,              -- 节点文本值
    energy REAL DEFAULT 1.0,          -- 存活度 (0~1)
    decay_rate REAL DEFAULT 0.01,
    precision TEXT DEFAULT 'exact',   -- exact|vague|gone
    importance REAL DEFAULT 0.5,      -- 初始重要性
    recall_count INTEGER DEFAULT 0,
    last_activated_at TEXT,
    merged_from TEXT,                 -- JSON array of IDs（融合来源）
    original_text TEXT,               -- 原文存档（尽量不用于检索）
    created_at_game_time TEXT NOT NULL,
    created_at TEXT DEFAULT (datetime('now'))
);

-- 边表（双向不对称权重）
CREATE TABLE memory_edges (
    id TEXT PRIMARY KEY,
    node_a TEXT NOT NULL,             -- 端点 A
    node_b TEXT NOT NULL,             -- 端点 B
    type TEXT NOT NULL,               -- happened_at|involved|located_at|felt|contains|associated_with|caused|mentioned
    weight_ab REAL DEFAULT 1.0,       -- A → B 的联想强度
    weight_ba REAL DEFAULT 1.0,       -- B → A 的联想强度
    decay_rate_ab REAL DEFAULT 0.01,
    decay_rate_ba REAL DEFAULT 0.01,
    last_traversed_ab TEXT,
    last_traversed_ba TEXT,
    created_at_game_time TEXT,
    created_at TEXT DEFAULT (datetime('now'))
);

-- 索引
CREATE INDEX idx_nodes_subject ON memory_nodes(subject_id, type);
CREATE INDEX idx_nodes_energy ON memory_nodes(subject_id, energy DESC);
CREATE INDEX idx_edges_a ON memory_edges(node_a);
CREATE INDEX idx_edges_b ON memory_edges(node_b);
```

### LanceDB（向量存储）

每个 NPC 一个表：`graph_vectors_{npc_id}`

Schema: `node_id`, `vector(1024)`, `type`, `value`, `energy`

用途：
- 节点融合候选检测（余弦相似度）
- 新事件入图时找到最接近的已有节点
- 场景线索到起点节点的映射（场景向量 → 最近邻节点）

## 10. 与现有架构对应

| 原设计概念 | 图模型对应 |
|-----------|-----------|
| 工作记忆 | 当前激活态 — 本次检索激活的节点 + 邻边（注入 LLM） |
| 短期记忆 | 原文 + 新创建节点（7天内不参与融合，保留完整精度） |
| 长期记忆 | 图节点（7天后开始衰减和融合） |
| 核心记忆 | energy 锁定为 1.0 且 decay_rate=0 的节点和边 |
| 印象 | 人物节点周围簇的统计摘要（高频共现标签） |
| NPC 状态 | 人物节点的实时属性（独立存储，不衰减） |
| 记忆衰减 | 节点 energy + 边 weight 每日衰减 |
| 记忆融合 | 同类型+高相似度节点合并 |
| 记忆复活 | 低 energy 节点被新激活扩散重新唤醒（energy 回升） |
| 每日反思 | 紧密簇 → LLM 生成 type=reflection 节点 |
| 情绪系统 | 情绪调节边代价（动态路由偏置） |

## 11. 实现顺序建议

1. **节点/边 CRUD** — SQLite + LanceDB 基础操作
2. **事件入图** — 新事件发生 → LLM 提取节点和边 → 写入数据库
3. **邻边查询** — 给定起点节点，返回排序后的邻边列表
4. **LLM 路由** — 策略选择器 + 路由 Prompt + 结构化输出
5. **记忆重建** — 选中节点集合 → LLM 生成自然语言
6. **每日演化** — 衰减 + 融合检测 + 反思生成 + 路径强化
7. **策略选择器调优** — 根据实际游戏体验调整各 NPC 的策略概率

---

**Why:** 表格式标签无法模拟人脑的联想结构。图模型允许双向联想、节点融合、激活扩散——更接近海马体的实际运作方式。LLM 路由方案避免了全图遍历的性能问题，同时保留了 LLM 在策略选择上的灵活性。

**How to apply:** 每个 NPC 维护独立的图实例（SQLite 节点表+边表 + LanceDB 向量表）。对话触发时，从起点节点查邻边 → LLM 路由选择 → 按结果查库 → LLM 重建。每日睡眠时运行图演化循环（衰减→融合→反思→强化）。不实时遍历全图。
