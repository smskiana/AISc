### 1.1 为什么 Python 后端而不是纯 C#？

| 对比维度 | Unity C# 直连 | Unity + Python 后端 |
|----------|--------------|-------------------|
| **Embedding 库** | C# 生态薄弱 | sentence-transformers 一行代码 |
| **向量数据库** | 无成熟方案 | LanceDB/ChromaDB 开箱即用 |
| **LLM SDK** | 需手写 HTTP 调用 | 官方 SDK，streaming 支持好 |
| **调试/测试** | 需开 Unity | Postman/curl 直接调 |
| **热更新** | 需重编译 | 直接改 Python 文件 |
| **生态** | ML/AI 生态约等于零 | 整个 Python AI 生态 |

> **结论**：Python 做 AI 层是唯一合理的选择。Unity 通过 `localhost` HTTP 与后端通信，相当于 Unity 是前端，Python 是 BFF（Backend For Frontend）。

## 2. 联想记忆图（Graph Memory Model）★ v0.5 重写

> 旧版四层模型（工作/短期/长期/核心记忆）已废弃。新版采用联想记忆图——每个 NPC 独立维护一个有向加权图，模拟人脑的联想记忆结构。

### 2.1 核心概念

```
记忆不是记录的集合，而是节点与边的网络。

  节点（Node）= 概念单元    "千早""花店""吃火锅""开心""2025年夏天"
  边（Edge）= 联想关系      "吃火锅 → 千早在场""花店 → 那天下了雨"
  权重 = 联想强度           强=几乎自动想到 / 弱=需要特定触发
```

**每个 NPC 一个独立的图**。鹿岛樱的图和千早的图完全独立，节点和边的演化各自进行。NPC 之间不共享记忆。

### 2.2 节点（Tag Node）

每个节点是一个带向量嵌入的概念单元。节点通过边互连，形成联想网络。

```
{
  id, type, value, vector(1024维), energy(存活度 0~1),
  decay_rate, precision(exact|vague|gone),
  merged_from, created_at_game_time
}
```

**"我"节点**：每个 NPC 的图中，代表自己的节点值为 `"我"`，不是名字。鹿岛樱的图中没有"鹿岛樱"这个人物节点——她不叫自己名字。`"我"` 节点的 energy 恒为 1.0，永不衰减，是图中最强的引力中心。

**11 种节点类型与衰减速度**：

| 类型 | 示例 | 衰减速度 | 认知依据 |
|------|------|----------|----------|
| **自我** | "我" | **不衰减** | 每个 NPC 图中有且仅有一个 |
| **人物** | 千早, 玩家 | 几乎不衰减 | 社交记忆是人类进化核心 |
| **时间-精确** | 2025.8.15, 10:30 | **最快** | 海马体时间细胞精度急剧下降 |
| **时间-模糊** | 2025年夏天 | 较稳定 | 模糊后不再衰减 |
| **时间-周期** | 秋天, 傍晚 | 很稳定 | 循环事件反复强化 |
| **地点-精确** | 花店门口 | 慢 | 空间记忆有专门的海马位置细胞 |
| **地点-模糊** | 花店, 街上 | 稳定 | 模糊后稳固 |
| **事件** | 吃火锅, 搬花盆 | 中等 | 情节记忆自然衰减 |
| **情绪** | 开心, 感激 | **极慢** | 杏仁核标记的情绪几乎终生保留 |
| **感官** | 辣, 很重, 雨声 | 快 | 感官细节最先丢失 |
| **物品** | 毛肚, 花盆 | 中等 | 依附于事件 |
| **话语** | 某句具体的话 | **最快** | 逐字记忆极不可靠 |
| **反思** | "小李帮了我很多次" | 慢 | 高层抽象比原始事件稳定 |

### 2.3 边（Association）

节点之间的指向关系。**每条边存两个方向的独立权重——联想是不对称的。**

```
{
  id, node_a, node_b, type,
  weight_ab(0~1),        // A → B 的联想强度
  weight_ba(0~1),        // B → A 的联想强度
  decay_rate_ab, decay_rate_ba,
  created_at_game_time
}
```

**为什么不对称**：
- 吃火锅 → 千早 weight=0.9 "吃火锅立刻想到千早"
- 千早 → 吃火锅 weight=0.3 "千早关联太多事，吃火锅只是其中一件"
- 触发线索（感官/地点/时间）到事件的联想通常更强——线索少，指向明确

**遍历代价 = 1 / weight**（取当前方向的权重）。cost 从 ~1.1（几乎免费）到 10+（极困难），NPC 的思考预算逐边消耗。

**11 种边类型**：

| 类型 | 适用范围 | 含义 | 典型不对称 |
|------|---------|------|-----------|
| **relationship** | 人物↔人物 | 直接人际关系 | 双向但不对等 |
| **similar_to** | **任意同类型↔同类型** | 模糊记忆中同类节点互相渗透 | 双向，向量相似度决定 |
| **sequenced** | 事件↔事件 | 同一时空的先后顺序 | 前→后强 / 后→前弱 |
| happened_at | 事件↔时间 | 事件发生的时间 | 事件→时间强 / 时间→事件弱 |
| involved | 事件↔人物 | 事件的参与者 | 事件→人物强 / 人物→事件弱 |
| located_at | 事件↔地点 | 事件发生的地点 | 事件→地点强 / 地点→事件中 |
| felt | 事件↔情绪 | 事件伴随的情绪 | 双向均强 |
| contains | 事件↔子事件/物品 | 组成关系 | 父→子强 / 子→父中 |
| associated_with | 任意↔任意 | 通用弱连接 | 取决于共现频率 |
| caused | 事件↔事件 | 明确因果关系 | 因→果强 / 果→因弱 |
| mentioned | 话语↔话题 | 话语涉及的话题 | 话语→话题强 / 话题→话语弱 |

### 2.4 与旧版四层模型的对应

| 旧版概念 | 图模型对应 |
|---------|-----------|
| 工作记忆 | 本次检索激活的节点集合（注入 LLM） |
| 短期记忆 | 原文 + 新创建节点（7天内不参与融合，保留完整精度） |
| 长期记忆 | 图节点（7天后开始衰减和融合，走 LLM 路由检索） |
| 核心记忆 | energy=1.0 且 decay_rate=0 的锁定节点和边 |

### 2.5 精度转换规则

节点的 `precision` 字段随 `energy` 下降而转换。**原地更新，不创建新节点。**

```
触发条件:
  exact → vague: energy < 0.3
  vague → gone:  energy < 0.1

转换时执行:
  1. LLM 轻量调用,将 value 重写为模糊版本
     例如: "Day3_10:30" → "那天上午"
           "花店.counter" → "花店"
           "感激" → "一种温暖的感觉"
     
  2. 向量重新生成（基于新的模糊 value）
     → 旧向量从 LanceDB 删除，新向量写入
     
  3. precision 字段更新
     → energy 不变（精度转换不改变存活度）

gone 状态:
  • 节点日常检索不可见
  • precision=gone, energy 保留在 0.05-0.1
  • 仍可通过复活机制被强线索重新激活
  • 复活后 precision 恢复为 vague（不会回到 exact）
```

### 2.6 节点融合规格

两个同类型节点 similar_to 边权重 > 0.85 时触发融合。

```
融合产物:
  vector    = weighted_avg(A.vector × A.energy, B.vector × B.energy)
  value     = LLM 轻量生成（如两个时间节点 → 一个模糊时间段）
  energy    = min(1.0, A.energy + B.energy)  ← 融合后更稳固
  precision = 取两者中更模糊的那个
  merged_from = [A.id, B.id]  ← 保留来源链

边冲突处理:
  A → X weight=0.7, B → X weight=0.5
  → 融合后 AB → X weight = max(0.7, 0.5) = 0.7
  → 同方向冲突取最大值（最熟悉的路径保留）

  A → X weight_ab=0.8, B → X weight_ba=0.3（方向不同）
  → 融合后 AB ↔ X: weight_ab=0.8, weight_ba=0.3
  → 不同方向各自保留

融合链:
  A+B→AB, AB+C→ABC
  → ABC.merged_from = [AB.id, C.id]
  → 递归展开 AB.merged_from = [A.id, B.id]
  → 完整来源: [A, B, C]
```

### 2.7 情绪节点去重规则

情绪节点与人物节点不同——情绪是**半去重**的。

```
同一种情绪可以在图中存在多次（不同事件产生不同"担心"实例），
但新事件入图时：
  • 如果新情绪与已有情绪节点向量相似度 > 0.9 → 复用已有节点
  • 如果 0.7-0.9 → 创建新节点 + 建立 similar_to 边连接两者
  • 如果 < 0.7 → 创建独立新节点

例: 千早送面包时的"担心" + 千早几天没来时的"担心"
   → 向量相似度 0.85
   → 两个独立的"担心"节点 + similar_to 0.7 边
   → 随时间推移 → similar_to 强化 → 可能融合为"那阵子一直在担心"
```

### 2.8 事件入图流程 ★ 延迟到睡眠时执行

白天对话中**不提取节点**。原文存入 `short_term_memories`，午夜睡眠时批量提取。

```
事件发生（玩家说话 / NPC 之间互动）
  │
  ▼
Step 1: 生成原始记忆文本
  "今天上午千早来送面包。她笑得有点勉强，说'没事没事'。"
  │
  ▼
Step 2: 短期存储（原文）
  写入 short_term_memories 表
  → 白天对话中全量注入 Prompt（不经过图）
  │
  ▼
（午夜 24:00 — 以下步骤在睡眠中执行）
  │
  ▼
Step 3: LLM 批量提取节点和边
  
  提取的节点:
    event:     "千早送面包，看起来有心事"
    time:      "Day3_10:00" (type=时间-精确, energy=1.0)
    place:     "花店.counter" (type=地点-精确, energy=1.0)
    person:    "千早" (已存在 → 不重复创建，只连边)
    person:    "我" (已存在)
    emotion:   "担心" (new, type=情绪, energy=0.9)
    sensory:   "面包很好吃" (type=感官, energy=0.5)
    item:      "南瓜面包" (type=物品, energy=0.6)
    quote:     "没事没事" (type=话语, energy=0.4)
  
  提取的边:
    事件 → 时间:     happened_at 0.9
    事件 → 地点:     located_at 0.85
    事件 → 千早:     involved 0.9
    事件 → "我":     involved 0.95
    事件 → 担心:     felt 0.7
    事件 → 面包好吃: felt 0.4 (弱:感官细节)
    事件 → 南瓜面包: contains 0.6
    事件 → "没事":   mentioned 0.5
    千早 → "没事":   mentioned 0.5 (千早说了这句话)
    "我" → 千早:     relationship 0.7 (已存在，本次微调)

  │
  ▼
Step 4: 写入数据库
  ├─ 新节点 → INSERT INTO memory_nodes
  ├─ 新边   → INSERT INTO memory_edges
  ├─ 向量   → INSERT INTO LanceDB graph_vectors_{npc_id}
  └─ 已有节点/边 → UPDATE energy/weight（如有变化）
  │
  ▼
Step 5: 触发已有节点关联
  • 新事件节点与图中已有节点的向量相似度检测
  • 相似度 > 0.6 且 < 0.85 → 建立 associated_with 边
  • 相似度 > 0.85 → 标记为融合候选（夜间演化时处理）
```

**节点去重规则**：人物节点全局唯一——"千早"节点在鹿岛樱的图中只有一个。新事件不创建重复的人物节点，直接在已有节点上连 involved 边。

**原文保留**：原始文本存入 `memory_nodes.original_text`，7 天内额外保留在 `short_term_memories` 用于全量注入。7 天后仅归档在节点中，日常检索不走原文。

## 3. 数据库设计

### 3.1 节点表（SQLite）

```sql
CREATE TABLE memory_nodes (
    id TEXT PRIMARY KEY,
    subject_id TEXT NOT NULL,         -- 哪个 NPC 的图
    type TEXT NOT NULL,               -- self|person|event|time_exact|time_vague|time_cycle|place_exact|place_vague|emotion|sensory|item|quote|reflection
    value TEXT NOT NULL,              -- 节点文本值
    energy REAL DEFAULT 1.0,          -- 存活度 (0~1)，<0.1 视为消失
    decay_rate REAL DEFAULT 0.01,
    precision TEXT DEFAULT 'exact',   -- exact|vague|gone
    importance REAL DEFAULT 0.5,
    recall_count INTEGER DEFAULT 0,
    last_activated_at TEXT,
    merged_from TEXT,                 -- JSON array，融合来源
    original_text TEXT,               -- 原文存档（尽量不用于检索）
    is_core INTEGER DEFAULT 0,        -- 是否为核心记忆（锁定不衰减）
    created_at_game_time TEXT NOT NULL,
    created_at TEXT DEFAULT (datetime('now'))
);
CREATE INDEX idx_nodes_subject ON memory_nodes(subject_id, type);
CREATE INDEX idx_nodes_energy ON memory_nodes(subject_id, energy DESC);
```

### 3.2 边表（SQLite）

```sql
CREATE TABLE memory_edges (
    id TEXT PRIMARY KEY,
    node_a TEXT NOT NULL,
    node_b TEXT NOT NULL,
    type TEXT NOT NULL,               -- relationship|similar_to|sequenced|happened_at|involved|located_at|felt|contains|associated_with|caused|mentioned
    weight_ab REAL DEFAULT 1.0,       -- A → B
    weight_ba REAL DEFAULT 1.0,       -- B → A
    decay_rate_ab REAL DEFAULT 0.01,
    decay_rate_ba REAL DEFAULT 0.01,
    last_traversed_ab TEXT,
    last_traversed_ba TEXT,
    created_at_game_time TEXT,
    created_at TEXT DEFAULT (datetime('now'))
);
CREATE INDEX idx_edges_a ON memory_edges(node_a);
CREATE INDEX idx_edges_b ON memory_edges(node_b);
```

### 3.3 向量存储（LanceDB）

每个 NPC 一个表：`graph_vectors_{npc_id}`。Schema: `node_id`, `vector(1024)`, `type`, `value`, `energy`。用途：节点融合候选检测（余弦相似度）、新事件入图时找最近邻已有节点、场景线索到起点节点的映射。

### 3.4 短期记忆原文表（保留）

```sql
-- 7天内的原文记忆，不参与图检索，全量注入 Prompt
CREATE TABLE short_term_memories (
    id TEXT PRIMARY KEY,
    subject_id TEXT NOT NULL,
    content TEXT NOT NULL,
    importance REAL DEFAULT 0.5,
    created_at_game_time TEXT NOT NULL,
    created_at TEXT DEFAULT (datetime('now'))
);
```
7 天后原文迁移到 `memory_nodes.original_text` 归档，短期表删除对应行。

## 4. 四阶段记忆退化模型 ★ v0.5 新增

> 超长久记忆不是一次性"忘了"，而是分阶段从"精确事件"逐步压缩成"模糊印象"，最终变成"知道发生过但细节全丢"。

### 4.1 退化全景

```
Stage 1: 模糊细节 (7-30天)      Stage 2: 同类渗透 (30-90天)
┌──────────────────────┐       ┌──────────────────────┐
│ 时间精确→模糊         │       │ similar_to 边建立      │
│ 地点精确→模糊         │       │ 同类节点互相指向        │
│ 话语节点消失          │       │ sequenced 边建立       │
│ 感官节点消失          │       │ 反思节点出现           │
│ 跨类型边仍在          │       │ 节点开始成团           │
└──────────────────────┘       └──────────────────────┘

Stage 3: 融合压缩 (90-365天)    Stage 4: 骨架残留 (1年+)
┌──────────────────────┐       ┌──────────────────────┐
│ 同类型节点融合         │       │ 原始事件 energy≈0     │
│ 反思节点成为主入口     │       │ 反思节点还在           │
│ 原始事件 energy 大幅降 │       │ 情绪只剩正/负两极      │
│ 只剩"发生过+感觉"      │       │ 核心记忆保留           │
│ sequenced 边开始衰减   │       │ "知道有这么回事"       │
└──────────────────────┘       └──────────────────────┘
```

### 4.2 Stage 1：模糊细节（7-30天）

**触发**：节点 energy 自然衰减到阈值。

| 变化 | 说明 |
|------|------|
| 时间-精确 → 时间-模糊 | "10:30"→"上午"→"那天" |
| 地点-精确 → 地点-模糊 | "花店.counter"→"花店" |
| 话语节点消失 | energy < 0.1，具体措辞忘记 |
| 感官节点消失 | energy < 0.1，感官细节丢失 |

**不变**：事件节点本身、人物/情绪节点、跨类型边（involved/felt/located_at）仍在。

### 4.3 Stage 2：同类渗透（30-90天）

**触发**：夜间演化检测同类节点向量相似度 > 0.65。

**核心机制 — similar_to 边**：适用于**任意同类型节点**。这是模糊记忆的物理基础。

```
时间团:  秋天 ↔ 上个月 ↔ 冬天 → "那阵子"
情绪团:  感激 ↔ 温暖 ↔ 安心 → "正面的感觉"
地点团:  花店 ↔ 面包店 ↔ 喫茶店 → "街上的店"
物品团:  毛肚 ↔ 牛肉 ↔ 豆腐 → "火锅菜"
事件团:  搬花盆 ↔ 修招牌 ↔ 浇水 → "帮过很多忙"
人物团:  和叶 ↔ 龙之介 → "都是安静的人"（不同于 relationship！）
```

**事件↔事件 special case**：
- `sequenced` 边：时间锚消失前自动建立，保留叙事链（"先搬花盆→再修招牌"）
- `similar_to` 边：向量相似 + 共享人物节点时建立（"同类帮忙事件"）
- 紧密簇 → 触发反思节点生成

### 4.4 Stage 3：融合压缩（90-365天）

**触发**：similar_to 边权重反复强化到 > 0.85。

**融合**：两个同类型高相似节点合并为一个：

```
三个时间 → "2025年"
四种正面情绪 → "好的感觉"（情绪团崩溃为大类）
多次帮忙事件 → 反思节点接管，原始事件 energy 降到 0.3
```

**关键：原始节点不删除，但变得不可达**：

```
反思 "小李帮了我很多忙" ← cost 1.3，每次都走到
  └─ contains 0.8 → 搬花盆(原始) ← cost 5+，日常走不到
                     修招牌(原始) ← cost 5+
                     浇水(原始)   ← cost 5+
```

除非玩家主动提起"那次搬花盆..."→ 强制激活原始事件 → 事件 energy +0.05（回忆强化）→ "啊...对，是有这么回事。我都快忘了。"

### 4.5 Stage 4：骨架残留（1年+）

**触发**：持续衰减，无新事件强化。

- 原始事件 energy < 0.1，几乎不可检索
- sequenced 边消失（叙事顺序彻底丢失）
- 反思节点之间出现 similar_to 边 → 更高层抽象
- 情绪团压缩为"正面"/"负面"两极
- **最终残留**："我" + 人物 + relationship 边 + 核心记忆（锁定）+ 高层反思节点 + 模糊情绪两极

### 4.6 各节点类型在四阶段中的命运

| 节点类型 | Stage 1 | Stage 2 | Stage 3 | Stage 4 |
|---------|---------|---------|---------|---------|
| **"我"** | 不动 | 不动 | 不动 | 不动 |
| **人物** | 不动 | 不动 | 不动 | 不动 |
| **时间-精确** | →模糊 | similar_to成团 | 融合为模糊时间 | 消失或极模糊 |
| **时间-模糊** | 不动 | similar_to成团 | 融合 | 可能消失 |
| **地点-精确** | →模糊 | similar_to成团 | 融合为"街上" | "街上" |
| **地点-模糊** | 不动 | similar_to成团 | 融合 | "街上" |
| **事件** | 不动 | sequenced+similar_to | 反思节点接管 | energy≈0 |
| **情绪** | 不动 | similar_to成团 | 融合为正面/负面 | 两极 |
| **感官** | **消失** | — | — | — |
| **话语** | **消失** | — | — | — |
| **物品** | 不动 | similar_to成团 | 融合为类别 | 消失或类别 |
| **反思** | — | 生成 | 成为主入口 | 仍在（慢衰减） |

### 4.7 记忆复活

低 energy 节点不一定永久消失。强线索可以重新激活它们。

```
复活触发条件:
  1. 新事件节点与"已消失"节点（energy < 0.1）向量相似度 > 0.85
  2. 激活扩散中，一条强边意外跨越了已消失节点（某个场景线索强制激活）
  3. 玩家明确提到某件事（"那次搬花盆你还记得吗？"）→ 强制激活

复活效果:
  • 节点 energy 恢复到 0.4（中等水平，不比新鲜时）
  • 节点关联的边 weight 恢复 50%
  • 该节点的 precision 可能仍为 vague（细节回不来）
  • 可能在对话中表现为突然的回忆:
    "啊...对，是有这么回事。我都快忘了。"
    "你这么一说...我想起来了。那天..."
  
复活后:
  • 该节点重新参与衰减和检索
  • 如果之后不再被激活 → 再次衰减消失（比第一次更快）
  • 如果频繁被激活 → energy 回升，边强化，重新稳定
```

**复活 ≠ 恢复原样**：复活的记忆是模糊的。precision 和 energy 不会回到初始水平。NPC 说"我想起来了"但说不出细节——这才是真实的。

### 4.8 一条记忆的完整一生

```
Day 1 — 事件发生
  "今天上午千早来送面包。笑得有点勉强，说'没事'。"
  图: 事件 + 时间精确 + 地点精确 + 情绪精确 + 话语 + 感官

Day 30 — Stage 1 模糊细节
  话语 "没事" → 消失  |  感官 "面包好吃" → 消失
  时间 → "那天上午"   |  地点 → "花店"
  → "千早那天来送面包，笑得有点勉强。好像在担心什么。"

Day 90 — Stage 2 同类渗透
  送面包×3 → similar_to 互联 → 反思 "她最近来送面包都不太对劲"
  情绪 "担心" + "不安" → 情绪团
  → "千早最近送面包的时候总感觉她不太开心。好几次了。"

Day 365 — Stage 3 融合压缩
  具体事件 energy < 0.3，反思节点成为入口
  情绪团 → "一种不太好的感觉"
  → "千早那段时间好像有什么心事。具体记不太清了。"

Day 1000 — Stage 4 骨架残留
  原始事件 energy < 0.1，多反思融合为更高层抽象
  → "千早有段时间状态不太好。后来慢慢好了。现在这样就够了。"
```

## 5. 印象系统（Impression System）★ v0.5 融入图模型

> 印象不再作为独立文本存储。印象是图中**人物节点的派生属性**——从该人物节点周围簇的统计摘要计算得出。

### 5.1 核心变更

| 旧版 | v0.5 |
|------|------|
| 印象存储为独立文本（"小李是个温柔的人..."） | 印象 = 人物节点周围簇的统计摘要 |
| 印象由 LLM 在睡眠时生成和更新 | 印象从图结构中实时推导 |
| 独立表 `graph_vectors_impressions` | 融入人物节点 + relationship 边 |

### 5.2 印象的图模型表达

```
鹿岛樱的图中，"玩家"人物节点的印象 = 以下要素的聚合：

1. relationship 边的权重和历史变化趋势
   "我" ←──→ 玩家  weight_ab=0.4 weight_ba=0.55
   → bond 数值，recent_trend

2. 指向"玩家"的所有 involved 边的统计
   有多少事件连向玩家？情绪分布如何？
   → dominant_trait（"可靠""有心事"等高频标签）

3. "玩家"节点与反思节点的关联
   反思 "小李帮了我很多忙" 包含了指向玩家的边
   → 印象的"based_on"来源

4. similar_to 边
   玩家 ↔ 其他人物（"他让我想起..."）
   → 印象中的比较性描述
```

### 5.3 印象在 Prompt 注入时的重建

印象不再从存储中读取，而是在每次对话时从图结构实时重建：

```
LLM 收到的人物节点邻域摘要：
  人物: 玩家
  relationship: bond=0.55, trend=warming
  最近10条 involved 事件的情绪: 感激×3, 温暖×2, 担心×1, 开心×2
  关联反思: "小李帮了我很多次""他最近好像有心事"
  关联 similar_to: (无)

→ LLM 在 System Prompt 中重建印象:
  "你对小李的感觉: 一个可靠但有心事的人。帮过你很多次，
   你很感激。最近觉得他状态好了些。"
```

### 5.4 设计原则（保留）

| 原则 | 说明 |
|------|------|
| **独立性** | 每个 NPC 的图独立，印象自然独立 |
| **不对称性** | relationship 边双向不同权重 → 印象不对称 |
| **模糊性** | 从统计摘要重建，天然模糊 |
| **缓慢性** | 边权重缓慢更新，印象不会突变 |
| **简洁性** | 重建时控制 ~50 tokens 内 |

### 5.5 印象的更新机制

印象在**两个时机**更新：

#### 时机 A：实时微调（每次重要互动后）

```
玩家和鹿岛樱的互动结束
  │
  ▼
判断：这次互动是否"有意义"？（importance ≥ 0.5）
  │
  ├── 否 → 不更新印象
  │
  └── 是 → 微调 impression_text
      │
      ▼
      提取本次互动的关键信息：
      "小李今天说他想把喫茶店改成深夜也能来的地方。
       他聊到这个的时候眼睛亮了，跟平时不一样。"
      │
      ▼
      与现有印象融合（LLM 轻量调用）：
      旧印象 + 新观察 → 更新后的印象
      "小李平时很安静，但聊到喫茶店的时候会变得很
       有热情。他可能找到了真正想做的事。"
      │
      ▼
      更新 affinity / trust / familiarity（数值小幅变化）
```

#### 时机 B：睡眠反思时（每日批量）

```
NPC 睡眠 → 反思阶段
  │
  ▼
遍历该 NPC 今天所有的互动对象
  │
  ▼
对每个对象，做一次"印象刷新"：
  "今天跟小李互动了 3 次。
   旧印象：小李是个温柔但逃避的人。
   今天的事：他修好了柜台的灯（动手能力强），
            聊到以前的工作时沉默了（有心事），
            给千早送了杯咖啡（对别人也很好）。
   请更新对小李的印象。保持模糊和主观。"
  │
  ▼
输出新印象 → 写入 impression_text
同时更新数值（亲和度、信任度、熟悉度）
```

### 5.6 印象在 Prompt 中的注入

印象是 Prompt 中**优先级最高**的注入内容之一：

```
# 你与对方的印象
你对{target_name}的总体印象：
{impression_text}

你对TA的感觉：
- 羁绊度：{bond_desc}  ← "很喜欢也很信任" / "还不错" / "普通" / "不太喜欢" / "讨厌"
- 信任等级：{confide_desc} ← "可以托付秘密" / "还算信得过" / "不太放心说私事" / "不熟"

# 记忆检索时使用
查询向量 = encode(impression_text + " " + current_context)
→ 用印象作为检索锚点，找到与"当前关系阶段"最匹配的记忆
```

### 5.7 与记忆的关系：互补而非替代

```
┌────────────────────────────────────────────────┐
│                 Prompt 组装时                    │
│                                                │
│  优先级 1: 印象（总是注入）                      │
│  "你对小李的总体感觉：..."  ← 1-3 句话，锚定基调  │
│                                                │
│  优先级 2: 相关记忆（语义检索注入）               │
│  "你记得和小李的这些事：..."  ← Top 5-8 条       │
│                                                │
│  优先级 3: 短期记忆（时间检索注入）               │
│  "最近你们之间发生了：..."  ← 最近 5-10 条       │
└────────────────────────────────────────────────┘

为什么印象在最上面？
→ 印象决定了 NPC 用什么样的"底色"来看待这些记忆。
   同样一句"小李帮我修了灯"——
   印象好 → "这年轻人真不错"
   印象差 → "他是不是想讨好我？"
```

## 6. NPC 状态模型

> 4 个数值维度 + 1 个自由文本心事。少而精，每个都驱动可感知的行为变化。

### 6.1 状态总览

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  即时状态（随时间/事件波动，玩家可直接感知）               │
│  ┌──────────┐  ┌──────────┐  ┌──────────────┐          │
│  │ 情绪      │  │ 精力      │  │ 社交开放度    │          │
│  │ Emotion   │  │ Energy   │  │ Sociability  │          │
│  │           │  │          │  │              │          │
│  │ 6 种状态  │  │ 0~100    │  │ 0~100        │          │
│  └──────────┘  └──────────┘  └──────────────┘          │
│                                                         │
│  持久关系（跨时间累积，玩家不可直接看到数值）              │
│  ┌──────────────────────────────────────┐               │
│  │ 羁绊度 Bond                          │               │
│  │ -1.0 ~ +1.0（对每个角色独立）          │               │
│  │ 衍生：confide_level（信任等级，行为体现）│               │
│  └──────────────────────────────────────┘               │
│                                                         │
│  隐藏心事（LLM 生成和维护，驱动自主行为）                 │
│  ┌──────────────────────────────────────┐               │
│  │ 当前心事 current_need（自由字符串）     │               │
│  └──────────────────────────────────────┘               │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 6.2 情绪（Emotion）

最直接影响对话语气和内容选择。

```
取值：开心 | 平静 | 低落 | 焦虑 | 生气 | 疲惫

驱动因素：
  • 近期记忆的情感标记（今日正面记忆多 → 偏开心）
  • 精力的连带效应（精力 < 20 → 容易"疲惫"或"低落"）
  • 互动的即时反馈（被夸了 → 开心；被冒犯了 → 生气）
  • 角色性格振幅（千早情绪波动大，鹿岛樱波动小）

行为影响：
  开心 → 话多、主动、容易答应请求
  平静 → 按性格行事（默认状态）
  低落 → 话少、反应慢、敏感、可能触发深度对话
  焦虑 → 心不在焉、可能主动求助
  生气 → 简短回应、可能拒绝互动
  疲惫 → 容易结束对话、可能闭店休息

表面 vs 真实：
  • 某些角色会隐藏真实情绪（鹿岛樱的心病、九条的过去）
  • 隐藏时，表面显示"平静"，真实情绪只在深度互动中暴露
  • Bond > 0.6 的角色更可能看到真实情绪
```

### 6.3 精力（Energy）

```
取值：0 ~ 100

自然消耗：
  • 开店营业：-15 / 游戏时段
  • 每次对话互动：-5 ~ -10（视对话长度和深度）
  • 情绪低落时消耗 × 1.5

自然恢复：
  • 睡眠：+60（一觉醒来基本回满）
  • 独处休息：+10 / 时段
  • 做喜欢的事：+10 ~ +15（取决于角色性格）

角色差异：
  千早：精力上限高、恢复快（热血笨蛋体质）
  鹿岛樱：精力上限较低（心脏病影响）
  龙之介：社交互动消耗翻倍（社恐）
  九条：精力消耗慢，但他本身精力就不高（颓废）

行为影响：
  > 70："今天精神很好！" → 对话活跃，容易触发长对话
  30~70：正常状态
  < 30："抱歉...今天有点累了" → 对话变短，可能婉拒互动
  < 10：NPC 主动收工闭店，回家休息
```

### 6.4 社交开放度（Sociability）

```
取值：0 ~ 100

计算方式：
  sociability = 性格基础值 + 精力修正 + 情绪修正 + 随机波动(±5)

  性格基础值：
    千早：80（社交牛逼症）
    鹿岛樱：50（不主动不拒绝）
    和叶：40（偏好安静但能应对）
    龙之介：20（社恐，主动回避社交）
    九条：35（想独处但职业身份要求他待人）

行为影响：
  > 70：NPC 主动找人聊天 / 串门 / 打招呼
        "哟！好久不见！进来坐！"
  30~70：接受互动，按正常节奏对话
  < 30：对话简短，一两个字回应，不主动延续话题
        "嗯。...还好。...没什么事的话我先忙了。"
  < 10：闭店/锁门/戴耳机/明确回避
        "今日休息"（门上挂着牌子）
        → 玩家如果强行打扰，可能触发特殊反应

玩法意义：
  玩家学会"读人"——
  看到龙之介缩在柜台后面戴着耳机 → 今天不打扰
  看到鹿岛樱在门口浇花还主动打招呼 → 今天可以多聊几句
```

### 6.5 羁绊度（Bond）

```
取值：-1.0 ~ +1.0（每个 NPC 对每个其他角色独立维护）

涵盖：
  • 情感亲近度（"喜不喜欢"）
  • 可靠感（"信不信得过"——但这是 Bond 的衍生，不独立追踪）

玩家感知：
  数值不可见，但从以下行为中推断：
  Bond < 0    → 回避、敷衍、明显的不耐烦
  Bond 0~0.2  → 礼貌但疏离，社交距离
  Bond 0.2~0.5 → 友好，愿意多聊几句
  Bond 0.5~0.7 → 亲近，主动关心，可以开玩笑
  Bond > 0.7  → 深度信任，可以倾诉秘密

更新：
  • 每次重要互动（importance ≥ 0.3）→ 微小变化 ±0.01~0.05
  • 睡眠反思时 → 可能因"想通了什么"而发生较大变化
  • 变化方向由互动内容决定，不是互动次数

衍生值：confide_level（信任等级）

  confide_level 不是独立维度，是 Bond 的产物：

  if bond < 0.2    → confide_level = 0（不透露任何秘密）
  if bond 0.2~0.5  → confide_level = 1（半遮半掩）
  if bond 0.5~0.7  → confide_level = 2（可以倾诉）
  if bond > 0.7    → confide_level = 3（完全信任）

  性格修正：
    龙之介即使 bond=0.7 也最多 confide=1（社恐天花板）
    千早即使 bond=0.3 也可能 confide=2（大嘴巴藏不住）

  秘密重量修正：
    轻度秘密（"我其实不喜欢吃胡萝卜"）→ confide 门槛低
    重度秘密（心脏病、搭档牺牲的真相）→ confide 门槛极高
```

### 6.6 当前心事（Current Need）— 自由字符串

```
类型：自由文本，由 LLM 在睡眠反思时生成和维护

不是一个枚举值，而是 LLM 以 NPC 第一人称写的一句话：

示例：
  "昨晚梦到父母了。今天想多给他们的照片前的花瓶换水。"
  "千早今天好像不开心，想找个机会问问她怎么了，但又怕太直接。"
  "小李上次提到喫茶店缺一种花。今天进货了新的满天星，想拿给他看看。"
  "没什么特别的事。今天就想安安静静地把店里那堆书整理完。"

注入方式：
  System Prompt 中增加一行：
  今天心里在想的事："{current_need}"

生命周期：
  生成 → 睡眠反思时，LLM 基于今日记忆和当前状态生成
  生效 → 第二天 NPC 的对话和自主行为会围绕心事展开
  终结 → 心事被满足（事情发生了）→ 生成新的
        → 心事被更重要的事覆盖
        → 连续 2-3 天未进展 → 自然淡化（写入记忆成为"想做但没做的事"）

特殊用途：
  心事是 NPC 自主互动的"发动机"——
  如果千早的心事是"想跟龙之介说他的画很好看"，
  她会在社交开放度足够时，主动往和果子店的方向走。
```

### 6.7 状态流转示例

```
鹿岛樱的一天：

8:00  起床。情绪:平静 精力:75 社交:45
      心事:"想给小李看看新到的百合。"
      
9:00  开店。精力-15 → 60
  
10:30 玩家路过，主动招呼。聊了10分钟，很开心。
      情绪:开心 精力-10 → 50 社交+10 → 55
      心事完成 → 改为无
      
13:00 独自看店，没人来。想起今天是妈妈的忌日。
      情绪:低落 精力:40 社交:15
      心事:"想一个人待着。"
      → 把店门半掩，不主动招呼
      
15:00 千早闯进来送面包。一开始不想说话，但千早的笑话让她笑了。
      情绪:回升至平静 精力-5 → 35 社交回升至30
      
18:00 关门。精力:25
  
24:00 睡眠反思 → 生成心事:
      "今天千早来的时候我一开始不想理她，有点过意不去。
       明天给她送一枝花吧。"
      精力恢复 → 85
```

## 8. AI 驱动的角色行为系统

> Python 后端决定 NPC **做什么**，Unity 决定**怎么做（动画/移动/表现）**。

### 8.1 核心架构

```
┌─────────────────────────────────────────────────────────────┐
│                    Python 后端（决策层）                      │
│                                                             │
│  状态(情绪/精力/社交/羁绊/心事) + 基底设定 + 时间/天气        │
│         │                                                   │
│         ▼                                                   │
│  ┌──────────────────┐    ┌─────────────────┐               │
│  │ 行为决策引擎      │───→│ Prompt 上下文    │               │
│  │ (规则+LLM混合)   │    │ 注入当前行为意图  │               │
│  └──────┬───────────┘    └─────────────────┘               │
│         │                                                   │
│         │  JSON 行为指令                                    │
│         ▼                                                   │
│  WebSocket ──── localhost:8000/ws/npc/{npc_id}             │
│         │                                                   │
└─────────┼───────────────────────────────────────────────────┘
          │
┌─────────▼───────────────────────────────────────────────────┐
│                    Unity（表现层）                            │
│                                                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                 │
│  │ 行为队列  │  │ 动画系统  │  │ 导航系统  │                 │
│  │ Queue    │  │ Animator │  │ NavMesh  │                 │
│  └──────────┘  └──────────┘  └──────────┘                 │
│       │              │              │                       │
│       └──────────────┼──────────────┘                       │
│                      ▼                                      │
│              NPC 在场景中执行动作                             │
│              (移动/播放动画/显示对话/交互)                    │
│                                                             │
│  执行完成 → 回调 Python → 可能生成记忆 → 可能触发新行为      │
└─────────────────────────────────────────────────────────────┘
```

### 8.2 核心设计：固定 ID 体系

**原则：LLM 只在预定义的 ID 目录中选择，永远不自由生成地点或行为名。**

```
LLM 的输出不是：                    LLM 的输出是：
"走到花店窗户边发呆"                action_id: "sit_rest"
                                   location_id: "flower_shop.window_seat"
                                   ↓
                                   Unity 查表：flower_shop.window_seat
                                   → 坐标 (12.5, 0, 8.3)
                                   → 有效行为: [sit_rest, read, stare_outside, drink]
                                   → 播放动画: "sit_bench_idle"
                                   → ✅ 验证通过，执行
```

### 8.3 行为指令协议（ID 化）

Python 发给 Unity 的指令，所有字段使用固定 ID：

```json
{
  "npc_id": "sakura",
  "command_id": "cmd_20260707_001",
  "timestamp": "2026-07-07T10:30:00Z",
  
  "behavior": {
    "action_id": "sit_rest",              // ← 只能从 Action 目录中选
    "location_id": "park.bench_01",        // ← 只能从 Location 目录中选
    "duration_sec": 60,                   // 预计持续时间（0=直到被打断）
    "interruptible": true
  },
  
  "context": {
    "reason": "精力有点低，出来坐坐",      // LLM 生成，但仅供调试
    "mood_visible": "平静地看着河面",      // 玩家可观察到的描述
    "thought_bubble": null                // 头顶气泡（可选）
  }
}
```

**校验机制（双重保险）**：

```
Python 侧（发送前）：                  Unity 侧（接收后）：
  action_id 必须在 ActionCatalog 中      location_id 必须在 LocationCatalog 中
  location_id 必须在 LocationCatalog 中  action_id 必须在该 location 的 valid_actions 中
  action_id 必须在该 location 的          npc_id 必须存在且活跃
    valid_actions 列表中                  → 任一校验失败 → 忽略指令 + 记录日志
  → 任一校验失败 → 不发送，fallback
    到默认行为（idle_stand at home）
```

## 9. 检索流程（LLM 路由 + 思考预算）★ v0.5 重写

> 旧版多路混合检索（语义+时间+关系+核心记忆→Rerank→Top-8）已废弃。新版采用 LLM 路由：LLM 不做全图遍历，只在起点节点的邻边中选择路径，后端按选择去库里查。

### 9.1 核心原则

**LLM 不做检索，LLM 做路由。** 算法负责从库里捞出邻边和权重，LLM 看这些信息决定走哪条路。引擎（数据库）负责实际移动。

### 9.2 思考能量预算（Think Budget）

检索深度不由跳数决定，由预算决定。预算耗尽 → 检索停止。

```
budget = base_budget × energy_factor × bond_factor × emotion_factor

base_budget（性格基线）:
  千早=3, 龙之介=4, 九条=5, 鹿岛樱=6, 和叶=8

energy_factor:
  > 70 → 1.2,  30-70 → 1.0,  < 30 → 0.4

bond_factor:
  > 0.6 → 1.5,  0.2-0.6 → 1.0,  < 0.2 → 0.6

emotion_factor:
  平静=1.2, 低落=1.3, 开心=0.9, 焦虑=0.5, 疲惫=0.3
```

**遍历代价 = 1 / weight**（取当前方向的权重）。强边 cost≈1.1，弱边 cost=3~10。同样预算下，沿强方向走 5 跳比沿弱方向走 2 跳更轻松——人脑的"省力优先"。

**情绪对边代价的瞬时调节**：

| 情绪 | 正面情绪边 cost | 负面情绪边 cost | 担忧对象边 cost |
|------|:---:|:---:|:---:|
| 低落 | ×1.5 | **×0.5** | — |
| 开心 | **×0.5** | ×1.5 | — |
| 焦虑 | ×2.0 | — | **×0.3** |

### 9.3 四种检索策略

同一起点，四种走法，四种结果。策略选择由性格+状态+上下文共同决定。

| 策略 | 走法 | 代价/跳 | 认知模式 | 典型产物 |
|------|------|---------|----------|----------|
| **广度优先** | 邻边各走一步，均匀扩散 | 中 | 自由联想 | "火锅！毛肚！辣的！和叶也在！" |
| **深度优先** | 选最强边一路走到底 | 高 | 沉思追忆 | "那天和叶很安静，在看乱步..." |
| **最佳优先** | 贪心走最强边 | 最低 | 安全应答 | "...还行。一个人吃惯了。" |
| **最轻松优先** | 永远走 cost 最低的边 | 近乎零 | 敷衍节能 | "......不太吃。" |

**策略选择器**（每次路由时重新计算）：

```
性格基线概率:
  千早:   广度40% 深度10% 最佳30% 轻松20%
  鹿岛樱: 广度20% 深度40% 最佳30% 轻松10%
  和叶:   广度30% 深度50% 最佳15% 轻松5%
  龙之介: 广度5%  深度15% 最佳30% 轻松50%
  九条:   广度5%  深度20% 最佳45% 轻松30%

状态修正（概率偏移）:
  精力<30 → 轻松大幅↑  深度几乎不可能
  情绪=开心 → 广度↑
  情绪=低落 → 深度↑（但路径被情绪过滤，只沿负面边深入）
  情绪=焦虑 → 深度↑但路径锁定（钻牛角尖）
  羁绊度>0.6 → 深度↑ 轻松↓
  羁绊度<0.2 → 轻松↑ 深度↓
  对话深度=deep → 深度强制↑ 轻松强制↓
```

### 9.4 完整调用链条

```
对话触发

Step 1: 图查询（SQLite，非 LLM）
  输入: 场景触发的起点节点 ID（玩家/花店/下雨...）
  输出: 直接邻边列表 [{target_summary, weight, cost, edge_type}]
  成本: 1 次 SQL 查询

Step 2: LLM 路由决策（轻量 Prompt，结构化输出）
  输入:
    - 邻边列表（带权重和摘要）
    - NPC 当前状态（精力/情绪/羁绊/预算）
    - 当前策略概率分布
  输出 JSON:
    - 选择的策略类型
    - 选中哪些路径继续
    - 是否继续深入下一跳
  成本: 1 次轻量 LLM 调用（不需流式）

Step 3: 图查询（SQLite，批量）
  输入: LLM 选中的节点 ID 列表
  输出: 节点的完整内容
  成本: 1 次 SQL batch 查询

Step 4: 如果 LLM 决定继续深入
  → 以新节点为起点，回到 Step 1
  → 直到 LLM 决定停止或预算耗尽

Step 5: 记忆重建（一次完整 LLM 调用）
  输入: 所有选中的节点 + 策略类型 + NPC 性格
  输出: 自然语言回忆/对话（流式）
  成本: 1 次完整 LLM 调用

总成本: 1-3 次轻量路由调用 + 1 次对话生成调用
```

### 9.5 邻边截断策略

中心节点（如"我"）可能有 200+ 条边。必须截断。

```
截断规则（SQL 层执行）:

SELECT * FROM memory_edges
WHERE node_a = :start_node_id OR node_b = :start_node_id
ORDER BY 
  GREATEST(weight_ab, weight_ba) DESC,  -- 权重高的优先
  last_traversed_at DESC                 -- 最近走过的其次
LIMIT 20;  -- 最多返回 20 条邻边

如果起点节点有多个（场景触发多个节点），
每个节点各自取 Top-15 条邻边，合并去重后传给 LLM。
```

### 9.6 LLM 路由输出 Schema

```json
{
  "strategy": "depth",              // breadth|depth|best|easy
  "strategy_reason": "他在问回忆，我想认真想一下",
  
  "selected_paths": [
    {
      "edge_id": "edge_042",
      "direction": "ab",            // 走 weight_ab 还是 weight_ba 方向
      "target_node_id": "node_搬花盆",
      "target_summary": "搬花盆",
      "cost": 1.7,
      "reason": "这条路最近走过，直接就想到了"
    }
  ],
  
  "skipped_paths": [
    {
      "edge_id": "edge_099",
      "reason": "禁区——指向父母忌日，日常不想提"
    }
  ],
  
  "continue": true,                 // 是否继续下一跳
  "continue_from_nodes": ["node_搬花盆"],  // 下一跳的起点
  
  "budget_remaining": 2.5,          // 预计剩余预算
  "stop_reason": null               // 如果停了，为什么
}
```

**路由降级策略**：
```
LLM 路由调用失败/超时:
  → 降级为规则路由: 取权重最高的前 3 条边，走最佳优先
  → 不深入（continue=false）
  → 日志记录降级事件

LLM 选择了不存在的节点:
  → 校验层过滤非法 node_id
  → 从合法选择中继续
  → 如果所有选择都非法 → 降级为规则路由

LLM 选择 "continue=true" 但 budget 已耗尽:
  → 后端强制 continue=false
  → 在 stop_reason 中记录 "budget_exhausted"
```

## 10. 技术选型详解

### 10.1 Embedding 模型：BAAI/BGE-large-zh-v1.5

| 项目 | 参数 |
|------|------|
| **模型** | `BAAI/bge-large-zh-v1.5` |
| **维度** | 1024 维 |
| **最大长度** | 512 tokens |
| **运行方式** | 本地 GPU (2-4GB VRAM) 或 CPU |
| **推理框架** | sentence-transformers / ONNX Runtime |
| **中文表现** | C-MTEB 榜单 Top-3，中文语义理解 SOTA |

备选方案：
- `text2vec-large-chinese` — 更轻量，768 维，CPU 可跑
- `BAAI/bge-m3` — 多语言，支持长文本 (8192 tokens)，但更重

> **建议**：先用 `bge-large-zh-v1.5`，GPU 够就用。如果没有独立显卡，降级到 `text2vec-large-chinese` 纯 CPU 也能跑。

### 10.2 向量数据库：LanceDB

| 对比 | LanceDB | ChromaDB | Milvus Lite | FAISS + SQLite |
|------|---------|----------|-------------|----------------|
| **部署方式** | 嵌入式（零配置） | 嵌入式 | 嵌入式 | 手动拼接 |
| **存储格式** | Lance (列存) | 自有格式 | 自有格式 | 内存索引 + SQLite |
| **磁盘占用** | 极小（列存压缩） | 中等 | 较大 | 中等 |
| **中文支持** | 无影响（只存向量） | 同左 | 同左 | 同左 |
| **过滤查询** | 原生 SQL 过滤 + 向量搜索 | 元数据过滤 | 标量过滤 | 手写 SQL |
| **无需服务进程** | ✅ | ✅ | ✅ | ✅ |
| **Python API** | 简洁 | 简洁 | 复杂 | 需自己封装 |

> **选择 LanceDB 的关键理由**：
> 1. 真正的嵌入式 — `pip install lancedb` 就完了，无需启动任何服务
> 2. 列式存储 — 向量 + 元数据一体化，一个文件全搞定
> 3. 支持增量写入 — 新记忆直接 append，不用重建索引
> 4. 单机游戏场景下，文件体积小 = 存档方便

### 10.3 云端 LLM：Claude API（主力）

| 用途 | 模型 | 说明 |
|------|------|------|
| **对话生成** | Claude Sonnet 5 | 快速、成本低，日常对话 |
| **关键剧情** | Claude Opus 4.8 | 高智能，重要转折点 |
| **反思/总结** | Claude Sonnet 5 | 睡眠时批量：融合value/精度转换/反思/梦话 |
| **事件→图提取** | Claude Sonnet 5 | 睡眠时批量执行，每NPC当日事件1次调用 |
| **路由决策** | Claude Sonnet 5（极轻量） | 白天对话中，结构化JSON输出 ~50 tokens |
| **重要性评估** | 本地小模型 / 规则 | 不需要云端 |

### 10.4 本地边缘模型（省成本 + 降延迟）

| 任务 | 本地方案 | 说明 |
|------|----------|------|
| **文本 Embedding** | BGE-large-zh | 节点向量化 + 场景线索映射 |
| **Reranker** | BGE-reranker-v2-m3 | 邻边排序时可选用 |
| **重要性评估** | 规则引擎 + 轻量分类模型 | 关键词匹配 |
| **情绪检测** | 规则 + 情感词典 | valence 计算 |

> 这些模型在 Python 后端启动时加载，常驻内存。内存占用约 2-4GB。

### 10.5 向量重新生成时机 ★ v0.5 新增

节点向量不是生成一次就永久不变。以下时机需要**重新生成**：

| 触发条件 | 说明 | 频率 |
|---------|------|------|
| 精度转换 (exact→vague) | value文本变了 → 向量语义变了，必须重建 | 每日睡眠时，按需 |
| 节点融合 | 两个节点合成一个新节点 → 新value → 新向量 | 每日睡眠时，按需 |
| 反思节点生成 | 新节点 → 第一次生成向量 | 每日睡眠时 |
| 新事件入图 | 新节点 → 第一次生成向量 | 每日睡眠时 |

不需要重新生成的场景：
- energy 衰减（value 没变）
- 边权重变化（向量不受影响）
- recall_count 更新（向量不受影响）

```
向量更新流程（睡眠时执行）:
  1. 收集所有需要更新向量的节点ID
  2. 批量生成: BGE.encode([node.value for node in nodes_to_update])
  3. 批量写入 LanceDB: 先删旧向量，再 insert 新向量
  4. LanceDB 追加式存储 → 旧向量仍占用空间 → 每月执行一次 compact
```

### 10.6 图规模预估 ★ v0.5 新增

```
单个 NPC 的图:

  游戏时间    节点数      边数       每日新增节点   SQLite大小
  ──────────────────────────────────────────────────────
  Day 1       ~15        ~30         —            < 50 KB
  Day 7       ~80        ~200        ~10           ~200 KB
  Day 30      ~250       ~700        ~8            ~1 MB
  Day 90      ~500       ~1500       ~5 (融合抵消)  ~3 MB
  Day 365     ~800       ~2500       ~2            ~5 MB
  5个NPC合计  ~4000      ~12500      —             ~25 MB

说明:
  • 每天新增节点逐渐减少（事件→反思→更高层反思，新事件不会无限增长）
  • 融合压缩抵消了部分增长（同类型节点合并后总数回落）
  • 边增长快于节点增长（similar_to 边在节点间建立全连接）
  • 核心内存占用在 LanceDB 向量（1024维 × ~4000节点 × 4字节 ≈ 16 MB）
  • 邻边查询（LIMIT 20）在万级边表中仍为毫秒级（有索引）

极端情况（1000天+）:
  • 反思节点相互融合 → 超高层抽象节点（"那几年"）
  • 原始事件 energy < 0.05 → 可考虑归档到单独的表（不参与日常检索）
  • 活跃节点稳定在 500-800 范围
```

## 11. Prompt 设计（核心）★ v0.5 更新

> 旧版原文记忆注入已废弃。新版 Prompt 从图节点+边重建上下文，利用标签的 precision 和 energy 控制注入细节的精确度。

### 11.1 System Prompt 模板

```
# Role
你是{name}，{age}岁，{gender}。
你的职业是{occupation}。

# Personality（从基底设定节点提取）
{personality_traits}
说话风格: {speech_style}

# 你今天的状态
时间: {game_time} | 地点: {location}
情绪: {emotion} | 精力: {energy}/100
心里在想: {current_need}

# 关于对话对方（从人物节点邻域实时重建）
你对{target_name}的感觉:
  {impression_rebuilt_from_graph}
  羁绊度: {bond_description}

# 规则
1. 用口语化、自然的中文对话
2. 回答不超过 80 字（除非在讲故事/深度对话）
3. 确定的事直接说，模糊的事用"好像""应该"——你的记忆本就不完美
4. 标签精确的事说细节，标签模糊的事说大概，标签消失的事不提
5. 你的情绪和态度与羁绊度一致
```

### 11.2 图节点 → Prompt 上下文重建

```
[近期记忆 — 7天内原文全量注入]
  short_term_memories 表内容（原文，不加修改）

[久远记忆 — 本次路由检索到的图节点 → LLM 重建]
  
  检索到的节点示例:
    搬花盆 (event, precision=vague, energy=0.6)
    那天躲雨 (event, precision=vague, energy=0.4)
    感激   (emotion, precision=exact, energy=0.8)
  
  → LLM 根据节点类型和precision重建:
    exact → 用确定语气
    vague → 用模糊语气（"好像是...""应该是..."）
    多个同类节点通过 similar_to 边互联 → 在模糊情绪团里扩散
    "上次下雨的时候...你帮我搬了花盆。具体记不太清了，但当时很感激。"

[核心记忆 — 锁定节点全量注入]
  从 memory_nodes 中取 is_core=1 的节点（energy=1.0, decay=0）
  这些定义了 NPC 是谁

[印象 — 从人物节点邻域实时重建]
  人物节点 + relationship 边权重 + 最近 N 条 involved 事件的情绪统计
  → LLM 生成 1-3 句自然语言印象
```

### 11.3 记忆注入的 Token 预算

```
总预算：~4000 tokens（预留给记忆+对话）

分配：
├─ System Prompt (角色+性格+状态):        ~400 tokens
├─ Impression (从图重建):                ~150 tokens
├─ Core Memories (锁定节点):             ~300 tokens
├─ Short-term (7天内原文):              ~1000 tokens
├─ Retrieved Graph Nodes (LLM重建):      ~800 tokens  ← 比旧版节省~400 tokens
├─ Current conversation history:         ~800 tokens
└─ User input + 预留输出空间:            ~550 tokens
```

## 14. 游戏时间系统

### 19.1 分层时间流速

```
探索时间（玩家自由走动时）：
  1 游戏分钟 = 3 现实秒
  1 游戏小时 = 3 现实分钟
  1 游戏天 ≈ 30 现实分钟（白天部分）
  → 有昼夜节奏，但不急促

对话时间（立绘对话模式中）：
  1 游戏分钟 = 30 现实秒（比探索慢 6 倍）
  → 聊天几乎不被时钟压迫
  → 但聊太久天色仍会渐暗（柔和的上限感）
  → NPC 的 routine 行为自动推迟到对话结束后

快进时间（玩家主动触发）：
  10 倍速快进
  → 屏幕上显示时钟转动 + 窗外光线变化
  → 玩家随时可停止
  → 可快进到预设时段（傍晚/明天）
```

### 19.2 三个核心规则

```
规则 1：对话不被打断
  立绘对话中，时间几乎暂停。
  NPC routine（如 17:30 送面包）→ 自动推迟到对话结束后补执行。
  → 玩家永远不需要"赶时间聊天"

规则 2：午夜强制睡眠 → 后台跑 LLM ★ v0.5 更新
  时钟走到 24:00 → 玩家强制回家睡眠（不能跳过）

### 15.2 图演化伪代码

```python
def nightly_graph_evolution(npc_id):
    graph = load_graph(npc_id)
    
    # 1. 事件入图
    today_events = get_today_events(npc_id)
    for event in today_events:
        nodes, edges = llm_extract_nodes_and_edges(event.original_text)
        graph.add_nodes(nodes)
        graph.add_edges(edges)
        generate_vectors(nodes)  # → LanceDB
    
    # 2. 衰减
    for node in graph.nodes:
        node.energy -= node.decay_rate * (1 - 0.3 * node.recall_count)
        if node.energy < 0.1 and not node.is_core:
            graph.mark_gone(node)
    
    for edge in graph.edges:
        edge.weight_ab -= edge.decay_rate_ab
        edge.weight_ba -= edge.decay_rate_ba
    
    # 3. 时间锚消失前 → 建立 sequenced 边
    for time_node in graph.get_dying_nodes(type__startswith="time"):
        linked_events = graph.get_nodes_pointing_to(time_node, via="happened_at")
        for i in range(len(linked_events) - 1):
            graph.add_edge(linked_events[i], linked_events[i+1], 
                          type="sequenced", weight=0.5)
        time_node.absorb_edges_to_nearest_neighbor()
        graph.remove(time_node)
    
    # 4. similar_to 聚类
    for node_type in NODE_TYPES:
        same_type = graph.nodes.filter(type=node_type)
        for a, b in pairs(same_type):
            sim = cosine(a.vector, b.vector)
            if sim > 0.85:
                graph.merge(a, b)  # 融合
            elif sim > 0.65:
                edge = graph.get_or_create_edge(a, b, type="similar_to")
                edge.weight_ab = min(edge.weight_ab + 0.05, sim * 0.9)
    
    # 5. 反思生成
    clusters = graph.detect_dense_clusters(min_edges=3)
    for cluster in clusters:
        reflection = llm_generate_reflection(cluster)
        graph.add_node(reflection)
        for event in cluster:
            graph.add_edge(reflection, event, type="contains", weight=0.7)
    
    # 6. 路径强化
    for node in graph.recently_activated_nodes_today:
        node.energy += 0.02 * node.recall_count_today
    for edge in graph.recently_traversed_edges_today:
        edge.weight += 0.01
    
    # 7. 短期→长期迁移
    migrate_old_short_term(npc_id, days=7)
    
    save_graph(npc_id, graph)
```

### 15.3 演化失败恢复策略

```
演化步骤是有序的。前一步失败不影响后续非依赖步骤。

恢复规则:
  
  Step 2a (衰减) 失败:
    → 严重。SQL 操作失败意味着数据库问题。
    → 终止本次演化，记录日志，下一天重试。
    → 衰减不是精确科学，晚一天衰减影响极小。
  
  Step 2c (事件间边) 失败:
    → 仅影响该时间节点关联的事件。其余事件继续。
    → 该时间节点不删除（保留到下一次演化）。
  
  Step 2d-2e (similar_to + 融合) LLM调用失败:
    → 该对节点跳过本次融合检测。
    → similar_to 边不建立/不强化。
    → 下次演化时重新检测（向量没变，仍会触发）。
  
  Step 2f (反思) LLM调用失败:
    → 该簇跳过本次反思生成。
    → 下次演化时重新检测（簇仍存在）。
    → 不影响其他簇。
  
  Step 1 (事件入图) LLM调用失败:
    → 最危险。当日事件不入图 = 记忆丢失。
    → 重试 3 次。仍失败 → 原文保留在 short_term_memories。
    → 第二天演化时重新尝试（原文还在7天窗口内）。
    → 7天后仍未入图 → 原文归档到 original_text，节点丢失。
  
  全局兜底:
    → 每个步骤前后记录 checkpoint。
    → 如果演化中途 Python 进程崩溃 → 下次启动时从 checkpoint 恢复。
    → 最坏情况：跳过本次演化的所有 LLM 步骤，只执行纯计算步骤。
```

## 17. 存档机制

### 19.1 核心问题

这个游戏的存档不是简单的"角色坐标 + 进度标记"。需要持久化的数据分布在三个完全不同的存储层：

```
┌──────────────────────────────────────────────────────────┐
│  存储层            │  内容                    │  格式      │
├────────────────────┼──────────────────────────┼───────────┤
│  LanceDB           │  长期记忆向量 + 元数据    │  Lance 文件 │
│  SQLite            │  短期记忆 / 印象 / 状态   │  .db 文件  │
│  Python 运行时      │  工作记忆 / 当前对话上下文 │  内存      │
│  Unity 场景         │  NPC 位置 / 动画状态      │  内存      │
└──────────────────────────────────────────────────────────┘
```

**存档的本质**：把分布在不同地方的数据**原子性地**冻结到磁盘上的一个文件夹里。

### 19.2 存档范围：什么要存，什么不存

```
✅ 必须存（游戏状态）：

  记忆层：
    ├─ 所有短期记忆（SQLite: short_term_memories 表）
    ├─ 所有长期记忆（LanceDB: 向量 + 元数据）
    ├─ 所有核心记忆（SQLite: core_memories 表 → 其实很少变，但标记可能变）
    └─ 所有反思记忆（type=reflection，已在 LanceDB 中）

  印象层：
    ├─ 所有印象条目（SQLite: impressions 表）
    │  包括 impression_text / bond / dominant_trait / recent_trend
    └─ 衍生值 confide_level（不存，从 bond 实时计算）

  NPC 状态层：
    ├─ 情绪 / 精力 / 社交开放度（每个 NPC 的当前即时状态）
    ├─ 羁绊度矩阵（每个 NPC 对其他所有人的 bond 值）
    └─ 当前心事 current_need（字符串）

  游戏层：
    ├─ 当前游戏时间（年/月/日/时）
    ├─ 玩家位置 / 姓名 / 进度
    ├─ NPC 当前位置（location_id）
    └─ 商店营业状态（开关门）

  队列层：
    └─ 每个 NPC 的当前行为队列（未完成的行为）


❌ 不存（可重建 / 静态配置）：

  ├─ 角色基底设定（偏好/兴趣/行为倾向/日常节奏）→ 配置文件，不改
  ├─ Location Catalog / Action Catalog → 配置文件，不改
  ├─ 工作记忆（当前对话上下文）→ 对话结束时已写入记忆，不需要存
  ├─ Embedding 模型文件 → 本地安装的，不是游戏数据
  └─ LLM API 配置 → 用户自己的设置
```

### 19.3 存档结构

```

### 19.4 存档流程

```
玩家触发存档（手动 or 自动触发点，如睡眠时）

┌─────────────────────────────────────────────────────────┐
│ Step 1: 暂停（~0.1 秒）                                  │
│   • 所有 NPC 行为队列标记为"存档暂停"                     │
│   • 正在进行中的对话 → 等待当前 LLM 回复完成              │
│   • 禁止新的 LLM 调用                                    │
│                                                         │
│ Step 2: 写入轻量状态（~0.05 秒）                          │
│   • game_state.json（游戏时间、玩家状态）                 │
│   • npc_states.json（所有 NPC 即时状态 + 羁绊度矩阵）     │
│   • 这两个文件很小（< 50KB），先写 → 如果失败损失最小     │
│                                                         │
│ Step 3: SQLite 热备份（~0.1-0.5 秒）                    │
│   sqlite3_backup_init() → 在线备份到存档文件夹           │
│   优势：不需要锁库，玩家可以继续玩着存档                   │
│                                                         │
│ Step 4: LanceDB 复制（~1-3 秒，视数据量）                 │
│   方案 A（推荐）：直接复制 LanceDB 数据目录               │
│     shutil.copytree(lancedb_path, save_lancedb_path)   │
│     → LanceDB 的 Lance 文件是追加写的，复制时不会损坏     │
│   方案 B（如果数据量太大）：增量备份                      │
│     只复制上次存档后有变更的表分区                        │
│                                                         │
│ Step 5: 写入 manifest（~0.01 秒）                        │
│   manifest.json → 存档完成的标志                         │
│   → 如果 manifest 不存在 = 存档不完整 = 无效             │
│                                                         │
│ Step 6: 恢复（~0.1 秒）                                  │
│   • 解除"存档暂停"                                       │
│   • 恢复行为队列                                         │
└─────────────────────────────────────────────────────────┘

总耗时：约 1.5-4 秒（大部分在 LanceDB 文件复制）
```

### 19.5 读档流程

```
玩家选择读档

┌─────────────────────────────────────────────────────────┐
│ Step 1: 验证存档完整性                                    │
│   ├─ manifest.json 存在？                                │
│   ├─ save_version 兼容？                                 │
│   ├─ memories.db 可打开？                                │
│   └─ lancedb_data/ 目录完整？                             │
│   任一失败 → 存档损坏 → 提示玩家                          │
│                                                         │
│ Step 2: 关闭当前运行数据                                  │
│   • 关闭当前 LanceDB 连接                                │
│   • 关闭当前 SQLite 连接                                 │
│   • 清空所有 NPC 行为队列                                │
│                                                         │
│ Step 3: 替换数据文件                                      │
│   • 用存档的 memories.db 替换运行中的 memories.db         │
│   • 用存档的 lancedb_data/ 替换运行中的向量数据            │
│                                                         │
│ Step 4: 重新加载                                          │
│   • Python 重新打开 SQLite 和 LanceDB                    │
│   • 读取 game_state.json → 恢复游戏时间、玩家状态         │
│   • 读取 npc_states.json → 恢复所有 NPC 即时状态          │
│   • 加载 Embedding 模型（已在内存中，不需要重新加载）      │
│                                                         │
│ Step 5: 通知 Unity                                        │
│   • WebSocket → Unity: "存档加载完成"                    │
│   • Unity 根据 location_id 重新放置所有 NPC              │
│   • NPC 行为 Tick 恢复运行                                │
│                                                         │
│ 总耗时：约 2-5 秒                                        │
└─────────────────────────────────────────────────────────┘
```

### 19.6 自动存档触发点

```
触发时机（不打断玩家体验的时间点）：

  ✅ 每个游戏日结束时（睡眠反思完成后自动存档）
     → 这是最自然的存档点，数据最完整
  ✅ 玩家手动存档（任何时候，在菜单中触发）
  ✅ 退出游戏时（自动存档到 "AutoSave" 槽位）
  ✅ 重要事件发生后（可选，如羁绊度跨越 0.5 阈值时）

  ❌ 对话进行中不自动存档
  ❌ NPC 行为执行中不自动存档（等当前行为完成）
```

### 19.7 存档版本兼容

```
manifest.save_version 用于处理未来版本的兼容性：

v1（首发）:
  - 5 个 NPC，基础记忆系统
  - 如果未来加了新 NPC / 新状态维度 / 新记忆类型

升级策略：
  def migrate_save(save_data, from_version, to_version):
      if from_version == 1 and to_version == 2:
          # 例：v2 新增了 confide_level 独立字段
          # → 从 bond 值推导默认值
          for npc in save_data["npc_states"]:
              npc["confide_level"] = derive_from_bond(npc["bond"])
      
      # 版本跨度太大 → 警告玩家但尽量加载
      if to_version - from_version > 5:
          log.warning("存档版本跨度过大，可能丢失部分数据")
```

### 19.8 存档文件大小估算

```
假设玩家玩了 30 个游戏日：

  短期记忆：~150 条 × 200B  ≈ 30 KB
  长期记忆：~600 条 × 2KB（含向量 1024×4B）≈ 1.2 MB
  印象数据：5 NPC × 6 目标 × 300B ≈ 9 KB
  向量索引（LanceDB 开销）：≈ 3-5 MB
  game_state / npc_states：≈ 30 KB
  SQLite 数据库文件：≈ 2-5 MB（含索引）

  每个存档槽位：约 8-12 MB
  10 个存档槽位：约 80-120 MB

→ 完全可接受。如果需要压缩，可以把 lancedb_data 打包为 .zip
  （压缩率约 60-70%，但加载时需要解压，增加 1-2 秒）
```

### 18.9 Unity 侧的配合

```
Unity 存档系统需要做的事（很少）：

1. 发送存档请求：
   WebSocket → Python: {"type": "SAVE_REQUEST", "slot": "Slot_01"}

2. 接收存档完成：
   Python → Unity: {"type": "SAVE_COMPLETE", "slot": "Slot_01", "success": true}

3. 发送读档请求：
   WebSocket → Python: {"type": "LOAD_REQUEST", "slot": "Slot_01"}

4. 接收 NPC 位置 + 状态：
   Python → Unity: {"type": "LOAD_COMPLETE", "graph_vectors_positions": [...], ...}
   → Unity 重新设置所有 GameObject 的位置

5. 存档列表显示：
   Unity 读取 SaveData/*/manifest.json → 展示存档列表 UI
   或者 Python 提供一个 GET /saves 接口返回列表

6. 删除存档：
   Unity → Python: {"type": "DELETE_SAVE", "slot": "Slot_01"}
   → Python 删除整个 Slot_01 文件夹
```

### 19.2 模式切换规则

```
什么时候用立绘模式（A）：
  ✅ 玩家主动点击 NPC → 进入对话
  ✅ 对话中玩家可以做选择、提问
  ✅ 深度互动，通常持续 5-20 轮对话
  ✅ LLM streaming 逐字显示

什么时候用气泡模式（B）：
  ✅ NPC 之间自主聊天
  ✅ NPC 路过跟玩家打招呼
  ✅ NPC 的喃喃自语（揉面时自言自语）
  ✅ NPC 主动走到玩家面前说一句简短的话
  ✅ 玩家未点击 NPC 但在附近 → 听到 NPC 之间的对话

模式切换：
  气泡模式中，玩家点击那个 NPC → 立即切换到立绘模式
  立绘模式中，玩家点"再见" → 回到气泡模式（如果 NPC 还在附近继续说话）
```

### 19.4 气泡模式（Mode B）— 距离衰减透明度

```
核心规则：气泡的不透明度 = f(玩家与声源的距离)

距离计算：
  声源位置 = 说话 NPC 的 GameObject 坐标
  听众位置 = 玩家摄像机的坐标
  distance = Vector3.Distance(source, listener)

透明度映射：
  distance ≤ 2m    → opacity = 1.0    ← 近处完全可见
  distance 2-8m    → opacity = 1.0 → 0.2 线性衰减
  distance 8-15m   → opacity = 0.2 → 0.05 接近消失
  distance > 15m   → opacity = 0      ← 太远不可见

视觉表现：
  ┌──────────────────────────────────────────────────┐
  │                                                  │
  │   远处的气泡（半透明）                              │
  │         ┌─────┐                                  │
  │         │ ... │  opacity 0.15                    │
  │         └─────┘                                  │
  │                    ┌──────────┐                  │
  │                    │ 早啊！     │ opacity 0.8     │
  │   玩家 📍          └──────────┘                  │
  │                      NPC（3m 外）                 │
  │                                                  │
  │                              ┌──────────────┐    │
  │                              │ 今天的面包... │ op 1.0│
  │                              └──────────────┘    │
  │                              NPC（1m 外）         │
  └──────────────────────────────────────────────────┘

气泡特性：
  • 跟随 NPC 头顶（World Space Canvas）
  • 离玩家越远越透明（移动时平滑过渡）
  • 不同类型的气泡用不同颜色/样式区分：
    - NPC 间聊天：白色气泡
    - NPC 自言自语：灰色气泡，虚线边框
    - NPC 对玩家说话：暖色气泡（淡黄），有轻微指向玩家的箭头
  • 气泡最长显示 5-8 秒后自动消失（除非玩家走近）
  • 多条气泡同时出现时，离玩家最近的在最上层
```

### 19.5 气泡持续时间和消失逻辑

```
气泡生命周期：

普通闲聊（greet）：显示 4 秒 → 淡出 1 秒 → 消失
问候（hi/早啊）：显示 3 秒 → 淡出 0.5 秒 → 消失
深度 NPC 对话（talk）：逐句显示，每句 3-6 秒 → 下一句替换
自言自语（special_talk_to_self）：显示 3 秒 → 淡出

玩家走近时：
  正在消失的气泡 → 如果玩家进入 5m 范围 → 淡入回来，重新计时
  已消失的气泡 → 不重新显示（除非 NPC 说新的）

多条气泡管理：
  每个 NPC 最多显示 1 个气泡
  新气泡替换旧气泡（旧气泡直接消失，不平滑过渡）
  不同 NPC 的气泡可同时显示（但总数不超过屏幕上 8 个）
```

## 19. 玩家记忆与对话选项生成

> 玩家记忆 ≠ NPC 记忆。玩家记忆是轻量的结构化日志，唯一用途是生成有上下文的对话选项。

### 19.1 与 NPC 记忆的对比

```
NPC 记忆系统（重）                  玩家记忆系统（轻）
─────────────────                  ─────────────────
  四层模型（工作/短期/长期/核心）      一层：结构化事实日志
  向量语义检索                        直接按 NPC 过滤 + 时间排序
  印象系统（模糊人际感知）             不需要（玩家自己判断）
  记忆衰减/遗忘/复活                  不衰减（玩家自己会忘，不需要模拟）
  反思日记                            不需要
  LanceDB + SQLite                   SQLite 一张表足矣
  每次对话注入 8+ 条记忆              每次对话注入 3-5 条，用于生成选项
```

### 19.2 玩家记忆的数据结构

```json
{
  "id": "pmem_042",
  "type": "learned_fact",              // learned_fact | witnessed_event | did_something | received_promise
  "about_npc": "sakura",              // 关于哪个 NPC（关联）
  "content": "鹿岛樱喜欢白玉丸子，不太喜欢咖啡。",
  "source": "sakura_told_me",         // 怎么知道的：graph_vectors_told / witnessed / did_myself / heard_from_other
  "game_time": "Day 3, 10:30",
  "importance": 0.6,                   // 0-1，影响是否生成选项
  "used_count": 1,                     // 被生成为对话选项的次数（控制复用频率）
  "last_used_at": "Day 4, 14:00"
}
```

### 19.3 玩家记忆的创建时机

```
自动创建（对话中 LLM 判断）：

  玩家跟鹿岛樱对话结束 →
    LLM 判断："这次对话中玩家学到了什么新信息？"
    
    例：鹿岛樱说"我每天早上五点半就起来了，给花浇水"
    → 创建记忆：
    {
      type: "learned_fact",
      about_npc: "sakura",
      content: "鹿岛樱每天早上五点半起床浇花，这是她一天中最安静的时间。",
      source: "sakura_told_me",
      importance: 0.4
    }

    例：鹿岛樱说"我心脏不太好，但请别告诉千早"
    → 创建记忆：
    {
      type: "learned_fact",
      about_npc: "sakura", 
      content: "鹿岛樱有心脏病，她让我保密，特别是对千早。",
      source: "sakura_told_me",
      importance: 0.95  ← 秘密！极高重要度
    }

手动触发：
  玩家做了某件事：
    → 帮鹿岛樱修招牌
    → 创建记忆：{type: "did_something", content: "帮鹿岛樱修好了花店的招牌",
                  about_npc: "sakura", importance: 0.7}

  玩家目睹了某件事：
    → 看到龙之介在偷偷画画
    → 创建记忆：{type: "witnessed_event", content: "看到龙之介在后台偷偷画插画，
                   画得很好。他好像不愿意被人看到。",
                  about_npc: "tatsunosuke", importance: 0.6}
```

### 19.4 对话选项生成流程

```
玩家点击鹿岛樱 → 进入立绘对话模式
  │
  ▼
┌─────────────────────────────────────────────────────────┐
│ Step 1: 检索相关玩家记忆                                  │
│                                                         │
│ SELECT * FROM player_memories                           │
│ WHERE about_npc = 'sakura'                              │
│ ORDER BY importance DESC, game_time DESC                 │
│ LIMIT 10                                                │
│                                                         │
│ 结果：                                                   │
│ 1. [imp 0.95] 鹿岛樱有心脏病，让我保密                    │
│ 2. [imp 0.7] 帮鹿岛樱修好了花店的招牌                     │
│ 3. [imp 0.6] 鹿岛樱喜欢白玉丸子，不喜欢咖啡               │
│ 4. [imp 0.4] 鹿岛樱每天早上五点半起床浇花                 │
│ 5. [imp 0.3] 鹿岛樱的花店有时会有流浪猫进来               │
│ ...                                                     │
└──────────────┬──────────────────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────────────────┐
│ Step 2: LLM 根据最近对话 + 玩家记忆 → 生成选项             │
│                                                         │
│ Prompt:                                                 │
│ "玩家正在跟鹿岛樱对话。上一句她说：'欢迎光临，今天想看     │
│  什么花？'                                               │
│                                                         │
│  玩家知道的关于鹿岛樱的事：                               │
│  - 她有心脏病，让我保密（不要直接提这个！）                │
│  - 我帮她修过花店的招牌                                  │
│  - 她喜欢白玉丸子，不喜欢咖啡                            │
│  - 她每天早上五点半起来浇花                              │
│                                                         │
│  请生成 3-4 个玩家可以选择的对话选项。选项应该：           │
│  1. 自然地引用玩家的记忆（让人觉得对话有连续性）           │
│  2. 不要每条都引用记忆，留一个通用的                      │
│  3. 不要提及 NPC 不愿意讨论的话题（如秘密）               │
│  4. 语气符合"从城市回来的年轻人"的身份"                   │
│                                                         │
│ LLM 输出：                                               │
│ [                                                       │
│   "樱姐早！上次修的招牌没再歪吧？",                        │
│   "今天有什么推荐的花吗？我想放一盆在喫茶店里。",          │
│   "（从包里拿出白玉丸子）路过点心店看到的，给你带了一份。", │
│   "（微笑）就是来看看你。"                                │
│ ]                                                       │
└─────────────────────────────────────────────────────────┘
```

### 19.5 选项的智能组合规则

```
每个对话回合生成 3-4 个选项，按以下比例：

  🗣️  1 个基于近期记忆的选项（引用过去互动）
     "上次修的招牌还好吗？"
     → used_count++，短期内不会重复生成

  💡 1 个基于 NPC 偏好的选项（投其所好）
     "给你带了白玉丸子。"
     → 需要相关记忆 + 物品在库存

  ❓ 1 个推进话题的选项（深入当前对话）
     "你说的那种花，好养吗？"
     → 基于 NPC 刚说的话

  😶 1 个通用选项（什么都不引用）
     "（微笑）就是来看看你。"
     → 玩家不想选上面三个时的安全选项

规则：
  • 涉及秘密的记忆 → 不生成选项，但可能生成一个隐晦的选项
    鹿岛樱的心脏病 → "樱姐，你最近气色不太好...要不要休息一下？"
    （不直接提"心脏病"，但表达关心）
  • used_count > 3 的记忆 → 暂时不生成选项（避免重复）
  • 同一会话中不重复引用同一记忆
```

### 19.6 特殊选项类型

```
除了对话选项，玩家记忆还会生成特殊行动选项：

  🎁 送礼选项（基于 NPC 偏好记忆）：
    "（送出白玉丸子）"
    条件：玩家库存有白玉丸子 + 记忆中鹿岛樱喜欢白玉丸子

  🤝 帮忙选项（基于之前的互动）：
    "需要我帮你搬这些花盆吗？"
    条件：观察到 NPC 附近有重物 + 之前帮过类似的忙

  🔍 追问选项（基于模糊记忆）：
    "樱姐，你上次说的那个...后来怎么样了？"
    条件：有一条超过 3 天但 importance > 0.5 的未跟进记忆

  🌙 提到梦话的选项（基于听到的梦话）：
    "樱姐，你昨晚好像做噩梦了？"
    条件：玩家昨晚听到了这个 NPC 的梦话
    → 高风险选项（可能 +bond 或 -bond）
```

### 19.7 对话结束后更新玩家记忆

```
对话结束 →
  LLM 判断本次对话中玩家获得了哪些新信息？

  例：鹿岛樱提到"周五是开花市的日子"
  → 新记忆：{type: "learned_fact", about_npc: "sakura",
              content: "每周五是花市的日子，鹿岛樱会去进货。",
              importance: 0.3}

  例：鹿岛樱说漏嘴"最近胸口有点闷"
  → 新记忆：{type: "learned_fact", about_npc: "sakura",
              content: "鹿岛樱最近胸口不舒服——但她没直接说。",
              importance: 0.8}

  同时更新旧记忆的 used_count：
  本次选项中被引用的记忆 → used_count++
```

### 19.8 数据存储

```
SQLite 新增一张表：

    id TEXT PRIMARY KEY,
    type TEXT NOT NULL,           -- learned_fact | witnessed_event | did_something | received_promise
    about_npc TEXT NOT NULL,      -- 关联 NPC ID
    content TEXT NOT NULL,        -- 记忆内容（中文文本）
    source TEXT,                  -- 怎么知道的
    game_time TEXT,               -- 游戏内时间
    importance REAL DEFAULT 0.5,  -- 0-1
    used_count INTEGER DEFAULT 0, -- 被生成为选项的次数

## 20. 玩家店铺与物品系统（V1 轻量版）

> V1 定位：家 + 社交据点。经营内容后续扩展。

### 20.1 店铺功能

```
店铺状态（玩家可选）：
  营业中 → NPC 可以进门。门口挂牌"OPEN"
  休息中 → 门半掩。NPC 通常不会进来（千早可能闯）
  已关门 → 挂"CLOSED"。NPC 不会打扰

玩家在店里可做的事：
  柜台后 → 泡咖啡/泡茶（简单动画）、触发快进面板、开关店门
  窗边座位 → 坐下、发呆、看书
  店里走动 → 调整装饰品位置、查看物品栏
```

### 20.2 物品系统

```
随身物品（可送人）：上限 5 个
店内库存（可招待 NPC）：无上限

获取途径：
  • 从商店街店铺购买（千早的面包、和果子）
  • NPC 赠送（鹿岛樱送花、千早送面包尾货）
  • 奶奶的遗物（初始就有几件）
  • 街上发现（河边石头、落花）

送礼触发：
  对话中自动生成送礼选项 ← 条件：库存有 + 记忆匹配 NPC 偏好
  送对 → 羁绊 +0.03~0.08
  送错 → 羁绊 +0.00~0.01（但不了解你的印象加深）
```

> 扩展接口：店铺菜单、收入、装修、NPC 点单、评价系统 → V2 经营扩展。

## 23. 情绪转换规则

### 23.1 情绪状态图

```
                    ┌──────────┐
           ┌───────→│   开心    │←────────┐
           │        └─────┬────┘         │
           │              │              │
           │         (自然回落)    (好事发生)
           │              │              │
           │              ▼              │
           │        ┌──────────┐         │
    (精力耗尽)      │   平静    │    (互动愉快)
           │        └─────┬────┘         │
           │              │              │
           ▼         (坏事/回忆)         │
    ┌──────────┐         │         ┌──────────┐
    │   疲惫    │         ▼         │   焦虑    │
    └──────────┘   ┌──────────┐    └──────────┘

           │       └─────┬────┘         │

                          │

                    └──────────┘
```

### 23.2 转换规则

```
平静 → 开心：
  触发：正面互动（被夸、被帮、收到喜欢的礼物）、好事发生（花开得好、面包烤成功）
  转换：直接跳转，无中间态

开心 → 平静：
  触发：自然回落（30-120 游戏分钟后）
  转换：平滑过渡（不是说变就变，而是渐渐回到日常状态）

平静 → 低落：
  触发：负面记忆被检索到、被忽视、别人提到伤心事、精力过低
  转换：直接跳转，无中间态

低落 → 平静：
  触发：自然恢复（60-180 游戏分钟）、被关心（成功互动）、做了喜欢的事
  转换：平滑过渡

平静 → 焦虑：
  触发：有心事但无法解决、担心某人、重要的事悬而未决
  转换：直接跳转

焦虑 → 平静：
  触发：心事解决、被安抚、事情有了结果
  转换：直接跳转

焦虑 → 生气：
  触发：焦虑的问题恶化、被人打断/冒犯、期待落空
  转换：直接跳转（但只有焦虑超过一定时长才可能转生气）

生气 → 平静：
  触发：自然消退（30-90 分钟）、被道歉、自己冷静下来
  转换：平滑过渡（不会瞬间消气）

任意 → 疲惫：
  触发：精力 < 20 时强制覆盖当前情绪
  转换：直接跳转（不是真正的情绪变化，是生理状态覆盖了情绪表达）

疲惫 → 平静：
  触发：精力恢复到 > 40
  转换：平滑过渡
```

不同 NPC 的情绪波动幅度不同：

  emotional_amplitude（0-1，越大情绪越容易波动）:

  千早：   0.8  ← 大喜大悲型。开心时蹦起来，低落时全世界都知道
  龙之介： 0.6  ← 主要是低落和焦虑，开心很少见但很真
  鹿岛樱： 0.3  ← 表面平静，情绪藏在里面。开心也只是微微一笑
  和叶：   0.4  ← 偏稳定，情绪变化慢但累积后也会爆发
  九条：   0.2  ← 面部表情几乎不变。"开心"表现为少喝一罐咖啡

振幅影响：
  • 情绪触发阈值：振幅高的人更容易被触发情绪变化
  • 情绪恢复速度：振幅低的人恢复得更慢（情绪越少波动越不容易走出来）
  • 表面 vs 真实：振幅低的人可能隐藏真实情绪
```

### 23.4 情绪对行为的量化影响

```
情绪对对话的影响（注入 Prompt）：
  开心 → "你今天说话的语气比平时轻快"
  低落 → "你的回答比平时短，偶尔走神"
  焦虑 → "你说话时会不自觉地搓手/看窗外"
  生气 → "你的语气变得尖锐/沉默"
  疲惫 → "你说话有气无力，经常叹气"

情绪对社交开放度的影响：
  开心 → sociability +20
  平静 → sociability +0
  低落 → sociability -15
  焦虑 → sociability -10（但可能主动找人求助）
  生气 → sociability -30（不想理人）
  疲惫 → sociability -25

情绪对精力消耗的影响：
  开心 → 正常消耗
  低落 → 消耗 ×1.3（情绪内耗）
  焦虑 → 消耗 ×1.5
  生气 → 消耗 ×1.2
  疲惫 → 已经是结果，不额外修正

情绪持续时间的角色差异：
  千早：生气 30 分钟后就忘了。低落 60 分钟后被自己治愈。
  鹿岛樱：低落可能持续一整天（如果没人发现的话）。
  九条：看起来永远在"疲惫"和"平静"之间。
```

### 23.5 实现：规则引擎 + LLM 辅助

```
规则引擎负责（确定性）：
  ✅ 精力阈值触发（精力<20 → 疲惫）
  ✅ 自然回落计时（开心→平静，生气→平静）
  ✅ 身体状态覆盖（生病/心脏病发作 → 强制情绪修正）

LLM 负责（需要上下文判断）：
  ✅ "这个互动让 NPC 开心还是低落？"
     → 对话结束后，LLM 输出 emotion_change: "开心" | "低落" | null
  ✅ "NPC 在隐藏真实情绪吗？"
     → 鹿岛樱可能表面"平静"，但内心"焦虑"
     → 不在数据库存两个字段。LLM 通过 Prompt 控制：
       Layer 2（情绪语气层）使用真实情绪来驱动措辞
       但 Prompt 中追加一句："你不想让别人看出来。表面保持平静。"
     → 结果：她说的话是"平静"的，但用词、停顿、回避话题的细微处会泄露焦虑
     → 只有羁绊度高的玩家能察觉到——不是看数值，是读对话
  ✅ "这个情绪应该持续多久？"
     → LLM 结合 NPC 性格和当前情境给出建议值

规则 + LLM 协作流程：
  1. 事件发生（对话结束 / 观察到某事 / 精力变化）
  2. 规则引擎先检查：有没有硬规则触发的情绪变化？
     （精力过低 → 疲惫，这是一定会发生的）
  3. 如果没有硬规则 → LLM 判断：
     "基于当前情境 + NPC 性格 + 记忆，这个事件改变情绪吗？"
  4. LLM 输出：{emotion, 是否隐藏, 预计持续时间}
  5. 写入 npc_states.emotion = 表面情绪（展示给玩家）
  6. 如果"是否隐藏"=true → Prompt Layer 2 追加隐藏指令
     玩家无法从数值面板看到真实情绪，只能从对话中感知
```

## 24. 角色 × 情绪 Prompt 差异化

### 24.1 设计目标

```
同一个事实，五个角色用五种方式说：

  事实：玩家帮鹿岛樱搬了花盆

  鹿岛樱（开心）："谢谢小李。这些花盆确实有点重...你手稳。"
  千早（开心）：  "哇！你力气不错嘛！比我搬得快！改天来面包店帮忙！"
  和叶（平静）：  "嗯。有效率。"（推眼镜，然后又看回书）
  龙之介（焦虑）："...那个...不用帮我也...啊你已经搬完了。谢谢。"
  九条（疲惫）：  "...年轻人就是有劲。"（继续喝咖啡）
```

### 24.2 Prompt 分层组装

```
System Prompt = 角色基底层 + 情绪语气层 + 情境注入层

Layer 1: 角色基底层（固定，每次对话都注入）
  来自第 7 章基底设定中的 speech 字段
  决定了"这个人正常时候怎么说话"

Layer 2: 情绪语气层（动态，根据当前情绪切换）
  决定了"今天说话的语气、长短、温度"
  情绪变了，这层就变

Layer 3: 情境注入层（动态，每次对话不同）
  印象 + 记忆 + 当前对话上下文
  决定了"这次对话的具体内容"
```

### 24.3 Layer 1: 角色基底说话风格（固定注入）

```yaml
鹿岛樱:
  base_tone: "温柔、轻声、话少但每个字都认真"
  sentence_style: "短句为主，偶尔用老派敬语。不说废话。"
  example: |
    你说话轻轻的。每句话都不长。
    开心时也是微笑，不会大笑。
    难过时话更少，但不冷漠——只是不知道该怎么说。
    你会用"没关系的"来安抚别人，即使你自己也需要被安抚。

千早:
  base_tone: "响亮、直率、想什么说什么、没有过滤机制"
  sentence_style: "句子忽长忽短，经常跑题。语气词多。"
  example: |
    你说话声音很大。想到什么说什么。
    开心的时候整条街都知道。
    不开心的时候整条街也知道。
    你不会拐弯抹角——"我觉得这个面包烤糊了"就是真的烤糊了。

和叶:
  base_tone: "理性、轻声但有逻辑、偶尔引用书里的话"
  sentence_style: "句子完整、有条理。会推眼镜。沉默比话多。"
  example: |
    你说话前会停顿一下，像是在组织语言。
    你不会说多余的话。但如果是你在意的事，你会说很多。
    紧张的时候会推眼镜。
    真正生气的时候不说话——你直接用沉默表达。

龙之介:
  base_tone: "极轻、极短、经常中断、不敢看对方眼睛"
  sentence_style: "单字或短句。充满停顿词。句子经常不完整。"
  example: |
    你声音很轻。说话时看地板。
    "...嗯。"是一个完整的回答。
    对信任的人，句子会变长一点点。
    被逼急了会说一长串，然后立刻后悔。
    你不是不想说话——你是不知道该说哪一句。

九条:
  base_tone: "慢、懒、声音低沉、话极少但偶尔一针见血"
  sentence_style: "句子短。不用敬语。经常用'随便''无所谓'。"
  example: |
    你说话很慢。不多说一个字。
    对猫说话比对人多。
    "...随便吧。"是你的万能回答。
    但如果你真的在意一件事，你会说得很直接、很准。
    不习惯被感谢——被感谢的时候你会沉默或转移话题。
```

### 24.4 Layer 2: 情绪语气层（动态切换）

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

开心
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
通用：话变多、愿意接话、容易答应请求

鹿岛樱 → "话稍微多了一点。会主动多说两句。笑起来眼睛弯弯的。"
千早   → "话多了一倍。音量大了 30%。可能会唱起歌来。"
和叶   → "推眼镜的次数减少了。说的句子比平时长。可能会推荐一本书。"
龙之介 → "能说出完整的句子了！虽然还是很轻。偶尔会看着对方。"
九条   → "少喝了一罐咖啡。开口的次数比平时多了一次。最大变化：嘴角有一个像素的弧度。"


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
低落
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
通用：话变少、反应慢、容易走神

鹿岛樱 → "几乎不说话。问三句答一句。笑容消失了。但她会说'没关系的'——这是谎话。"
千早   → "话变少了（但还是比正常人正常状态多）。揉面的力气变小了。会叹气。"
和叶   → "长时间的沉默。书页很久没翻过。如果有人问'怎么了'，他说'没什么'。"
龙之介 → "完全缩回壳里。单字回应都没有了。只摇头或点头。"
九条   → "多喝了一罐咖啡。跟猫说话的时间变长了。对人说话的时间约等于零。"


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
焦虑
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
通用：心不在焉、会反复确认小事、可能主动求助

鹿岛樱 → "说话时会不自觉地捏手指。会反复检查花的摆放。偶尔走神。"
千早   → "揉面的时候用力过猛。话反而变多了——她在用说话掩盖焦虑。"
和叶   → "书拿反了五分钟没发现。深夜散步的时间比平时长。"
龙之介 → "躲进后台不出来。如果必须对话→语速变快但全是'对不起对不起'。"
九条   → "喂猫的时间比平时早了两个小时。在派出所里反复整理同一份文件。"


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
生气
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
通用：话变短或变冲、拒绝互动、可能说伤人的话然后后悔

鹿岛樱 → "沉默。但这次不是温柔的沉默——是冰冷的沉默。如果开口，每个字都像针。"
千早   → "声音大了三倍。直接怼。'这不对！'——然后三分钟后气消了来道歉。"
和叶   → "极度冷静但句句带刺。用逻辑当武器。吵完可能会离开现场。"
龙之介 → "不会对别人发火。他在后台撕画纸。如果被逼急了→突然爆发一串话→然后哭。"
九条   → "语气变得更慢更有压迫感。不说废话了——直接点出问题的核心。让人害怕。"


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
疲惫
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
通用：有气无力、句子断裂、容易结束对话

鹿岛樱 → "声音比平时更轻。会靠在柜台上。'抱歉...今天有点累了...'"
千早   → "难得的安静。揉面的动作变慢。还是会笑——但笑得很勉强。"
和叶   → "在沙发上缩成一团。咖啡凉了也没发现。'...明天再说吧。'"
龙之介 → "几乎不可见——他根本不会从后台出来。"
九条   → "在椅子上睡着了。被叫醒也只是'...嗯。'然后继续睡。"
```

### 26.2 成本估算

```
每次 LLM 调用的 Token 消耗（基于设计的 Prompt 预算）：

  立绘对话（单轮）：
    输入:  ~3500 tokens (System + 印象 + 记忆 + 上下文)
    输出:  ~150 tokens  (NPC 一句话 + 选项)
    单轮:  ~3650 tokens

  一次完整对话（平均 8 轮）：
    输入:  8 × 3500 = 28,000 tokens
    输出:  8 × 150 = 1,200 tokens
    一次对话: ~29,200 tokens

  睡眠反思（每游戏日一次 × 5 NPC）：
    输入:  5 × 1500 = 7,500 tokens
    输出:  5 × 500 = 2,500 tokens
    每日:  ~10,000 tokens

  对话选项生成（每次对话附带）：
    输入:  ~800 tokens
    输出:  ~100 tokens
    每次:  ~900 tokens

每游戏日（约 30 现实分钟）调用量：
  假设玩家对话 3-5 次 + 自主对话 2-3 次 + 反思 1 次
  → 约 8-10 次 LLM 调用
  → 约 100,000-150,000 tokens

每现实小时（约 2 个游戏日）：
  → 约 200,000-300,000 tokens

成本（Claude API 2026 年参考价格）：
  Sonnet: 输入 $3/M tok | 输出 $15/M tok
  → 每现实小时约 $0.80-1.50 (Sonnet)

  Opus:   输入 $15/M tok | 输出 $75/M tok
  → 深度剧情用 Opus 时约 $0.50/次（很少触发）
  
  合计：玩家玩 1 小时 ≈ 花费 $1-2
        玩 10 小时   ≈ 花费 $10-20
        （约等于一杯奶茶到一顿饭的钱）

成本优化选项（游戏内提示）：
  • 使用 DeepSeek 替代 Claude → 成本约 1/5
  • 使用本地 Ollama + Qwen/Llama → 免费（但对话质量下降）
  • Embedding 永远走本地 → 不产生费用
```

### 26.3 玩家首次启动流程（含 API 配置）

```
首次启动 → 主菜单 → "新游戏"

  1. 检测 API Key 是否已配置

  3. API Key 存储：
     存储在本地文件: backend/.env（gitignore）
     格式: CLAUDE_API_KEY=sk-ant-xxx
     不上传到任何服务器
```

## 32. 2026-07-11 图记忆实现追平注记（追加，不覆盖原文）

<!--

时间语义修正与长测执行记录追加。

### 32.1 当前总口径：轻量图 + 向量数据层

当前实现不再追求把所有记忆内容完整压进 SQLite 图节点表。

近期执行口径改为：

1. **图是联想索引层**
   - SQLite `memory_nodes` 只保留节点 ID、所属 NPC 和创建时间。
   - SQLite `memory_edges` 保留关系类型、双向 `clarity_ab / clarity_ba`、`target_importance` 和遍历痕迹。
   - 图负责回答“从谁自然想到什么”“这条联想有多清晰”“这个方向是否比反方向更容易”。

2. **向量库是节点数据层**
   - LanceDB 优先；不可用时降级到 SQLite 向量表。
   - 节点内容、类型、importance、created_day、archived 与 embedding 存在向量层。
   - 向量层负责语义近邻、节点内容读取、强制回忆与图结果不足时的兜底。

3. **遗忘优先发生在边上**
   - 普通联想边的 clarity 会在午夜衰减。
   - 被检索实际走过的边会恢复少量 clarity。
   - 边衰减到阈值以下后删除；孤立节点再被归档。
   - 这意味着“记忆还在，但日常不容易自然想起”，比直接删除文本更适合当前原型。

4. **近期不实现完整全概念认知图**
   - `energy / precision / exact→vague→gone` 作为远期设计保留。
   - `similar_to` 大规模建边暂不作为近期必做项。
   - “节点融合 + 基础边继承”已在 2026-07-11 补齐闭环，但仍不扩为完整四阶段退化主机制。

### 32.2 当前数据库口径

当前实现中的长期记忆由两层共同组成：

```text
SQLite
  memory_nodes:
    id, subject_id, created_at_game_time

  memory_edges:
    id, node_a, node_b, type,
    clarity_ab, clarity_ba,
    target_importance,
    last_traversed_ab, last_traversed_ba,
    created_at_game_time

LanceDB / SQLite vector fallback
  node_id, vector, type, value,
  importance, created_day, archived
```

因此，旧章节中 `memory_nodes.type / value / energy / precision / original_text / is_core` 等完整字段应理解为**远期理想 schema**，不是当前必须追齐的近期实现。

当前实现中较重要的边类型是：

| 边类型 | 当前作用 |
|------|------|
| `relationship` | `我` 与熟人人物节点之间的长期社交骨架 |
| `involved` | 人物 / 自我 与事件之间的主要可达路径 |
| `felt` | 事件与情绪节点之间的情绪支撑 |
| `located_at` | 事件与地点之间的场景线索 |
| `happened_at` | 事件与时间之间的时间线索 |
| `mentioned` | 话语、话题、事件间的轻关联 |

### 32.3 当前检索模式

当前检索已经分为三种模式：

| 模式 | 使用场景 | 口径 |
|------|------|------|
| `player_dialogue` | 玩家与 NPC 对话 | 从 `我 + 玩家人物节点` 出发，最多 2 跳，优先玩家背景、喫茶店、奶奶、樱桥通旧事，可带出 NPC 私人旧事 |
| `npc_dialogue` | NPC ↔ NPC 日常闲聊 | 最多 1 跳，优先对方、眼前场景、商店街日常，避免一上来跳到沉重私人往事 |
| `nightly_impression` | 午夜基准印象刷新 | 最多 2 跳，比白天更宽，优先目标人物、近期互动、核心关系和长期旧事 |

共同流程：

1. 找起点节点：`我` 节点 + 目标人物节点。
2. 从当前前沿节点读取方向性邻边。
3. 本地按 clarity、边类型、目标人物、地点、关键词做预排序。
4. 必要时用轻量 LLM 在候选边中选路。
5. 图结果不足时才做少量向量兜底。
6. 最终节点按目标相关性重排，并重建为 prompt 中的“相关回忆”。

当前原则是：**图优先，向量兜底，不回到全局向量 Top-K 主导。**

2026-07-11 增强：

1. 检索会写入 `memory_retrieval_logs`，记录图命中、向量兜底、LLM 路由调用、本地路由跳过、融合节点命中和耗时。
2. 当本地边排序足够明确时，可跳过每跳 LLM 路由，降低白天检索成本。
3. `/api/npc/{npc_id}/graph?detail=true` 可用于查看节点、边、融合来源和最近检索诊断。

### 32.4 玩家背景与时间语义

玩家背景已经从 prompt 硬编码下沉为所有 NPC 的 Day 0 长期记忆。

初始长期记忆包括：

1. 小李小时候在樱桥通长大，后来去了城市。
2. 街上听说小李准备重新开奶奶留下的喫茶店。
3. 奶奶去世后喫茶店关了很久。
4. 樱桥通是熟人社会，街上消息传得快。

这些记忆会：

1. 写入 `memory_nodes`。
2. 写入向量层。
3. 连接到 `self` 和 `player person` 节点。
4. 参与图路由、向量兜底、遗忘和夜间印象整理。

检索输出必须保留时间标签：

```text
- [Day 0] 听说: 小李准备重新开奶奶留下的喫茶店...
- [第5天] 记得: 小李今天真的把招牌挂回去了...
```

Prompt 解释规则：

1. 较新的记忆优先代表现在。
2. 较旧的记忆只代表当时知道的事。
3. “听说 / 准备 / 可能”不能被改写成亲眼所见或已经完成。
4. 如果新记忆更新旧说法，以新记忆为准。

### 32.5 夜间流程的当前边界

午夜当前主链是：

1. 边 clarity 衰减。
2. 短期记忆提取入图与向量层。
3. 轻量图演化。
4. 夜间多跳图路由生成熟人基准印象。
5. 短期记忆清理。

当前已经验证：

1. 后端可脱离前端独立长测。
2. 30 分钟墙钟跑测可推进到第 12 天，午夜平均耗时约 37 秒。
3. 图规模在 12 个游戏日后仍处于每 NPC 数十到一百多节点 / 边的可控范围。
4. 检索样例能取出近期互动、玩家背景、NPC 之间的目标相关事件。

2026-07-11 增强后，玩家真实行动可通过 `PLAYER_EVENT` 写入见证 NPC 的短期记忆，再由午夜提取进入图和向量层。该入口用于让 Day N 真实事件自然更新 Day 0 传闻，不作为独立硬编码事实库。

后续性能优化优先级：

1. 降低白天每跳 LLM 路由频率，优先使用本地 scoring。
2. 缓存夜间印象检索结果或批量读取节点数据。
3. 对行为计划异常与重复触发做去重，减少无意义短期记忆。

### 32.6 近期明确暂缓的旧设计项

以下内容保留为远期方向，不再作为当前原型必须追齐的缺口：

| 旧设计项 | 当前处理 |
|------|------|
| 全节点 `energy / precision` 生命周期 | 用边 clarity 衰减 + archived 归档替代 |
| 大规模 `similar_to` 边 | 暂不建边；相似度主要用于融合候选或向量兜底 |
| 四阶段完整退化 | 保留设计语义，近期不实现完整状态机 |
| 复杂思考预算与四策略选择器 | 先用检索 mode 参数和本地排序表达深浅差异 |
| 反思独立模块 | 不独立拆；可并入图演化或印象整理 |
| 印象独立主系统 | 不作为独立真相源；基准印象由图 + 短期记忆夜间刷新 |

### 32.7 后续设计优先级

如果继续推进记忆系统，建议顺序是：

1. **巩固轻量图语义**
   - 检查边类型是否真的影响检索结果。
   - 给 `felt / involved / relationship` 保持足够稳定的创建质量。

2. **优化检索成本**
   - 白天先本地选边，LLM 路由只在候选冲突明显或深度对话中启用。
   - 记录每次检索是否使用了向量兜底，作为质量与成本指标。

3. **验证融合边界**
   - 基础节点融合已补旧边继承 / 重连规则：融合节点继承旧邻边，旧图节点退场，旧向量节点 archived。
   - `memory_merge_sources` 已记录融合来源；后续重点不是继续扩概念，而是做长测可达性统计和融合质量评估。

4. **补玩家真实行动入图**
   - 当玩家真的拿钥匙、打扫、挂招牌、开店时，写入新的 Day N 记忆。
   - 让新记忆自然覆盖 Day 0 “听说准备”的旧语义。

5. **再考虑远期完整退化**
   - 只有当原型闭环稳定且长测证明需要更细遗忘表现时，再回到 `precision / similar_to / 四阶段退化`。
